# Control Panel 优化清单

## 背景

`control-panel` 用于辅助启动同级目录下的 `metersphere` 多服务项目，核心职责包括：

- 启停 Java 后端服务
- 构建并发布多个前端模块
- 展示实时日志与构建进度
- 作为本地开发辅助控制台，降低多模块联调成本

当前项目已经具备可用的控制面板能力，但在“进程控制、构建链路、状态同步、日志性能、仓库卫生”几个方向仍有较明确的优化空间。

## 优先级高

### 1. 进程管理升级为可控进程模型

**问题**

- 启动服务依赖命令字符串与 shell 包装，进程模型不稳定。
- 停止服务依赖模糊匹配，存在误杀或漏杀风险。
- 服务状态判断容易依赖进程扫描结果，准确性不足。

**建议**

- 直接使用 `spawn('./mvnw', ['-f', pom, 'spring-boot:run'])` 启动服务。
- 跟踪真实子进程与 PID，而不是 shell 包装进程。
- 使用 PID、端口、健康检查的组合方式确认服务状态。
- 为停止、重启、批量启动建立统一的进程组管理策略。

**收益**

- 降低误判和误杀概率。
- 提高“启动/停止/重启”结果的可预测性。
- 为后续健康检查串行启动打基础。

**当前状态**

- 已完成第一阶段重构：改为直接拉起子进程，补充 PID、端口、`pgrep` 兜底检测。
- 后续仍建议继续补齐“按健康状态串行启动”和更细粒度的失败恢复策略。

## 2. 构建取消做成真正可中断

**问题**

- UI 触发取消后，如果只是更新状态但不终止 `npm install` 或 `npm run build` 子进程，会导致页面与真实执行状态不一致。
- 构建日志与构建任务状态会出现漂移。

**建议**

- 对构建进程建立独立跟踪表。
- 取消构建时主动终止对应 `npm` 子进程或进程组。
- 将取消结果统一写回构建状态和构建历史。

**收益**

- 构建状态更可信。
- 避免后台残留构建任务占用 CPU / IO。
- 提升操作反馈的一致性。

**当前状态**

- 已完成第一阶段重构：取消操作已接入真实构建进程终止逻辑，状态会同步更新为 `cancelled`。

## 3. 构建默认跳过重复安装依赖

**问题**

- 每次构建都先执行一次 `npm install`，本地迭代成本过高。
- 对开发机网络、缓存和包管理器状态依赖较强，影响稳定性。

**建议**

- 默认仅执行 `npm run build`。
- 仅在以下场景执行依赖安装：
  - 首次构建
  - `node_modules` 缺失
  - lockfile 变更
  - 用户主动勾选“强制安装依赖”
- 有 lockfile 时优先使用 `npm ci`。

**收益**

- 显著减少重复构建耗时。
- 降低网络问题对构建成功率的影响。
- 更符合本地高频开发场景。

**当前状态**

- 已完成第一阶段优化：默认检测本地依赖目录，存在时跳过安装；需要时优先执行 `npm ci`。
- 后续可继续细化到 lockfile 变更检测，而不是仅检测 `node_modules`。

## 4. 配置源收敛为单一元数据来源

**问题**

- 服务信息和可构建模块信息分散在多处代码中。
- 新增模块时需要修改后端校验、控制器和前端常量，维护成本高。

**建议**

- 以 `config.json` 为基础配置源。
- 在后端统一生成：
  - 服务目录
  - 前端模块目录
  - 校验映射
- 前端通过 API 动态获取服务与模块列表，而不是硬编码。

**收益**

- 新增或调整模块时只需要改一个地方。
- 避免前后端配置不一致。
- 降低后续扩展成本。

**当前状态**

- 已完成第一阶段收敛：服务目录和构建模块目录已经由后端统一生成，前端改为通过接口拉取。

## 优先级中

### 5. 批量启动改为“按健康状态推进”

**问题**

- 当前批量启动如果只是固定等待几秒，再启动下一个服务，对 Spring Boot 服务链路并不稳定。
- 机器性能或环境不同会导致依赖服务还未准备完成就启动下游服务。

**建议**

- 启动一个服务后，轮询对应端口或健康接口。
- 确认服务健康后再启动下一个依赖服务。
- 对超时、失败、半启动状态做明确提示。

**收益**

- 提高整套服务启动成功率。
- 减少“明明点了启动全部但最后有几个服务挂掉”的情况。

**当前状态**

- 尚未完成，建议作为下一阶段重点。

### 6. 日志链路性能优化

**问题**

- 前端将日志以大字符串形式存储，过滤和搜索时频繁 `split/join`，日志多时会卡顿。
- 服务端仍存在同步写日志的实现方式，可能阻塞事件循环。

**建议**

- 前端改用日志行数组或 ring buffer。
- 搜索、过滤、渲染拆分为更细粒度的数据流。
- 对长日志列表使用虚拟滚动。
- 服务端改用 `createWriteStream` 持久化日志。

**收益**

- UI 在高日志吞吐下更平滑。
- 降低内存抖动和主线程阻塞。

**当前状态**

- 已完成部分体验修复：`build` 和 `service` 日志筛选状态已拆开，避免互相串扰。
- 日志存储结构和服务端写盘方式仍建议继续优化。

### 7. 充分利用 WebSocket，减少轮询

**问题**

- 当前已经接入 WebSocket，但部分状态仍依赖前端定时轮询刷新。
- 在服务状态和构建状态高频变化时，轮询存在延迟和冗余请求。

**建议**

- 把服务状态变化也纳入 WebSocket 推送。
- 前端轮询仅保留为连接失败或断线恢复时的兜底机制。
- 在关键事件点做增量更新，而不是全量刷新。

**收益**

- 页面状态更实时。
- 减少无意义请求和状态抖动。

**当前状态**

- 尚未完全完成，当前仍保留轮询兜底。

### 8. 日志筛选状态按类型隔离

**问题**

- 构建日志和服务日志共用筛选条件时，切换页面体验会互相影响。

**建议**

- 以日志类型为维度分别存储 `logLevel` 和 `searchTerm`。

**收益**

- 减少切换 Tab 时的认知干扰。

**当前状态**

- 已完成。

## 优先级低但值得做

### 9. 构建产物复制改用 Node 原生文件 API

**问题**

- 基于 `rm -rf`、`cp` 的复制方案依赖 shell，路径处理和平台兼容性较弱。

**建议**

- 使用 `fs.rm`、`fs.cp` 和 `path.resolve` 统一处理目标路径。

**收益**

- 路径更安全，可维护性更好。
- 降低 shell 命令拼接风险。

**当前状态**

- 已完成第一阶段替换。

### 10. 健康检查并行化

**问题**

- 批量健康检查串行执行时，服务数量一多整体耗时明显上升。

**建议**

- 改为 `Promise.allSettled` 并行探测。
- 尽量解析 actuator 返回内容，而不只是判断 HTTP 200。

**收益**

- 更快得到全量服务健康状态。
- 健康信息更准确。

**当前状态**

- 尚未完成。

### 11. 开发/生产职责与文档对齐

**问题**

- 项目文档中仍有部分历史描述，例如日志通道仍写成 SSE，而当前主通道已经是 WebSocket。
- 部分开发/生产行为在文档和实现层面还不够统一。

**建议**

- 更新 `README`，统一描述日志与实时状态同步机制。
- 明确区分开发模式与生产模式的静态资源职责。

**收益**

- 降低维护者误解。
- 提升项目上手效率。

**当前状态**

- 尚未系统整理，建议后续与接口文档一起统一更新。

### 12. 仓库卫生治理

**问题**

- 本地运行会生成 `logs`、`.pids`、`frontend/dist` 等产物。
- 如果未忽略，会污染工作区和提交记录。

**建议**

- 补充 `.gitignore`。
- 明确哪些内容属于运行时产物，哪些属于源码。

**收益**

- 让提交更聚焦。
- 降低误提交概率。

**当前状态**

- 已完成第一阶段处理：已补充 `.gitignore` 忽略常见运行时产物。

### 13. Redis 降级策略简化

**问题**

- 对于本地辅助启动器，Redis 并不是强依赖。
- 默认密码硬编码也不够理想。

**建议**

- 默认以内存缓存为主，Redis 作为可选增强。
- 去掉默认敏感信息，改为显式配置或环境变量注入。

**收益**

- 降低本地运行门槛。
- 提高安全性与可移植性。

**当前状态**

- 尚未完成，建议在缓存层下一轮整理时统一推进。

## 推荐后续顺序

建议按以下顺序推进后续优化：

1. 批量启动接入健康检查等待机制
2. 服务状态改为 WebSocket 推送为主、轮询为辅
3. 日志存储结构升级为 ring buffer / 行数组
4. 服务端日志写盘切换到流式写入
5. 健康检查并行化与 actuator 内容解析
6. Redis 依赖降级为可选模式
7. 更新 `README` 与 API/运行机制说明

## 备注

本清单用于记录 `control-panel` 当前阶段的优化方向与推进状态，后续可以按阶段继续拆成：

- 稳定性优化
- 性能优化
- 可维护性优化
- 文档与工程卫生优化
