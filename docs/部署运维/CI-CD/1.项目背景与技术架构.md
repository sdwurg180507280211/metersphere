## 一、项目概况

### 1.1 基本信息

- **项目名称**：MeterSphere测试管理平台
- **当前版本**：v2.10.23-lts
- **架构类型**：微服务架构（Spring Cloud）
- **部署方式**：模块化Docker Compose
- **团队规模**：独立负责开发+运维

### 1.2 为什么选择微服务架构？

**业务背景**：
- MeterSphere是一个复杂的测试管理平台
- 涵盖接口测试、性能测试、测试跟踪等多个领域
- 各模块业务独立性强，开发周期不同

**微服务带来的价值**：
1. **业务解耦**：接口测试模块变更不影响性能测试模块
2. **独立部署**：可以单独更新某个服务，降低部署风险
3. **技术选型灵活**：不同服务可以使用不同技术栈
4. **团队协作**：多人开发时减少代码冲突

**面临的挑战**：
- 服务间通信复杂度增加
- 运维难度提升（需要管理9个服务）
- 分布式事务处理
- 服务监控和故障排查

## 二、技术架构全景

### 2.1 架构分层

```
┌─────────────────────────────────────────┐
│         用户层（浏览器）                │
│         HTTP/HTTPS访问                  │
└──────────────┬──────────────────────────┘
               │ 端口8081
               ↓
┌─────────────────────────────────────────┐
│      Gateway网关层（统一入口）          │
│  - 路由转发                             │
│  - 前端页面服务                         │
│  - 负载均衡                             │
└──────────────┬──────────────────────────┘
               │ 服务发现
               ↓
┌─────────────────────────────────────────┐
│      Eureka注册中心（服务治理）         │
│  - 服务注册                             │
│  - 健康检查                             │
│  - 服务发现                             │
└──────────────┬──────────────────────────┘
               │
        ┌──────┴──────┐
        ↓             ↓
┌─────────────┐  ┌────────────────────┐
│system-setting│  │  7个业务服务       │
│ (基础服务)  │  │  - api-test        │
└─────────────┘  │  - test-track      │
                 │  - performance-test│
                 │  - 等等...         │
                 └────────────────────┘
```

**设计思路**：
- **网关统一入口**：简化外部访问，隐藏内部服务结构
- **Eureka服务治理**：自动服务发现，无需手动配置服务地址
- **基础服务优先**：system-setting作为其他服务的依赖，优先启动

### 2.2 微服务模块划分

#### 框架服务层（2个）

**1. Eureka - 服务注册中心**
- **职责**：管理所有微服务的注册和发现
- **端口**：8761（内部访问）
- **内存**：512MB
- **为什么需要**：微服务架构中，服务实例动态变化，需要统一的注册中心

**2. Gateway - API网关**
- **职责**：统一入口、路由转发、前端页面服务
- **端口**：8000（容器内）→ 8081（对外）
- **内存**：800MB
- **为什么重要**：
  - 唯一对外暴露的服务，安全性更高
  - 集成前端页面，用户只需访问一个地址
  - 统一处理跨域、认证等横切关注点

#### 业务服务层（7个）

**1. system-setting - 系统设置（基础服务）**
- **职责**：用户管理、权限管理、系统配置
- **端口**：8001
- **为什么是基础服务**：其他服务依赖它的用户和权限信息

**2. api-test - 接口测试服务**
- **职责**：HTTP接口测试、Mock服务
- **端口**：8004
- **内存**：1GB（需要运行JMeter）

**3. performance-test - 性能测试服务**
- **职责**：负载测试、压力测试
- **端口**：8005

**4. test-track - 测试跟踪服务**
- **职责**：测试用例管理、测试计划
- **端口**：8003

**5. project-management - 项目管理服务**
- **职责**：项目、版本、环境管理
- **端口**：8002

**6. report-stat - 报表统计服务**
- **职责**：测试报告、数据统计
- **端口**：8006

**7. workstation - 工作台服务**
- **职责**：个人工作台、消息通知
- **端口**：8007

**服务划分原则**：
- 按业务领域划分（DDD - 领域驱动设计）
- 每个服务有独立的数据库表
- 服务间通过REST API或消息队列通信

## 三、技术栈详解

### 3.1 后端技术栈

#### 核心框架
```
Java 17
├── 为什么选Java 17？
│   ├── LTS长期支持版本（支持到2029年）
│   ├── 性能提升（GC优化、JIT改进）
│   └── 新语法特性（Records、Pattern Matching）
│
Spring Boot 3.2.12
├── 为什么选3.x？
│   ├── 原生支持Java 17
│   ├── 性能优化（启动速度提升30%）
│   └── 原生镜像支持（GraalVM）
│
Spring Cloud 2023.0.0
└── 微服务全家桶
    ├── Eureka（服务发现）
    ├── Gateway（网关）
    └── OpenFeign（服务调用）
```

**版本选择理由**：
- **稳定性优先**：选择LTS版本和稳定版本
- **向后兼容**：Spring Boot 3.x平滑升级路径
- **社区活跃**：遇到问题容易找到解决方案

#### 数据存储

**MySQL - 主数据库**
- **为什么选MySQL**：
  - 成熟稳定，运维经验丰富
  - 事务支持完善（ACID）
  - 生态丰富（监控、备份工具多）
- **使用外部实例**：独立部署，便于备份和扩展

**Redis - 缓存**
- **用途**：
  - Session共享（多实例负载均衡）
  - 热点数据缓存（用户信息、配置信息）
  - 分布式锁
- **为什么不用内存数据库**：Redis持久化能力强，重启不丢数据

**MinIO - 对象存储**
- **用途**：存储测试附件、报告文件、JMeter脚本
- **为什么不用文件系统**：
  - 支持分布式，易于扩展
  - S3兼容API，便于迁移
  - 自带Web管理界面

#### 消息队列

**Kafka**
- **用途**：
  - JMeter测试指标收集（JMETER_METRICS）
  - 测试日志收集（JMETER_LOGS）
  - 测试报告生成（JMETER_REPORTS）
- **为什么选Kafka而不是RabbitMQ**：
  - 高吞吐量（百万级QPS）
  - 适合日志、指标等大数据量场景
  - 持久化能力强

### 3.2 前端技术栈

**Vue.js - 微前端架构**
```
各服务独立前端项目
├── api-test/frontend
├── test-track/frontend
├── performance-test/frontend
└── ...

构建后集成到后端
├── 前端打包 → dist/
└── 复制到 backend/src/main/resources/static/
```

**为什么采用微前端？**
1. **前后端解耦**：前端团队独立开发
2. **按需加载**：只加载当前模块的前端代码
3. **独立部署**：更新某个模块前端不影响其他模块
4. **技术栈灵活**：不同模块可以使用不同前端框架

**为什么不用独立的前端项目？**
- 简化部署：不需要额外的Nginx服务器
- 统一入口：通过Gateway统一提供服务
- 减少跨域问题：前后端同域名

### 3.3 构建与部署技术栈

**Maven - 项目构建**
```
revision变量统一版本管理
├── 父POM定义 ${revision} = 2.10.23
├── 子模块继承版本号
└── Jenkins构建时动态设置版本

多模块并行构建
├── framework（框架层）
├── api-test（业务层）
├── test-track（业务层）
└── ... 9个模块并行编译
```

**Docker - 容器化**
```
分层构建优化
├── 依赖层（BOOT-INF/lib）    → 很少变化
├── 元数据层（META-INF）      → 很少变化
└── 代码层（BOOT-INF/classes）→ 频繁变化

多架构支持
├── amd64（x86服务器）
└── arm64（ARM服务器、Mac M1）
```

**Jenkins - CI/CD**
```
Pipeline as Code
├── Jenkinsfile（版本控制）
├── 定时触发（每天凌晨1点）
└── 企业微信通知

构建流程
├── SDK构建（可选）
├── Maven多模块构建
├── Docker镜像构建
└── 推送到阿里云镜像仓库
```

## 四、部署架构

### 4.1 部署方式选择

**为什么选择Docker Compose而不是Kubernetes？**

| 对比项 | Docker Compose | Kubernetes |
|--------|----------------|------------|
| 学习曲线 | 简单，几小时上手 | 陡峭，需要几周 |
| 运维复杂度 | 低，一个人可维护 | 高，需要专业团队 |
| 资源消耗 | 小，适合中小规模 | 大，需要多台服务器 |
| 适用场景 | 单机或少量服务器 | 大规模集群 |

**项目实际情况**：
- 独立负责运维
- 服务规模：9个微服务
- 服务器资源：1-3台
- 结论：Docker Compose更合适

### 4.2 模块化Compose设计

**传统方式的问题**：
```yaml
# 所有服务在一个文件
docker-compose.yml（2000行）
  ├── eureka
  ├── gateway
  ├── api-test
  └── ... 其他7个服务
```

❌ 更新一个服务影响所有服务  
❌ 配置混乱，难以维护  
❌ 无法灵活启停服务

**模块化设计的优势**：
```bash
docker-compose-base.yml           # 基础设施
docker-compose-eureka.yml         # Eureka独立文件
docker-compose-gateway.yml        # Gateway独立文件
docker-compose-api-test.yml       # 每个服务独立文件
...
```

✅ 独立更新：只重启需要更新的服务  
✅ 配置清晰：每个文件只关注一个服务  
✅ 灵活组合：调试时只启动需要的服务  
✅ 差异化配置：不同服务不同资源限制

### 4.3 外部基础设施

**为什么MySQL、Redis、Kafka不用Docker部署？**

**优势**：
1. **稳定性更高**：独立部署，不受容器重启影响
2. **性能更好**：直接使用宿主机资源，无虚拟化损耗
3. **备份更方便**：传统备份工具和流程可直接使用
4. **运维经验丰富**：团队对传统部署方式更熟悉

**配置方式**：
```bash
install.conf
├── MS_EXTERNAL_MYSQL=true    # 使用外部MySQL
├── MS_EXTERNAL_REDIS=true    # 使用外部Redis
└── MS_EXTERNAL_KAFKA=true    # 使用外部Kafka
```

## 五、关键设计决策

### 5.1 为什么采用Eureka而不是Consul/Nacos？

| 对比项 | Eureka | Consul | Nacos |
|--------|--------|--------|-------|
| 学习成本 | 低（Spring生态） | 中等 | 中等 |
| 稳定性 | 成熟稳定 | 稳定 | 较新 |
| 社区支持 | Spring Cloud官方 | HashiCorp | 阿里巴巴 |
| 配置管理 | 无（需要Config） | 有 | 有 |

**选择理由**：
- Spring Cloud生态首选
- Netflix经过大规模生产验证
- 社区资料丰富，问题易解决

### 5.2 为什么采用Jenkins而不是GitLab CI？

**GitLab CI优势**：
- 与代码仓库集成更好
- YAML配置简单
- 开箱即用

**Jenkins优势**：
- 插件生态极其丰富
- 灵活性更高
- 社区更大，资料更多

**选择Jenkins的原因**：
- 项目已有Jenkins环境
- 需要与现有系统集成
- 未来扩展性考虑

### 5.3 为什么前端集成到后端而不是独立部署？

**独立部署方式**：
```
前端（Nginx） + 后端（Spring Boot）
优点：前后端完全解耦
缺点：需要额外Nginx，跨域问题，运维成本增加
```

**集成部署方式**：
```
前端打包 → 后端static目录 → Gateway统一服务
优点：简化部署，无跨域，运维简单
缺点：前端更新需要重新打包后端镜像
```

**为什么选择集成部署**：
- 独立运维，追求简单
- 更新频率不高（每周1-2次）
- 减少运维负担

## 七、总结

### 7.1 架构亮点

1. **微服务架构**：业务解耦，独立部署
2. **模块化Compose**：灵活管理，降低运维复杂度
3. **外部基础设施**：稳定性和性能最优
4. **微前端集成**：简化部署，降低运维成本
5. **多架构支持**：兼容x86和ARM服务器

### 7.2 适用场景

✅ 适合：
- 中小型团队（1-5人）
- 服务规模：5-20个微服务
- 服务器规模：1-5台
- 更新频率：每周1-2次

❌ 不适合：
- 大规模集群（100+服务）
- 高频部署（每天10+次）
- 需要弹性伸缩的场景
