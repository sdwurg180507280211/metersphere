## 一、已完成的工作

```

├── functions.sh          ✅ 公共函数库（200+行）

├── update-service.sh     ✅ 单服务更新

├── health-check.sh       ✅ 健康检查

├── view-logs.sh          ✅ 查看日志
```

```bash
# 开发人员
git checkout -b feature/new-api
# 开发代码...
git commit -m "feat: 新增API接口"
git push origin feature/new-api

# 合并到develop
git checkout develop
git merge feature/new-api
git push origin develop

# 自动化流程（无需人工干预）
1. Jenkins检测到develop分支变化
2. 自动构建Maven项目
3. 自动构建Docker镜像
4. 自动推送到阿里云
5. 自动部署到测试环境
6. 企业微信通知：部署成功

# 测试人员
访问测试环境验证新功能
```

```bash
# 运维人员
git checkout master
git merge develop
git tag -a v2.10.24 -m "Release v2.10.24"
git push origin master --tags

# Jenkins自动构建并等待审批
1. 构建镜像（v2.10.24）
2. 推送到Registry
3. 等待人工确认部署

# 运维确认
进入Jenkins → 点击"继续" → 确认部署

# 自动部署到生产
1. SSH到生产服务器
2. 执行滚动更新
3. 健康检查验证

监控10分钟，确认无问题
```

```bash
# 通知

# 后续
排查问题 → 修复 → 重新发布
```

## 八、最佳实践总结

### 8.1 部署原则

1. **小步快跑**：频繁小版本发布，而非大版本
2. **测试先行**：先测试环境，后生产环境
3. **观察期**：部署后观察10-15分钟
4. **保留回退**：保留最近3个版本的镜像
5. **文档同步**：部署和代码同步更新文档

### 8.2 脚本使用技巧

**日常更新单个服务**：
```bash
# 快速更新develop版本到测试环境
./scripts/update-service.sh api-test develop

# 更新生产环境到指定版本
./scripts/update-service.sh gateway v2.10.24
```

**全量更新**：
```bash
# 更新所有服务到新版本
./scripts/update-all.sh v2.10.24

# 自动按顺序更新，失败可选择继续或中止
```

**故障处理**：
```bash
# 第一时间回滚
./scripts/rollback.sh <服务名> backup

# 查看日志排查问题
./scripts/view-logs.sh <服务名>

# 健康检查确认
./scripts/health-check.sh
```

### 8.3 Jenkins使用技巧

**参数化构建**：
```
Build with Parameters
├── buildSdk=false（日常构建）
├── deployToTest=true（自动部署）
└── runTests=false（跳过测试，加快构建）
```

**手动触发部署**：
```
develop分支构建完成后：
└── 如需重新部署，点击"Replay"重新执行Deploy阶段
```

**查看构建历史**：
```
Pipeline → 查看各阶段耗时
找出瓶颈，针对性优化
```

## 九、故障预案

### 9.1 部署失败预案

**现象**：deploy.sh执行失败

**处理步骤**：
```
1. 查看错误信息（脚本会显示失败的服务）
2. 查看服务日志：./scripts/view-logs.sh <服务>
3. 如无法快速修复：./scripts/rollback.sh <服务> backup
4. 修复问题后重新部署
```

**预计恢复时间**：5-10分钟

### 9.2 Jenkins构建失败预案

**现象**：Pipeline失败

**处理步骤**：
```
1. 查看Console Output
2. 常见问题：
   - Maven依赖下载失败 → 重新触发
   - Docker推送失败 → 检查网络
   - SSH连接失败 → 检查密钥
3. 修复问题后Rebuild
```

### 9.3 服务异常预案

**现象**：某服务health check失败

**处理步骤**：
```
1. ./scripts/view-logs.sh <服务>
2. docker stats 查看资源使用
3. 如是新版本问题：./scripts/rollback.sh <服务> backup
4. 如是环境问题：检查MySQL/Redis/Kafka连接
```
