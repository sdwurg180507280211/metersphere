## 一、整体设计理念

### 1.1 Pipeline as Code的价值

**传统方式的问题**：
```
Jenkins Web界面配置
├── 配置不可追溯
├── 团队协作困难
├── 迁移成本高
└── 容易误操作
```

**Pipeline as Code的优势**：
```
Jenkinsfile纳入版本控制
├── ✅ 配置可追溯（Git历史记录）
├── ✅ 代码审查（PR review）
├── ✅ 分支隔离（不同分支不同配置）
└── ✅ 易于迁移（新Jenkins只需导入代码）
```

### 1.2 MeterSphere的Pipeline结构

```groovy
pipeline {
    agent         // 在哪里执行
    triggers      // 什么时候执行
    environment   // 环境变量
    stages        // 做什么事情
    post          // 执行后做什么
}
```

**设计原则**：
1. **简单清晰**：每个阶段职责单一
2. **可重用**：通过环境变量参数化
3. **容错性**：失败重试机制
4. **可观测**：详细日志和通知

## 二、触发策略设计

### 2.1 定时触发 vs 事件触发

**当前策略：pollSCM定时检查**
```groovy
triggers {
    pollSCM('0 1 * * *')  // 每天凌晨1点检查代码变化
}
```

**为什么选择定时触发？**
- ✅ 可预期的构建时间（凌晨服务器空闲）
- ✅ 避免频繁构建（节省资源）
- ✅ 稳定性更好（不依赖Webhook）

**Webhook触发的问题**：
- ❌ 每次提交都触发（浪费资源）
- ❌ 依赖网络连通性（防火墙问题）
- ❌ 需要配置Webhook（增加复杂度）

**适用场景选择**：
| 场景 | 推荐方式 |
|------|---------|
| 开发分支频繁提交 | pollSCM定时 |
| 发布分支需要立即构建 | 手动触发 |
| 紧急修复需要快速部署 | 手动触发 |

### 2.2 触发时机的权衡

```
凌晨1点构建的考虑：
├── 服务器负载低（白天业务高峰）
├── 网络带宽充足（Docker镜像拉取）
├── 构建失败有时间修复（上班前发现问题）
└── 不影响开发工作（开发人员已下班）
```

## 三、版本管理策略

### 3.1 分支与版本号映射

**设计思路**：
```
分支命名规则          构建版本号
├── v2.10.23-lts  →  2.10.23
├── v2.10.24      →  2.10.24
├── develop       →  develop
└── feature/xxx   →  feature/xxx
```

**代码实现思路**：
```groovy
if (分支名以v开头) {
    版本号 = 去掉v前缀
} else {
    版本号 = 分支名
}
```

**为什么这样设计？**

**场景1：发布版本**
```bash
# 运维操作
git checkout -b v2.10.23-lts
git push origin v2.10.23-lts

# Jenkins自动
镜像标签：registry/eureka:v2.10.23-lts
Maven版本：2.10.23
```
- 版本号清晰（镜像回滚时知道对应哪个版本）
- 与Git标签一致（便于追溯）

**场景2：开发分支**
```bash
# 开发操作
git checkout -b feature/new-api
git push origin feature/new-api

# Jenkins自动
镜像标签：registry/eureka:feature/new-api
Maven版本：feature/new-api
```
- 方便测试（不同特性分支独立测试）
- 不污染版本号（开发镜像和发布镜像区分）

### 3.2 Maven revision变量

**传统方式的痛点**：
```xml
<!-- 每个pom.xml都要硬编码版本号 -->
<version>2.10.23</version>

发布新版本时：
├── 需要修改所有pom.xml
├── 容易遗漏某个模块
└── Git提交历史混乱
```

**revision变量的优势**：
```xml
<!-- 父POM定义 -->
<version>${revision}</version>
<properties>
    <revision>2.10</revision>  <!-- 默认版本 -->
</properties>

<!-- 构建时动态设置 -->
mvn clean package -Drevision=2.10.23
```

**价值**：
- ✅ 单点修改（只改父POM）
- ✅ 构建时灵活（Jenkins动态设置）
- ✅ 减少Git提交（不需要每次修改版本号）

## 四、SDK优先构建策略

### 4.1 什么是SDK？

**MeterSphere的SDK结构**：
```
framework/sdk-parent/
├── domain/          # 数据模型（实体类、DTO）
├── sdk/             # 通用工具类
├── xpack-interface/ # 企业版接口
└── jmeter/          # JMeter组件
```

**为什么需要SDK？**
- 避免重复代码（9个服务共用的代码）
- 统一数据模型（确保数据结构一致）
- 版本统一管理（SDK升级，所有服务同步升级）

### 4.2 SDK构建时机

**什么时候构建SDK？**

**场景1：SDK代码变更**
```bash
修改了domain模块 → 触发SDK构建 → 发布到Maven私有仓库
```

**场景2：首次搭建环境**
```bash
新Jenkins环境 → 先构建SDK → 再构建服务
```

**场景3：版本发布**
```bash
发布v2.10.23 → 先发布SDK 2.10.23 → 再构建服务
```

**什么时候不构建SDK？**
- 只修改业务服务代码
- 日常开发调试
- 紧急修复某个服务

### 4.3 SDK与服务的依赖关系

```
构建顺序：
1️⃣ SDK构建并发布到Maven仓库
2️⃣ 各服务从Maven仓库下载SDK依赖
3️⃣ 各服务编译构建

为什么分两步？
├── 解耦：SDK变更不影响服务构建流程
├── 复用：多个服务共用同一个SDK版本
└── 加速：服务构建时直接下载，不需要重新编译SDK
```

## 五、分层构建的设计思路

### 5.1 为什么要解压JAR包？

**传统Dockerfile的问题**：
```dockerfile
COPY target/app.jar /app.jar
CMD ["java", "-jar", "/app.jar"]
```

**问题分析**：
```
Docker镜像层结构：
├── 基础镜像层（openjdk）
└── 应用层（app.jar）   ← 50MB，任何变更都要重新构建

代码变更 → 整个JAR包重新构建 → Docker缓存失效 → 重新上传50MB
```

**分层构建的优势**：
```dockerfile
COPY lib/     /app/lib/        ← 依赖层，40MB，很少变化
COPY META-INF /app/META-INF    ← 元数据，1MB，很少变化
COPY classes  /app/classes     ← 代码层，9MB，经常变化
```

**Docker缓存利用**：
```
只修改业务代码：
├── lib层（40MB）    → 使用缓存 ✅
├── META-INF层（1MB） → 使用缓存 ✅
└── classes层（9MB）  → 重新构建 ⚠️

上传时间：从50秒降低到10秒（减少80%）
```

### 5.2 JAR解压的时机

**在Jenkins中解压（当前方案）**：
```bash
# 构建阶段
jar -xf target/app.jar -d target/dependency/

# Dockerfile中
COPY target/dependency/lib /app/lib
```

**优点**：
- Dockerfile简单
- 构建过程可控
- 便于注入xpack包

**在Dockerfile中解压（备选方案）**：
```dockerfile
COPY target/app.jar /tmp/
RUN jar -xf /tmp/app.jar -d /app/
```

**缺点**：
- 多一次解压操作（构建时间增加）
- xpack注入复杂

### 5.3 xpack注入机制

**xpack是什么？**
- MeterSphere企业版功能包
- 以JAR包形式提供
- 需要注入到BOOT-INF/lib目录

**注入时机选择**：
```
方案1：源码阶段注入
└── ❌ 需要修改Maven配置，侵入性强

方案2：JAR包构建后注入
└── ❌ 需要重新打包JAR，复杂

方案3：Docker构建前注入（当前方案）
└── ✅ 解压后直接复制到lib目录，简单高效
```

**实现思路**：
```bash
# 1. 解压JAR
jar -xf target/app.jar

# 2. 复制xpack
cp xpack.jar BOOT-INF/lib/

# 3. Docker构建
docker build ...
```

## 六、多架构构建策略

### 6.1 为什么需要多架构？

**使用场景**：
```
x86服务器（amd64）
├── 传统服务器
└── 云服务器（阿里云、腾讯云）

ARM服务器（arm64）
├── Mac M1/M2开发机
├── 华为鲲鹏服务器
└── AWS Graviton实例
```

**不支持多架构的问题**：
```
Mac M1开发 → 构建amd64镜像 → 本地无法运行
服务器部署 → 只有arm64镜像 → 无法部署
```

### 6.2 buildx的工作原理

**传统docker build**：
```bash
docker build -t myapp .
# 只构建当前平台架构（Mac M1 → arm64）
```

**buildx多架构**：
```bash
docker buildx build \
    --platform linux/amd64,linux/arm64 \
    -t myapp:v1 \
    --push .
```

**工作流程**：
```
1️⃣ 在当前机器启动两个构建器
   ├── amd64构建器（通过QEMU模拟）
   └── arm64构建器（原生）

2️⃣ 并行构建两个架构的镜像
   ├── myapp:v1-amd64
   └── myapp:v1-arm64

3️⃣ 创建manifest（镜像索引）
   └── myapp:v1 指向两个架构镜像

4️⃣ Docker pull时自动选择
   └── x86服务器 → 自动拉取amd64镜像
```

### 6.3 构建时间的权衡

**单架构构建**：
```
eureka（amd64）：2分钟
gateway（amd64）：3分钟
总计：9个服务 × 2.5分钟 = 22.5分钟
```

**多架构构建**：
```
eureka（amd64+arm64）：4分钟
gateway（amd64+arm64）：6分钟
总计：9个服务 × 5分钟 = 45分钟
```

**时间增加一倍，值得吗？**
- ✅ 值得：支持更多部署场景
- ✅ 值得：Mac M1开发体验更好
- ✅ 值得：未来ARM服务器趋势

**优化方案**：
- 凌晨构建，时间充足
- 并行构建9个服务
- 实际耗时：约30分钟（并行）

## 七、失败重试机制

### 7.1 为什么需要重试？

**网络不稳定**：
```
Docker镜像推送失败
├── 网络波动
├── Registry临时不可用
└── 带宽限制
```

**资源竞争**：
```
并行构建9个服务
├── 磁盘IO竞争
├── 网络带宽竞争
└── CPU资源竞争
```

### 7.2 重试策略设计

**简单重试（当前方案）**：
```groovy
for (int i = 0; i < 10; i++) {
    try {
        构建并推送镜像
        break  // 成功则退出
    } catch (Exception e) {
        sleep 10  // 等待10秒
        continue  // 继续重试
    }
}
```

**重试参数的权衡**：
```
重试次数：10次
├── 太少：网络问题导致失败
├── 太多：浪费时间
└── 10次：平衡点（约2分钟）

重试间隔：10秒
├── 太短：问题还未恢复
├── 太长：等待时间过长
└── 10秒：给予网络恢复时间
```

### 7.3 失败处理策略

**全部失败怎么办？**
```
重试10次后仍失败
├── 构建停止
├── 企业微信通知运维
├── 保留构建日志
└── 手动排查问题
```

**部分服务失败？**
```
9个服务，2个失败
├── 方案1：全部失败 ← 当前方案（保证一致性）
├── 方案2：部分成功 ← 可能导致版本不一致
└── 方案3：失败服务单独重试 ← 增加复杂度
```

## 八、通知机制设计

### 8.1 通知时机

**成功通知**：
```
构建成功 + 镜像推送成功
├── 分支名
├── 提交ID（前8位）
├── 构建时间
└── 镜像标签
```

**失败通知**：
```
任何阶段失败
├── 失败阶段
├── 错误信息
├── Pipeline链接
└── @相关人员
```

### 8.2 通知渠道选择

**企业微信 vs 钉钉 vs 邮件**

| 渠道 | 优点 | 缺点 |
|------|------|------|
| 企业微信 | 实时性好，移动端方便 | 需要企业账号 |
| 钉钉 | 功能丰富 | 消息容易被忽略 |
| 邮件 | 正式，有记录 | 不够及时 |

**当前选择：企业微信**
- 公司已有企业微信
- 即时通知，快速响应
- 支持Markdown格式

### 8.3 通知内容设计

**信息密度的权衡**：
```
过少：
├── "构建成功"
└── ❌ 缺少关键信息

过多：
├── 完整日志（1000行）
└── ❌ 信息过载，难以阅读

合适：
├── 构建状态
├── 关键信息（分支、版本、时间）
└── ✅ 一眼看出重点
```

## 十、最佳实践总结

### 10.1 设计原则

1. **简单优先**：能简单实现就不要复杂
2. **渐进式**：从基础开始，逐步完善
3. **可维护**：代码清晰，注释充分
4. **可观测**：日志详细，通知及时
5. **容错性**：失败重试，优雅降级

### 10.2 避坑指南

**❌ 不要过度优化**
```
问题：构建时间30分钟
过度：花3天优化到25分钟
合理：凌晨构建，时间充足，不需优化
```

**❌ 不要盲目追新**
```
问题：Jenkins 2.x用得好好的
盲目：升级到最新版，遇到兼容性问题
合理：稳定版本，除非有明确收益
```

**❌ 不要忽视通知**
```
问题：构建失败了不知道
后果：开发调试时才发现镜像没更新
合理：失败立即通知，及时处理
```
