### 二、当前已确认事实
#### 1. 字段传递范围
- 字段对接范围以 `docs/需求平台字段确认清单2.md` 为准
- 核心字段：`dmpNum`（唯一主键）、`name1`（需求名称）、`reqManagerName`（需求负责人）、`actName`（当前环节）、`operationType`（操作类型）、`createTime`（需求提出时间）
- 辅助字段：`parentWfinstCode`、`reqFatherClass`、`reqSonClass`、`systemName`、`upTime`、`assigneeName`、`createdept`、`createUser1`、`deptName`、`startUserName`
- 这些字段用于需求池展示、查询、筛选和追溯

#### 2. RocketMQ 信息
| 方向 | Topic | 发送者 ID | 消费者 ID | 说明 |
|------|-------|-----------|-----------|------|
| 全流程平台 → MeterSphere | `topic-requirement-to-metersphere` | `producer-requirement-to-metersphere` | `consumer-requirement-to-metersphere` | 需求同步消息，通过 `operationType` 区分新建、更新、取消等动作 |
| MeterSphere → 全流程平台 | `topic-metersphere-to-requirement` | `producer-metersphere-to-requirement` | `consumer-metersphere-to-requirement` | 测试计划状态、负责人、排期等回传消息 |

#### 3. 研发时间窗口
- 当前时间为 **2026年3月10日**。
- 目标开发完成时间为 **2026年6月30日**。
- 需要特别说明：**2026年6月30日属于 2026 年第二季度（Q2）末**。
- 因此，本文档统一按 **2026-03-10 ~ 2026-06-30** 的节奏进行规划。

#### 4. 产品方向
- **在测试计划旁边新增”需求池”能力，用于同步需求平台的需求数据**
- 需求池负责”接收、展示、创建测试计划入口”；测试计划继续负责”测试执行、排期、报告与过程沉淀”
- 需求池不做资源隔离，全局可见
- 创建测试计划时不指定项目，自动划分到当前项目下

#### 5. 核心业务规则
- **一个需求强制对应一个测试计划**
- 需求池状态：未创建（PENDING）、已创建（CREATED）、已取消（CANCELLED）
- 未创建状态：显示”创建测试计划”按钮
- 已创建状态：按钮置灰，显示关联的测试计划
- 已取消状态：按钮置灰，需求平台取消的需求
- 需求更新时只同步需求池，不同步已创建的测试计划
- 需求取消后，测试人员需手动更改对应测试计划状态为”已取消”

### 三、范围与边界
#### 本期范围
- 需求平台 → MeterSphere 的需求同步（通过 RocketMQ）
- MeterSphere 需求池设计（列表、详情、筛选、分页）
- 需求池与测试计划的 1:1 关联关系

- MeterSphere → 需求平台的状态回传（测试报告链接等）
- 幂等、审计、历史初始化设计

#### 本期不包含
- 测试人员分配/领取流程（需求池不做分配，直接创建测试计划）
- 需求池资源隔离（需求池全局可见，不分项目）
- 需求与测试计划的数据同步（需求更新不同步到测试计划）
- 复杂的状态映射配置平台（采用固定映射规则）

### 四、关键需求（轻量级 Spec / EARS）
1. **WHEN** 全流程平台向 `topic-requirement-to-metersphere` 发布包含 `operationType` 的需求同步消息，**THE SYSTEM SHALL** 以 `dmpNum` 作为唯一业务键，在 MeterSphere 需求池中新增或更新对应记录。
2. **WHEN** 接收到 `operationType=CREATED` 的消息，**THE SYSTEM SHALL** 创建需求池记录，状态为 PENDING（未创建测试计划）。
3. **WHEN** 接收到 `operationType=UPDATED` 的消息，**THE SYSTEM SHALL** 更新需求池的需求信息，不改变测试计划关联状态。
4. **WHEN** 接收到 `operationType=CANCELLED` 的消息，**THE SYSTEM SHALL** 将需求池状态更新为 CANCELLED，不自动取消已创建的测试计划。
5. **WHEN** 测试人员在需求池中点击”创建测试计划”按钮，**THE SYSTEM SHALL** 检查需求状态，仅允许 PENDING 状态的需求创建测试计划。
6. **WHEN** 创建测试计划成功，**THE SYSTEM SHALL** 自动绑定需求编号，将需求池状态更新为 CREATED，并记录关联的测试计划 ID。
7. **WHEN** 同一需求尝试创建第二个测试计划，**THE SYSTEM SHALL** 拒绝操作并提示”该需求已创建测试计划”。
8. **WHEN** 测试计划关键状态发生变化，**THE SYSTEM SHALL** 通过 `topic-metersphere-to-requirement` 回传 `dmpNum`、`plan_status`、时间字段、负责人字段以及测试报告链接。
9. **WHEN** RocketMQ 消息重复投递或乱序，**THE SYSTEM SHALL** 基于业务键与时间戳进行幂等控制，忽略旧消息。

### 五、核心业务流程

#### 5.1 总体流程

```mermaid
flowchart LR
    A[全流程平台] -->|发送需求消息| B[topic-requirement-to-metersphere]
    B --> C[MeterSphere 需求同步消费者]
    C --> D[需求池 PENDING]
    D --> E[点击创建测试计划按钮]
    E --> F[创建测试计划]
    F --> G[需求池状态 CREATED]
    G --> H[测试执行与状态更新]
    H --> I[topic-metersphere-to-requirement]
    I --> J[全流程平台]
```

核心要点：
- **需求池是同步入口**：需求先进入需求池，状态为 PENDING
- **测试计划是执行载体**：点击创建按钮后，创建测试计划并绑定需求编号
- **一对一强关联**：一个需求只能创建一个测试计划
- **状态独立演进**：需求更新不影响已创建的测试计划

#### 5.2 需求池定位
需求池放在”测试计划”旁边，承担以下职责：
- 展示来自需求平台的需求信息
- 显示当前同步状态、最后同步时间
- 提供”创建测试计划”入口
- 记录与测试计划的关联关系
- 作为需求侧与测试侧之间的缓冲层
- 全局可见，不做资源隔离，不分项目

#### 5.3 需求池状态定义
| 状态 | 说明 | 按钮状态 |
|------|------|----------|
| PENDING（未创建） | 需求已同步，尚未创建测试计划 | 显示”创建测试计划”按钮 |
| CREATED（已创建） | 已创建测试计划 | 按钮置灰，显示关联的测试计划 |
| CANCELLED（已取消） | 需求平台取消的需求 | 按钮置灰 |

#### 5.4 需求池列表建议字段
| 字段 | 来源 | 用途 | 备注 |
|------|------|------|------|
| 需求编号 | `dmpNum` | 唯一业务键 | 点击可查看详情 |
| 需求名称 | `name1` | 展示 | 需求池主标题 |
| 需求状态 | `pool_status` | 状态展示 | PENDING/CREATED/CANCELLED |
| 主流程编码 | `parentWfinstCode` | 辅助追踪 | 用于排障和定位 |
| 需求大类 / 子类 | `reqFatherClass` / `reqSonClass` | 分类筛选 | 便于统计 |
| 所属系统 | `systemName` | 筛选 | 便于系统维度查看 |
| 当前环节 | `actName` | 外部状态展示 | 展示需求平台当前流程节点 |
| 需求负责人 | `reqManagerName` | 需求侧责任人 | 展示需求平台的负责人 |
| 当前处理人 | `assigneeName` | 外部流程跟踪 | 展示需求平台当前处理人 |
| 需求申请部门 | `createdept` | 来源追溯 | |
| 需求申请人 | `createUser1` | 来源追溯 | |
| 需求提出时间 | `createTime` | 排序 / 审计 | |
| 预计上线时间 | `upTime` | 排期参考 | |
| 关联测试计划 | `linked_plan_id` | 执行入口 | 展示计划名称/跳转链接 |
| 最后同步时间 | 系统生成 | 审计 | 消费成功时间 |
| 操作 | 系统生成 | 创建测试计划按钮 | 根据状态控制按钮状态 |

#### 5.5 创建测试计划流程
**触发方式**：在需求池列表中，点击”创建测试计划”按钮

**前置条件**：
- 需求池状态为 PENDING（未创建）
- 需求未被取消

**创建流程**：
1. 点击按钮后，弹出创建测试计划窗口
2. 窗口中自动绑定：
   - 需求编号（只读，不可修改）
   - 计划名称（默认=需求名称，可修改）
3. 填写其他测试计划必填字段（负责人、排期等）
4. 创建成功后：
   - 测试计划的 `requirement_number` = 需求的 `dmpNum`
   - 需求池状态更新为 CREATED
   - 需求池记录关联的测试计划 ID
   - 测试计划自动划分到当前项目下

**并发控制**：
- 数据库层面：`test_plan.requirement_number` 添加唯一索引
- 业务层面：创建前检查需求池状态
- 如果并发创建，后创建的请求会失败并提示”该需求已创建测试计划”

### 六、消息与字段设计
#### 6.1 当前确认的消息通道
| 方向 | Topic | 主要内容 |
|------|-------|----------|
| 全流程平台 → MeterSphere | `topic-requirement-to-metersphere` | 需求同步消息，通过 `operationType` 区分新建、更新、取消、全量初始化 |
| MeterSphere → 全流程平台 | `topic-metersphere-to-requirement` | 需求同步结果、计划状态、负责人、排期、报告链接 |

#### 6.2 MeterSphere 接收字段建议处理方式
| 字段 | 当前建议 | 说明 |
|------|----------|------|
| `dmpNum` | 必填唯一键 | 需求池主键 / 幂等主键 |
| `name1` | 必填展示字段 | 需求标题 |
| `reqManagerName` | 展示字段 | 需求侧负责人，先不等同测试负责人 |
| `actName` | 外部流程节点字段 | 用于展示与状态映射，不直接判断操作类型 |
| `operationType` | 必填操作类型字段 | 用于区分 `CREATED` / `UPDATED` / `CANCELLED` / `FULL_SYNC` |
| `createTime` | 必填时间字段 | 需求创建时间 |
| `parentWfinstCode` | 可选追踪字段 | 辅助排障和检索 |
| `reqFatherClass` / `reqSonClass` | 可选分类字段 | 统计、筛选 |
| `systemName` | 可选筛选字段 | 系统维度分配参考 |
| `upTime` | 可选排期字段 | 排期参考，不直接代表测试截止时间 |
| `assigneeName` | 外部当前处理人 | 仅用于展示外部流程进展 |
| `createdept` / `createUser1` / `deptName` / `startUserName` | 审计与追溯字段 | 在详情页展示即可 |

#### 6.3 建议补充但尚未最终确认的入参字段
为保证幂等和可追踪性，建议全流程平台在消息体中补充以下字段：

| 字段 | 是否建议 | 用途 |
|------|----------|------|
| `operationType` | 必填 | 区分 `CREATED` / `UPDATED` / `CANCELLED` / `FULL_SYNC` |
| `eventTime` | 建议必填 | 记录消息产生时间 |
| `version` | 建议必填 | 幂等和乱序处理 |
| `traceId` | 建议必填 | 跨系统追踪 |
| `operator` | 建议选填 | 标记外部触发人 |

如果短期内无法提供 `version`，则至少需要：
- `dmpNum` + `eventTime`；
- 或 `dmpNum` + 外部系统更新时间；
- 并在消费侧保留最近一次有效消息时间作为兜底规则。

#### 6.4 MeterSphere 回传字段建议
当前建议回传字段与 `docs/需求平台字段确认清单2.md` 中“测试平台标准回传字段”保持一致：
- `dmpNum`
- `plan_status`
- `planned_start_time`
- `planned_end_time`
- `actual_start_time`
- `actual_end_time`
- `principal_users`
- `plan_share_url`

建议补充两个平台通用字段：
- `syncTime`：本次回传时间；
- `traceId`：便于跨系统排障。

### 七、数据落地建议
#### 7.1 需求池主表设计
需求池主表至少包含：
- **外部业务主键**：`dmp_num`（唯一索引）
- **需求基础信息**：`requirement_name`、`req_father_class`、`req_son_class`、`system_name`、`create_time`、`up_time`
- **需求平台字段**：`req_manager_name`、`assignee_name`、`act_name`、`parent_wfinst_code`、`created_dept`、`create_user`、`dept_name`、`start_user_name`
- **需求池状态**：`pool_status`（PENDING/CREATED/CANCELLED）
- **关联关系**：`linked_plan_id`（关联的测试计划ID）
- **同步字段**：`last_sync_time`、`event_time`、`trace_id`
- **审计字段**：`created_at`、`updated_at`

#### 7.2 测试计划表扩展
测试计划表需要新增：
- **需求编号字段**：`requirement_number`（唯一索引，存储 `dmpNum`）
- **状态扩展**：在现有状态枚举中新增 `CANCELLED`（已取消）

#### 7.3 关联关系处理
- **V1 采用 1:1 强关联**：一个需求只能创建一个测试计划
- 需求池表的 `linked_plan_id` 记录关联的测试计划
- 测试计划表的 `requirement_number` 记录来源需求编号
- 通过数据库唯一索引保证一对一关系

### 十、结论
- **核心方向已明确**：MeterSphere 新增需求池，作为需求同步入口，测试计划继续作为执行层
- **核心流程已定稿**：需求先入池（PENDING） → 点击创建测试计划 → 需求池状态变为 CREATED → 测试执行 → 回传状态
- **一对一强关联**：一个需求只能创建一个测试计划，通过数据库唯一索引保证
- **需求池不做资源隔离**：全局可见，不分项目；创建测试计划时自动划分到当前项目
- **需求更新不同步测试计划**：需求池数据更新不影响已创建的测试计划
- **消息通道已确认**：`topic-requirement-to-metersphere` 与 `topic-metersphere-to-requirement` 作为双向通信基线
- **下一步工作**：进入详细设计阶段，补充接口定义、DDL、状态映射表和异常补偿细节
