# 项目创建接口调用与命令行模拟（Metersphere /setting/project/add）

## 1. 我在做：先回答“创建项目只会调用一个接口吗？”
目的是：区分**前端发起的 HTTP 请求数量**与**后端内部会触发的初始化/微服务调用**。
如果不这样做,就无法实现：你用命令行复现时会误判“只打一条 HTTP 就等价于完整创建流程”。

### 1.1 前端（Network 里看到的）请求
- 浏览器创建项目主要就是一次请求：
  - `POST http://<host>:<port>/setting/project/add`

说明：`/setting` 是前端模块前缀（路由/网关路径），实际后端 controller 映射路径是 `/project/add`。

### 1.2 后端（代码）真实入口与内部动作
后端入口在：
- `system-setting/backend/src/main/java/io/metersphere/controller/SystemProjectController.java`
  - `@RequestMapping("/project")`
  - `@PostMapping("/add") public Project addProject(@RequestBody AddProjectRequest project, HttpServletRequest request)`

该方法内部做两件关键事：
1. `systemProjectService.addProject(project)`
   - **真正创建项目**（插表、初始化等都在 service 内）
2. 后端内部额外调用微服务（前端不会看到第二个 network 请求，但后端会发起服务间调用）：
   - `microService.getForData(MicroServiceName.API_TEST, "/api/definition/mock-environment/" + p.getId())`
   - 失败会记录日志，但不影响接口返回 200（代码里 try/catch）

结论：
- **前端HTTP层面**：通常看到一次创建项目请求是正常的
- **后端业务层面**：创建项目会触发一系列初始化与可能的微服务调用，不是“只做一次 insert”

## 4. 我在做：明确必须携带的请求头（来自 Network 截图）
目的是：命令行必须带齐鉴权/CSRF/上下文 header，才能模拟浏览器成功。
如果不这样做,就无法实现：后端会返回 401/403 或 CSRF 校验失败。

关键 headers（至少需要这些）：
- `Content-Type: application/json`
- `X-AUTH-TOKEN: <从浏览器复制>`
- `CSRF-TOKEN: <从浏览器复制>`
- `WORKSPACE: <workspaceId>`
- `PROJECT: <projectId>`

说明：
- 获取方式：浏览器 Network 里复制；或从前端本地存储（localStorage）里取登录信息。
- 前端请求拦截器会自动附带这些 header（可参考：`framework/sdk-parent/frontend/src/plugins/request.js`）。

## 6. 我在做：解释最容易踩坑的字段 issueConfig（嵌套 JSON 字符串）
目的是：避免你命令行复现失败。
如果不这样做,就无法实现：`issueConfig` 转义错误会导致 400 或后端解析异常。

- `issueConfig` 在请求体里是一个 **String**
- 但这个字符串的内容本身又是 JSON（如 `{"jiraKey":"CLM",...}`）
- 因此在外层 JSON 中必须对内层 JSON 的双引号进行转义

## 7. 我在做：提供命令行模拟方案（内网无 curl 时默认 PowerShell）
目的是：你提到内网没有 curl，因此文档默认给出**不依赖 curl**的可用方案。

### 7.1 默认方案：PowerShell（系统自带，最不易转义出错）
我在做：补充 PowerShell 语法注意点（你截图里的 `UnexpectedToken` 就是这里踩坑）。
目的是：避免命令在“解析阶段”就失败，根本发不出 HTTP 请求。
如果不这样做,就无法实现：你会一直卡在 PowerShell 报语法错误。

- 关键点：PowerShell 的哈希表 `@{}` **不能用逗号 `,` 分隔键值对**，要用“换行”或分号 `;`。

我在做：补充 PowerShell 版本/模块差异说明（你截图里的 `ConvertTo-Json` 找不到就是这个问题）。
目的是：避免你按文档执行时因为 PowerShell 版本不同而误以为接口不可用。
如果不这样做,就无法实现：脚本在本地就会报 `CommandNotFoundException`，请求不会发送。

- 检查 PowerShell 版本：
  - `$PSVersionTable.PSVersion`
  - 或 `$Host.Version`
- 结论：如果提示 `ConvertTo-Json` 不存在，优先使用第 7.2 节的“读文件发送”方案（不依赖该 cmdlet）。

```powershell
$headers = @{
  "Content-Type" = "application/json"
  "X-AUTH-TOKEN" = "<从浏览器复制>"
  "CSRF-TOKEN"   = "<从浏览器复制>"
  "WORKSPACE"    = "75fd1cc9-2624-4373-8012-1baed75cc7d3"
  "PROJECT"      = "<从浏览器复制或当前projectId>"
}

$bodyObj = @{
  apiTemplateId  = "1bb1fae3-5d83-45a1-80ab-31135d91de39"
  caseTemplateId = "02e89340-22e0-4cd3-9402-a79c1e22f088"
  createUser     = "admin"
  issueConfig    = '{"jiraKey":"CLM","jiraIssueTypeId":"10102","jiraStoryTypeId":"10102"}'
  issueTemplateId = $null
  name           = "jira 保全项目-变更需求"
  platform       = "Jira"
  protocol       = "http"
  workspaceId    = "75fd1cc9-2624-4373-8012-1baed75cc7d3"
}

$body = ($bodyObj | ConvertTo-Json -Compress)
Invoke-RestMethod -Method Post -Uri "http://10.0.131.125:8081/setting/project/add" -Headers $headers -ContentType "application/json" -Body $body
```

### 7.2 PowerShell（更稳）：把 JSON 存文件，再读取发送
我在做：提供文件化的请求体方式，避免在命令行里处理长 JSON/转义。
目的是：减少手工输入错误，尤其是 `issueConfig` 这种嵌套 JSON 字符串字段。
如果不这样做,就无法实现：命令行里极易因为引号/转义写错导致 400。

我在做：强调该方案同时兼容“没有 `ConvertTo-Json` 的旧版 PowerShell”。
目的是：在内网精简环境下也能稳定发出请求。
如果不这样做,就无法实现：你会被 PowerShell 版本差异卡住，无法复现接口。

1) 创建 `payload.json`（内容用第 5.1 节的 JSON）

2) 执行：
```powershell
$headers = @{
  "X-AUTH-TOKEN" = "<从浏览器复制>"
  "CSRF-TOKEN"   = "<从浏览器复制>"
  "WORKSPACE"    = "75fd1cc9-2624-4373-8012-1baed75cc7d3"
  "PROJECT"      = "<从浏览器复制或当前projectId>"
}

$body = Get-Content -Raw -Encoding UTF8 .\payload.json
Invoke-RestMethod -Method Post -Uri "http://10.0.131.125:8081/setting/project/add" -Headers $headers -ContentType "application/json" -Body $body
```

### 7.2.1 可选：旧版 PowerShell 生成 JSON（当你不想手工写 payload.json）
我在做：提供一个不依赖 `ConvertTo-Json` 的 JSON 生成方式（使用 .NET 序列化）。

  createUser      = "admin"

  issueTemplateId = $null

### 7.2.2 兜底方案：没有 Invoke-RestMethod 时（PowerShell 2.0/精简环境）
我在做：提供一个完全不依赖 `Invoke-RestMethod` / `ConvertTo-Json` 的调用方式（直接用 .NET 发 HTTP）。
目的是：兼容你截图里这种 `Invoke-RestMethod` 也不存在的环境。
如果不这样做,就无法实现：命令会一直报 `CommandNotFoundException`，请求根本发不出去。

执行前建议先确认命令是否存在：
- `powershell -Command "$Host.Version"`
- `powershell -Command "Get-Command Invoke-RestMethod"`

#### 7.2.2.1 推荐：System.Net.WebClient（最短）
我在做：用 `WebClient.UploadString` 直接发送 JSON 字符串。
目的是：脚本最短、最少依赖。
如果不这样做,就无法实现：你需要升级 PowerShell 才能调用接口。

```powershell
# 说明：建议使用 payload.json（最稳），避免在命令行里拼 JSON
$url = "http://10.0.131.125:8081/setting/project/add"

$wc = New-Object System.Net.WebClient
$wc.Headers.Add("Content-Type", "application/json")
$wc.Headers.Add("X-AUTH-TOKEN", "<从浏览器复制>")
$wc.Headers.Add("CSRF-TOKEN", "<从浏览器复制>")
$wc.Headers.Add("WORKSPACE", "75fd1cc9-2624-4373-8012-1baed75cc7d3")
$wc.Headers.Add("PROJECT", "<从浏览器复制或当前projectId>")

# 说明：PowerShell 2.0 不支持 Get-Content -Raw，这里用 .NET 读取文件（兼容性更好）
$payloadPath = (Resolve-Path .\payload.json).Path
$body = [System.IO.File]::ReadAllText($payloadPath, [System.Text.Encoding]::UTF8)
$resp = $wc.UploadString($url, "POST", $body)
$resp
```

#### 7.2.2.2 可选：HttpWebRequest（可控性更强）
我在做：提供更底层的 `HttpWebRequest` 示例，便于你后续加超时、打印状态码。
目的是：当你需要更精细的错误定位（比如 401/403/400）时更方便。
如果不这样做,就无法实现：某些网络/代理问题下难以定位失败原因。

```powershell
$url = "http://10.0.131.125:8081/setting/project/add"
$payloadPath = (Resolve-Path .\payload.json).Path
$body = [System.IO.File]::ReadAllText($payloadPath, [System.Text.Encoding]::UTF8)
$bytes = [System.Text.Encoding]::UTF8.GetBytes($body)

$req = [System.Net.HttpWebRequest]::Create($url)
$req.Method = "POST"
$req.ContentType = "application/json"
$req.Headers.Add("X-AUTH-TOKEN", "<从浏览器复制>")
$req.Headers.Add("CSRF-TOKEN", "<从浏览器复制>")
$req.Headers.Add("WORKSPACE", "75fd1cc9-2624-4373-8012-1baed75cc7d3")
$req.Headers.Add("PROJECT", "<从浏览器复制或当前projectId>")
$req.ContentLength = $bytes.Length

$stream = $req.GetRequestStream()
$stream.Write($bytes, 0, $bytes.Length)
$stream.Close()

try {
  $resp = $req.GetResponse()
  $reader = New-Object System.IO.StreamReader($resp.GetResponseStream())
  $reader.ReadToEnd()
} catch [System.Net.WebException] {
  # 说明：这里可以拿到状态码与响应体，便于定位 401/403/400
  $errResp = $_.Exception.Response
  if ($errResp -ne $null) {
    $statusCode = [int]$errResp.StatusCode
    $reader = New-Object System.IO.StreamReader($errResp.GetResponseStream())
    $errBody = $reader.ReadToEnd()
    "HTTP Status: $statusCode"
    $errBody
  } else {
    throw
  }
}
```
