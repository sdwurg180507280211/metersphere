## 一、快速开始

### 1.1 前置条件

**服务器要求**：
- Linux系统（Ubuntu 20.04+ / CentOS 7+）
- Docker 20.10+
- docker-compose 2.0+
- 8C16G内存（最低配置）

**Jenkins要求**：
- Jenkins 2.300+
- 安装插件：SSH Agent、Pipeline、Credentials Binding

**权限要求**：
- 部署用户具有Docker权限
- SSH免密登录到部署服务器

### 1.2 五分钟快速部署

```bash
# 1. 克隆项目到服务器
cd /opt
git clone <your-git-repo> metersphere
cd metersphere

# 2. 创建scripts目录并复制脚本
mkdir -p scripts
# 将scripts/*复制到这里

# 3. 配置环境
cp .env.example install.conf
vim install.conf  # 修改配置项

# 4. 执行部署
chmod +x scripts/*.sh
./scripts/deploy.sh test v2.10.23-lts

# 5. 验证
./scripts/health-check.sh
```

## 二、脚本使用说明

### 2.1 deploy.sh - 主部署脚本

**功能**：一键部署或更新所有服务

**用法**：
```bash
./scripts/deploy.sh [环境] [版本号]

# 示例
./scripts/deploy.sh test v2.10.23-lts    # 部署测试环境
./scripts/deploy.sh prod v2.10.23-lts    # 部署生产环境
```

**执行流程**：
```
1. 前置检查（Docker、配置文件）
2. 创建备份点
3. 检查基础设施（MySQL、Redis、Kafka）
4. 按顺序更新所有服务
5. 健康检查
6. 发送企业微信通知
```

**预计耗时**：10-15分钟

### 2.2 update-service.sh - 单服务更新

**功能**：更新指定的单个服务

**用法**：
```bash
./scripts/update-service.sh <服务名> [版本号]

# 示例
./scripts/update-service.sh api-test v2.10.23-lts
./scripts/update-service.sh gateway develop
```

**执行流程**：
```
1. 备份当前镜像
2. 拉取最新镜像
3. 停止旧容器
4. 启动新容器
5. 健康检查
6. 失败自动回滚
```

**预计耗时**：2-3分钟

### 2.3 rollback.sh - 回滚脚本

**功能**：快速回滚到上一个版本

**用法**：
```bash
./scripts/rollback.sh <服务名> [版本号]

# 示例1：回滚到最近的backup版本
./scripts/rollback.sh gateway backup

# 示例2：回滚到指定版本
./scripts/rollback.sh api-test v2.10.22
```

**预计耗时**：2-3分钟

### 2.4 health-check.sh - 健康检查

**功能**：检查所有服务健康状态和资源使用

**用法**：
```bash
./scripts/health-check.sh
```

**输出示例**：
```
✅ eureka: 健康
✅ gateway: 健康
✅ system-setting: 健康
✅ api-test: 健康
...

资源使用：
NAME              CPU%    MEM USAGE / LIMIT
eureka            2.5%    450MB / 512MB
gateway           5.2%    650MB / 800MB
api-test          8.1%    850MB / 1GB
```

### 2.5 start-all.sh / stop-all.sh - 启停服务

**功能**：按正确顺序启动或停止所有服务

**用法**：
```bash
./scripts/start-all.sh    # 启动
./scripts/stop-all.sh     # 停止
```

### 2.6 view-logs.sh - 查看日志

**功能**：方便查看各服务日志

**用法**：
```bash
./scripts/view-logs.sh                  # 交互式选择
./scripts/view-logs.sh gateway          # 直接查看gateway日志
./scripts/view-logs.sh all              # 查看所有服务日志
```

## 三、Jenkins集成

### 3.1 Jenkins配置步骤

**步骤1：创建Jenkins凭证**

在Jenkins中配置以下凭证：
```
1. SSH Private Key
   - ID: metersphere-deploy-key
   - 用途：SSH连接到部署服务器
   
2. Secret Text
   - ID: wechat-bot-webhook
   - 内容：企业微信Webhook URL
   
3. Secret Text  
   - ID: test-server-ip
   - 内容：测试服务器IP
   
4. Secret Text
   - ID: prod-server-ip
   - 内容：生产服务器IP
```

**步骤2：配置Maven settings.xml**

```
Jenkins → Manage Jenkins → Managed files
→ Add a new Config → Global Maven settings.xml
→ ID: metersphere-maven
```

**步骤3：创建Pipeline任务**

```
1. New Item → Pipeline
2. 名称：MeterSphere-CI/CD
3. Pipeline script from SCM
4. Repository URL：<your-git-repo>
5. Script Path：Jenkinsfile.enhanced
```

**步骤4：配置构建触发器**

Jenkins任务已配置`pollSCM('0 1 * * *')`，每天凌晨1点自动检查代码变化并构建。

### 3.2 构建参数说明

| 参数 | 默认值 | 说明 |
|------|--------|------|
| buildSdk | false | 是否只构建SDK |
| deployToTest | true | develop分支是否自动部署到测试环境 |
| runTests | false | 是否运行单元测试 |
| frontendLink | "" | 前端链接（可选） |

### 3.3 分支与部署策略

```
develop分支：
├── 自动构建镜像
├── 推送到Registry
└── 自动部署到测试环境（如deployToTest=true）

master分支：
├── 自动构建镜像
├── 推送到Registry
└── 手动触发部署到生产环境（需审批）

feature/*分支：
├── 自动构建镜像
└── 不自动部署
```

## 四、部署服务器配置

### 4.1 创建部署用户

```bash
# 在部署服务器上执行

# 1. 创建deploy用户
sudo useradd -m -s /bin/bash deploy

# 2. 添加到docker组
sudo usermod -aG docker deploy

# 3. 配置sudo免密（可选）
echo "deploy ALL=(ALL) NOPASSWD: ALL" | sudo tee /etc/sudoers.d/deploy

# 4. 切换到deploy用户
sudo su - deploy
```

### 4.2 配置SSH免密登录

```bash
# 在Jenkins服务器上生成密钥对
ssh-keygen -t rsa -b 4096 -C "jenkins@metersphere"

# 复制公钥到部署服务器
ssh-copy-id deploy@<test-server-ip>

# 测试连接
ssh deploy@<test-server-ip> "docker ps"
```

### 4.3 部署目录结构

```bash
/opt/metersphere/
├── bin/                              # 官方脚本
├── scripts/                          # 自动化脚本（我们创建的）
│   ├── functions.sh
│   ├── deploy.sh
│   ├── update-service.sh
│   ├── rollback.sh
│   ├── start-all.sh
│   ├── stop-all.sh
│   ├── health-check.sh
│   └── view-logs.sh
├── docker-compose-*.yml              # Compose配置文件
├── install.conf                      # 环境配置
├── conf/                             # 应用配置
├── logs/                             # 日志目录
├── data/                             # 数据目录
└── backups/                          # 备份目录（自动创建）
```

## 五、日常运维操作

### 5.1 代码发布流程

**场景1：发布新功能到测试环境**

```bash
# 开发机操作
git checkout develop
git merge feature/new-function
git push origin develop

# Jenkins自动执行：
# 1. 检测到代码变化（pollSCM或立即触发）
# 2. 构建Maven项目
# 3. 构建Docker镜像
# 4. 推送到阿里云Registry
# 5. 自动部署到测试环境
# 6. 企业微信通知

# 验证
curl http://test-server:8081/actuator/health
```

**场景2：发布到生产环境**

```bash
# 开发机操作
git checkout master
git merge develop
git tag -a v2.10.24 -m "Release v2.10.24"
git push origin master --tags

# Jenkins执行（到Docker push阶段）
# 然后手动操作：
# 1. 进入Jenkins → MeterSphere-CI/CD → 最新构建
# 2. 点击"Deploy to Production"阶段的"继续"按钮
# 3. 确认部署
# 4. 等待部署完成
# 5. 企业微信通知

# 验证
curl http://prod-server:8081/actuator/health
```

### 5.2 手动部署操作

**全量部署**：
```bash
ssh deploy@test-server
cd /opt/metersphere
./scripts/deploy.sh test v2.10.23-lts
```

**单服务更新**：
```bash
ssh deploy@test-server
cd /opt/metersphere
./scripts/update-service.sh api-test v2.10.23-lts
```

**查看服务状态**：
```bash
ssh deploy@test-server
cd /opt/metersphere
./scripts/health-check.sh
```

### 5.3 故障处理

**服务异常快速回滚**：
```bash
# 1. 确认哪个服务有问题
./scripts/health-check.sh

# 2. 回滚该服务
./scripts/rollback.sh api-test backup
```

**查看服务日志**：

## 七、最佳实践

### 7.1 部署建议

✅ **DO**：
- 先在测试环境验证
- 选择低峰期部署
- 部署后观察10分钟
- 保留最近3个版本
- 重要发布前备份数据库

❌ **DON'T**：
- 直接在生产部署未测试的版本
- 高峰期部署
- 部署后立即离开
- 频繁删除旧镜像
- 跳过备份步骤

### 7.2 版本管理

```bash
# 发布版本命名规范
v主版本.次版本.修订号

# 示例
v2.10.23  # 当前版本
v2.10.24  # bug修复
v2.11.0   # 新功能
v3.0.0    # 重大升级
```

### 7.3 回滚决策

**什么时候需要回滚？**
- 新版本有严重bug影响使用
- 性能明显下降
- 健康检查持续失败
- 数据异常

**回滚时间要求**：
- 发现问题后3分钟内完成回滚
- 回滚后立即验证功能
- 通知受影响用户

## 八、故障排查

### 8.1 部署失败

**现象**：deploy.sh执行失败

**排查步骤**：
```bash
# 1. 查看具体错误信息
# 部署脚本会显示哪个服务失败

# 2. 查看服务日志
./scripts/view-logs.sh <失败的服务>

# 3. 检查容器状态
docker ps -a | grep <失败的服务>

# 4. 检查镜像是否存在
docker images | grep <失败的服务>

# 5. 手动启动测试
docker-compose -f docker-compose-<服务>.yml up
```

**常见原因**：
- 镜像拉取失败（网络问题）
- 依赖服务未启动（顺序问题）
- 配置错误（install.conf）
- 端口冲突

### 8.2 服务启动超时

**现象**：wait_healthy超时

**排查**：
```bash
# 1. 查看容器是否启动
docker ps | grep <服务名>

# 2. 查看启动日志
docker logs <服务名> --tail 100

# 3. 进入容器排查
docker exec -it <服务名> sh

# 4. 检查端口监听
docker exec <服务名> netstat -tunlp

# 5. 检查健康检查配置
docker inspect <服务名> | grep -A 10 Health
```

### 8.3 Jenkins构建失败

**现象**：Jenkins Pipeline失败

**排查步骤**：
```
1. 查看Console Output
   - 找到具体的错误信息
   
2. 常见问题：
   - Maven依赖下载失败 → 检查settings.xml
   - Docker构建失败 → 检查Docker daemon
   - 推送镜像失败 → 检查Registry凭证
   - SSH连接失败 → 检查SSH密钥

3. 重新触发构建
   - 点击"Rebuild"
```

## 九、进阶配置

### 9.1 多环境管理

**创建环境配置**：
```bash
# 测试环境
cp .env.example install.conf.test
vim install.conf.test

# 生产环境
cp .env.example install.conf.prod
vim install.conf.prod

# 切换环境
```

### 9.2 自动化测试集成

**在部署后运行自动化测试**：
```bash
# 在deploy.sh的最后添加

    log_info "运行自动化测试..."
    ./scripts/run-tests.sh
fi
```

**集成Prometheus**：

```bash

```

### Q2：部署失败如何回滚？

