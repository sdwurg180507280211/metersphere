# 概念说明

本文档解释背景知识，适合新人阅读。

## 1. jenkins 用户 vs deploy 用户

这是两个不同的 Linux 用户，职责分离：

| 维度 | jenkins 用户 | deploy 用户 |
|------|--------------|-------------|
| **职责** | 构建：编译、打包、推送镜像 | 部署：拉取镜像、启停服务 |
| **工作目录** | `/var/lib/jenkins/workspace/` | `/opt/metersphere/` |
| **典型命令** | `mvn`, `npm`, `docker build`, `docker push` | `docker-compose pull`, `docker-compose up` |
| **权限需求** | Docker 构建权限、ACR 推送权限 | Docker 运行权限、部署目录写权限 |

### 为什么要分离？

1. **安全**：构建环境被攻破不影响部署环境
2. **隔离**：构建机需要 JDK/Maven/Node，部署机只需 Docker
3. **审计**：出问题能快速定位是构建阶段还是部署阶段

### 执行流程中的用户切换

```
jenkins 用户                          deploy 用户
    │                                     │
    ├─ mvn clean package                  │
    ├─ docker build                       │
    ├─ docker push                        │
    │                                     │
    └─────── SSH 连接 ────────────────────┤
                                          ├─ docker-compose pull
                                          ├─ docker-compose up
                                          │
```

## 2. Pipeline Job vs Multibranch Pipeline

### Pipeline Job

- **一个 Job = 一个配置**
- 适合：固定分支、固定参数
- 手动指定分支或通过参数传入

```
metersphere-build (Pipeline Job)
└── #1, #2, #3... (每次构建)
```

### Multibranch Pipeline

- **一个母 Job + 每分支一个子 Job**
- 适合：多分支并行、自动发现新分支
- 自动扫描仓库创建/删除分支 Job

```
metersphere-multibranch (Multibranch Pipeline)
├── develop (子 Job)
│   └── #1, #2, #3...
├── develop-v2.10.26 (子 Job)
│   └── #1, #2...
└── master (子 Job)
    └── #1...
```

### BRANCH_NAME 变量差异

| Job 类型 | env.BRANCH_NAME |
|----------|-----------------|
| Pipeline | 可能为 null（需从 GIT_BRANCH 获取） |
| Multibranch | 有值（如 `develop-v2.10.26`） |

**兼容写法**：优先用 `BRANCH_NAME`，fallback 到 `GIT_BRANCH`。

## 3. 代码仓库 vs 镜像仓库

这是两个完全不同的东西：

| 维度 | 代码仓库 (Gitee/GitHub) | 镜像仓库 (ACR/DockerHub) |
|------|-------------------------|--------------------------|
| **存储内容** | 源代码 | Docker 镜像 |
| **访问协议** | Git (HTTPS/SSH) | Docker Registry API |
| **典型操作** | `git clone`, `git push` | `docker pull`, `docker push` |
| **凭证类型** | SSH Key 或 Personal Access Token | 用户名 + 密码 |

### 本项目的仓库

- **代码仓库**：Gitee（存放 Java/前端代码、Jenkinsfile）
- **镜像仓库**：阿里云 ACR 个人版（存放构建好的 Docker 镜像）

### 常见误区

❌ "阿里云仓库"不明确，可能指 ACR 也可能指代码托管
✅ 明确说"镜像仓库"或"代码仓库"

❌ 用 ACR 凭证去拉 Git 代码
✅ Git 用 `gitee-token`，Docker 用 `aliyun-acr-personal`

## 4. ACR 仓库结构

```
Registry (登录地址)
└── crpi-xxx.cn-beijing.personal.cr.aliyuncs.com

Namespace (命名空间)
└── metersphere-edy

Repository (仓库/镜像名)
├── eureka
├── gateway
└── api-test

Tag (标签/版本)
├── develop
├── develop-v2.10.26
└── v2.10.26-lts
```

完整镜像名：`Registry/Namespace/Repository:Tag`

```
crpi-xxx.cn-beijing.personal.cr.aliyuncs.com/metersphere-edy/eureka:develop-v2.10.26
```

### VPC 内网地址

ECS 在 VPC 内时，使用带 `-vpc` 的域名更快：

```
公网：crpi-xxx.cn-beijing.personal.cr.aliyuncs.com
VPC：crpi-xxx-vpc.cn-beijing.personal.cr.aliyuncs.com
```

## 5. Jenkinsfile 路径

Jenkinsfile 是流水线脚本，存放在代码仓库中。

### 默认位置

Jenkins 默认找仓库根目录的 `Jenkinsfile`。

### 自定义位置

本项目用 `Jenkinsfile.enhanced`，需在 Job 配置中指定：

```
Build Configuration → Script Path → Jenkinsfile.enhanced
```

### 为什么不直接叫 Jenkinsfile？

保留原始 `Jenkinsfile` 作为参考，增强版用新名字便于区分。

## 6. Jenkins 凭证类型

| 类型 | 用途 | 示例 |
|------|------|------|
| Username with password | 用户名密码认证 | ACR 登录、Gitee Token |
| SSH Username with private key | SSH 连接 | 部署服务器 |
| Secret text | 单个敏感值 | 服务器 IP、API Key |
| Secret file | 敏感文件 | 证书、配置文件 |

### 在 Jenkinsfile 中使用

```groovy
// Username with password
withCredentials([usernamePassword(credentialsId: 'xxx', usernameVariable: 'USER', passwordVariable: 'PASS')])

// SSH
sshagent(['xxx'])

// Secret text
environment {
    MY_SECRET = credentials('xxx')
}
```

## 7. Docker buildx 多架构构建

### 为什么需要多架构？

- `linux/amd64`：Intel/AMD 服务器
- `linux/arm64`：M1/M2 Mac、ARM 服务器

一次构建同时生成两种架构镜像，部署时自动拉取匹配的版本。

### 关键命令

```bash
docker buildx build \
    --platform linux/amd64,linux/arm64 \
    --push \
    -t 镜像名:标签 \
    .
```

> 注意：多架构构建必须带 `--push`，不能先 build 再 push。
