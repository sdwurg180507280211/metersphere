## 一、部署策略概览

### 1.1 MeterSphere的部署特点

**9个微服务的协调**：
```
挑战：
├── 服务间有依赖关系
├── 启动顺序有要求
├── 某个服务失败影响整体
└── 需要统一版本管理

目标：
├── 零停机部署
├── 快速回滚
├── 可观测性强
└── 运维简单
```

### 1.2 部署策略对比

| 策略 | 停机时间 | 复杂度 | 资源消耗 | 适用场景 |
|------|---------|--------|---------|---------|
| 停机部署 | 5-10分钟 | 低 | 低 | 维护窗口期 |
| 滚动更新 | 0 | 中 | 中 | 日常更新 |
| 蓝绿部署 | 0 | 高 | 高（2倍） | 重要发布 |
| 金丝雀发布 | 0 | 很高 | 中 | 风险验证 |

**MeterSphere当前方案：滚动更新**
- 原因：平衡了停机时间、复杂度和资源消耗
- 未来：可考虑蓝绿部署（生产环境）

## 二、启动顺序设计

### 2.1 为什么顺序很重要？

**错误的顺序示例**：
```
api-test先启动
├── 尝试注册到Eureka
├── Eureka还未启动
├── 注册失败，反复重试
├── 占用CPU和网络资源
└── 最终超时失败

正确的顺序：
Eureka先启动 → api-test启动 → 注册成功
```

### 2.2 服务依赖关系图

```
┌─────────────┐
│   外部依赖  │
│ MySQL/Redis │
│   /Kafka    │
└──────┬──────┘
       │
       ↓
┌─────────────┐
│   Eureka    │  ← 第1层：服务注册中心
│  注册中心   │
└──────┬──────┘
       │
       ↓
┌─────────────┐
│system-setting│ ← 第2层：基础服务
│  基础服务    │     （用户、权限）
└──────┬──────┘
       │
       ↓
┌─────────────┐
│   Gateway   │  ← 第3层：API网关
│    网关     │
└──────┬──────┘
       │
       ├─────────┬─────────┬─────────┐
       ↓         ↓         ↓         ↓
  ┌────────┐ ┌────────┐ ┌────────┐ ...
  │api-test│ │test-   │ │perfor- │     ← 第4层：业务服务
  │        │ │track   │ │mance   │     （可并行启动）
  └────────┘ └────────┘ └────────┘
```

### 2.3 启动脚本设计

**核心思想：分层启动 + 健康等待**

```bash
#!/bin/bash
# 完整启动脚本

echo "=== MeterSphere服务启动 ==="

# 第0层：基础设施检查
echo "[0/4] 检查基础设施..."
check_mysql
check_redis
check_kafka

# 第1层：Eureka
echo "[1/4] 启动Eureka..."
docker-compose -f docker-compose-eureka.yml up -d
wait_healthy "eureka" 120  # 等待最多120秒

# 第2层：system-setting
echo "[2/4] 启动system-setting..."
docker-compose -f docker-compose-system-setting.yml up -d
wait_healthy "system-setting" 60

# 第3层：Gateway
echo "[3/4] 启动Gateway..."
docker-compose -f docker-compose-gateway.yml up -d
wait_healthy "gateway" 60

# 第4层：业务服务（并行）
echo "[4/4] 启动业务服务..."
启动其他6个服务（并行）

echo "✅ 所有服务启动完成"
```

**健康等待函数**：
```bash
wait_healthy() {
    local service=$1
    local timeout=$2
    local count=0
    
    while [ $count -lt $timeout ]; do
        if docker exec $service nc -z localhost 8000 2>/dev/null; then
            echo "✓ $service 已就绪"
            return 0
        fi
        echo "等待 $service 启动... ($count/$timeout)"
        sleep 1
        count=$((count + 1))
    done
    
    echo "✗ $service 启动超时"
    return 1
}
```

## 三、滚动更新实战

### 3.1 滚动更新的核心原则

**原则1：逐个更新**
```
不要：
└── 同时更新所有服务（可能全部失败）

要做：
└── 按顺序一个一个更新（失败影响可控）
```

**原则2：等待就绪**
```
不要：
└── 启动后立即更新下一个（服务可能未就绪）

要做：
└── 健康检查通过后再更新下一个
```

**原则3：保留回滚**
```
不要：
└── 更新后立即删除旧镜像（无法回滚）

要做：
└── 保留最近3个版本的镜像
```

### 3.2 单服务更新流程

```bash
#!/bin/bash
# update-service.sh <服务名>

SERVICE=$1

echo "=== 更新 ${SERVICE} 服务 ==="

# 步骤1：备份当前镜像（可选但推荐）
echo "[1/6] 备份当前镜像..."
CURRENT_IMAGE=$(docker inspect ${SERVICE} --format='{{.Config.Image}}')
docker tag ${CURRENT_IMAGE} ${CURRENT_IMAGE}-backup-$(date +%Y%m%d)

# 步骤2：拉取最新镜像
echo "[2/6] 拉取最新镜像..."
docker-compose -f docker-compose-${SERVICE}.yml pull

# 步骤3：停止旧容器（不删除）
echo "[3/6] 停止旧容器..."
docker-compose -f docker-compose-${SERVICE}.yml stop

# 步骤4：启动新容器
echo "[4/6] 启动新容器..."
docker-compose -f docker-compose-${SERVICE}.yml up -d

# 步骤5：健康检查
echo "[5/6] 健康检查..."
if wait_healthy ${SERVICE} 120; then
    echo "✅ ${SERVICE} 更新成功"
    
    # 步骤6：清理旧容器
    echo "[6/6] 清理旧容器..."
    docker-compose -f docker-compose-${SERVICE}.yml rm -f
else
    echo "❌ ${SERVICE} 启动失败，开始回滚..."
    rollback_service ${SERVICE}
    exit 1
fi
```

### 3.3 全量更新流程

```bash
#!/bin/bash
# update-all-services.sh

# 定义更新顺序
SERVICES=(
    "eureka"
    "system-setting"
    "gateway"
    "api-test"
    "test-track"
    "performance-test"
    "project-management"
    "report-stat"
    "workstation"
)

echo "=== 开始滚动更新所有服务 ==="
echo "预计耗时：约15分钟"

for service in "${SERVICES[@]}"; do
    echo ""
    echo "────────────────────────────────"
    echo "正在更新：${service}"
    echo "────────────────────────────────"
    
    if ./update-service.sh ${service}; then
        echo "✅ ${service} 更新完成"
    else
        echo "❌ ${service} 更新失败"
        echo "停止后续更新"
        exit 1
    fi
    
    # 短暂等待，确保服务稳定
    sleep 5
done

echo ""
echo "✅ 所有服务更新完成"
```

### 3.4 更新时的流量管理

**Gateway如何做到零停机？**

```
方案1：健康检查 + 优雅关闭
├── 新Gateway启动
├── 健康检查通过
├── 旧Gateway收到SIGTERM
├── 旧Gateway停止接受新请求
├── 旧Gateway处理完现有请求（最多30秒）
└── 旧Gateway退出

效果：
└── 用户无感知，无请求丢失
```

**Spring Boot优雅关闭配置**：
```yaml
# application.yml
server:
  shutdown: graceful

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
```

## 四、回滚策略

### 4.1 快速回滚的必要性

**场景：新版本有严重bug**
```
09:00 发布新版本v2.10.24
09:05 用户报告无法登录
09:06 确认是新版本问题
09:07 开始回滚
09:10 回滚完成，服务恢复

关键：3分钟内完成回滚
```

### 4.2 回滚方案设计

**方案1：使用备份镜像（推荐）**
```bash
#!/bin/bash
# rollback-service.sh <服务名>

SERVICE=$1

echo "=== 回滚 ${SERVICE} 服务 ==="

# 找到备份镜像（当天的backup）
BACKUP_IMAGE=$(docker images | grep ${SERVICE} | grep backup | grep $(date +%Y%m%d) | awk '{print $1":"$2}' | head -1)

if [ -z "$BACKUP_IMAGE" ]; then
    echo "❌ 未找到备份镜像"
    exit 1
fi

echo "发现备份镜像：$BACKUP_IMAGE"

# 修改docker-compose.yml中的镜像
sed -i "s|image:.*|image: $BACKUP_IMAGE|" docker-compose-${SERVICE}.yml

# 重启服务
docker-compose -f docker-compose-${SERVICE}.yml up -d

echo "✅ 回滚完成"
```

**方案2：使用特定版本标签**
```bash
# 直接指定旧版本
MS_IMAGE_TAG=v2.10.23-lts docker-compose -f docker-compose-api-test.yml up -d
```

### 4.3 回滚验证

```bash

# 2. 检查镜像版本

# 4. 健康检查

# 5. 功能验证
```

## 六、故障排查与应急

### 6.1 常见问题排查

**问题1：服务启动失败**

```bash
# 排查步骤
1. 查看容器状态
   docker ps -a | grep api-test

2. 查看启动日志
   docker logs api-test

3. 常见原因：
   ├── 端口被占用
   ├── 依赖服务未启动
   ├── 配置错误
   └── 资源不足

4. 进入容器排查
   docker exec -it api-test sh
```

**问题2：服务间通信失败**

```bash
# 排查步骤
1. 检查网络
   docker network inspect ms-network

2. 测试连通性
   docker exec gateway ping api-test
   docker exec gateway nc -zv api-test 8004

3. 检查Eureka注册
   curl http://eureka:8761/eureka/apps

4. 常见原因：
   ├── 服务未注册到Eureka
   ├── 网络配置错误
   └── 防火墙规则
```

**问题3：性能问题**

```bash
# 排查步骤
1. 查看资源使用
   docker stats

2. 查看JVM堆内存
   docker exec api-test jmap -heap 1

3. 查看线程
   docker exec api-test jstack 1

4. 常见原因：
   ├── 内存不足（OOM）
   ├── 线程池满
   ├── 数据库慢查询
   └── 网络延迟
```

### 6.2 应急预案

**预案1：单服务故障**
```
影响：该服务不可用，其他服务正常
处理：
1. 快速回滚该服务
2. 通知用户（如需要）
3. 排查问题
4. 修复后重新发布
```

**预案2：Gateway故障**
```
影响：整个系统不可用
处理：
1. 立即回滚Gateway
2. 通知所有用户
3. 紧急修复
4. 验证后重新发布
```

**预案3：Eureka故障**
```
影响：服务发现失败，新请求受影响
处理：
1. 已注册的服务短期内可正常工作
2. 尽快重启Eureka
3. 确认所有服务重新注册
```

### 7.2 部署原则

**1. 小步快跑**
```
不要：
└── 一次发布10个新功能

要做：
└── 每次发布1-2个功能，频繁发布
```

**2. 先测试后生产**
```
流程：
开发环境 → 测试环境 → 预生产环境 → 生产环境
```

**3. 高峰期避免发布**
```
时间选择：
├── 工作日晚上（用户少）
├── 周末（如需要）
└── 避开周一早上、月初等高峰
```

**4. 观察期**
```
发布后：
├── 观察10分钟（检查日志、监控）
├── 确认无问题后离开
└── 保持待命（1小时内）
```

## 八、性能优化

### 8.1 资源优化

**合理的资源分配**：
```yaml
# 根据实际负载调整
eureka: 512MB        # 轻量级
gateway: 800MB       # 中等负载
api-test: 1GB        # 重量级（JMeter）
```

**JVM参数优化**：
```bash
# 堆内存设置（容器内存的70-80%）
-Xms700m -Xmx700m（容器1GB）

# GC选择
-XX:+UseG1GC（推荐）
```

### 8.2 启动优化

**并行启动**：
```bash
# 同一层的服务可以并行
docker-compose -f docker-compose-api-test.yml up -d &
docker-compose -f docker-compose-test-track.yml up -d &
docker-compose -f docker-compose-performance-test.yml up -d &
wait  # 等待所有后台任务完成
```

**启动时间优化**：
```
优化前：
├── Eureka: 2分钟
├── Gateway: 3分钟
└── 总计：约15分钟

优化后：
├── Eureka: 1分钟（JVM参数优化）
├── Gateway: 2分钟（依赖包缓存）
└── 总计：约8分钟（并行启动）
```

### 8.3 网络优化

**减少跨服务调用**：
```
不好：
└── Gateway → api-test → test-track → system-setting
    （链式调用，延迟累加）

好的：
└── Gateway → 并行调用 → api-test、test-track
    （并行调用，延迟最小）
```

## 十一、总结

### 11.1 核心要点

1. **顺序重要**：Eureka → system-setting → Gateway → 业务服务
2. **等待就绪**：健康检查通过后再继续
3. **保留回滚**：旧版本镜像保留3天
4. **监控到位**：实时监控服务状态
5. **快速响应**：故障3分钟内回滚

### 11.2 运维成熟度

**Level 1：手工运维**
- 手动执行命令
- 无监控
- 出问题再处理

**Level 2：脚本化（当前）**
- 自动化脚本
- 基础监控
- 预案准备

**Level 3：自动化（目标）**

## 附录：完整脚本示例

**master-deploy.sh（主部署脚本）**：
```bash
#!/bin/bash
# MeterSphere主部署脚本
# 用途：一键部署或更新所有服务

set -e  # 遇到错误立即退出

# 导入配置
source ./install.conf

# 导入函数库
source ./scripts/functions.sh

# 主函数
main() {
    echo "=== MeterSphere部署工具 ==="
    echo "1. 全新部署"
    echo "2. 更新所有服务"
    echo "3. 更新单个服务"
    echo "4. 回滚服务"
    echo "5. 健康检查"
    echo "0. 退出"
    
    read -p "请选择操作：" choice
    
    case $choice in
        1) fresh_deploy ;;
        2) update_all ;;
        3) update_single ;;
        4) rollback_service ;;
        5) health_check ;;
        0) exit 0 ;;
        *) echo "无效选择"; exit 1 ;;
    esac
}

main
```
