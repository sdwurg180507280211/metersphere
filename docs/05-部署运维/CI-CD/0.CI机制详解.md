## 一、CI基本概念

### 1.1 什么是CI（持续集成）？

**CI的定义**：
```
持续集成（Continuous Integration）是一种软件开发实践
├── 开发人员频繁地将代码集成到主分支
├── 每次集成都通过自动化构建和测试来验证
└── 尽早发现集成错误，快速反馈
```

**CI的核心价值**：
1. **快速反馈**：代码提交后立即发现问题
2. **降低风险**：小步快跑，避免大爆炸式集成
3. **提高质量**：自动化测试保证代码质量
4. **减少冲突**：频繁集成减少代码冲突

### 1.2 CI的核心流程

**标准CI流程**：
```
代码提交
    ↓
自动触发构建
    ↓
代码检查（可选）
    ├── 静态代码分析
    ├── 代码规范检查
    └── 安全漏洞扫描
    ↓
编译构建
    ├── 依赖下载
    ├── 代码编译
    └── 打包生成制品
    ↓
自动化测试
    ├── 单元测试
    ├── 集成测试
    └── 测试覆盖率
    ↓
生成制品
    ├── JAR包
    ├── Docker镜像
    └── 版本标签
    ↓
通知结果
    ├── 构建成功通知
    └── 构建失败告警
```

**CI vs CD的区别**：

| 概念 | 含义 | 范围 |
|------|------|------|
| **CI** | 持续集成 | 代码提交 → 构建 → 测试 → 生成制品 |
| **CD** | 持续交付/部署 | CI + 自动部署到环境 |
| **CD (Delivery)** | 持续交付 | 自动部署到测试环境，生产需手动 |
| **CD (Deployment)** | 持续部署 | 自动部署到所有环境（包括生产） |

**MeterSphere项目的定位**：
- ✅ **CI部分**：已完整实现（代码提交 → 构建 → 镜像推送）
- 🔄 **CD部分**：部分实现（develop分支自动部署测试环境，master分支需手动审批）

## 二、当前项目的CI实现

### 2.1 pollSCM定时触发机制

**当前配置**：
```groovy
triggers {
    pollSCM('0 1 * * *')  // 每天凌晨1点检查代码变化
}
```

**pollSCM的工作原理**：

```
1. 定时轮询
   └── Jenkins在指定时间（凌晨1点）主动检查Git仓库

2. 比较差异
   ├── 获取上次构建的提交ID
   ├── 获取当前仓库的最新提交ID
   └── 比较两者是否相同

3. 决策触发
   ├── 有新提交 → 自动触发构建
   └── 无新提交 → 跳过本次构建（节省资源）

4. 执行构建
   └── 按照Jenkinsfile定义的流程执行
```

**为什么选择pollSCM？**

**优势**：
- ✅ **可预期**：固定时间执行，便于规划资源
- ✅ **节省资源**：只在有代码变化时构建，避免无效构建
- ✅ **稳定性好**：不依赖Webhook网络连通性
- ✅ **适合开发分支**：开发分支频繁提交，但不需要立即构建

**劣势**：
- ❌ **延迟较高**：最多延迟24小时（如果刚提交完代码）
- ❌ **不够实时**：无法立即反馈构建结果

**适用场景**：
- 开发分支（develop）：频繁提交，但不需要立即构建
- 长期维护分支（v2.10.23-lts）：定期检查是否有更新
- 资源受限环境：避免频繁构建占用资源

### 2.2 触发时机和频率

**Cron表达式说明**：
```
'0 1 * * *'
│ │ │ │ │
│ │ │ │ └── 星期几（0-7，0和7都表示周日）
│ │ │ └──── 月份（1-12）
│ │ └────── 日期（1-31）
│ └──────── 小时（0-23）
└────────── 分钟（0-59）
```

**常见配置示例**：
```
'0 1 * * *'      # 每天凌晨1点
'H/5 * * * *'    # 每5分钟检查一次
'0 */2 * * *'    # 每2小时检查一次
'0 1 * * 1-5'    # 工作日凌晨1点
```

**为什么选择凌晨1点？**

**考虑因素**：
1. **服务器负载低**：白天业务高峰，凌晨服务器空闲
2. **网络带宽充足**：Docker镜像拉取需要较大带宽
3. **构建失败有时间修复**：凌晨发现问题，上班前有时间处理
4. **不影响开发工作**：开发人员已下班，不会影响开发

**实际效果**：
- 如果当天有代码提交，会在第二天凌晨1点自动构建
- 如果当天无代码提交，不会触发构建（节省资源）
- 构建结果会在凌晨1-2点完成，上班时可以看到结果

## 三、CI完整流程详解

### 3.1 Preparation阶段：环境准备

**目的**：准备构建所需的环境变量和版本号

**执行内容**：
```
1. 版本号处理
   ├── 如果分支名以"v"开头（如v2.10.23）
   │   └── 去掉"v"前缀，得到版本号"2.10.23"
   └── 如果分支名不以"v"开头（如develop）
       └── 直接使用分支名作为版本号

2. 构建参数处理
   ├── buildSdk：是否只构建SDK
   ├── frontendLink：前端链接（可选）
   └── 设置环境变量供后续阶段使用

3. 输出信息
   └── 打印版本号和构建参数，便于调试
```

**版本号映射示例**：
```
分支名              →  版本号（REVISION）
v2.10.23-lts    →  2.10.23
v2.10.24        →  2.10.24
develop         →  develop
feature/new-api →  feature/new-api
```

**为什么这样设计？**
- 发布版本（v2.10.23）：版本号清晰，便于追溯
- 开发分支（develop）：使用分支名，避免版本号混乱
- 特性分支（feature/*）：独立测试，不影响主版本号

### 3.2 SDK阶段：条件构建SDK

**触发条件**：`BUILD_SDK = true`（通过参数buildSdk控制）

**SDK是什么？**
```
framework/sdk-parent/
├── domain/          # 数据模型（实体类、DTO）
├── sdk/             # 通用工具类
├── xpack-interface/ # 企业版接口
└── jmeter/          # JMeter组件
```

**为什么需要单独构建SDK？**
- **避免重复代码**：9个微服务共用的代码
- **统一数据模型**：确保数据结构一致
- **版本统一管理**：SDK升级，所有服务同步升级

**构建流程**：
```
1. 部署父POM
   └── mvn deploy -N -Drevision=${REVISION}

2. 构建SDK模块
   └── mvn clean deploy -Drevision=${REVISION}
       -pl framework,framework/sdk-parent,...

3. 发布到Maven私有仓库
   └── 供后续服务构建时使用

4. 前端代码处理（可选）
   └── 如果提供了frontendLink，复制前端代码
```

**什么时候需要构建SDK？**
- SDK代码变更时
- 首次搭建环境时
- 版本发布前

**什么时候不需要构建SDK？**
- 只修改业务服务代码
- 日常开发调试
- 紧急修复某个服务

### 3.3 Build/Test阶段：Maven构建

**触发条件**：`BUILD_SDK = false`（正常构建流程）

**构建流程**：
```
1. Maven编译打包
   └── mvn clean package -Drevision=${REVISION}
       ├── 下载依赖
       ├── 编译代码
       └── 打包JAR文件

2. JAR分层解压（框架层）
   ├── framework/eureka
   └── framework/gateway
   └── jar -xf ../*.jar → target/dependency/

3. JAR分层解压（业务层）
   ├── api-test/backend
   ├── performance-test/backend
   ├── project-management/backend
   └── ...（共7个服务）
   └── jar -xf ../*.jar → target/dependency/

4. xpack注入
   └── 从Maven仓库复制xpack.jar到BOOT-INF/lib/
```

**为什么要解压JAR包？**

**传统方式的问题**：
```
Dockerfile:
COPY target/app.jar /app.jar

问题：
├── 整个JAR包50MB作为一个Docker层
├── 任何代码变更都要重新上传50MB
└── 无法利用Docker缓存机制
```

**分层构建的优势**：
```
Dockerfile:
COPY lib/     /app/lib/        # 依赖层，40MB，很少变化
COPY META-INF /app/META-INF    # 元数据，1MB，很少变化
COPY classes  /app/classes     # 代码层，9MB，经常变化

效果：
├── 只修改业务代码时
│   ├── lib层（40MB）→ 使用缓存 ✅
│   ├── META-INF层（1MB）→ 使用缓存 ✅
│   └── classes层（9MB）→ 重新构建 ⚠️
└── 上传时间：从50秒降低到10秒（减少80%）
```

**xpack注入机制**：

**xpack是什么？**
- MeterSphere企业版功能包
- 以JAR包形式提供
- 需要注入到BOOT-INF/lib目录

**注入时机**：
```
1. 解压JAR包
   └── jar -xf target/app.jar

2. 从Maven仓库获取xpack
   └── cp ${LOCAL_REPOSITORY}/.../xpack.jar

3. 复制到lib目录
   └── cp xpack.jar BOOT-INF/lib/

4. Docker构建时使用
   └── COPY target/dependency/BOOT-INF/lib/ /app/lib/
```

### 3.4 Docker构建阶段：多架构镜像

**构建流程**：
```
1. 遍历所有服务
   └── 9个服务：eureka, gateway, api-test, ...

2. 构建多架构镜像
   ├── --platform linux/amd64,linux/arm64
   ├── 支持x86服务器（传统服务器）
   └── 支持ARM服务器（Mac M1、华为鲲鹏）

3. 推送镜像
   └── 推送到阿里云镜像仓库
       registry.cn-qingdao.aliyuncs.com/metersphere/${IMAGE_NAME}:${TAG}

4. 失败重试
   └── 最多重试10次，每次间隔10秒
```

**多架构构建的必要性**：

**使用场景**：
```
x86服务器（amd64）
├── 传统服务器
└── 云服务器（阿里云、腾讯云）

ARM服务器（arm64）
├── Mac M1/M2开发机
├── 华为鲲鹏服务器
└── AWS Graviton实例
```

**不支持多架构的问题**：
- Mac M1开发 → 构建amd64镜像 → 本地无法运行
- 服务器部署 → 只有arm64镜像 → 无法部署

**构建时间对比**：
```
单架构构建：9个服务 × 2.5分钟 = 22.5分钟
多架构构建：9个服务 × 5分钟 = 45分钟（并行后约30分钟）

时间增加一倍，但值得：
├── 支持更多部署场景
├── Mac M1开发体验更好
└── 未来ARM服务器趋势
```

**失败重试机制**：

**为什么需要重试？**
- 网络不稳定：Docker镜像推送可能失败
- 资源竞争：并行构建9个服务，可能资源竞争
- Registry临时不可用：镜像仓库可能临时故障

**重试策略**：
```
for (int i = 0; i < 10; i++) {
    try {
        构建并推送镜像
        break  // 成功则退出
    } catch (Exception e) {
        sleep 10  // 等待10秒
        continue  // 继续重试
    }
}
```

**重试参数权衡**：
- 重试次数：10次（约2分钟）
- 重试间隔：10秒（给予网络恢复时间）
- 全部失败：停止构建，发送通知

### 3.5 通知机制：企业微信通知

**通知时机**：
- **构建成功**：所有阶段完成后通知
- **构建失败**：任何阶段失败立即通知

**通知内容**：
```
构建成功：
├── 项目名称
├── 分支名
├── 版本号
├── 提交ID（前8位）
├── 构建号
└── 构建耗时

构建失败：
├── 上述信息
├── 失败阶段
└── 构建日志链接
```

**通知渠道**：
- 企业微信机器人（当前使用）
- 支持Markdown格式
- 实时推送，移动端方便查看

## 四、触发策略对比

### 4.1 不同触发方式

| 触发方式 | 配置方式 | 优点 | 缺点 | 适用场景 |
|---------|---------|------|------|---------|
| **pollSCM定时** | `pollSCM('0 1 * * *')` | ✅ 稳定可靠<br>✅ 节省资源<br>✅ 可预期 | ❌ 延迟较高<br>❌ 不够实时 | 开发分支、长期维护分支 |
| **Webhook推送** | Git仓库配置Webhook | ✅ 实时触发<br>✅ 立即反馈 | ❌ 依赖网络<br>❌ 每次提交都触发 | 发布分支、需要立即构建 |
| **手动触发** | Jenkins界面点击 | ✅ 可控性强<br>✅ 按需构建 | ❌ 需要人工操作<br>❌ 容易忘记 | 紧急修复、发布版本 |
| **定时构建** | `cron('0 1 * * *')` | ✅ 固定时间<br>✅ 可预期 | ❌ 无变化也构建<br>❌ 浪费资源 | 定期测试、每日构建 |

### 4.2 混合策略建议

**推荐方案**：
```
开发分支（develop）
└── pollSCM定时（每天凌晨1点）
    └── 有代码变化才构建，节省资源

发布分支（master）
└── Webhook推送 + 手动触发
    ├── 合并到master时立即构建
    └── 紧急情况可手动触发

特性分支（feature/*）
└── 手动触发
    └── 需要测试时手动构建

长期维护分支（v2.10.23-lts）
└── pollSCM定时（每周检查一次）
    └── 定期检查是否有更新
```

### 4.3 如何配置Webhook触发

**GitHub配置**：
```
1. 仓库设置 → Webhooks → Add webhook
2. Payload URL: http://jenkins-server/github-webhook/
3. Content type: application/json
4. Events: Just the push event
5. Active: ✓
```

**Jenkinsfile配置**：
```groovy
triggers {
    // 移除pollSCM，使用Webhook
    // pollSCM('0 1 * * *')
    
    // 或者两者结合：Webhook优先，pollSCM作为备份
    pollSCM('H/30 * * * *')  // 每30分钟检查一次（作为备份）
}
```

**Jenkins任务配置**：
```
Build Triggers:
☑ GitHub hook trigger for GITScm polling
```

## 五、构建流程关键技术点

### 5.1 版本号管理策略

**Maven revision变量机制**：

**传统方式的痛点**：
```xml
<!-- 每个pom.xml都要硬编码版本号 -->
<version>2.10.23</version>

发布新版本时：
├── 需要修改所有pom.xml
├── 容易遗漏某个模块
└── Git提交历史混乱
```

**revision变量的优势**：
```xml
<!-- 父POM定义 -->
<version>${revision}</version>
<properties>
    <revision>2.10</revision>  <!-- 默认版本 -->
</properties>

<!-- 构建时动态设置 -->
mvn clean package -Drevision=2.10.23
```

**价值**：
- ✅ 单点修改（只改父POM）
- ✅ 构建时灵活（Jenkins动态设置）
- ✅ 减少Git提交（不需要每次修改版本号）

### 5.2 JAR分层解压的时机选择

**在Jenkins中解压（当前方案）**：
```bash
# 构建阶段
jar -xf target/app.jar -d target/dependency/

# Dockerfile中
COPY target/dependency/lib /app/lib
```

**优点**：
- Dockerfile简单
- 构建过程可控
- 便于注入xpack包

**在Dockerfile中解压（备选方案）**：
```dockerfile
COPY target/app.jar /tmp/
RUN jar -xf /tmp/app.jar -d /app/
```

**缺点**：
- 多一次解压操作（构建时间增加）
- xpack注入复杂

### 5.3 多架构构建的工作原理

**buildx的工作原理**：
```
1. 在当前机器启动两个构建器
   ├── amd64构建器（通过QEMU模拟）
   └── arm64构建器（原生）

2. 并行构建两个架构的镜像
   ├── myapp:v1-amd64
   └── myapp:v1-arm64

3. 创建manifest（镜像索引）
   └── myapp:v1 指向两个架构镜像

4. Docker pull时自动选择
   └── x86服务器 → 自动拉取amd64镜像
   └── ARM服务器 → 自动拉取arm64镜像
```

**构建时间影响**：
- 单架构：每个服务约2.5分钟
- 多架构：每个服务约5分钟（并行后实际约3分钟）
- 总时间：从22.5分钟增加到约30分钟

**是否值得？**
- ✅ 值得：支持更多部署场景
- ✅ 值得：Mac M1开发体验更好
- ✅ 值得：未来ARM服务器趋势

## 六、失败处理机制

### 6.1 Docker推送失败重试

**失败原因**：
- 网络不稳定：Docker镜像推送可能失败
- Registry临时不可用：镜像仓库可能临时故障
- 资源竞争：并行构建9个服务，可能资源竞争

**重试策略**：
```
重试次数：10次
重试间隔：10秒
总重试时间：约2分钟

如果10次都失败：
├── 构建停止
├── 企业微信通知运维
├── 保留构建日志
└── 手动排查问题
```

**重试参数权衡**：
- 太少：网络问题导致失败
- 太多：浪费时间
- 10次：平衡点（约2分钟）

### 6.2 构建失败通知

**通知内容**：
```
⚠️ 构建失败

项目: MeterSphere-CI/CD
分支: develop
版本: develop
提交: a1b2c3d4
构建号: #123
失败阶段: Docker build & push
查看日志: http://jenkins/.../console
```

**通知时机**：
- 任何阶段失败立即通知
- 不等待后续阶段执行

### 6.3 日志保留和问题排查

**日志位置**：
- Jenkins Console Output：完整的构建日志
- 构建历史：保留最近N次构建日志
- 企业微信通知：包含日志链接

**问题排查步骤**：
```
1. 查看企业微信通知
   └── 获取失败阶段和日志链接

2. 查看Jenkins Console Output
   └── 找到具体的错误信息

3. 常见问题：
   ├── Maven依赖下载失败 → 检查网络和settings.xml
   ├── Docker构建失败 → 检查Docker daemon状态
   ├── 镜像推送失败 → 检查Registry连接和权限
   └── 资源不足 → 检查服务器资源使用情况
```

## 七、CI优化建议

### 7.1 增加自动化测试

**当前状态**：
- Build/Test阶段使用 `-DskipTests` 跳过测试
- 没有测试覆盖率报告

**优化建议**：
```
1. 增加Unit Tests阶段
   └── 运行单元测试，生成测试报告

2. 增加集成测试
   └── 测试服务间集成

3. 测试覆盖率
   └── 使用JaCoCo生成覆盖率报告

4. 质量门禁
   └── 测试覆盖率低于阈值时构建失败
```

**Jenkinsfile.enhanced已包含**：
```groovy
stage('Unit Tests') {
    when {
        expression { params.runTests == true }
    }
    steps {
        mvn test -Drevision=${REVISION}
    }
    post {
        always {
            junit '**/target/surefire-reports/*.xml'
        }
    }
}
```

### 7.2 代码质量检查集成

**当前状态**：
- 没有代码质量检查
- 已有GitHub Actions的SonarCloud配置

**优化建议**：
```
1. 集成SonarQube
   ├── 静态代码分析
   ├── 代码规范检查
   └── 安全漏洞扫描

2. 质量门禁
   └── 代码质量不达标时构建失败

3. 质量报告
   └── 在Jenkins中展示质量报告
```

### 7.3 构建性能优化

**当前可以优化的点**：

**1. Maven依赖缓存**：
```
当前：每次构建重新下载依赖（15分钟）
优化：持久化Maven本地仓库
预期：依赖下载时间降低到1分钟
```

**2. 并行构建**：
```
当前：串行构建9个服务
优化：并行构建多个服务
预期：构建时间减少50%
```

**3. 增量构建**：
```
当前：每次都clean构建
优化：只构建变更的模块
预期：构建时间减少30%
```

### 7.4 缓存策略

**Maven依赖缓存**：
```groovy
environment {
    MAVEN_OPTS = '-Dmaven.repo.local=/var/jenkins_home/.m2/repository'
}
```

**Docker层缓存**：
```
已实现：通过JAR分层解压利用Docker缓存
效果：代码变更时只上传变更层（减少80%上传时间）
```

## 八、最佳实践总结

### 8.1 设计原则

**1. 简单优先**：
```
能简单实现就不要复杂
├── pollSCM定时触发（简单、稳定）
├── 模块化构建（清晰、可控）
└── 失败重试（简单、有效）
```

**2. 渐进式**：
```
从基础开始，逐步完善
├── 当前：基础CI流程
├── 下一步：增加测试阶段
├── 再下一步：代码质量检查
└── 持续优化
```

**3. 可维护**：
```
代码清晰，注释充分
├── Jenkinsfile结构清晰
├── 每个阶段职责单一
└── 环境变量统一管理
```

**4. 可观测**：
```
日志详细，通知及时
├── 每个阶段都有日志输出
├── 构建结果实时通知
└── 失败信息详细明确
```

**5. 容错性**：
```
失败重试，优雅降级
├── Docker推送失败自动重试
├── 构建失败立即通知
└── 保留日志便于排查
```

### 8.2 避坑指南

**❌ 不要过度优化**：
```
问题：构建时间30分钟
过度：花3天优化到25分钟
合理：凌晨构建，时间充足，不需优化
```

**❌ 不要盲目追新**：
```
问题：Jenkins 2.x用得好好的
盲目：升级到最新版，遇到兼容性问题
合理：稳定版本，除非有明确收益
```

**❌ 不要忽视通知**：
```
问题：构建失败了不知道
后果：开发调试时才发现镜像没更新
合理：失败立即通知，及时处理
```

**❌ 不要跳过测试**：
```
问题：为了快速构建，跳过所有测试
后果：问题积累，后期修复成本高
合理：至少运行关键测试，逐步完善
```

### 8.3 演进方向

**阶段1：完善测试**（当前可做）
```
+ 单元测试
+ 集成测试
+ 代码覆盖率报告
```

**阶段2：代码质量**（下一步）
```
+ SonarQube质量检查
+ 安全漏洞扫描
+ 依赖版本检查
```

**阶段3：构建优化**（持续优化）
```
+ Maven依赖缓存
+ 并行构建优化
+ 增量构建
```

**阶段4：持续交付**（未来）
```
+ 自动部署到测试环境（已部分实现）
+ 灰度发布
+ 自动回滚
```
