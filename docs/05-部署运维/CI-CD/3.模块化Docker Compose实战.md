> 📋 系列文档第3篇：模块化部署架构设计与实践  
> 🎯 重点：为什么模块化、如何设计、运维实践  
> 📅 更新时间：2025年11月29日

## 一、为什么需要模块化Compose？

### 1.1 传统单文件Compose的痛点

**场景还原**：
```yaml
# docker-compose.yml（2000行）
version: '3.8'
services:
  eureka:
    image: ...
    environment: ...
    # 100行配置
  
  gateway:
    image: ...
    environment: ...
    # 150行配置
  
  api-test:
    # 又是100行
  
  # ... 其他7个服务
  # 配置混在一起，难以维护
```

**遇到的问题**：

**问题1：更新风险高**
```
需求：更新api-test服务
操作：docker-compose up -d
结果：所有服务都重启了（包括不需要更新的）
影响：正在进行的测试任务中断
```

**问题2：配置混乱**
```
现象：找gateway的配置
操作：搜索2000行文件
耗时：5分钟
效率：低下
```

**问题3：调试不便**
```
需求：只想启动eureka+gateway调试
操作：注释掉其他服务（手动修改yaml）
风险：容易遗漏，提交时忘记恢复
```

### 1.2 模块化带来的价值

**核心理念**：一个服务一个配置文件

```bash
/opt/metersphere/
├── docker-compose-base.yml       # 基础设施（50行）
├── docker-compose-eureka.yml     # Eureka（30行）
├── docker-compose-gateway.yml    # Gateway（50行）
└── docker-compose-api-test.yml   # API测试（40行）
```

**解决的问题**：

✅ **独立更新**
```bash
# 只更新api-test
docker-compose -f docker-compose-api-test.yml pull
docker-compose -f docker-compose-api-test.yml up -d
# 其他服务不受影响
```

✅ **配置清晰**
```bash
# 想看gateway配置
cat docker-compose-gateway.yml
# 50行，一目了然
```

✅ **灵活调试**
```bash
# 只启动部分服务
docker-compose -f docker-compose-base.yml up -d
docker-compose -f docker-compose-eureka.yml up -d
docker-compose -f docker-compose-gateway.yml up -d
# 不需要修改yaml文件
```

## 二、模块化设计思路

### 2.1 分层设计

**三层架构**：
```
基础设施层（docker-compose-base.yml）
├── 网络定义（ms-network）
├── 卷定义（ms-conf, ms-logs, ms-data）
└── 所有服务共用

服务层（docker-compose-{service}.yml）
├── 各服务独立配置
├── 继承基础设施层的网络和卷
└── 可以单独启停

外部依赖层（可选）
├── docker-compose-mysql.yml
├── docker-compose-redis.yml
└── 根据需要启动
```

**为什么这样分层？**

**基础设施单独管理**：
- 网络和卷很少变化
- 所有服务依赖这些基础设施
- 单独定义，避免重复

**服务独立管理**：
- 每个服务独立演进
- 更新互不影响
- 配置职责清晰

### 2.2 命名规范

**文件命名**：
```bash
docker-compose-{服务名}.yml

示例：
docker-compose-eureka.yml       # 注册中心
docker-compose-gateway.yml      # 网关
docker-compose-api-test.yml     # API测试服务
```

**为什么用这种命名？**
- 一眼看出是哪个服务
- 按字母排序，相关服务在一起
- 脚本中易于批量处理（docker-compose-*.yml）

**容器命名**：
```yaml
container_name: eureka        # 简洁
container_name: gateway       # 不加前缀
container_name: api-test      # 与服务名一致
```

**为什么不加前缀？**
```bash
# 不好的命名
ms-eureka, ms-gateway, ms-api-test
├── 输入麻烦
└── docker logs ms-eureka（多余字符）

# 好的命名
eureka, gateway, api-test
├── 简洁
└── docker logs eureka（一目了然）
```

### 2.3 变量化配置

**统一配置文件：install.conf**
```bash
# 镜像配置
MS_IMAGE_PREFIX=registry.cn-qingdao.aliyuncs.com/metersphere
MS_IMAGE_TAG=v2.10.23-lts

# 网络配置
MS_DOCKER_SUBNET=172.30.10.0/24
MS_SERVER_PORT=8081

# 基础设施
MS_MYSQL_HOST=172.21.183.234
MS_MYSQL_PORT=3307
MS_REDIS_HOST=172.21.183.234
```

**设计理念**：
```
所有可变的配置都通过变量
├── 环境切换（测试/生产）→ 改install.conf
├── 版本切换 → 改MS_IMAGE_TAG
└── 网络调整 → 改MS_DOCKER_SUBNET
```

**好处**：
- 集中管理，一处修改，处处生效
- 不需要修改yaml文件
- 便于脚本化管理

## 三、网络设计

### 3.1 自定义网络的必要性

**默认网络的问题**：
```
使用Docker默认网络
├── 服务间通过IP访问（IP会变化）
├── 多个项目网络冲突
└── 无法自定义子网
```

**自定义网络的优势**：
```yaml
networks:
  ms-network:
    driver: bridge
    ipam:
      config:
        - subnet: ${MS_DOCKER_SUBNET}
```

**价值**：
```
1️⃣ 服务发现
   └── 通过服务名访问（http://eureka:8761）

2️⃣ 网络隔离
   └── MeterSphere网络与其他项目隔离

3️⃣ 子网可控
   └── 172.30.10.0/24（避免与宿主机网络冲突）
```

### 3.2 服务间通信

**场景：gateway需要调用api-test**

```java
// 不需要配置IP地址
String url = "http://api-test:8004/api/test";
```

**工作原理**：
```
1️⃣ Docker内置DNS
   └── api-test → 172.30.10.5

2️⃣ 容器名即主机名
   └── ping api-test（可以ping通）

3️⃣ 网络隔离
   └── 只有同一网络的容器可以互相访问
```

### 3.3 端口设计

**内部端口 vs 对外端口**

```yaml
gateway:
  ports:
    - '8081:8000'  # 宿主机8081 → 容器8000

eureka:
  # 不暴露端口
  # 只在内部网络访问
```

**设计理念**：
```
最小暴露原则
├── 只有gateway对外暴露（8081）
├── 其他服务内部访问
└── 减少攻击面，提高安全性
```

**端口映射的权衡**：
```
方案1：所有服务都映射端口
├── 优点：调试方便
└── 缺点：端口冲突，安全风险

方案2：只映射必要端口（当前方案）
├── 优点：安全，端口管理简单
└── 缺点：调试需要进入容器
```

## 四、存储设计

### 4.1 bind挂载 vs 命名卷

**命名卷（volume）**：
```yaml
volumes:
  mydata:
    # Docker管理，位置不可控
```

**bind挂载（当前方案）**：
```yaml
volumes:
  ms-conf:
    driver_opts:
      type: none
      device: /opt/metersphere/conf
      o: bind
```

**为什么选择bind挂载？**

**优势1：配置易于编辑**
```bash
# 直接在宿主机编辑
vim /opt/metersphere/conf/application.yml
# 重启服务生效
docker-compose -f docker-compose-gateway.yml restart
```

**优势2：日志易于查看**
```bash
# 直接查看宿主机日志
tail -f /opt/metersphere/logs/gateway/gateway.log
# 不需要docker logs
```

**优势3：备份简单**
```bash
# 直接打包目录
tar -czf backup.tar.gz /opt/metersphere/data
# 命名卷需要先导出
```

### 4.2 存储规划

**三类数据**：
```
conf/  配置文件
├── 各服务的application.yml
├── 日志配置（logback.xml）
└── 频繁修改，需要易于编辑

logs/  日志文件
├── gateway/gateway.log
├── api-test/api-test.log
└── 需要实时查看和分析

data/  数据文件
├── JMeter脚本
├── 测试报告
└── 需要持久化和备份
```

**设计考虑**：
```
为什么分开？
├── 职责分离（配置、日志、数据各司其职）
├── 权限管理（日志只读，配置可写）
└── 备份策略（数据备份，日志归档）
```

### 4.3 权限管理

**宿主机目录权限**：
```bash
# 创建目录并设置权限
mkdir -p /opt/metersphere/{conf,logs,data}
chown -R 1000:1000 /opt/metersphere

# 为什么是1000？
# Docker容器内默认用户UID通常是1000
```

**容器内权限**：
```yaml
# Dockerfile中
USER appuser  # UID 1000

# 映射关系
宿主机1000:1000 ←→ 容器appuser:appuser
```

## 五、健康检查设计

### 5.1 为什么需要健康检查？

**没有健康检查的问题**：
```
docker-compose up -d
├── 容器启动了
├── 但是服务还在初始化
├── 依赖它的服务立即启动
└── 结果：依赖服务连接失败
```

**有健康检查的好处**：
```yaml
api-test:
  depends_on:
    system-setting:
      condition: service_healthy  # 等待健康才启动
```

### 5.2 健康检查方式选择

**方式1：HTTP检查**
```yaml
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
```
- 优点：准确（服务真正可用）
- 缺点：需要安装curl（镜像变大）

**方式2：端口检查（当前方案）**
```yaml
healthcheck:
  test: ["CMD", "nc", "-zv", "localhost", "8000"]
```
- 优点：轻量（nc工具小）
- 缺点：只检查端口，不检查服务

**为什么选择端口检查？**
```
微服务环境的特点：
├── 服务很多（9个）
├── 每个都安装curl（镜像增大100MB+）
├── 端口可用基本代表服务可用
└── 性价比高
```

### 5.3 健康检查参数调优

```yaml
healthcheck:
  interval: 6s      # 检查间隔
  timeout: 10s      # 超时时间
  retries: 50       # 重试次数
  start_period: 0s  # 启动宽限期
```

**参数含义**：
```
总等待时间 = retries × (interval + timeout)
         = 50 × (6 + 10)
         = 800秒
         = 约13分钟
```

**为什么这样设置？**
```
Eureka启动慢
├── 需要初始化
├── 需要注册服务
├── 需要等待心跳
└── 约1-2分钟才能就绪

Gateway启动更慢
├── 等待Eureka
├── 初始化路由
├── 加载配置
└── 约2-3分钟
```

## 六、依赖关系管理

### 6.1 启动顺序的重要性

**错误的启动顺序**：
```
api-test先启动 → 连接system-setting失败 → 反复重试 → 资源浪费
```

**正确的启动顺序**：
```
1️⃣ Eureka（注册中心）
2️⃣ system-setting（基础服务）
3️⃣ Gateway（网关）
4️⃣ 其他业务服务（并行）
```

### 6.2 depends_on的两种用法

**简单依赖（不推荐）**：
```yaml
api-test:
  depends_on:
    - system-setting  # 只等待启动，不等待就绪
```

**健康依赖（推荐）**：
```yaml
api-test:
  depends_on:
    system-setting:
      condition: service_healthy  # 等待健康检查通过
```

**差别**：
```
简单依赖：
└── system-setting容器启动后，api-test立即启动
    └── 可能system-setting还在初始化

健康依赖：
└── system-setting健康检查通过后，api-test才启动
    └── 确保system-setting真正可用
```

### 6.3 循环依赖的避免

**错误示例**：
```yaml
gateway:
  depends_on:
    - api-test

api-test:
  depends_on:
    - gateway
```
- 结果：都启动不了

**正确设计**：
```
明确依赖关系：
├── Gateway不依赖具体服务
├── Gateway通过Eureka发现服务
└── 服务注册到Eureka即可
```

## 七、资源限制策略

### 7.1 为什么需要资源限制？

**不限制的风险**：
```
某个服务内存泄漏
├── 占用所有内存
├── 宿主机OOM
└── 所有服务崩溃
```

**限制的好处**：
```yaml
mem_limit: 1073741824  # 1GB
```
- 单个服务崩溃不影响其他服务
- 资源可控，便于规划

### 7.2 内存限制的设定

**不同服务的内存需求**：
```
eureka: 512MB
├── 轻量级服务
└── 只做注册发现

gateway: 800MB
├── 需要缓存路由信息
└── 承载所有流量

api-test: 1GB
├── 需要运行JMeter
└── 内存需求较高
```

**如何确定内存大小？**
```
1️⃣ 运行一段时间，观察实际使用
   └── docker stats api-test

2️⃣ 预留30%余量
   └── 实际700MB → 设置1GB

3️⃣ 根据业务量调整
   └── 高峰期观察，按需扩容
```

### 7.3 CPU限制（可选）

```yaml
cpus: '2.0'  # 限制使用2个CPU核心
```

**是否需要CPU限制？**
```
需要限制：
├── 多租户环境
├── 防止某服务占满CPU
└── 确保公平调度

不需要限制（当前）：
├── 单一项目
├── CPU通常不是瓶颈
└── 简化配置
```

## 八、运维实战

### 8.1 日常运维命令

**查看服务状态**：
```bash
# 查看所有服务
docker ps -a

# 查看特定服务
docker ps | grep api-test
```

**查看服务日志**：
```bash
# 实时日志
docker-compose -f docker-compose-api-test.yml logs -f

# 最近100行
docker-compose -f docker-compose-api-test.yml logs --tail 100

# 或者直接查看宿主机日志
tail -f /opt/metersphere/logs/api-test/api-test.log
```

**重启服务**：
```bash
# 重启单个服务
docker-compose -f docker-compose-gateway.yml restart

# 重启所有服务
for f in docker-compose-*.yml; do
    docker-compose -f $f restart
done
```

### 8.2 服务更新流程

**标准更新流程**：
```bash
#!/bin/bash
SERVICE=$1

echo "=== 更新 ${SERVICE} 服务 ==="

# 1. 拉取最新镜像
docker-compose -f docker-compose-${SERVICE}.yml pull

# 2. 停止服务（保留容器）
docker-compose -f docker-compose-${SERVICE}.yml stop

# 3. 备份旧镜像（可选）
OLD_TAG=$(docker ps -a --filter "name=${SERVICE}" --format "{{.Image}}")
docker tag ${OLD_TAG} ${OLD_TAG}-backup

# 4. 启动新服务
docker-compose -f docker-compose-${SERVICE}.yml up -d

# 5. 查看日志
docker-compose -f docker-compose-${SERVICE}.yml logs -f --tail 50
```

**为什么用stop而不是down？**
```
docker-compose stop:
└── 停止容器，保留网络和卷

docker-compose down:
└── 停止容器，删除网络
    └── 可能影响其他服务
```

### 8.3 故障排查

**服务启动失败**：
```bash
# 1. 查看容器状态
docker ps -a | grep api-test

# 2. 查看启动日志
docker logs api-test

# 3. 进入容器排查
docker exec -it api-test sh

# 4. 检查健康状态
docker inspect api-test | grep -A 10 Health
```

**服务间通信失败**：
```bash
# 1. 检查网络
docker network ls
docker network inspect ms-network

# 2. 测试连通性
docker exec gateway ping api-test

# 3. 检查端口
docker exec gateway nc -zv api-test 8004
```

**性能问题排查**：
```bash
# 1. 查看资源使用
docker stats --no-stream

# 2. 查看容器进程
docker top api-test

# 3. 查看宿主机资源
top
free -h
df -h
```

### 8.4 批量操作脚本

**启动所有服务**：
```bash
#!/bin/bash
SERVICES=(
    "base"
    "eureka"
    "system-setting"
    "gateway"
    "api-test"
    "test-track"
    "performance-test"
    "project-management"
    "report-stat"
    "workstation"
)

for service in "${SERVICES[@]}"; do
    echo "启动 ${service}..."
    docker-compose -f docker-compose-${service}.yml up -d
    sleep 5
done
```

**停止所有服务**：
```bash
#!/bin/bash
for f in docker-compose-*.yml; do
    [ "$f" != "docker-compose-base.yml" ] && \
    docker-compose -f $f stop
done
```

## 九、最佳实践

### 9.1 配置管理

✅ **DO**：
- 使用install.conf统一管理变量
- 配置文件纳入版本控制
- 敏感信息使用环境变量

❌ **DON'T**：
- 硬编码IP地址在yaml中
- 直接提交密码到Git
- 每个文件重复定义变量

### 9.2 版本管理

✅ **DO**：
- 使用具体版本标签（v2.10.23）
- 保留最近3个版本的镜像
- 测试环境先验证

❌ **DON'T**：
- 使用latest标签（难以回滚）
- 频繁删除旧镜像（丢失回滚选项）
- 直接在生产更新

### 9.3 监控告警

✅ **DO**：
- 配置健康检查
- 监控容器资源使用
- 定期查看日志

❌ **DON'T**：
- 依赖容器状态判断服务健康
- 忽视资源限制
- 等出问题再查日志

## 十、总结

### 10.1 模块化的核心价值

1. **降低风险**：独立更新，影响范围可控
2. **提高效率**：配置清晰，运维简单
3. **灵活性强**：按需启动，调试方便
4. **易于维护**：职责清晰，易于理解

### 10.2 适用场景

✅ **适合**：
- 微服务项目（5-20个服务）
- 需要频繁更新某些服务
- 团队规模小（1-5人）

❌ **不适合**：
- 单体应用
- 所有服务总是一起更新
- 已有复杂的编排系统
