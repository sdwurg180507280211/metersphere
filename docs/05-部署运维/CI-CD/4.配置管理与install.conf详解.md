## 一、为什么需要统一配置管理？

### 1.1 分散配置的痛点

**场景还原**：
```bash
# 修改MySQL地址
1. 修改docker-compose-gateway.yml（MS_MYSQL_HOST）
2. 修改docker-compose-api-test.yml（MS_MYSQL_HOST）
3. 修改docker-compose-test-track.yml（MS_MYSQL_HOST）
# ... 修改9个文件
4. 发现某个文件忘改了
5. 服务启动失败，排查30分钟才发现
```

**问题分析**：
- ❌ 配置重复（9个文件都写MySQL地址）
- ❌ 易出错（修改时容易遗漏）
- ❌ 难维护（找某个配置要搜索多个文件）
- ❌ 不一致（可能新旧值混用）

### 1.2 统一配置的价值

**核心思想**：所有配置集中在一个文件

```bash
install.conf（统一配置文件）
├── 镜像配置
├── 网络配置
├── 基础设施配置
└── 资源限制配置

docker-compose-*.yml（引用配置）
└── 使用 ${变量名} 引用
```

**价值**：
- ✅ 单点修改（只改install.conf）
- ✅ 不易出错（只有一个地方）
- ✅ 易于查找（配置集中）
- ✅ 保证一致（所有服务同步）

## 二、install.conf设计思路

### 2.1 配置分类

**按用途分类**：
```bash
# 1. 镜像相关
MS_IMAGE_PREFIX=...    # 镜像仓库地址
MS_IMAGE_TAG=...       # 版本标签
MS_JMETER_IMAGE=...    # JMeter镜像

# 2. 网络相关
MS_DOCKER_SUBNET=...   # 容器网络子网
MS_SERVER_PORT=...     # 对外端口
MS_TCP_MOCK_PORT=...   # Mock端口范围

# 3. 基础设施
MS_MYSQL_HOST=...      # MySQL地址
MS_REDIS_HOST=...      # Redis地址
MS_KAFKA_HOST=...      # Kafka地址

# 4. 开关配置
MS_EXTERNAL_MYSQL=...  # 是否使用外部MySQL
MS_EXTERNAL_REDIS=...  # 是否使用外部Redis

# 5. 资源限制
MS_EUREKA_MEM_LIMIT=...  # Eureka内存限制
MS_GATEWAY_MEM_LIMIT=... # Gateway内存限制
```

**为什么这样分类？**
- 职责清晰（一眼看出是什么配置）
- 便于维护（相关配置在一起）
- 易于理解（分类逻辑清晰）

### 2.2 命名规范

**统一前缀：MS_**
```bash
MS_MYSQL_HOST    # ✅ 清晰，一看就是MeterSphere的配置
MYSQL_HOST       # ❌ 可能与系统环境变量冲突
```

**语义化命名**：
```bash
MS_MYSQL_HOST    # ✅ 明确是主机地址
MS_MYSQL_IP      # ❌ IP限制了IPv6的可能性
MS_DB_HOST       # ❌ DB太泛，不知道是什么数据库
```

**层级结构**：
```bash
MS_服务_属性_子属性
    │    │    └── 具体属性
    │    └── 配置项
    └── 服务名

示例：
MS_GATEWAY_MEM_LIMIT    # Gateway的内存限制
MS_API_MEM_LIMIT        # API服务的内存限制
```

### 2.3 默认值设计

**为什么需要默认值？**
```yaml
mem_limit: ${MS_EUREKA_MEM_LIMIT:-536870912}
                                    └── 默认512MB
```

**场景**：
```
install.conf中没有配置MS_EUREKA_MEM_LIMIT
├── 使用默认值536870912（512MB）
├── 服务正常启动
└── 避免因配置缺失导致启动失败
```

**默认值的设定原则**：
```
1️⃣ 保守原则
   └── 内存默认值够用但不浪费

2️⃣ 安全原则
   └── 端口默认不对外暴露

3️⃣ 便利原则
   └── 开发环境默认配置即可用
```

## 三、配置项详解

### 3.1 镜像配置

**MS_IMAGE_PREFIX**
```bash
MS_IMAGE_PREFIX=registry.cn-qingdao.aliyuncs.com/metersphere
```

**设计考虑**：
```
为什么用变量？
├── 场景1：从阿里云切换到腾讯云
│   └── 只改这一个配置
├── 场景2：使用私有Registry
│   └── 改为内部Registry地址
└── 场景3：本地开发
    └── 改为localhost:5000
```

**MS_IMAGE_TAG**
```bash
MS_IMAGE_TAG=v2.10.23-lts
```

**版本标签策略**：
```
生产环境：
└── v2.10.23-lts（具体版本，便于回滚）

测试环境：
└── v2.10.23-lts or develop（灵活切换）

开发环境：
└── develop or feature/xxx（最新代码）
```

### 3.2 网络配置

**MS_DOCKER_SUBNET**
```bash
MS_DOCKER_SUBNET=172.30.10.0/24
```

**子网选择原则**：
```
避免冲突：
├── 不用172.17.0.0/16（Docker默认网段）
├── 不用192.168.1.0/24（常见局域网网段）
└── 选择172.30.x.x（相对冷门）

为什么/24？
├── 254个可用IP（够用）
├── 便于管理（不会太分散）
└── 性能考虑（广播域不宜太大）
```

**MS_SERVER_PORT**
```bash
MS_SERVER_PORT=8081
```

**端口选择考虑**：
```
为什么8081？
├── 不用8080（常被其他服务占用）
├── 不用80（需要root权限）
├── 不用3000（Node.js常用）
└── 8081（相对安全，不冲突）
```

### 3.3 基础设施配置

**为什么分离主机、端口、用户、密码？**

```bash
# 好的设计
MS_MYSQL_HOST=172.21.183.234
MS_MYSQL_PORT=3307
MS_MYSQL_USER=root
MS_MYSQL_PASSWORD=Password123@mysql

# 不好的设计
MS_MYSQL_URL=jdbc:mysql://root:password@172.21.183.234:3307/db
```

**分离的好处**：
```
1️⃣ 灵活性
   └── 只改密码，不改其他

2️⃣ 安全性
   └── 密码单独管理，不与URL混在一起

3️⃣ 复用性
   └── 多个服务复用相同的主机配置
```

**连接字符串的拼接**：
```yaml
environment:
  SPRING_DATASOURCE_URL: jdbc:mysql://${MS_MYSQL_HOST}:${MS_MYSQL_PORT}/${MS_MYSQL_DB}?参数
  SPRING_DATASOURCE_USERNAME: ${MS_MYSQL_USER}
  SPRING_DATASOURCE_PASSWORD: ${MS_MYSQL_PASSWORD}
```

### 3.4 外部服务开关

**MS_EXTERNAL_MYSQL**
```bash
MS_EXTERNAL_MYSQL=true   # 使用外部MySQL
MS_EXTERNAL_MYSQL=false  # 使用Docker部署的MySQL
```

**设计思路**：
```
灵活性：
├── 开发环境：false（Docker一键启动）
├── 测试环境：true（使用测试数据库）
└── 生产环境：true（使用高可用集群）

使用方式：
if [ "${MS_EXTERNAL_MYSQL}" = "false" ]; then
    docker-compose -f docker-compose-mysql.yml up -d
fi
```

### 3.5 资源限制配置

**为什么用字节而不是GB？**

```bash
# 当前方式
MS_EUREKA_MEM_LIMIT=536870912  # 512MB

# 为什么不用
MS_EUREKA_MEM_LIMIT=512M
MS_EUREKA_MEM_LIMIT=0.5G
```

**原因**：
```
1️⃣ 精确控制
   └── 字节数精确，不会产生歧义

2️⃣ Docker原生
   └── mem_limit参数接受字节数

3️⃣ 计算方便
   └── 512MB = 512 * 1024 * 1024 = 536870912
```

## 四、环境差异管理

### 4.1 多环境策略

**方案1：多个配置文件（推荐）**
```bash
install.conf.dev    # 开发环境
install.conf.test   # 测试环境
install.conf.prod   # 生产环境

# 使用时
cp install.conf.test install.conf
```

**方案2：环境变量覆盖**
```bash
# install.conf（默认开发环境）
MS_MYSQL_HOST=localhost

# 生产环境启动时
MS_MYSQL_HOST=prod-mysql-server docker-compose up -d
```

**方案3：配置模板**
```bash
# install.conf.template
MS_MYSQL_HOST={{MYSQL_HOST}}

# 部署时替换
sed "s/{{MYSQL_HOST}}/192.168.1.100/g" install.conf.template > install.conf
```

### 4.2 配置差异点

**开发环境**：
```bash
MS_IMAGE_TAG=develop           # 最新开发版
MS_SERVER_PORT=8081            # 本机端口
MS_EXTERNAL_MYSQL=false        # Docker部署
MS_MYSQL_HOST=mysql            # 容器名
MS_GATEWAY_MEM_LIMIT=536870912 # 512MB（节省资源）
```

**测试环境**：
```bash
MS_IMAGE_TAG=v2.10.23-lts      # 稳定版本
MS_SERVER_PORT=8081            # 测试服务器端口
MS_EXTERNAL_MYSQL=true         # 外部测试库
MS_MYSQL_HOST=192.168.1.100    # 测试数据库服务器
MS_GATEWAY_MEM_LIMIT=858993459 # 800MB
```

**生产环境**：
```bash
MS_IMAGE_TAG=v2.10.23-lts      # 生产版本
MS_SERVER_PORT=80              # 标准HTTP端口
MS_EXTERNAL_MYSQL=true         # 外部生产库
MS_MYSQL_HOST=db-prod-master   # 生产数据库主库
MS_GATEWAY_MEM_LIMIT=2147483648 # 2GB（性能优先）
```

## 五、敏感信息管理

### 5.1 密码管理策略

**不要做的事**：
```bash
# ❌ 直接写在install.conf
MS_MYSQL_PASSWORD=admin123

# ❌ 提交到Git
git add install.conf
git commit -m "add config"  # 密码泄露！
```

**应该做的事**：

**方案1：环境变量**
```bash
# install.conf不包含密码
MS_MYSQL_PASSWORD=${MYSQL_PASSWORD}

# 服务器环境变量
export MYSQL_PASSWORD=SecurePassword123

# 启动服务
docker-compose up -d
```

**方案2：独立密钥文件**
```bash
# install.conf
source /opt/secrets/mysql.env

# /opt/secrets/mysql.env（不纳入Git）
MS_MYSQL_PASSWORD=SecurePassword123
```

**方案3：密钥管理系统**
```bash
# 从HashiCorp Vault获取
MS_MYSQL_PASSWORD=$(vault kv get secret/mysql-password)
```

### 5.2 .gitignore配置

```bash
# .gitignore
install.conf          # 实际配置（包含敏感信息）
*.env                 # 环境变量文件
secrets/              # 密钥目录

# 提交到Git的
install.conf.example  # 配置模板（不含敏感信息）
```

### 5.3 配置模板示例

**install.conf.example**：
```bash
# ===== 镜像配置 =====
MS_IMAGE_PREFIX=registry.cn-qingdao.aliyuncs.com/metersphere
MS_IMAGE_TAG=v2.10.23-lts

# ===== MySQL配置 =====
MS_MYSQL_HOST=<your-mysql-host>
MS_MYSQL_PORT=3306
MS_MYSQL_USER=root
MS_MYSQL_PASSWORD=<your-password>  # 请修改为实际密码

# ===== 使用说明 =====
# 1. 复制此文件：cp install.conf.example install.conf
# 2. 修改占位符为实际值
# 3. 不要提交install.conf到Git
```

## 六、配置验证

### 6.1 启动前检查

**检查脚本示例**：
```bash
#!/bin/bash
# check-config.sh

echo "检查配置文件..."

# 1. 检查文件是否存在
if [ ! -f "install.conf" ]; then
    echo "❌ install.conf不存在"
    echo "请复制install.conf.example并修改"
    exit 1
fi

# 2. 检查必要配置是否为空
source install.conf

check_var() {
    if [ -z "$2" ]; then
        echo "❌ $1 未配置"
        exit 1
    else
        echo "✅ $1 = $2"
    fi
}

check_var "MS_IMAGE_PREFIX" "$MS_IMAGE_PREFIX"
check_var "MS_IMAGE_TAG" "$MS_IMAGE_TAG"
check_var "MS_MYSQL_HOST" "$MS_MYSQL_HOST"

# 3. 检查格式
if [[ ! "$MS_MYSQL_PORT" =~ ^[0-9]+$ ]]; then
    echo "❌ MS_MYSQL_PORT格式错误（应为数字）"
    exit 1
fi

echo "✅ 配置检查通过"
```

### 6.2 配置冲突检测

**检查端口冲突**：
```bash
# 检查端口是否被占用
PORT=${MS_SERVER_PORT}
if lsof -Pi :$PORT -sTCP:LISTEN -t >/dev/null ; then
    echo "❌ 端口${PORT}已被占用"
    lsof -i :$PORT
    exit 1
fi
```

**检查网络冲突**：
```bash
# 检查子网是否冲突
SUBNET=${MS_DOCKER_SUBNET}
if docker network ls | grep -q "$SUBNET"; then
    echo "❌ 网段${SUBNET}已被使用"
    exit 1
fi
```

## 七、配置管理最佳实践

### 7.1 配置分离原则

```
分离层次：
1️⃣ 不变配置（镜像仓库地址）
   └── 写在install.conf

2️⃣ 环境配置（MySQL地址）
   └── 不同环境的install.conf

3️⃣ 敏感配置（密码）
   └── 环境变量或密钥管理系统

4️⃣ 运行时配置（JVM参数）
   └── 应用配置文件（application.yml）
```

### 7.2 版本管理

✅ **DO**：
- 提交install.conf.example到Git
- 不同环境维护不同的install.conf
- 配置变更记录在CHANGELOG

❌ **DON'T**：
- 提交包含密码的install.conf
- 所有环境共用一个配置
- 直接修改生产配置

### 7.3 文档化

**配置说明文档**：
```markdown
# install.conf配置说明

## MS_IMAGE_TAG
- 作用：指定镜像版本
- 可选值：v2.10.23-lts, develop, feature/xxx
- 建议：
  - 生产环境使用具体版本号
  - 测试环境可使用develop
  
## MS_SERVER_PORT
- 作用：Gateway对外端口
- 默认值：8081
- 注意：
  - 不要使用80（需要root权限）
  - 不要使用已占用端口
```

## 八、配置自动化

### 8.1 配置生成脚本

```bash
#!/bin/bash
# generate-config.sh

ENV=$1  # dev, test, prod

cat > install.conf <<EOF
# MeterSphere ${ENV} 环境配置
# 生成时间: $(date)

MS_IMAGE_TAG=$(get_image_tag $ENV)
MS_MYSQL_HOST=$(get_mysql_host $ENV)
# ... 根据环境生成配置
EOF

echo "配置文件已生成：install.conf"
```

### 8.2 配置同步

**从配置中心同步**：
```bash
#!/bin/bash
# sync-config.sh

# 从Nacos/Consul/etcd获取配置
curl http://config-server/metersphere/prod > install.conf

# 验证配置
./check-config.sh

# 重启服务
./restart-services.sh
```

## 九、故障排查

### 9.1 配置问题排查

**服务启动失败**：
```bash
# 1. 检查配置是否加载
docker-compose config

# 2. 查看实际使用的值
docker inspect gateway | grep MS_MYSQL_HOST

# 3. 验证连接
docker exec gateway ping $MS_MYSQL_HOST
```

**配置不生效**：
```bash
可能原因：
1️⃣ 没有source install.conf
2️⃣ 环境变量被覆盖
3️⃣ docker-compose没有重启
4️⃣ 配置文件路径错误
```

### 9.2 常见错误

**错误1：变量未定义**
```bash
# 错误信息
invalid interpolation format for "image"

# 原因
docker-compose.yml中使用了${MS_IMAGE_PREFIX}
但install.conf没有定义或没有source

# 解决
source install.conf
docker-compose up -d
```

**错误2：特殊字符未转义**
```bash
# 错误
MS_MYSQL_PASSWORD=Pass@word

# 原因
@ 可能被解释为其他含义

# 解决
MS_MYSQL_PASSWORD='Pass@word'  # 加引号
```

## 十、总结

### 10.1 统一配置的价值

1. **降低复杂度**：配置集中，易于管理
2. **减少错误**：单点修改，避免遗漏
3. **提高效率**：快速切换环境
4. **便于审计**：配置变更可追溯

### 10.2 配置管理原则

- **集中管理**：统一配置文件
- **分层设计**：不变/环境/敏感分离
- **安全第一**：敏感信息不提交Git
- **文档完善**：每个配置有说明
- **自动化**：脚本生成和验证
