# 高级搜索记忆功能 - 项目隔离升级说明

## 一、改动概述

将高级搜索记忆功能从"用户+模块"隔离升级为"用户+项目+模块"三重隔离。

### 存储键格式变更

**原格式**: `ADV_SEARCH_{userId}_{moduleKey}`
**新格式**: `ADV_SEARCH_{userId}_{projectId}_{moduleKey}`

---

### 2.1 核心工具类

## 三、使用方法

### 3.1 旧版组件使用方式（MsTableAdvSearchBar）

```vue
<template>
  <MsTableAdvSearchBar
    :condition.sync="condition"
    :custom-fields="customFields"
    module-key="ISSUE_LIST"
    :project-id="currentProjectId"
    @search="search"
  />
</template>

<script>
export default {
  data() {
    return {
      condition: {},
      currentProjectId: '', // 从路由或store获取当前项目ID
    };
  },
  mounted() {
    // 获取当前项目ID
    this.currentProjectId = this.$route.params.projectId ||
                           this.$store.state.projectId;
  },
  methods: {
    search(condition) {
      // 执行搜索
    }
  }
}
</script>
```

### 3.2 新版组件使用方式（MsTableAdvSearch）

```vue
<template>
  <MsTableAdvSearch
    :condition.sync="condition"
    :custom-fields="customFields"
    module-key="TEST_CASE_LIST"
    :project-id="projectId"
    @search="handleSearch"
  />
</template>

<script>
export default {
  computed: {
    projectId() {
      return this.$store.getters.currentProjectId;
    }
  },
  methods: {
    handleSearch(condition) {
      // 执行搜索逻辑
    }
  }
}
</script>
```

## 四、关键要点

### 4.1 必需参数
- `module-key`: 模块标识符（如 ISSUE_LIST、TEST_CASE_LIST）
- `project-id`: 当前项目ID

**重要**: 两个参数都必须提供，否则搜索记忆功能不会启用。

### 4.2 获取项目ID的方式

**方式1 - 从路由参数获取**:
```javascript
this.$route.params.projectId
```

**方式2 - 从Vuex Store获取**:
```javascript
this.$store.state.projectId
// 或
this.$store.getters.currentProjectId
```

**方式3 - 从sessionStorage获取**:
```javascript
sessionStorage.getItem('project_id')
```

### 4.3 向后兼容性

- 如果不传 `project-id`，搜索记忆功能将不启用
- 原有不使用搜索记忆的页面无需修改
- 旧的localStorage数据不会自动迁移，用户需要重新设置搜索条件

## 五、业务价值

### 5.1 隔离效果
- 用户A在项目1的搜索条件 ≠ 用户A在项目2的搜索条件
- 用户A在项目1的搜索条件 ≠ 用户B在项目1的搜索条件
- 同一项目内，缺陷列表的搜索条件 ≠ 用例列表的搜索条件

### 5.2 使用场景
1. 测试人员在不同项目间切换时，保持各自的搜索习惯
2. 同一用户在不同项目中关注的字段可能不同
3. 避免项目间搜索条件互相干扰

---

## 六、注意事项

### 6.1 数据迁移
旧格式的localStorage数据不会自动迁移到新格式，用户首次使用时需要重新设置搜索条件。

### 6.2 localStorage容量
每个项目+模块组合都会占用localStorage空间，建议定期清理不活跃项目的数据。

### 6.3 项目ID变更
如果项目ID发生变更（如项目重建），搜索记忆数据将丢失。

---

## 七、测试建议

### 7.1 功能测试
1. 在项目A设置搜索条件，切换到项目B，验证搜索条件不会带过去
2. 在项目A设置搜索条件，刷新页面，验证搜索条件能正确恢复
3. 点击"重置"按钮，验证localStorage中的数据被清除

### 7.2 边界测试
1. 不传 `project-id`，验证搜索记忆功能不启用
2. 传入空字符串作为 `project-id`，验证功能不启用
3. 快速切换项目，验证不会出现数据混乱

---

## 八、后续优化建议

1. **数据清理机制**: 定期清理超过30天未使用的搜索记忆数据
2. **数据迁移工具**: 提供工具将旧格式数据迁移到新格式
3. **跨设备同步**: 考虑将搜索记忆存储到服务端，实现跨设备同步
