# 前端构建与微前端资源加载机制

## 概述

MeterSphere 采用 qiankun 微前端架构，Gateway 作为统一入口，各业务模块前端由各自后端 serve。理解这个机制是前端开发调试的前提。

## SDK 前端的两类内容

```
framework/sdk-parent/frontend/
│
├── 【类型1：共享组件/工具】→ 被业务模块 import，打包进各模块的 dist
│   └── src/
│       ├── components/             # 核心共享组件（MsTableHeader、MsDialog 等）
│       ├── api/                    # 共享 API 封装
│       ├── assets/                 # 共享静态资源（图片、字体）
│       ├── utils/                  # 工具函数
│       ├── mixins/                 # Vue mixins
│       ├── plugins/                # Vue 插件
│       ├── store/                  # Pinia 状态管理
│       ├── styles/                 # 共享样式
│       ├── i18n/                   # 国际化
│       ├── icons/                  # 图标
│       ├── model/                  # 数据模型
│       ├── directive/              # Vue 指令
│       ├── filters/                # Vue 过滤器
│       ├── chart/                  # 图表组件
│       ├── index.js                # SDK 导出入口
│       └── micro-app.js            # 微前端配置
│
├── 【类型2：独立页面/静态资源】→ 由 Gateway 直接 serve
│   ├── public/
│   │   ├── favicon.ico
│   │   ├── index.html              # 主入口 HTML（qiankun 主应用）
│   │   ├── error/
│   │   │   └── 404.html
│   │   └── js/
│   └── src/business/               # 独立页面（构建后进 dist/）
│       ├── login/                  # 登录页
│       ├── error-page/             # 错误页面
│       └── app-layout/             # 主布局框架
│
└── dist/                           # 构建产物 → cp 到 gateway/static/
```

### 共享组件的引用机制

各业务模块 `package.json` 中声明：

```json
"metersphere-frontend": "file:../../framework/sdk-parent/frontend"
```

`file:` 协议的本质是 npm 在 `node_modules/metersphere-frontend` 下创建 symlink，指向 SDK 前端源码目录。业务模块执行 `npm run build` 时，webpack 顺着 symlink 直接读取 SDK 的 `.vue` 和 `.js` 源文件，与业务模块自己的代码一起编译打包进同一个 dist。

**因此改了共享组件后，不需要先构建 SDK 本身，不需要 npm link，不需要发布。**

## 前端资源加载全链路

### 第一步：浏览器访问 Gateway

```
浏览器 → Gateway(8000) → 返回 gateway/static/index.html
                          这是 SDK 前端构建的产物
                          里面包含 qiankun 微前端框架
```

### 第二步：qiankun 注册子应用

`micro-app.js` 从 Eureka 拿到所有微服务列表，然后注册子应用：

```javascript
// 开发模式：直接访问 webpack dev server
entry: '//127.0.0.1:4001'  // 端口 = 后端端口 - 4000

// 生产模式：通过 Gateway 代理
entry: 'http://localhost:8000/setting/'
entry: 'http://localhost:8000/track/'
```

### 第三步：Gateway 路由前端资源（关键机制）

`SessionFilter.java` 中的核心逻辑：

```java
// 转发 css js 到具体的模块
if (path.startsWith("/css") || path.startsWith("/js")) {
    for (String prefix : PREFIX) {  // PREFIX = {"/setting", "/track", "/api", ...}
        if (path.contains(prefix)) {
            return addPrefix(prefix, exchange, chain);
        }
    }
}
```

请求链路示例：

```
浏览器请求: GET /js/setting-4480.46a8bc11.js
                    ↓
Gateway SessionFilter 扫描路径，发现文件名包含 "setting"
                    ↓
改写路径为: /setting/js/setting-4480.46a8bc11.js
                    ↓
Spring Cloud Discovery Locator 匹配 serviceId=setting
                    ↓
转发到 system-setting 后端(8001)
                    ↓
Spring Boot 从 classpath:/static/js/setting-4480.46a8bc11.js 返回文件
```

同理 `track-11.xxx.js` → 路由到 test-track(8003)。

### 完整请求链路图

```
浏览器
  │
  ├─ GET /                          → Gateway 返回 index.html（SDK 前端主应用）
  ├─ GET /js/app.xxx.js             → Gateway 自己 serve（无模块前缀关键字）
  │
  ├─ qiankun 加载子应用入口
  │   ├─ GET /setting/              → 路由到 system-setting(8001) 返回 index.html
  │   └─ GET /track/                → 路由到 test-track(8003) 返回 index.html
  │
  ├─ 子应用 JS/CSS（文件名含模块关键字）
  │   ├─ GET /js/setting-4480.js    → SessionFilter 加 /setting 前缀 → 8001 serve
  │   └─ GET /js/track-11.js        → SessionFilter 加 /track 前缀 → 8003 serve
  │
  └─ API 请求
      ├─ POST /release-note/list/1/10  → Gateway 路由到注册了该端点的微服务
      └─ GET /release-note/recent/5    → 同上（Controller 在 SDK 层，所有微服务都有）
```

## 为什么修改前端后必须重启微服务

```
Spring Boot 启动
    ↓
扫描 src/main/resources/static/ → 加载到 classpath 缓存
    ↓
所有 /static/** 请求从缓存中读取
    ↓
你在运行期间 cp 新文件到 static/ 目录
    ↓
❌ classpath 缓存里还是旧文件，新文件不会被读到
    ↓
必须重启 → 重新扫描 static/ → 缓存更新 → ✅ 新文件生效
```

## 前端修改后的完整操作流程

```
改前端代码 → npm run build → cp dist/* 到后端 static/ → 重启对应微服务 → 刷新浏览器
```

### 三种场景对照表

| 改了什么 | 构建命令 | cp 目标 | 重启谁 |
|---------|---------|---------|--------|
| SDK 独立页面（登录页、404页等） | `cd framework/sdk-parent/frontend && npm run build` | `cp -r dist/* ../../gateway/src/main/resources/static/` | Gateway(8000) |
| system-setting 前端 | `cd system-setting/frontend && npm run build` | `cp -r dist/* ../backend/src/main/resources/static/` | system-setting(8001) |
| test-track 前端 | `cd test-track/frontend && npm run build` | `cp -r dist/* ../backend/src/main/resources/static/` | test-track(8003) |
| api-test 前端 | `cd api-test/frontend && npm run build` | `cp -r dist/* ../backend/src/main/resources/static/` | api-test(8004) |
| performance-test 前端 | `cd performance-test/frontend && npm run build` | `cp -r dist/* ../backend/src/main/resources/static/` | performance-test(8005) |
| SDK 共享组件（被业务模块 import 的） | 去使用该组件的业务模块 `npm run build` | cp 到对应业务模块后端 static/ | 对应业务模块微服务 |

### 开发模式的替代方案

如果使用 `npm run serve`（webpack dev server），前端资源由 dev server 直接 serve（端口 = 后端端口 - 4000），支持热更新，不需要 build、cp、重启。适合日常开发调试。

## 后端修改的操作流程

| 改了什么 | 操作 |
|---------|------|
| SDK 后端（framework/sdk-parent 下的 Java 代码） | IDEA 里重启所有引用了该代码的业务微服务（IDEA 会自动重新编译依赖模块） |
| 业务模块后端（如 test-track/backend） | IDEA 里只重启该业务微服务 |
| Gateway 后端 | 重启 Gateway(8000) |
| 数据库迁移脚本 | 重启包含该迁移脚本的微服务（Flyway 在启动时自动执行） |
