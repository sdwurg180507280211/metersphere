## 分析结果

### 所有文件都应该放在 test-track 模块

**原因：**

1. **业务归属**：
   - 所有文件都操作 `test_case_template`（测试用例模板表）
   - 所有 `custom_field` 的 scene 都是 `TEST_CASE`（测试用例场景）
   - 所有 `custom_field_template` 的 scene 都是 `TEST_CASE`
   - 这些都属于测试跟踪（test-track）模块的业务范围

2. **表归属**：
   - 根据数据库表结构说明文档，`test_case_template` 属于"测试跟踪（Test Track）模块"
   - `custom_field` 和 `custom_field_template` 虽然由 project-management 管理，但这里的字段都是 TEST_CASE 场景，属于测试用例功能

3. **模块职责**：
   - **project-management**：管理项目、模板的通用管理（包括用例模板、缺陷模板、接口模板的 CRUD）
   - **test-track**：测试用例的业务逻辑和数据初始化
   - 这些迁移脚本是测试用例模板的**数据初始化**，应该属于 test-track

4. **目录结构**：
   - test-track 模块已有 `2.10.23.ddl` 目录
   - 应该放在 `test-track/backend/src/main/resources/db/migration/2.10.23.ddl/` 目录下

## 建议操作

将所有 5 个文件移动到 test-track 模块：

```bash
# 目标目录
test-track/backend/src/main/resources/db/migration/2.10.23.ddl/

# 需要移动的文件
- V3__2.10.23_release.sql
- V4__2.10.23_release.sql
- V5__2.10.23_release.sql
- V6__2.10.23_release.sql
- V7__fix_global_template_fields.sql
```

## 注意事项

1. **Flyway 版本号**：
   - test-track 模块使用 `track_version` 表记录迁移
   - 需要确保版本号不与现有迁移冲突
   - 当前 test-track 最新版本是 V16（2.10.23/ddl/V16__test_case_query_index.sql）
   - 这些文件的版本号（V3-V7）可能需要调整

2. **版本号调整建议**：
   - 如果 test-track 的版本号已经到 V16，这些文件应该使用 V17-V21
   - 或者根据实际版本号情况调整

3. **依赖关系**：
   - V7 依赖 V3、V5、V6 的执行结果
   - 需要保持执行顺序

---

**结论**：所有 5 个文件都应该移动到 test-track 模块。
