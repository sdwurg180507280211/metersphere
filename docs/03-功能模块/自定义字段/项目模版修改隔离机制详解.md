# 项目模版修改隔离机制详解

## 文档说明

本文档详细说明 MeterSphere 中一个非常重要但容易被误解的机制：**在项目设置中修改全局模版时，系统如何实现修改隔离，避免影响其他项目**。

---

## 🎯 核心问题

**问题**：当我在一个项目下修改全局模版的字段关联关系时，是不是全局修改？会影响其他项目吗？

**答案**：❌ **不是全局修改**，✅ **不会影响其他项目**

---

## 🔒 隔离机制原理

### 1. 系统设计思想

MeterSphere 采用了一种**写时复制（Copy-on-Write）**的机制来处理项目级别的模版修改：

- 当项目使用全局模版时，多个项目共享同一个模版定义
- 当某个项目需要修改模版时，系统自动为该项目创建专属模版
- 其他项目继续使用原来的全局模版，不受影响

### 2. 核心代码逻辑

**后端实现**：`IssueTemplateService.update()`

```java
public void update(UpdateIssueTemplateRequest request) {
    // 判断：编辑的是全局模版还是项目模版？
    if (request.getGlobal() != null && request.getGlobal()) {
        // 情况1：编辑的是全局模版
        String originId = request.getId();  // 原全局模版ID
        
        // 步骤1：创建一个新的项目模版（复制全局模版的内容）
        String newTemplateId = add(request);
        
        // 步骤2：更新项目的 issue_template_id，指向新的项目模版
        projectService.updateIssueTemplate(originId, newTemplateId, request.getProjectId());
        
        // 结果：该项目使用新的项目模版，其他项目仍使用全局模版
    } else {
        // 情况2：编辑的是项目模版
        // 直接更新项目模版，不影响其他项目
        checkExist(request);
        customFieldTemplateService.deleteByTemplateId(request.getId());
        IssueTemplate template = new IssueTemplate();
        BeanUtils.copyBean(template, request);
        template.setUpdateTime(System.currentTimeMillis());
        issueTemplateMapper.updateByPrimaryKeySelective(template);
        customFieldTemplateService.create(request.getCustomFields(), request.getId(),
                TemplateConstants.FieldTemplateScene.ISSUE.name());
    }
}
```

## 📊 操作场景对比

### 场景1：在项目设置中修改全局模版

**操作路径**：项目设置 > 模版管理 > 编辑全局模版

| 步骤 | 操作 | 系统行为 | 数据库变化 |
|------|------|---------|-----------|
| 1 | 打开项目A的模版管理 | 显示全局模版和项目模版 | 无变化 |
| 2 | 点击编辑全局模版（Local-default） | 打开编辑对话框 | 无变化 |
| 3 | 添加/删除字段，点击保存 | 触发 `update()` 方法 | 开始变化 |
| 4 | 系统检测到编辑的是全局模版 | `request.getGlobal() == true` | 无变化 |
| 5 | 系统创建项目A的专属模版 | 复制全局模版内容 | 插入新的项目模版记录 |
| 6 | 系统更新项目A的模版ID | `project.issue_template_id = 新模版ID` | 更新 project 表 |
| 7 | 完成 | 项目A使用新的项目模版 | 其他项目不受影响 |

**数据库变化示例**：

```sql
-- 操作前：项目A使用全局模版
SELECT 
  p.name AS 项目名,
  p.issue_template_id AS 模版ID,
  it.name AS 模版名,
  CASE WHEN it.global = 1 THEN '全局模版' ELSE '项目模版' END AS 模版类型
FROM project p
JOIN issue_template it ON p.issue_template_id = it.id
WHERE p.name = '项目A';

-- 结果：
-- 项目名 | 模版ID                               | 模版名  | 模版类型
-- 项目A  | 5d7c87d2-f405-4ec1-9a3d-71b514cdfda3 | default | 全局模版

-- 在项目A中编辑全局模版（添加字段）

-- 操作后：项目A使用项目模版
SELECT 
  p.name AS 项目名,
  p.issue_template_id AS 模版ID,
  it.name AS 模版名,
  CASE WHEN it.global = 1 THEN '全局模版' ELSE '项目模版' END AS 模版类型
FROM project p
JOIN issue_template it ON p.issue_template_id = it.id
WHERE p.name = '项目A';

-- 结果：
-- 项目名 | 模版ID                               | 模版名  | 模版类型
-- 项目A  | 新生成的UUID                          | default | 项目模版

-- 其他项目仍然使用全局模版
SELECT 
  p.name AS 项目名,
  p.issue_template_id AS 模版ID,
  it.name AS 模版名,
  CASE WHEN it.global = 1 THEN '全局模版' ELSE '项目模版' END AS 模版类型
FROM project p
JOIN issue_template it ON p.issue_template_id = it.id
WHERE p.name IN ('项目B', '项目C');

-- 结果：
-- 项目名 | 模版ID                               | 模版名  | 模版类型
-- 项目B  | 5d7c87d2-f405-4ec1-9a3d-71b514cdfda3 | default | 全局模版
-- 项目C  | 5d7c87d2-f405-4ec1-9a3d-71b514cdfda3 | default | 全局模版
```

### 场景2：在系统设置中修改全局模版

**操作路径**：系统设置 > 工作空间 > 模版管理 > 编辑全局模版

| 步骤 | 操作 | 系统行为 | 影响范围 |
|------|------|---------|---------|
| 1 | 打开系统设置的模版管理 | 显示全局模版 | 无影响 |
| 2 | 点击编辑全局模版 | 打开编辑对话框 | 无影响 |
| 3 | 添加/删除字段，点击保存 | 直接修改全局模版 | ⚠️ 影响所有使用该模版的项目 |

**关键区别**：
- 项目设置中修改 → 创建项目模版 → 只影响当前项目
- 系统设置中修改 → 直接修改全局模版 → 影响所有项目

```
用户在项目A中编辑全局模版
│
├─ 前端发送请求
│  └─ PUT /field/template/issue/update
│     └─ 请求参数：
│        {
│          id: "5d7c87d2-f405-4ec1-9a3d-71b514cdfda3",  // 全局模版ID
│          global: true,                                  // 标识这是全局模版
│          projectId: "项目A的ID",
│          customFields: [...]                            // 修改后的字段列表
│        }
│
├─ 后端处理请求
│  └─ IssueTemplateService.update()
│     │
│     ├─ 判断：request.getGlobal() == true
│     │  └─ 是全局模版，需要隔离修改
│     │
│     ├─ 步骤1：创建项目模版
│     │  └─ add(request)
│     │     ├─ 生成新的模版ID
│     │     ├─ 设置 global = false（项目模版）
│     │     ├─ 设置 projectId = 项目A的ID
│     │     ├─ 复制全局模版的内容
│     │     └─ 插入 issue_template 表
│     │        └─ 返回新模版ID
│     │
│     ├─ 步骤2：更新项目的模版ID
│     │  └─ projectService.updateIssueTemplate(originId, newTemplateId, projectId)
│     │     └─ UPDATE project 
│     │        SET issue_template_id = 新模版ID
│     │        WHERE id = 项目A的ID
│     │
│     └─ 步骤3：创建字段关联
│        └─ customFieldTemplateService.create(customFields, newTemplateId, "ISSUE")
│           └─ INSERT INTO custom_field_template (...)
│
└─ 结果
   ├─ 项目A：使用新的项目模版（包含修改后的字段）
   ├─ 项目B：仍使用全局模版（不受影响）
   └─ 项目C：仍使用全局模版（不受影响）
```

## ⚠️ 重要注意事项

### 1. 修改入口的区别

| 修改入口 | 路径 | 影响范围 | 是否隔离 |
|---------|------|---------|---------|
| **项目设置** | 项目设置 > 模版管理 | 只影响当前项目 | ✅ 隔离修改 |
| **系统设置** | 系统设置 > 工作空间 > 模版管理 | 影响所有项目 | ❌ 全局修改 |

### 2. 隔离机制的触发条件

隔离机制只在以下情况触发：
- ✅ 在项目设置中编辑全局模版
- ❌ 在项目设置中编辑项目模版（直接修改，不触发隔离）
- ❌ 在系统设置中编辑全局模版（直接修改，不触发隔离）

### 3. 性能影响

- 创建项目模版是一个轻量级操作
- 只复制模版定义和字段关联关系
- 不复制历史数据
- 对系统性能影响极小

### 4. 数据一致性

- 项目模版创建后，与全局模版完全独立
- 后续对全局模版的修改不会影响已创建的项目模版
- 项目模版的修改也不会影响全局模版

## 🔍 常见问题

### Q1: 为什么要设计这种隔离机制？

**A**: 这是一种**防御性设计**，目的是：
1. 避免误操作影响所有项目
2. 允许项目级别的定制化
3. 保持系统的灵活性和安全性

### Q2: 如果我想让修改影响所有项目怎么办？

**A**: 在**系统设置**中修改全局模版，而不是在项目设置中修改。

### Q3: 项目模版创建后，能否恢复使用全局模版？

**A**: 可以，在项目设置中重新选择全局模版即可。但之前的项目模版不会自动删除。

### Q4: 如何判断项目使用的是全局模版还是项目模版？

**A**: 查询数据库：

```sql
SELECT 
  p.name AS 项目名,
  CASE 
    WHEN it.global = 1 THEN '全局模版'
    WHEN it.global = 0 THEN '项目模版'
  END AS 模版类型
FROM project p
JOIN issue_template it ON p.issue_template_id = it.id
WHERE p.name = '你的项目名';
```

### Q5: 项目模版会占用很多存储空间吗？

**A**: 不会。项目模版只存储模版定义和字段关联关系，数据量很小（通常几KB）。

## 📝 最佳实践

### 1. 推荐做法

- ✅ 大部分项目直接使用全局模版
- ✅ 需要定制时，在项目设置中修改（自动隔离）
- ✅ 定期清理无用的项目模版
- ✅ 在系统设置中统一管理全局模版

### 2. 不推荐做法

- ❌ 为每个项目都创建专属模版（增加维护成本）
- ❌ 在系统设置中频繁修改全局模版（影响所有项目）
- ❌ 不了解隔离机制就随意修改模版

### 3. 操作建议

| 场景 | 推荐操作 | 原因 |
|------|---------|------|
| 新建项目 | 使用全局模版 | 简单、统一 |
| 项目需要特殊字段 | 在项目设置中添加字段 | 自动隔离，不影响其他项目 |
| 所有项目都需要新字段 | 在系统设置中添加到全局模版 | 统一管理 |
| 测试新字段 | 在测试项目中添加 | 隔离测试，不影响生产项目 |

## 🎯 总结

### 核心要点

1. **隔离机制**：在项目设置中修改全局模版时，系统自动创建项目专属模版
2. **写时复制**：只有在修改时才创建副本，读取时共享全局模版
3. **影响范围**：项目级别的修改只影响当前项目，不影响其他项目
4. **修改入口**：项目设置 = 隔离修改，系统设置 = 全局修改

### 关键结论

- ✅ 在项目设置中修改全局模版是**安全的**
- ✅ 系统会自动创建项目专属模版，实现**隔离修改**
- ✅ 其他项目仍然使用原来的全局模版，**不受影响**
- ⚠️ 只有在**系统设置**中修改全局模版，才会影响所有项目

### 设计优势

1. **安全性**：防止误操作影响所有项目
2. **灵活性**：允许项目级别的定制化
3. **易用性**：用户无需关心底层实现，系统自动处理
4. **可维护性**：清晰的隔离边界，便于问题排查
