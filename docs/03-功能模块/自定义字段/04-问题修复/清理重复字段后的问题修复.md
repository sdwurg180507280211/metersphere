# 清理重复字段后的问题修复

## 问题现象

清理 `custom_field_template` 表中的重复字段关联后，在某些项目（如"测试项目"）下创建缺陷时，"缺陷所属系统"和"处理人"字段无法指定值。

## 问题分析

### 1. 根本原因

项目引用的缺陷模板不存在，导致无法加载字段配置。

**数据库查询结果**：

```sql
SELECT 
    p.id as project_id,
    p.name as project_name,
    p.issue_template_id,
    it.name as template_name
FROM project p
LEFT JOIN issue_template it ON p.issue_template_id = it.id
WHERE p.name = '测试项目';
```

结果：
```
project_id: 5d75b4c6-a431-4743-b0b3-1876e36f2905
project_name: 测试项目
issue_template_id: c52e4242-b753-47a9-9384-e9e236d9a436
template_name: NULL  <- 模板不存在！
```

### 2. 问题来源

- 模板 `c52e4242-b753-47a9-9384-e9e236d9a436` 在 `issue_template` 表中不存在
- 但该模板的字段关联在 `custom_field_template` 表中还存在
- 这个问题**不是清理脚本导致的**，模板记录在清理前就已经被删除了

**证据**：
- 备份文件中包含该模板的字段关联记录
- 但 `issue_template` 表中从未有过该模板记录
- 可能是之前手动删除模板时，没有级联删除字段关联

### 3. 影响范围

查询所有引用不存在模板的项目：

```sql
SELECT 
    p.id,
    p.name,
    p.issue_template_id
FROM project p
LEFT JOIN issue_template it ON p.issue_template_id = it.id
WHERE p.issue_template_id IS NOT NULL 
AND it.id IS NULL;
```

## 修复方案

### 方案一：指向全局默认模板（已实施）

将项目的 `issue_template_id` 修改为全局默认模板 `5d7c87d2-f405-4ec1-9a3d-71b514cdfda3`。

**优点**：
- 简单快速
- 全局默认模板包含所有必需字段
- 不需要重建模板

**执行步骤**：

```sql
-- 修改测试项目的模板引用
UPDATE project 
SET issue_template_id = '5d7c87d2-f405-4ec1-9a3d-71b514cdfda3'
WHERE id = '5d75b4c6-a431-4743-b0b3-1876e36f2905';
```

**验证**：

```sql
-- 验证模板字段配置
SELECT 
    cf.name as field_name,
    cft.`order`,
    cft.default_value
FROM custom_field_template cft
INNER JOIN custom_field cf ON cft.field_id = cf.id
WHERE cft.template_id = '5d7c87d2-f405-4ec1-9a3d-71b514cdfda3'
AND cf.name IN ('缺陷所属系统', '处理人', '严重程度', '状态')
ORDER BY cft.`order`;
```

结果：
```
缺陷所属系统 - order: NULL, default_value: ""
严重程度     - order: NULL, default_value: "P0"
处理人       - order: NULL, default_value: ""
状态         - order: NULL, default_value: "new"
```

✅ 所有必需字段都存在！

### 方案二：批量修复所有问题项目

如果有多个项目引用了不存在的模板，可以批量修复：

```sql
-- 查找所有引用不存在模板的项目
SELECT 
    p.id,
    p.name,
    p.issue_template_id as old_template_id,
    '5d7c87d2-f405-4ec1-9a3d-71b514cdfda3' as new_template_id
FROM project p
LEFT JOIN issue_template it ON p.issue_template_id = it.id
WHERE p.issue_template_id IS NOT NULL 
AND it.id IS NULL;

-- 批量修复
UPDATE project p
LEFT JOIN issue_template it ON p.issue_template_id = it.id
SET p.issue_template_id = '5d7c87d2-f405-4ec1-9a3d-71b514cdfda3'
WHERE p.issue_template_id IS NOT NULL 
AND it.id IS NULL;
```

### 方案三：清理孤立的字段关联

清理 `custom_field_template` 表中引用不存在模板的记录：

```sql
-- 查看孤立的字段关联
SELECT 
    cft.id,
    cft.template_id,
    COUNT(*) as field_count
FROM custom_field_template cft
LEFT JOIN issue_template it ON cft.template_id = it.id
WHERE it.id IS NULL
GROUP BY cft.template_id;

-- 删除孤立的字段关联（可选）
DELETE cft
FROM custom_field_template cft
LEFT JOIN issue_template it ON cft.template_id = it.id
WHERE it.id IS NULL;
```

**注意**：这个操作是可选的，不影响功能，只是清理冗余数据。

## 预防措施

### 1. 添加外键约束

为了防止将来再次出现类似问题，建议添加外键约束：

```sql
-- 为 project 表添加外键约束
ALTER TABLE project
ADD CONSTRAINT fk_project_issue_template
FOREIGN KEY (issue_template_id) 
REFERENCES issue_template(id)
ON DELETE SET NULL;

-- 为 custom_field_template 表添加外键约束
ALTER TABLE custom_field_template
ADD CONSTRAINT fk_custom_field_template_issue_template
FOREIGN KEY (template_id) 
REFERENCES issue_template(id)
ON DELETE CASCADE;
```

**说明**：
- `project.issue_template_id` 使用 `ON DELETE SET NULL`：模板删除时，项目的模板引用设为 NULL
- `custom_field_template.template_id` 使用 `ON DELETE CASCADE`：模板删除时，自动删除相关的字段关联

### 2. 删除模板前的检查

在删除模板前，应该检查是否有项目在使用：

```sql
-- 检查模板是否被使用
SELECT 
    p.id,
    p.name
FROM project p
WHERE p.issue_template_id = '要删除的模板ID';
```

如果有项目在使用，应该先将项目指向其他模板，再删除。

## 总结

- **问题原因**：项目引用的模板不存在，不是清理脚本导致的
- **修复方法**：将项目指向全局默认模板 `5d7c87d2-f405-4ec1-9a3d-71b514cdfda3`
- **预防措施**：添加外键约束，防止将来出现孤立引用
