## 1. 一句话定义：自定义字段系统在做什么

**自定义字段 = 定义字段（custom_field） + 把字段挂到模板上（custom_field_template） + 在业务资源上存值（如 custom_field_issues），最终实现“同一套字段能力在不同项目/模板/场景可配置、可展示、可存储”。**

---

## 2. 关键概念与字段层级（按项目实际）

### 2.1 资源隔离层级（字段侧）

- **全局层（Global）**
  - `custom_field.global = 1`
  - `custom_field.project_id = NULL`
  - 作用：所有项目共享（但是否显示取决于是否关联到模板）

- **项目层（Project）**
  - `custom_field.global = 0`
  - `custom_field.project_id = 具体项目ID`
  - 作用：仅当前项目可见/可配

### 2.2 system/global 组合语义（四种组合）

- `system=1, global=1`：**全局系统字段**（系统级核心字段，通常不可删除）
- `system=0, global=1`：全局自定义字段（理论存在，通常通过脚本创建；是否允许删除取决于权限与实现）
- `system=1, global=0`：项目系统字段（较少见）
- `system=0, global=0`：项目自定义字段（UI常规创建的字段）

---

## 3. 数据模型（核心表与关系）

### 3.1 四条主链路

- `workspace`（工作空间）
- `project`（项目）
- `issue_template / test_case_template / api_template`（模板）
- `custom_field`（字段定义）

### 3.2 字段显示的决定因素：custom_field_template

**关键结论：字段是否显示，不是由 `global=1` 决定，而是由 `custom_field_template` 是否存在关联决定。**

- `custom_field`：只定义“字段是什么”
- `custom_field_template`：定义“字段在哪个模板里展示、顺序/必填/默认值是什么”

---

## 4. “全局共享”的正确做法（以需求号/手机/地址为例）

### 4.1 全局共享的最小闭环（必须满足）

- **Step 1**：在 `custom_field` 插入字段
  - 关键：`global=1`、`project_id=NULL`（若要系统级保护，再加 `system=1`）
- **Step 2**：把字段关联到全局模板（`custom_field_template` 关联到 `issue_template.global=1` 等）
- **Step 3（关键）**：把字段关联到所有项目级模板（`issue_template.global=0`）

### 4.2 常见误区

- 误区：只插入 `custom_field(global=1)` 就以为所有项目都能看到
  - 实际：没有 `custom_field_template` 关联就不会显示

---

## 5. UI 创建“全局系统字段”方案（项目内文档结论）

### 5.1 现状与约束

- 传统实现中，UI 创建字段的后端逻辑往往会强制 `global=false`，因此 UI 只能创建项目级字段。

### 5.2 UI 创建全局系统字段的核心思路

- **后端**：新增创建全局系统字段的入口（如 `/custom/field/add/global`）
  - 权限：仅系统管理员可用（与 `SYSTEM_SETTING:READ` 等权限点匹配）
  - 创建后自动关联：
    - 全局模板（多平台 default 模板）
    - 所有项目模板
  - 删除保护：系统字段不可删

- **前端**：
  - 管理员可见“创建系统字段”入口/或“字段范围”选项
  - `isGlobal=true` 时走全局接口

---

## 6. 模板复制（项目级批量配置手段）

### 6.1 适用场景

- 少量项目快速初始化
- 标准模板推广（各项目后续可独立微调）
- 系统字段的“选项补充”场景（慎选复制模式）

### 6.2 重要限制

- **复制后的模板不会自动成为目标项目默认模板**
  - 复制完成后仍需逐个项目手动把新模板设置为默认模板，否则新字段不会在创建/编辑页出现。

### 6.3 追加 vs 覆盖

- **追加模式**：
  - 对系统字段：追加选项（更安全）
  - 对同名自定义字段：通常会生成 `_copy_xxx` 副本
- **覆盖模式**：
  - 可能覆盖目标项目已有字段的类型/选项（风险高）

## 7. 国际化（i18n）在字段显示中的策略

### 7.1 静态国际化（现状）

- 系统字段通常依赖 `SYSTEM_FIELD_NAME_MAP`（如 `table-constants.js`）将中文字段名映射到 i18n key
- i18n key 需要在 `zh-CN.js / en-US.js / zh-TW.js` 中存在

### 7.2 常见问题：新增字段显示空白 / i18n key 缺失

- 如果走“半自动 i18n key（remark 保存 key）”但未在 i18n 文件补齐翻译：
  - UI 可能显示为空或出现 i18n 警告

### 7.3 推荐的最小可用策略

- **系统字段（内置/白名单）**：继续静态 i18n
- **新增字段**：
  - 优先直接显示字段名（fallback），避免空白
  - 若要完整多语言：再引入“动态国际化”（DB 存翻译 + 前端加载）

---

## 8. 发布/打包（当修改涉及前端映射与国际化时）

### 8.1 需要打包的典型情况

- 修改 `framework/sdk-parent/frontend` 中：
  - `SYSTEM_FIELD_NAME_MAP` 映射
  - i18n 翻译文件

### 8.2 依赖顺序（最关键）

- 先构建 `framework/sdk-parent/frontend`（前端 SDK）
- 再构建依赖它的业务模块（例如 `test-track`、必要时 `project-management`）

---

## 9. 排查清单（最常用）

- 字段定义是否存在：`custom_field` 是否有记录（system/global/project_id 是否正确）
- 字段是否挂到模板：`custom_field_template` 是否有对应 `template_id` 关联
- 项目当前使用哪个模板：`project.issue_template_id` / `project.case_template_id`
- 前端是否因 i18n 缺失导致“显示空白”：检查 `SYSTEM_FIELD_NAME_MAP` 与 i18n 文件
