# 高级搜索/筛选逻辑对比（原有 vs 修改后）

> 目标：解释为什么“新增系统字段（custom_field.system=1）在页面可显示但筛选无效”，并沉淀一套可复用的参数组装与后端解析规则。
>
> 适用范围：
> - 缺陷管理（Issue）列表高级搜索
> - 功能用例（TestCase）列表高级搜索
> - 任何基于 `custom_field` 体系动态生成搜索项的列表页

---

## 1. 核心结论（一句话）

- **内置字段（built-in）**：后端在主表（如 `issues` / `test_case`）上直接拼 SQL 条件，前端参数形态是：
  - `combine.<key> = { operator, value }`
- **模板字段（custom_field 体系，包含 system=1 系统字段 + system=0 普通字段）**：后端通过 `custom_field_*` 存值表（如 `custom_field_issues` / `custom_field_test_case`）过滤资源 id，前端参数形态必须是：
  - `combine.customs = [{ id, operator, value, type }, ...]`

> 只要把“模板字段”误放到了 `combine.<fieldId>`（非 `customs`），后端通常不识别该 key，表现为：**高级搜索可选但筛选无效**。

---

## 2. 背景：为什么新增系统字段会踩坑

### 2.1 字段来源的两条链路

- **内置字段**：前端列表页通过 `search-components.js` 预置（例如 `name/updateTime/platform/creator`），后端 MyBatis `combine` SQL 直接使用主表字段。
- **模板字段**：前端列表页通过 `getAdvSearchCustomField(condition, template.customFields)` 动态把模板字段追加到高级搜索项中。
  - 这批字段来源于 `custom_field` / `custom_field_template`
  - system=1 只是字段属性（权限/保护/i18n 等），**查询仍然属于模板字段逻辑**

### 2.2 问题现象

- 新增系统字段（通过 SQL 插入 `custom_field.system=1` 并绑定到模板）后：
  - 高级搜索能显示该字段
  - 但筛选不生效（请求能发出，但后端不按该字段过滤）
- 原有系统字段（例如“状态/处理人”等）筛选正常

### 2.3 根因

- 原有高级搜索组装逻辑对“哪些字段走 `customs`”采用了**硬编码 label 白名单**（例如“状态/处理人/严重程度”等），导致：
  - 不在白名单的新字段（即使是系统字段）被错误放入 `combine.<fieldId>`
  - 后端不识别该 `<fieldId>` key -> 无效

---

## 3. 前端逻辑对比

本次涉及两个前端高级搜索组件：

- **旧版高级搜索（缺陷管理等使用）**：
  - 文件：`framework/sdk-parent/frontend/src/components/search/MsTableAdvSearchBar.vue`
- **新版高级搜索（功能用例等使用）**：
  - 文件：`framework/sdk-parent/frontend/src/components/new-ui/MsTableAdvSearch.vue`

> 两者 UI 形态不同，但本质都是“遍历 condition.components，把有值的组件组装成 condition.combine”。

### 3.1 原有逻辑（问题逻辑）

- 旧版 `MsTableAdvSearchBar.vue` / 新版 `MsTableAdvSearch.vue` 在 `setCondition()` 中：
  - 仅当 `component.custom === true` **或** label 命中硬编码白名单（如“状态/处理人/严重程度”等）时，才走 `handleCustomField()` 写入 `combine.customs`。
  - 否则写入：`combine[component.key] = {operator, value}`。

**后果**：
- 新增系统字段通常 `custom=false`（因为 `custom = !field.system`），且 label 不在白名单
- -> 被错误写入 `combine[字段id]`
- -> 后端忽略 -> 筛选无效

### 3.2 修改后逻辑（修复逻辑）

#### 3.2.1 白名单集中维护

- 文件：`framework/sdk-parent/frontend/src/components/search/search-components.js`
- 新增导出：`BUILTIN_ADV_SEARCH_KEYS`
  - 该集合是 **search-components.js 中所有预置高级搜索对象的 key 去重全集**
  - 作为“内置字段 key 白名单”的单一事实来源

#### 3.2.2 统一判定规则

- 两个高级搜索组件统一使用：
  - `BUILTIN_ADV_SEARCH_KEY_SET = new Set([...BUILTIN_ADV_SEARCH_KEYS, workspaceId, projectId, creatorName, operator, operatorName])`
- 在 `setCondition()` 中：
  - `key ∈ BUILTIN_ADV_SEARCH_KEY_SET` -> 写入 `combine[key]`
  - `key ∉ BUILTIN_ADV_SEARCH_KEY_SET` -> 一律视为模板字段（custom_field 体系）-> 写入 `combine.customs[]`

**效果**：
- 新增系统字段（system=1）不在白名单 -> 必然进入 `customs` -> 后端可识别
- 新增普通字段（system=0）同理
- 既有内置字段（name/updateTime/...）仍走 `combine[key]` 不受影响

## 4. 后端逻辑：内置字段 vs 自定义字段（SQL 实现）

### 4.1 缺陷列表（Issue）

- Controller：`test-track/backend/src/main/java/io/metersphere/controller/IssuesController.java`
  - `POST /issues/list/{goPage}/{pageSize}`
- Service：`test-track/backend/src/main/java/io/metersphere/service/IssuesService.java`
  - 调用 `extIssuesMapper.getIssues(request)`
- MyBatis：`test-track/backend/src/main/java/io/metersphere/base/mapper/ext/ExtIssuesMapper.xml`

#### 4.1.1 内置字段（combine.key）

在 `ExtIssuesMapper.xml` 的 `<sql id="combine">` 中：
- 例如：
  - `issues.update_time ...`（updateTime）
  - `issues.create_time ...`（createTime）
  - `issues.platform ...`（platform）
  - `issues.creator ...`（creator）

#### 4.1.2 模板字段（combine.customs）

在 `ExtIssuesMapper.xml` 的 `<if test="${condition}.customs != null ...">` 分支中：
- 遍历 `combine.customs`，通过 `custom_field_issues` 过滤 `issues.id`
- 多选字段会被 `ServiceUtils.setBaseQueryRequestCustomMultipleFields()` 预处理为 `JSON_CONTAINS` 片段

### 4.2 功能用例列表（TestCase）

- Service：`test-track/backend/src/main/java/io/metersphere/service/TestCaseService.java`
  - `ServiceUtils.setBaseQueryRequestCustomMultipleFields(request)`
  - `extTestCaseMapper.list(request)`
- MyBatis：`test-track/backend/src/main/java/io/metersphere/base/mapper/ext/ExtTestCaseMapper.xml`

#### 4.2.1 内置字段（combine.key）

`ExtTestCaseMapper.xml` 的 `<sql id="combine">` 中直接拼：
- `test_case.update_time ...`
- `test_case.create_time ...`
- `test_case.name ...`
- `test_case.priority ...`
- `test_case.status ...`
- `test_case.maintainer ...`
- 等

#### 4.2.2 模板字段（combine.customs）

`ExtTestCaseMapper.xml` 的 `<if test="${condition}.customs != null ...">` 分支：
- 遍历 `combine.customs`
- 通过 `custom_field_test_case` 表过滤 `test_case.id`
- 不同字段类型会走不同列：
  - `text_value`（richText/textarea）
  - `value`（其它类型）
  - 多选用 `${custom.value}`（预拼 JSON_CONTAINS）

## 5. 典型请求参数示例

### 5.1 正确示例（新增系统字段应进入 customs）

```json
{
  "combine": {
    "updateTime": { "operator": "between", "value": [1766332800000, 1766505600000] },
    "customs": [
      {
        "id": "issue-priority-field-2024-12-01-001",
        "operator": "in",
        "value": ["重要"],
        "type": "select"
      }
    ]
  }
}
```

### 5.2 错误示例（新增系统字段误放 combine.fieldId -> 无效）

```json
{
  "combine": {
    "updateTime": { "operator": "between", "value": [1766332800000, 1766505600000] },
    "issue-priority-field-2024-12-01-001": { "operator": "in", "value": ["重要"] }
  }
}
```

## 6. 排查清单（快速定位“能选但筛选无效”）

1. **看前端请求 payload 的 combine**
   - 该字段是否在 `combine.customs[]`？
   - 如果在 `combine.<fieldId>`，后端大概率不会识别。
2. **确认页面使用的高级搜索组件**
   - 缺陷管理：`MsTableAdvSearchBar.vue`
   - 功能用例：`MsTableAdvSearch.vue`
3. **确认后端 MyBatis 是否有 customs 分支**
   - Issue：`custom_field_issues`
   - TestCase：`custom_field_test_case`
4. **多选字段是否被预处理**
   - `ServiceUtils.setBaseQueryRequestCustomMultipleFields()` 会把多选 value 变成 `JSON_CONTAINS` 片段
