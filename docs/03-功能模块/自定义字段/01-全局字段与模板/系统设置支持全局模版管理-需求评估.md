# 系统设置支持全局模版和字段管理 - 需求评估

## 需求描述

在系统设置模块中，增加全局模版和全局字段的创建/编辑功能，允许管理员统一管理所有项目共享的模版和字段。

---

## 📊 现状分析

### 1. 当前架构

| 资源类型 | 当前管理位置 | 可操作范围 | 影响范围 |
|---------|------------|-----------|---------|
| 全局系统字段 | 系统预置 | ❌ 不可修改 | 所有项目 |
| 全局自定义字段 | 项目设置 > 自定义字段 | ✅ 可创建/编辑 | 所有项目 |
| 全局系统模版 | 系统预置 | ❌ 不可修改 | 所有项目 |
| 全局自定义模版 | ❌ 不支持 | ❌ 不可创建 | - |
| 项目字段 | 项目设置 > 自定义字段 | ✅ 可创建/编辑 | 当前项目 |
| 项目模版 | 项目设置 > 模版管理 | ✅ 可创建/编辑 | 当前项目 |

### 2. 当前数据统计

```
全局字段：13个
├─ 系统字段：6个（严重程度、处理人、状态、复测次数、缺陷所属系统、需求号）
└─ 自定义字段：7个（优先级、关闭时间、系统模块、缺陷产生原因等）

全局模版：4个
├─ 系统模版：4个（Local-default、JIRA-默认模版、Tapd-默认模版、禅道-默认模版）
└─ 自定义模版：0个
```

### 3. 当前问题

| 问题 | 描述 | 影响 |
|------|------|------|
| 全局字段管理分散 | 全局自定义字段在项目设置中创建 | 不直观，容易混淆 |
| 无全局自定义模版 | 不支持创建全局自定义模版 | 无法统一管理通用模版 |
| 权限管理不清晰 | 项目管理员可以创建全局字段 | 权限边界模糊 |
| 缺少统一入口 | 没有统一的全局资源管理入口 | 管理成本高 |

---

## 🎯 需求方案

### 方案1：系统设置增加全局资源管理

**功能位置**：系统设置 > 工作空间 > 全局资源管理

**功能模块**：

```
全局资源管理
├─ 全局字段管理
│  ├─ 全局系统字段（只读，查看）
│  └─ 全局自定义字段（创建、编辑、删除）
│
└─ 全局模版管理
   ├─ 全局系统模版（只读，查看）
   └─ 全局自定义模版（创建、编辑、删除）
```

### 方案2：保持现状，优化项目设置

**优化点**：
- 在项目设置中明确标识全局字段
- 增加全局字段的统一管理视图
- 优化权限控制

## 🔧 技术实现方案

### 1. 前端实现

**新增模块**：`system-setting/frontend/src/business/workspace/global-resource/`

```
global-resource/
├─ GlobalFieldList.vue          # 全局字段列表
├─ GlobalFieldEdit.vue          # 全局字段编辑
├─ GlobalTemplateList.vue       # 全局模版列表
├─ GlobalTemplateEdit.vue       # 全局模版编辑
└─ GlobalResourceSetting.vue    # 全局资源管理主页
```

**路由配置**：

```javascript
{
  path: 'global-resource',
  name: 'GlobalResource',
  component: () => import('@/business/workspace/global-resource/GlobalResourceSetting'),
  meta: {
    permissions: ['WORKSPACE:READ+GLOBAL_RESOURCE']
  }
}
```

### 2. 后端实现

**新增Controller**：`system-setting/backend/src/main/java/io/metersphere/controller/GlobalResourceController.java`

```java
@RestController
@RequestMapping("/global/resource")
public class GlobalResourceController {
    
    // 全局字段管理
    @GetMapping("/field/list")
    @RequiresPermissions("WORKSPACE:READ+GLOBAL_RESOURCE")
    public List<CustomField> listGlobalFields() {
        // 查询所有全局字段
    }
    
    @PostMapping("/field/add")
    @RequiresPermissions("WORKSPACE:READ+EDIT_GLOBAL_RESOURCE")
    public void addGlobalField(@RequestBody CustomField field) {
        // 创建全局字段
        field.setGlobal(true);
        // ...
    }
    
    // 全局模版管理
    @GetMapping("/template/list")
    @RequiresPermissions("WORKSPACE:READ+GLOBAL_RESOURCE")
    public List<IssueTemplate> listGlobalTemplates() {
        // 查询所有全局模版
    }
    
    @PostMapping("/template/add")
    @RequiresPermissions("WORKSPACE:READ+EDIT_GLOBAL_RESOURCE")
    public void addGlobalTemplate(@RequestBody IssueTemplate template) {
        // 创建全局模版
        template.setGlobal(true);
        template.setSystem(false);  // 自定义模版
        // ...
    }
}
```

### 3. 权限设计

| 权限 | 说明 | 角色 |
|------|------|------|
| `WORKSPACE:READ+GLOBAL_RESOURCE` | 查看全局资源 | 系统管理员、工作空间管理员 |
| `WORKSPACE:READ+EDIT_GLOBAL_RESOURCE` | 编辑全局资源 | 系统管理员 |
| `WORKSPACE:READ+DELETE_GLOBAL_RESOURCE` | 删除全局资源 | 系统管理员 |

### 4. 数据库变更

**无需变更**，现有表结构已支持：
- `custom_field` 表的 `global` 字段标识全局字段
- `issue_template` 表的 `global` 字段标识全局模版

## 🌐 国际化处理方案

### 1. 国际化现状

MeterSphere 的国际化配置存储在 JS 文件中，无法通过页面动态管理：

**国际化文件位置**：
```
framework/sdk-parent/frontend/src/i18n/lang/
├─ zh-CN.js          # 简体中文
├─ zh-TW.js          # 繁体中文
└─ en-US.js          # 英文
```

**字段国际化示例**：
```javascript
// zh-CN.js
custom_field: {
  field_name: '字段名称',
  field_type: '字段类型',
  issue_processor: '处理人',
  issue_status: '状态',
  // ...
}
```

### 2. 国际化挑战

| 挑战 | 描述 | 影响 |
|------|------|------|
| 无法页面管理 | 国际化配置在JS文件中，需要修改代码 | 🔴 高 |
| 开发者操作 | 需要开发者手动添加国际化配置 | 🟡 中 |
| 部署依赖 | 修改国际化需要重新构建和部署 | 🟡 中 |
| 维护成本 | 每个字段需要维护3种语言 | 🟡 中 |

### 3. 解决方案

#### 方案A：页面提示 + 文档说明（推荐）

**实现方式**：

1. **创建成功后显示提示**：

```vue
<template>
  <el-dialog title="创建成功" :visible.sync="successDialogVisible">
    <el-alert
      title="国际化配置提示"
      type="info"
      :closable="false"
      show-icon>
      <template slot>
        <p>字段创建成功！为了支持多语言，请按以下步骤添加国际化配置：</p>
        <ol>
          <li>打开国际化配置文件：
            <el-tag size="mini">framework/sdk-parent/frontend/src/i18n/lang/zh-CN.js</el-tag>
          </li>
          <li>在 <code>custom_field</code> 对象中添加：
            <pre><code>{{ i18nCode }}</code></pre>
          </li>
          <li>同样在 <code>zh-TW.js</code> 和 <code>en-US.js</code> 中添加对应翻译</li>
          <li>重新构建前端项目</li>
        </ol>
        <el-button type="text" @click="copyI18nCode">复制配置代码</el-button>
      </template>
    </el-alert>
  </el-dialog>
</template>

<script>
export default {
  data() {
    return {
      successDialogVisible: false,
      i18nCode: ''
    }
  },
  methods: {
    showSuccessTip(fieldName) {
      // 生成国际化配置代码
      const key = this.generateI18nKey(fieldName);
      this.i18nCode = `${key}: '${fieldName}',  // 简体中文\n` +
                      `${key}: '${fieldName}',  // 繁体中文（需要翻译）\n` +
                      `${key}: '${fieldName}',  // 英文（需要翻译）`;
      this.successDialogVisible = true;
    },
    generateI18nKey(fieldName) {
      // 将字段名转换为国际化key
      // 例如："处理人" -> "issue_processor"
      return fieldName.toLowerCase().replace(/\s+/g, '_');
    },
    copyI18nCode() {
      // 复制到剪贴板
      navigator.clipboard.writeText(this.i18nCode);
      this.$message.success('已复制到剪贴板');
    }
  }
}
</script>
```

2. **在列表页显示国际化状态**：

```vue
<el-table-column label="国际化状态" width="120">
  <template slot-scope="scope">
    <el-tag v-if="scope.row.i18nConfigured" type="success" size="mini">
      已配置
    </el-tag>
    <el-tooltip v-else content="需要手动添加国际化配置" placement="top">
      <el-tag type="warning" size="mini">
        未配置
      </el-tag>
    </el-tooltip>
  </template>
</el-table-column>
```

3. **提供国际化配置文档**：

在帮助文档中提供详细的国际化配置指南。

#### 方案B：数据库存储国际化（长期方案）

**实现方式**：

1. **新增国际化表**：

```sql
CREATE TABLE i18n_config (
  id VARCHAR(50) PRIMARY KEY,
  resource_type VARCHAR(20),  -- 'field', 'template', 'option'
  resource_id VARCHAR(50),    -- 关联的资源ID
  locale VARCHAR(10),         -- 'zh-CN', 'zh-TW', 'en-US'
  key VARCHAR(100),           -- 国际化key
  value VARCHAR(255),         -- 翻译文本
  create_time BIGINT,
  update_time BIGINT
);
```

2. **前端动态加载**：

```javascript
// 从后端API加载国际化配置
async loadI18nConfig() {
  const response = await getI18nConfig();
  this.$i18n.mergeLocaleMessage('zh-CN', response.zhCN);
  this.$i18n.mergeLocaleMessage('zh-TW', response.zhTW);
  this.$i18n.mergeLocaleMessage('en-US', response.enUS);
}
```

3. **页面管理国际化**：

在创建字段时，同时填写多语言翻译。

**优缺点对比**：

| 方案 | 优点 | 缺点 | 工作量 |
|------|------|------|--------|
| 方案A：页面提示 | 实现简单，不改变现有架构 | 需要手动修改代码 | 2人天 |
| 方案B：数据库存储 | 完全页面化管理，无需修改代码 | 改动较大，需要重构国际化机制 | 10人天 |

### 4. 推荐方案

**阶段1（MVP）**：采用方案A - 页面提示

**理由**：
- 实现简单，不改变现有架构
- 满足基本需求
- 开发工作量小

**阶段2（长期优化）**：逐步迁移到方案B - 数据库存储

**理由**：
- 提供更好的用户体验
- 完全页面化管理
- 降低维护成本

### 5. 国际化配置指南

**步骤1：生成国际化Key**

规则：`资源类型_字段名`

示例：
- 字段名："处理人" → Key: `issue_processor`
- 字段名："优先级" → Key: `issue_priority`
- 字段名："所属系统" → Key: `issue_associated_system`

**步骤2：添加国际化配置**

```javascript
// zh-CN.js
custom_field: {
  issue_processor: '处理人',
  issue_priority: '优先级',
  issue_associated_system: '所属系统',
}

// zh-TW.js
custom_field: {
  issue_processor: '處理人',
  issue_priority: '優先級',
  issue_associated_system: '所屬系統',
}

// en-US.js
custom_field: {
  issue_processor: 'Processor',
  issue_priority: 'Priority',
  issue_associated_system: 'Associated System',
}
```

**步骤3：重新构建**

```bash
cd framework/sdk-parent/frontend
npm run build
```

### 6. 国际化最佳实践

| 实践 | 说明 | 示例 |
|------|------|------|
| 统一命名规则 | 使用下划线分隔，全小写 | `issue_processor` |
| 分类管理 | 按模块分组 | `custom_field.xxx`, `template.xxx` |
| 注释说明 | 添加注释说明用途 | `// 缺陷处理人` |
| 保持一致 | 相同含义使用相同翻译 | "处理人"统一翻译为"Processor" |
| 及时更新 | 创建字段后立即添加国际化 | 避免遗漏 |
