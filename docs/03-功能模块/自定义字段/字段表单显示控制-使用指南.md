# 字段表单显示控制 - 使用指南

## 一、功能概述

### 1.1 核心问题

在 MeterSphere 中，某些自定义字段（如"复测次数"）需要在列表页显示，但不应在创建/编辑表单中显示。传统做法是在每个页面组件中硬编码过滤逻辑，导致：

- **代码重复**：多个页面需要重复相同的过滤逻辑
- **难以维护**：新增字段需要修改多个文件
- **扩展性差**：无法灵活配置不同场景的显示规则

### 1.2 解决方案

通过抽象出统一的字段过滤机制，实现：

- **集中配置**：所有字段显示规则在一个地方管理
- **灵活扩展**：支持按名称、类型、自定义函数等多种过滤方式
- **场景隔离**：不同业务场景（缺陷、用例、接口）独立配置

### 1.3 重要说明

**创建页和编辑页共用同一组件**：
- 在 MeterSphere 中，创建缺陷和编辑缺陷使用的是同一个组件（`IssueEditDetail.vue`）
- 通过 `type` 参数区分：`'add'`（创建）、`'edit'`（编辑）、`'copy'`（复制）
- 字段过滤逻辑在组件初始化时执行，**会同时影响创建和编辑页面**
- 如果需要区分创建和编辑的显示规则，可以在 `customCheck` 函数中根据 `type` 判断

## 二、使用场景

### 2.1 典型场景

| 场景 | 说明 | 示例 |
|------|------|------|
| **只读统计字段** | 系统自动计算，用户不应手动编辑 | 复测次数、关联用例数 |
| **列表展示字段** | 仅用于列表展示，表单中不需要 | 某些汇总字段 |
| **权限控制字段** | 根据权限决定是否显示 | 管理员专用字段 |
| **条件显示字段** | 根据业务状态动态显示 | 已关闭缺陷的某些字段 |

### 2.2 应用范围

- **缺陷管理**：创建/编辑缺陷时过滤特定字段
- **用例管理**：创建/编辑用例时过滤特定字段
- **接口管理**：创建/编辑接口时过滤特定字段
- **其他业务模块**：任何需要字段显示控制的场景

## 三、配置方式

### 3.1 配置文件位置

**文件路径**：`test-track/frontend/src/business/utils/sdk-utils.js`

**配置对象**：`FIELD_DISPLAY_CONFIG`

### 3.2 配置结构

每个场景的配置包含三个维度：

```javascript
{
  scene: {
    excludeNames: [],      // 按字段名称过滤
    excludeTypes: [],      // 按字段类型过滤
    customCheck: null      // 自定义判断函数
  }
}
```

## 四、配置方法详解

### 4.1 按字段名称过滤（最常用）

**适用场景**：明确知道哪些字段不显示

**配置方式**：
- 在 `excludeNames` 数组中添加字段名称
- 支持多个字段名称

**示例场景**：
- 复测次数：系统自动累加，用户不应编辑
- 关联用例数：自动计算，无需手动输入
- 创建时间：系统自动记录

**注意事项**：
- 字段名称必须与模板中的字段名称完全一致
- 支持中英文字段名
- 建议使用常量定义字段名，避免拼写错误

---

### 4.2 按字段类型过滤

**适用场景**：某一类字段都不应在表单中显示

**配置方式**：
- 在 `excludeTypes` 数组中添加字段类型
- 支持多个字段类型

**示例场景**：
- 所有富文本字段：表单中不显示，仅在详情页显示
- 所有只读字段：系统字段且只读的都不显示
- 所有计算字段：类型为"计算"的字段都不显示

**常见字段类型**：
- `richText`：富文本
- `int`：整数
- `float`：浮点数
- `select`：下拉选择
- `multipleSelect`：多选
- `date`：日期
- `member`：成员选择

**注意事项**：
- 类型过滤会影响所有该类型的字段
- 如果只需要过滤部分类型字段，建议使用 `customCheck`

---

### 4.3 自定义判断函数（最灵活）

**适用场景**：需要复杂逻辑判断字段是否显示

**配置方式**：
- 在 `customCheck` 中定义函数
- 函数接收字段对象作为参数
- 返回 `false` 表示不显示，`true` 表示显示

**示例场景**：

**场景1：系统字段且只读的不显示**
```javascript
customCheck: (field) => {
  return !(field.system && field.readonly);
}
```

**场景2：特定前缀的字段不显示**
```javascript
customCheck: (field) => {
  return !field.name.startsWith('统计_');
}
```

**场景3：根据字段属性组合判断**
```javascript
customCheck: (field) => {
  // 系统字段 + 整数类型 + 只读 = 不显示
  if (field.system && field.type === 'int' && field.readonly) {
    return false;
  }
  return true;
}
```

**场景4：根据业务状态判断**
```javascript
customCheck: (field) => {
  // 如果缺陷已关闭，某些字段不显示
  if (this.form && this.form.status === 'closed') {
    return field.name !== '处理人';
  }
  return true;
}
```

**场景5：区分创建和编辑页面**
```javascript
customCheck: (field) => {
  // 注意：需要在组件中通过闭包传递 type
  // 创建页面不显示某些字段，编辑页面显示
  if (this.type === 'add' && field.name === '处理人') {
    return false;  // 创建时不显示处理人
  }
  return true;
}
```

**注意**：由于 `customCheck` 函数在配置对象中定义，无法直接访问组件的 `this.type`。如果需要区分创建和编辑，可以：
- 方式1：在组件中动态设置 `customCheck`，通过闭包传递 `type`
- 方式2：在字段对象中添加标记，通过 `customCheck` 判断字段属性
- 方式3：使用不同的场景配置（如 `issue_add`、`issue_edit`）

**注意事项**：
- 自定义函数中的 `this` 上下文可能受限，需要谨慎使用
- 复杂逻辑建议提取为独立函数
- 函数应保持纯函数特性，避免副作用

---

### 4.4 组合使用

**适用场景**：需要多种条件组合判断

**执行顺序**：
1. 先检查 `excludeNames`（字段名称）
2. 再检查 `excludeTypes`（字段类型）
3. 最后执行 `customCheck`（自定义函数）

**逻辑关系**：
- 任一条件匹配，字段即不显示
- 所有条件都不匹配，字段才显示

**示例**：
```javascript
{
  excludeNames: ['复测次数'],           // 明确排除的字段
  excludeTypes: ['richText'],          // 排除所有富文本
  customCheck: (field) => {            // 排除系统只读字段
    return !(field.system && field.readonly);
  }
}
```

## 五、在页面中使用

### 5.1 基本用法

**步骤**：
1. 导入 `filterFieldsForForm` 函数
2. 在初始化模板时调用函数过滤字段
3. 传入场景类型（如 'issue'、'testCase'）

**关键时机**：
- 模板初始化时（`init` 方法）
- 表单重置时（`resetFormForContinue` 方法）
- 模板切换时（`templateChange` 方法）

**注意事项**：
- 必须在模板克隆后立即过滤
- 需要同时过滤当前模板和模板快照
- 确保所有使用模板的地方都经过过滤
- **创建和编辑共用同一组件**：过滤逻辑会同时影响创建和编辑页面

### 5.3 区分创建和编辑页面

**场景**：某些字段在创建时不显示，但在编辑时需要显示

**实现方式**：

**方式1：动态设置 customCheck（推荐）**
```javascript
// 在组件中动态设置配置
init(template, data) {
  // 克隆模板
  this.issueTemplate = this.cloneIssueTemplate(template);
  
  // 根据 type 动态设置过滤规则
  const config = FIELD_DISPLAY_CONFIG.issue;
  if (this.type === 'add') {
    // 创建时额外过滤某些字段
    const originalCustomCheck = config.customCheck;
    config.customCheck = (field) => {
      // 先执行原有逻辑
      if (originalCustomCheck && !originalCustomCheck(field)) {
        return false;
      }
      // 创建时不显示处理人
      if (field.name === '处理人') {
        return false;
      }
      return true;
    };
  }
  
  // 过滤字段
  this.issueTemplate.customFields = filterFieldsForForm(
    this.issueTemplate.customFields, 
    'issue'
  );
}
```

**方式2：使用不同场景配置**
```javascript
// 在配置中定义两个场景
const FIELD_DISPLAY_CONFIG = {
  issue_add: {
    excludeNames: ['复测次数', '处理人'],  // 创建时不显示
    excludeTypes: [],
    customCheck: null
  },
  issue_edit: {
    excludeNames: ['复测次数'],  // 编辑时只过滤复测次数
    excludeTypes: [],
    customCheck: null
  }
};

// 在组件中根据 type 选择场景
const scene = this.type === 'add' ? 'issue_add' : 'issue_edit';
this.issueTemplate.customFields = filterFieldsForForm(
  this.issueTemplate.customFields, 
  scene
);
```

### 5.2 扩展新场景

**步骤**：
1. 在 `FIELD_DISPLAY_CONFIG` 中添加新场景配置
2. 在页面组件中调用 `filterFieldsForForm` 时传入新场景名
3. 根据业务需求配置过滤规则

**示例**：
- 用例管理场景：添加 `testCase` 配置
- 接口管理场景：添加 `api` 配置
- 测试计划场景：添加 `testPlan` 配置

## 六、最佳实践

### 6.1 配置原则

**单一职责**：
- 每个场景的配置只关注该场景的字段显示规则
- 避免跨场景的配置污染

**可读性优先**：
- 字段名称使用常量定义，避免硬编码
- 复杂逻辑提取为独立函数并添加注释

**向后兼容**：
- 新增配置不影响已有功能
- 默认行为是显示所有字段

---

### 6.2 命名规范

**场景命名**：
- 使用小写字母和下划线
- 与业务模块名称保持一致
- 示例：`issue`、`test_case`、`api_definition`

**字段名称**：
- 建议使用常量定义
- 与后端 `SystemCustomField` 常量保持一致
- 示例：`'复测次数'` 对应 `SystemCustomField.ISSUE_RETEST_COUNT`

---

### 6.3 性能考虑

**过滤时机**：
- 在模板初始化时过滤，避免重复过滤
- 不要在渲染过程中频繁调用过滤函数

**缓存策略**：
- 过滤后的模板可以缓存
- 避免每次渲染都重新过滤

---

## 七、常见问题

### 7.1 字段仍然显示

**可能原因**：
- 字段名称拼写错误
- 场景类型不匹配
- 过滤函数未正确调用
- 模板克隆时机不对

**排查步骤**：
1. 检查字段名称是否与配置一致
2. 确认场景类型是否正确传入
3. 验证过滤函数是否被调用
4. 检查模板克隆和过滤的顺序

---

### 7.2 字段意外隐藏

**可能原因**：
- 字段类型被误过滤
- 自定义函数逻辑错误
- 配置过于宽泛

**排查步骤**：
1. 检查 `excludeTypes` 是否包含该字段类型
2. 验证 `customCheck` 函数逻辑
3. 临时移除配置，确认字段是否正常显示

---

### 7.3 国际化问题

**注意事项**：
- 字段名称使用原始名称（中文），不使用国际化 key
- 如果字段名称可能变化，建议使用字段 ID 或自定义函数

---

## 八、扩展指南

### 8.1 添加新场景

**步骤**：
1. 在 `FIELD_DISPLAY_CONFIG` 中添加新场景配置
2. 在对应页面组件中导入并使用 `filterFieldsForForm`
3. 根据业务需求配置过滤规则

**示例**：为用例管理添加场景
```javascript
// 1. 添加配置
const FIELD_DISPLAY_CONFIG = {
  issue: { /* ... */ },
  testCase: {
    excludeNames: ['用例状态'],  // 用例状态不在表单中显示
    excludeTypes: [],
    customCheck: null
  }
};

// 2. 在用例编辑页面使用
import {filterFieldsForForm} from "@/business/utils/sdk-utils";
// ...
this.testCaseTemplate.customFields = filterFieldsForForm(
  this.testCaseTemplate.customFields, 
  'testCase'
);
```

---

### 8.2 添加新的过滤维度

**扩展思路**：
- 在配置对象中添加新属性（如 `excludeIds`、`excludeScenes`）
- 在 `shouldDisplayFieldInForm` 函数中添加对应的判断逻辑
- 保持向后兼容，新属性为可选

**示例**：按字段 ID 过滤
```javascript
// 配置中添加
excludeIds: ['field-id-1', 'field-id-2'],

// 函数中添加判断
if (config.excludeIds && config.excludeIds.includes(field.id)) {
  return false;
}
```

---

### 8.3 动态配置

**场景**：需要根据运行时条件动态决定字段显示

**实现方式**：
- 在 `customCheck` 函数中访问组件状态
- 使用闭包传递上下文信息
- 或者将配置改为函数，返回配置对象

**注意事项**：
- 动态配置可能影响性能
- 需要处理配置变化时的重新过滤

## 九、总结

### 9.1 核心价值

- **集中管理**：所有字段显示规则统一配置
- **灵活扩展**：支持多种过滤方式和场景
- **易于维护**：修改配置即可，无需改动页面代码

### 9.2 适用场景

- 系统自动计算的字段不应在表单中显示
- 列表展示字段不需要在表单中编辑
- 根据权限或业务状态动态控制字段显示
- 需要统一管理多个页面的字段显示规则

### 9.3 使用建议

- **简单场景**：使用 `excludeNames` 按名称过滤
- **类型过滤**：使用 `excludeTypes` 批量过滤
- **复杂逻辑**：使用 `customCheck` 自定义函数
- **组合使用**：根据实际需求组合多种方式
