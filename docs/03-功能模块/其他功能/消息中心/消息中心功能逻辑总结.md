# 消息中心功能逻辑总结

## 一、功能概述

消息中心是 MeterSphere 的全局通知功能，位于页面右上角顶栏（铃铛图标），以抽屉（Drawer）形式展示。用户可以在此查看系统内的各类通知消息。

## 二、UI 交互逻辑

### 2.1 入口位置

- 位于全局顶栏右侧，组件路径：`HeaderRightMenus.vue` → `MsNotification`
- 顶栏右侧组件排列顺序：用户头像 → 语言切换 → 工作空间 → 任务中心 → **消息中心（铃铛）** → 帮助指引

### 2.2 铃铛图标状态

| 状态 | 表现 |
|------|------|
| 有未读消息 | 铃铛图标右上角显示红色圆点（`el-badge is-dot`） |
| 无未读消息 | 铃铛图标无标记 |

### 2.3 抽屉面板

- 点击铃铛图标打开右侧抽屉（550px 宽度）
- 打开时自动调用 `readAll()` 将所有消息标记为已读，并清除红点
- 包含两个 Tab 页：
  - **@提到我的**（`MENTIONED_ME`）：被 @ 提及的通知
  - **系统通知**（`SYSTEM_NOTICE`）：系统自动推送的通知
- 底部提示：**仅显示最近 3 个月的站内消息**

### 2.4 消息列表展示

每条消息包含：
- 左侧圆形头像（取用户名首字，紫色背景；评审类消息显示"系统"图标）
- 操作描述：`{用户名} {操作} {资源类型}: {资源名称}`
- 右下角显示创建时间
- 资源名称可点击跳转到对应页面

### 2.5 分页

- 每页 20 条
- 顶部显示总消息数和页码导航（下拉选页 + 左右翻页按钮）

## 三、实时推送机制（WebSocket）

### 3.1 连接建立

```
WebSocket 端点: /websocket/notification/count/{userId}
```

- 页面加载时（`created` 钩子）自动建立 WebSocket 连接
- 每个用户维护一个 `Timer`，每 **10 秒** 轮询一次未读消息数
- 支持同一用户多个 Session（多标签页）

### 3.2 推送内容

服务端每 10 秒推送一次 JSON 消息：

```json
{
  "count": 3,       // 未读消息数（最近1小时内的）
  "now": 1707123456000  // 服务器当前时间戳
}
```

### 3.3 前端收到推送后的行为

1. 更新 `noticeCount`（控制红点显示）
2. 如果有未读消息，调用 `showNotification()` 弹出桌面通知
3. 如果抽屉已打开，自动刷新两个 Tab 的数据

### 3.4 桌面弹窗通知

- 仅对 **10 秒内** 新产生的未读消息弹出 `$notify` 通知
- 弹出后自动调用 `read(id)` 标记该条为已读
- 评审类消息格式：`{资源类型}: {资源名} 包含脚本评审`
- 普通消息格式：`{用户名} {操作} {资源类型}: {资源名}`

## 四、后端架构

### 4.1 数据模型（notification 表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Long | 主键 |
| type | String | 消息类型：`MENTIONED_ME` / `SYSTEM_NOTICE` |
| receiver | String | 接收者用户 ID |
| title | String | 消息标题 |
| status | String | 状态：`READ` / `UNREAD` |
| create_time | Long | 创建时间（毫秒时间戳） |
| operator | String | 操作者用户 ID |
| operation | String | 操作类型（如 CREATE、UPDATE、DELETE、EXECUTE_SUCCESSFUL 等） |
| resource_id | String | 关联资源 ID |
| resource_type | String | 资源类型（如 TEST_PLAN_TASK、DEFECT_TASK 等） |
| resource_name | String | 资源名称 |
| content | String | 消息内容（JSON 格式的参数 Map） |

### 4.2 API 接口

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/notification/list/all/{goPage}/{pageSize}` | 分页查询通知列表（支持按 type 过滤） |
| GET | `/notification/read/{id}` | 标记单条为已读 |
| GET | `/notification/read/all` | 标记当前用户所有消息为已读 |
| POST | `/notification/count` | 统计未读消息数 |

### 4.3 查询逻辑

- `listNotification`：查询当前用户最近 **90 天**（3 个月）的消息，按创建时间倒序
- `countNotification`：统计最近 **1 小时** 内的未读消息数（用于 WebSocket 推送）
- 支持按 `type`（MENTIONED_ME / SYSTEM_NOTICE）过滤
- 支持按 `title` 模糊搜索

## 五、消息产生机制

### 5.1 站内通知发送器

核心类：`InSiteNoticeSender extends AbstractNoticeSender`

发送流程：
1. 业务操作触发事件（如创建缺陷、执行测试等）
2. 根据消息通知配置（`MessageDetail`）确定接收者
3. 构建 `Notification` 对象并写入数据库
4. WebSocket 定时器检测到新消息后推送给前端

### 5.2 接收者类型确定规则

| 配置的接收者 | 实际接收者 | 通知类型 |
|-------------|-----------|---------|
| `EXECUTOR`（执行人） | 从参数中取 executor / userIds | SYSTEM_NOTICE |
| `CREATOR`（创建人） | 从参数中取 creator / createUser / userId | SYSTEM_NOTICE |
| `MAINTAINER`（维护人） | 从参数中取 maintainer / userIds | SYSTEM_NOTICE |
| `FOLLOW_PEOPLE`（关注人） | 查询关注关系表 | SYSTEM_NOTICE |
| `PROCESSOR`（处理人） | 从参数中取 processor | SYSTEM_NOTICE |
| 具体用户 ID（被 @ 的人） | 直接使用该 ID | **MENTIONED_ME** |

- 发送时会排除操作者自己（`excludeSelf`）

### 5.3 触发场景

消息通知由各业务模块的事件触发，常见场景包括：
- 接口用例执行完成（`ApiTestCaseService`）
- 性能测试执行完成（`PerformanceTestService`）
- 环境审核（`BaseEnvironmentService`）
- 以及通过消息通知配置的所有事件（缺陷创建/更新、用例评审、测试计划执行等）

## 六、资源跳转映射

| REVIEW_TASK | `/track/review/view/{id}` |
| DEFECT_TASK | `/track/issue` |
| API_AUTOMATION_TASK | `/api/automation?resourceId={id}` |
| API_DEFINITION_TASK | `/api/definition?caseId={id}` 或 `?resourceId={id}` |
| PERFORMANCE_TEST_TASK | `/performance/test/edit/{id}` |
| TRACK_TEST_CASE_TASK | `/track/case/edit/{id}` |
| JENKINS_TASK | 根据 content 中的 reportUrl 动态判断 |
| SWAGGER_TASK | `/api/definition` |
| UI_AUTOMATION_TASK | `/ui/automation` |
| ENV_TASK | `/project/env?resourceId={id}` |

## 八、什么情况下会收到通知（消息设置）

消息通知是**按项目配置**的，在 `项目管理 → 更多选项 → 消息设置` 页面进行配置。只有配置了通知规则后，对应事件才会触发通知。

### 8.1 通知配置结构

每条通知规则包含 5 个要素：

| 要素 | 说明 |
|------|------|
| 事件 | 触发通知的操作类型（如创建、更新、删除等） |
| 接收人 | 谁收到通知（可选：创建人、执行人、维护人、关注人、处理人、或指定用户） |
| 接收方式 | 站内信、邮件、钉钉机器人、企业微信机器人、飞书、Webhook |
| Webhook | 钉钉/企业微信/飞书的 Webhook 地址 |
| 模板 | 通知内容模板（支持变量替换） |

### 8.2 可配置的通知事件（完整清单）

#### 一、测试跟踪任务通知

| 子模块 | 可配置事件 |
|--------|-----------|
| 首页 | 关闭定时任务 |
| 功能用例 | 创建、更新、删除、评论 |
| 用例评审 | 创建、更新、删除、评论、评审完成 |
| 测试计划 | 创建、更新、删除、执行完成、执行成功、执行失败 |
| 缺陷管理 | 创建、更新、删除、评论 |
| 测试报告 | 删除 |

#### 二、接口测试任务通知

| 子模块 | 可配置事件 |
|--------|-----------|
| 首页 | 关闭定时任务 |
| 接口定义 | API 创建、API 更新、API 删除、CASE 创建、CASE 更新、CASE 删除、MOCK 创建、MOCK 更新、MOCK 删除、CASE 执行成功、CASE 执行失败 |
| 接口自动化 | 创建、更新、删除、执行成功、执行失败 |
| 接口报告 | 删除 |

#### 三、性能测试任务通知

| 子模块 | 可配置事件 |
|--------|-----------|
| 性能测试 | 创建、更新、删除、执行完成 |
| 测试报告 | 删除 |

#### 四、Jenkins 任务通知

| 子模块 | 可配置事件 |
|--------|-----------|
| Jenkins 任务 | 执行成功、执行失败 |

### 8.3 接收方式说明

| 方式 | 标识 | 说明 |
|------|------|------|
| 站内信 | IN_SITE | 消息中心铃铛通知（本文档主要描述的功能） |
| 邮件 | EMAIL | 发送邮件通知 |
| 钉钉机器人 | NAIL_ROBOT | 通过钉钉 Webhook 推送 |
| 企业微信机器人 | WECHAT_ROBOT | 通过企业微信 Webhook 推送 |
| 飞书 | LARK | 通过飞书 Webhook 推送 |
| Webhook | WEBHOOK | 自定义 Webhook 推送 |

### 8.4 关键结论

- **默认不发送任何通知**：必须在消息设置中手动创建通知规则
- 通知规则是项目级别的，不同项目可以有不同的配置
- 一个事件可以配置多条规则（不同接收人、不同接收方式）
- 站内信（IN_SITE）是消息中心铃铛收到通知的前提条件
- 如果只配置了邮件/钉钉等方式，消息中心不会显示对应通知

## 九、数据流总结

```
用户操作（创建缺陷/执行测试/评审等）
    ↓
业务 Service 触发事件
    ↓
AbstractNoticeSender.getRealUserIds() 确定接收者 + 类型
    ↓
InSiteNoticeSender.sendAnnouncement() 写入 notification 表
    ↓
NotificationWebSocket（每10秒轮询）检测到未读消息
    ↓
WebSocket 推送 {count, now} 给前端
    ↓
前端收到推送 → 显示红点 + 弹出桌面通知
    ↓
用户点击铃铛 → 打开抽屉 → 调用 readAll() 标记已读 → 加载消息列表
    ↓
用户点击消息中的资源名 → 根据 resourceType 跳转到对应页面
```
