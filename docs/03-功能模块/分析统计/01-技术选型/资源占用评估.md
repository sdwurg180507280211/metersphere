# 分析统计模块 - 资源占用评估

## 功能定位

基于你的描述，analytics-stat 模块主要包含以下功能：

1. **综合查询** - 跨表、跨库的复杂查询
2. **SQL查询台** - 在线 SQL 执行和结果展示
3. **统计图表** - 数据可视化（ECharts、Chart.js 等）
4. **各种统计** - 报表生成、数据分析、趋势分析

### 对比分析

analytics-stat 的内存占用与 **report-stat** 相当，原因：

1. **相似的功能定位**
   - 都涉及数据查询和统计
   - 都需要处理大量数据
   - 都需要生成报表和图表

2. **内存占用因素**
   - 数据查询：需要缓存查询结果
   - 统计计算：需要临时存储中间结果

3. **优化空间**
   - 使用流式处理减少内存占用
   - 合理的分页策略
   - 及时释放大对象
   - 使用 Redis 缓存减轻 JVM 压力

## 内存优化建议

### 1. 查询优化

#### 分页查询
```java
// 避免一次性加载大量数据
@GetMapping("/query")
public PageResult<Map<String, Object>> query(
    @RequestParam String sql,
    @RequestParam(defaultValue = "1") int page,
    @RequestParam(defaultValue = "100") int size
) {
    // 使用 PageHelper 分页
    PageHelper.startPage(page, size);
    List<Map<String, Object>> results = sqlService.executeQuery(sql);
    return PageResult.of(results);
}
```

#### 流式处理
```java
// 大数据量导出使用流式处理
@GetMapping("/export")
public void export(HttpServletResponse response) {
    response.setContentType("application/vnd.ms-excel");
    
    try (OutputStream out = response.getOutputStream()) {
        // 使用 EasyExcel 流式写入，不占用大量内存
        EasyExcel.write(out, DataDTO.class)
            .sheet("数据")
            .doWrite(() -> {
                // 分批查询，每次 1000 条
                return dataService.queryBatch(1000);
            });
    }
}
```

### 2. 缓存优化

#### 使用 Caffeine 本地缓存
```java
@Configuration
public class CacheConfig {
    @Bean
    public Cache<String, Object> queryCache() {
        return Caffeine.newBuilder()
            .maximumSize(1000)              // 最多缓存 1000 个查询
            .expireAfterWrite(10, TimeUnit.MINUTES)  // 10分钟过期
            .recordStats()                  // 记录统计信息
            .build();
    }
}
```

#### 使用 Redis 缓存大对象
```java
@Service
public class StatisticsService {

    public StatisticsResult getStatistics(String type, LocalDate date) {
        // 复杂统计计算

    }
}
```

### 3. 对象池化

#### 使用对象池减少 GC 压力
```java
// 使用 Apache Commons Pool
public class QueryResultPool extends BasePooledObjectFactory<QueryResult> {
    @Override
    public QueryResult create() {
        return new QueryResult();
    }
    
    @Override
    public PooledObject<QueryResult> wrap(QueryResult obj) {
        return new DefaultPooledObject<>(obj);
    }
    
    @Override
    public void passivateObject(PooledObject<QueryResult> p) {
        p.getObject().clear();  // 清空数据，准备复用
    }
}
```

### 4. GC 调优

#### 推荐 G1 GC 配置
```bash
java -Xms2048m \
     -Xmx4096m \
     -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=200 \
     -XX:G1HeapRegionSize=16m \
     -XX:InitiatingHeapOccupancyPercent=45 \
     -XX:+ParallelRefProcEnabled \
     -XX:+HeapDumpOnOutOfMemoryError \
     -XX:HeapDumpPath=/opt/metersphere/logs/analytics-stat/ \
     -jar analytics-stat.jar
```

**参数说明**：
- `MaxGCPauseMillis=200`: 最大 GC 停顿时间 200ms
- `G1HeapRegionSize=16m`: G1 区域大小 16MB
- `InitiatingHeapOccupancyPercent=45`: 堆占用 45% 时触发并发标记
- `HeapDumpOnOutOfMemoryError`: OOM 时自动 dump 堆

## 总结

### 内存占用预估

| 场景 | 堆内存配置 | 容器内存限制 | 说明 |
|------|-----------|-------------|------|
| **开发环境** | 512m-1536m | 2GB | 单用户测试 |
| **测试环境** | 1024m-2560m | 4GB | 10-20 并发 |
| **生产环境（中）** | 2048m-4096m | 6GB | 50-100 并发 |
| **生产环境（高）** | 3072m-6144m | 8GB | 100+ 并发 |

### 关键结论

1. **典型内存占用**: **2-4 GB**（生产环境）
2. **与 report-stat 相当**: 功能相似，内存占用相近
3. **优化空间大**: 通过分页、流式处理、缓存优化可以显著降低内存占用
4. **成本可控**: 单实例月成本 ¥400-800（云服务器）

### 优化建议优先级

1. ⭐⭐⭐ **分页查询** - 立即实施，效果显著
2. ⭐⭐⭐ **流式处理** - 大数据导出必须使用
3. ⭐⭐ **Redis 缓存** - 减轻 JVM 压力
4. ⭐⭐ **GC 调优** - 提升性能和稳定性
5. ⭐ **对象池化** - 高并发场景考虑

---

**评估结论**: analytics-stat 模块内存占用在 **2-4 GB** 之间，与现有的 report-stat 模块相当，属于**中等偏上**的资源占用水平，通过合理的优化策略可以有效控制内存使用。

**推荐配置**: 生产环境使用 **8GB 内存、4核 CPU** 的服务器，可以支撑 **50-100 并发用户**的正常使用。
