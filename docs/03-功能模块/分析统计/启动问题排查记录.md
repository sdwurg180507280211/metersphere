# 分析统计微服务启动问题排查记录

## 问题描述

analytics-stat微服务启动失败，报错：
```
Caused by: java.lang.IllegalArgumentException: Realms collection argument cannot be empty
```

## 问题分析

### 1. 初步排查

虽然在`@SpringBootApplication`注解中排除了`ShiroConfig`和`ShiroAutoConfiguration`：

```java
@SpringBootApplication(exclude = {
        QuartzAutoConfiguration.class,
        LdapAutoConfiguration.class,
        Neo4jAutoConfiguration.class,
        ShiroConfig.class,
        ShiroAutoConfiguration.class
})
```

但Shiro的其他自动配置类仍在加载：
- `ShiroWebAutoConfiguration`
- `ShiroBeanAutoConfiguration`
- `ShiroWebFilterConfiguration`
- `ShiroAnnotationProcessorAutoConfiguration`

### 2. 根本原因

SDK依赖中包含了两个Shiro Spring Boot Starter：
1. `shiro-spring-boot-web-starter`
2. `shiro-spring-boot-starter`

这些starter会自动触发Shiro的自动配置，即使在Application类中排除了部分配置类，Spring Boot的自动配置机制仍会加载其他Shiro相关的配置类。

### 3. 为什么需要排除Shiro

在微服务架构中：
- **Gateway**：负责统一认证和路由
- **system-setting**：负责用户管理和权限管理
- **业务微服务**（如analytics-stat、workflow-service）：只需要处理业务逻辑，不需要独立的认证体系

如果每个微服务都引入Shiro，会导致：
1. 资源浪费（每个服务都要初始化Shiro）
2. 架构混乱（认证逻辑分散）
3. 维护困难（多处配置需要同步）

### 4. 为什么project-management能正常启动

project-management模块能正常启动是因为：
- pom.xml中**没有排除Shiro依赖**
- 启动类中**也没有排除Shiro的自动配置**
- 它**需要Shiro**来做认证和权限控制

而analytics-stat和workflow-service是新建的微服务，我们希望它们不包含认证逻辑，所以需要排除Shiro。

## 解决方案

### 方案一：在pom.xml中排除Shiro依赖（推荐）

在`analytics-stat/backend/pom.xml`中排除SDK带来的Shiro依赖：

```xml
<dependency>
    <groupId>io.metersphere</groupId>
    <artifactId>sdk</artifactId>
    <version>${revision}</version>
    <exclusions>
        <!-- 排除Shiro依赖：分析统计微服务不需要独立的认证体系，认证由Gateway统一处理 -->
        <exclusion>
            <groupId>org.apache.shiro</groupId>
            <artifactId>shiro-spring-boot-web-starter</artifactId>
        </exclusion>
        <exclusion>
            <groupId>org.apache.shiro</groupId>
            <artifactId>shiro-spring-boot-starter</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

同样在`workflow-service/backend/pom.xml`中添加相同的排除配置。

### 方案二：保留Shiro依赖（不推荐）

如果不排除Shiro依赖，需要：
1. 移除Application类中的`exclude`配置
2. 配置Shiro的Realm和SecurityManager
3. 但这违背了微服务架构的设计原则

## 参考案例

Gateway模块也采用了相同的方式排除Shiro依赖（参见`framework/gateway/pom.xml`）。

## 总结

**关键点**：
1. 在Application类中排除配置类（`exclude`）只是部分解决方案
2. **必须在pom.xml中排除Shiro的starter依赖才能彻底解决问题**
3. 微服务架构中，业务服务不应该包含认证框架，认证应该由Gateway统一处理
4. 如果业务服务需要认证功能（如project-management），则保留Shiro依赖并正常配置
