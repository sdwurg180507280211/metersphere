# 用户组权限过滤功能 - 测试指南

## 功能说明

在缺陷管理页面实现用户组权限过滤：
- **开发人员组（developer）**：只能看到处理人是自己的缺陷
- **测试人员组（tester）**：只能看到创建人是自己的缺陷
- **其他用户组**：能看到所有缺陷

## 前提条件

1. 当前项目绑定的缺陷模板中只绑定了一个"处理人"字段
2. 用户已分配到对应的用户组（developer 或 tester）

## 测试场景

### 场景1：开发人员组权限过滤

**测试步骤**：
1. 使用开发人员组账号登录
2. 进入缺陷管理页面
3. 观察列表数据

**预期结果**：
- 只显示"处理人"是当前用户的缺陷
- 翻页后仍保持过滤条件
- 页面切换后返回，重新施加过滤

### 场景2：测试人员组权限过滤

**测试步骤**：
1. 使用测试人员组账号登录
2. 进入缺陷管理页面
3. 观察列表数据

**预期结果**：
- 只显示"创建人"是当前用户的缺陷
- 翻页后仍保持过滤条件
- 页面切换后返回，重新施加过滤

### 场景3：高级搜索清除权限过滤

**测试步骤**：
1. 使用开发人员或测试人员账号登录
2. 进入缺陷管理页面（此时应施加权限过滤）
3. 点击"搜索"或"重置"按钮
4. 观察列表数据

**预期结果**：
- 搜索/重置后能看到所有符合条件的缺陷
- 不再受权限过滤限制

### 场景4：其他用户组无权限过滤

**测试步骤**：
1. 使用非 developer/tester 用户组账号登录
2. 进入缺陷管理页面
3. 观察列表数据

**预期结果**：
- 显示所有缺陷，无权限过滤

### 场景5：翻页保持权限过滤

**测试步骤**：
1. 使用开发人员或测试人员账号登录
2. 进入缺陷管理页面（此时应施加权限过滤）
3. 翻页到第2页、第3页
4. 观察列表数据

**预期结果**：
- 每一页都只显示符合权限过滤的缺陷
- 翻页不会清除权限过滤条件

### 场景6：页面切换后重新施加过滤

**测试步骤**：
1. 使用开发人员或测试人员账号登录
2. 进入缺陷管理页面（此时应施加权限过滤）
3. 切换到其他页面（如测试用例）
4. 返回缺陷管理页面
5. 观察列表数据

**预期结果**：
- 返回后重新施加权限过滤
- 只显示符合权限的缺陷

## 技术验证点

### 1. 网络请求验证

打开浏览器开发者工具 → Network 标签：

**初始加载时的请求顺序**：
1. `GET /issues/user/group/{projectId}/{userId}` - 获取用户组
2. `GET /user/project/member/list` - 获取项目成员
3. `GET /issues/template/project/{projectId}` - 获取缺陷模板
4. `POST /issues/list/{page}/{pageSize}` - 获取缺陷列表

**验证 filters 参数**：
- 开发人员组：`filters: { "custom_single-{fieldId}": ["userId"] }`
- 测试人员组：`filters: { "creator": ["userId"] }`

### 2. 前端状态验证

在浏览器控制台执行：

```javascript
// 查看当前用户组
this.currentUserGroupId

// 查看过滤条件
this.page.condition.filters

// 查看记录的过滤条件key
this.userGroupFilterKeys
```

### 3. 性能验证

- 权限过滤逻辑执行时间 < 1ms（纯内存操作）
- 只在页面激活时执行1次
- 不影响列表渲染性能

## 常见问题

### Q1: 开发人员看不到任何缺陷？

**原因**：当前项目没有处理人是该用户的缺陷

**解决**：创建一个缺陷并设置处理人为当前用户

### Q2: 搜索后仍然只能看到自己的缺陷？

**原因**：可能是缓存问题或代码未正确清除过滤条件

**解决**：
1. 刷新页面
2. 检查 `clearUserGroupFilter()` 方法是否被正确调用
3. 检查浏览器控制台是否有错误

### Q3: 翻页后权限过滤失效？

**原因**：`page.condition.filters` 被清空

**解决**：
1. 检查 `handlePageChange()` 方法是否设置了 `pageRefresh = true`
2. 检查 MsTable 组件是否正确处理 `pageRefresh` 属性

### Q4: 页面切换后权限过滤失效？

**原因**：`activated()` 钩子未正确执行

**解决**：
1. 检查路由是否使用了 `<keep-alive>`
2. 检查 `activated()` 中是否重置了状态并重新施加过滤

## 回滚方案

如果需要回滚此功能：

1. 删除 `test-track/frontend/src/api/user-group.js`
2. 删除 `IssuesController.getUserGroupInProject()` 方法
3. 删除 `IssuesService.getUserGroupInProject()` 方法
4. 删除 `ExtIssuesMapper.getUserGroupInProject()` 方法和SQL
5. 在 `IssueList.vue` 中删除：
   - `currentUserGroupId` 数据属性
   - `userGroupFilterKeys` 数据属性
   - `applyUserGroupFilter()` 方法
   - `clearUserGroupFilter()` 方法
   - `activated()` 中的用户组获取和过滤逻辑
