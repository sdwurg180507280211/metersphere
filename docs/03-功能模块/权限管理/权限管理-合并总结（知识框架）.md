## 1. 一句话定义：权限管理到底能做什么

**权限管理 = 给“用户组(角色)”分配“权限点(能力)”，并把该能力绑定到“资源范围(system/workspace/project)”，从而控制用户是否能进入某功能、是否能调用某接口、是否能对某资源执行某类操作。**

---

## 2. 权限管理的四件事（从使用角度）

### 2.1 控制“看得见”

- **菜单/模块可见**：用户是否能进入某模块（如缺陷管理、用例、系统设置）
- **页面入口可见**：某些 Tab、某些路由入口是否可见

> 本质：前端“入口收敛”（体验层），后端仍需兜底。

### 2.2 控制“点得动 / 做得了”

- **操作级权限**：READ / CREATE / EDIT / DELETE / EXPORT / IMPORT / RUN / BATCH 等
- 典型权限点来自：`PermissionConstants.java`
  - 例如：`PROJECT_TRACK_ISSUE:READ+EDIT`

> 本质：后端接口拦截（安全边界）+ 前端按钮/控件禁用（体验）。

### 2.3 控制“在哪个范围内有效”（资源隔离）

- 同一个用户在不同项目里可以是不同角色
  - 项目 A：`project_admin`
  - 项目 B：`project_member`
- 关键在于：`user_group.source_id`
  - `system`：系统级
  - `workspaceId`：工作空间级
  - `projectId`：项目级

> 关键是“同一用户在不同资源下可能属于不同用户组”。

### 2.4 支持“可配置化二开”（新增权限点）

- 当现有粒度不够细：
  - **推荐**：新增权限点（让 UI 可勾选/取消勾选）
  - **兜底**：Service 层增加业务规则拦截（硬规则/半硬规则）

---

## 3. 权限管理不能直接做什么（边界/误区）

### 3.1 不能天然做到“字段级 + 页面级”可见性

- “字段绑定到缺陷模板”通常是**模板级**：
  - 字段一旦绑定，新增/编辑都会有（列表是否展示可能另有列设置）
- 模板绑定阶段一般不能选择：只在新增页/只在编辑页/只在列表页

> 要做页面级字段控制，通常走：页面渲染策略/列表列设置/二开改造，而不是权限系统原生配置。

### 3.2 复杂业务规则不是权限系统的天然表达

示例：
- 某状态只能某角色修改
- 某字段在某状态只读

通常需要：
- 权限点拆分（可配置，推荐）
- 或 Service 层规则校验（兜底）
- 或工作流/状态机机制（如果模块支持）

## 4. 运转机制（从 UI 到 DB 到鉴权）

### 4.1 UI：系统设置 -> 用户组与权限

你在页面里做的两类配置：
- **给用户组配置权限点**：本质写入 `user_group_permission`
- **给用户分配用户组（并绑定资源）**：本质写入 `user_group`（重点是 `source_id`）

### 4.2 前端：负责体验

- 控制菜单/按钮/入口的显示与禁用
- 目的：减少无权限用户误点

> 重要：前端不是安全边界。

### 4.3 后端：安全边界（必须兜底）

- **Controller 层**：`@RequiresPermissions(PermissionConstants.xxx)`
- **Service 层**：防御性校验（防绕过/防未来新入口遗漏）
- **资源归属**：`@CheckOwner(...)` 或基于可访问资源集合校验

---

## 5. 资源隔离（Scope）与数据模型（落地到表）

### 5.1 三层资源作用域

- **System（系统级）**：`source_id = 'system'`
- **Workspace（工作空间级）**：`source_id = workspaceId`
- **Project（项目级）**：`source_id = projectId`

### 5.2 权限的核心公式（项目内约定）

**权限是否成立 =（用户属于哪些用户组）+（用户组包含哪些权限点）+（这些关系绑定在哪个资源）**

### 5.3 常见表（用于理解落库）

- `group`
  - 用户组（角色）定义
  - `id`：如 `project_admin` / `ws_admin` / `super_group`
  - `type`：SYSTEM / WORKSPACE / PROJECT（决定该组可绑定的资源层级）
- `user_group`
  - 用户属于某用户组，并绑定资源范围
  - `user_id` + `group_id` + `source_id`
- `user_group_permission`
  - 用户组拥有哪些权限点（能力）

---

## 6. 后端鉴权最佳实践（避免踩坑）

### 6.1 系统级操作：优先判断超级管理员组

- 系统级高危操作（例如“创建全局系统字段”）建议使用：判断 `SessionUser.groups` 是否包含 `UserGroupConstants.SUPER_GROUP`
- 不要依赖不存在的 `SessionUtils.isAdmin()` 之类方法

### 6.2 业务级操作：权限点 + 资源维度

- 推荐统一口径：`SessionUtils.hasPermission(workspaceId, projectId, permission)`
- Controller 层声明式拦截：`@RequiresPermissions(PermissionConstants.xxx)`
- Service 层做兜底校验：避免未来新增入口绕过 Controller

### 6.3 资源归属：必须与权限点一起考虑

- 仅有权限点但无资源范围，等同于“没法落地到多租户隔离”
- 常见实现：`@CheckOwner(...)` / BaseCheckPermissionService / “可访问资源集合”校验

## 6. 操作日志权限控制（实战案例）

### 6.1 操作日志权限架构

MeterSphere 操作日志采用**分层权限控制**机制，确保不同用户组只能访问其权限范围内的日志数据。

#### **权限定义**
```java
// PermissionConstants.java
public static final String SYSTEM_OPERATING_LOG_READ = "SYSTEM_OPERATING_LOG:READ";
public static final String WORKSPACE_OPERATING_LOG_READ = "WORKSPACE_OPERATING_LOG:READ";
```

#### **权限检查**
```java
// OperatingLogController.java
@PostMapping("/list/{goPage}/{pageSize}")
@RequiresPermissions(value = {
    PermissionConstants.WORKSPACE_OPERATING_LOG_READ, 
    PermissionConstants.SYSTEM_OPERATING_LOG_READ
}, logical = Logical.OR)  // OR 逻辑：满足任一权限即可
public Pager<List<OperatingLogDTO>> list(...) {
    return PageUtils.setPageInfo(page, operatingLogService.list(request));
}
```

### 6.2 用户组权限映射

#### **系统级用户组**（`type = 'SYSTEM'`）

| 用户组ID | 用户组名称 | 权限范围 | 操作日志权限 |
|---------|-----------|---------|-------------|
| `super_group` | 超级管理员 | 全系统 | ✅ `SYSTEM_OPERATING_LOG:READ` |
| `admin` | 系统管理员 | 全系统 | ✅ `SYSTEM_OPERATING_LOG:READ` |

#### **工作空间级用户组**（`type = 'WORKSPACE'`）

| 用户组ID | 用户组名称 | 权限范围 | 操作日志权限 |
|---------|-----------|---------|-------------|
| `ws_admin` | 工作空间管理员 | 指定工作空间 | ✅ `WORKSPACE_OPERATING_LOG:READ` |
| `ws_member` | 工作空间成员 | 指定工作空间 | ❌ 无操作日志权限 |

#### **项目级用户组**（`type = 'PROJECT'`）

| 用户组ID | 用户组名称 | 权限范围 | 操作日志权限 |
|---------|-----------|---------|-------------|
| `project_admin` | 项目管理员 | 指定项目 | ⚠️ 默认无权限（可手动配置） |
| `project_member` | 项目成员 | 指定项目 | ⚠️ 默认无权限（可手动配置） |

#### **权限配置说明**

**重要**：项目级用户组**可以手动配置**操作日志权限，并非绝对禁止：

##### **配置方式**
```sql
-- 给项目管理员添加工作空间操作日志权限
INSERT INTO user_group_permission (id, group_id, permission_id, module_id)
VALUES (UUID(), 'project_admin', 'WORKSPACE_OPERATING_LOG:READ', 'WORKSPACE');

-- 给项目成员添加工作空间操作日志权限  
INSERT INTO user_group_permission (id, group_id, permission_id, module_id)
VALUES (UUID(), 'project_member', 'WORKSPACE_OPERATING_LOG:READ', 'WORKSPACE');
```

##### **配置后的权限效果**
- **项目管理员 + 操作日志权限**：可查看所在项目及其工作空间的操作日志
- **项目成员 + 操作日志权限**：可查看参与项目及其工作空间的操作日志
- **数据过滤依然生效**：只能访问权限范围内的资源，不会越权

##### **推荐配置策略**
1. **工作空间管理员**：默认拥有 `WORKSPACE_OPERATING_LOG:READ`
2. **项目管理员**：建议配置 `WORKSPACE_OPERATING_LOG:READ`（便于项目管理）
3. **项目成员**：根据需要配置（一般不建议，避免信息泄露）

### 6.3 数据过滤机制

#### **SQL 层面的权限过滤**
```xml
<!-- BaseOperatingLogMapper.xml -->
<select id="list" resultType="io.metersphere.log.vo.OperatingLogDTO">
    SELECT t.id, t.project_id, t.oper_user, ...
    FROM operating_log t
    LEFT JOIN project t2 ON t.project_id = t2.id
    <where>
        <!-- 工作空间权限过滤 -->
        <if test="request.workspaceId != null and request.workspaceId !=''">
            AND t2.workspace_id = #{request.workspaceId}
        </if>
        <!-- 项目权限过滤 -->
        <if test="request.projectIds != null and request.projectIds.size > 0 ">
            t.project_id IN (#{projectId})
        </if>
    </where>
</select>
```

#### **服务层的权限控制**
```java
// OperatingLogService.java
public List<OperatingLogDTO> list(OperatingLogRequest request) {
    // 根据权限级别过滤模块
    if (StringUtils.equals(level, OperatorLevel.WORKSPACE)) {
        modules.addAll(OperLogModule.WORKSPACE_MODULES);
    } else if (StringUtils.equals(level, OperatorLevel.SYSTEM)) {
        modules.addAll(OperLogModule.SYSTEM_MODULES);
        modules.addAll(OperLogModule.WORKSPACE_MODULES);
    }
    request.setLevelModules(modules);
    
    // 查询数据
    List<OperatingLogDTO> list = baseOperatingLogMapper.list(request);
}
```

### 6.4 实际权限效果

#### **超级管理员**（拥有 `SYSTEM_OPERATING_LOG:READ`）
- ✅ 查看所有工作空间的操作日志
- ✅ 查看系统级模块的操作日志
- ✅ 查看工作空间级模块的操作日志

#### **工作空间管理员**（拥有 `WORKSPACE_OPERATING_LOG:READ`）
- ✅ 只查看当前工作空间的操作日志
- ✅ 查看工作空间级模块的操作日志
- ❌ 不能查看其他工作空间的操作日志
- ❌ 不能查看系统级模块的操作日志

#### **普通用户**（无相关权限）
- ❌ 无法访问操作日志页面
- ❌ API 调用会返回 403 权限错误

### 6.5 权限分配示例

#### **数据库配置**
```sql
-- 系统管理员权限
INSERT INTO user_group_permission (id, group_id, permission_id, module_id)
VALUES (UUID(), 'admin', 'SYSTEM_OPERATING_LOG:READ', 'SYSTEM');

-- 工作空间管理员权限  
INSERT INTO user_group_permission (id, group_id, permission_id, module_id)
VALUES (UUID(), 'ws_admin', 'WORKSPACE_OPERATING_LOG:READ', 'WORKSPACE');

-- 用户组分配
INSERT INTO user_group (id, user_id, group_id, source_id, create_time, update_time)
VALUES (UUID(), 'user_id', 'admin', 'system', unix_timestamp() * 1000, unix_timestamp() * 1000);
```

#### **权限验证查询**
```sql
-- 查看用户组权限分配
SELECT 
    u.name as user_name,
    g.name as group_name,
    g.type as group_type,
    ugp.permission_id,
    ugp.module_id
FROM user u
JOIN user_group ug ON u.id = ug.user_id
JOIN `group` g ON ug.group_id = g.id
LEFT JOIN user_group_permission ugp ON g.id = ugp.group_id
WHERE u.id = '用户ID';
```

### 6.6 最佳实践

#### **权限设计原则**
1. **接口层权限检查**：使用 `@RequiresPermissions` 注解
2. **数据层权限过滤**：根据工作空间、项目 ID 过滤数据
3. **业务层权限控制**：根据用户权限级别过滤模块
4. **OR 逻辑权限**：满足任一权限即可访问

#### **开发注意事项**
- 权限检查采用 `Logical.OR`，满足任一权限即可
- 数据过滤在 SQL 层面实现，确保数据安全
- 不同用户组的权限范围严格区分，避免权限泄露
- 权限配置需要与用户组类型匹配（SYSTEM/WORKSPACE/PROJECT）

## 7. 二次开发最短路径（简明步骤）

### 5.1 当你觉得“权限粒度不够”

优先级建议：
1. **新增权限点（推荐）**
   - 权限点可配置、可审计、可运营
2. **Service 层规则拦截（兜底）**
   - 改动小、见效快，但偏硬规则，建议后续演进为权限点

### 5.2 新增权限点的标准流程

1. 在 `PermissionConstants.java` 增加常量
2. 找到需要保护的后端接口：加 `@RequiresPermissions`
3. Service 层再补防御性校验（建议）
4. 让“用户组与权限”页面能配置到该权限点（权限树/权限列表来源）
5. 前端按该权限点控制按钮/入口
6. 用两类用户做验证：
   - UI 不可见/不可点
   - 绕过 UI 调接口也必须失败

## 8. 验证方法（建议固定成回归步骤）

- **准备测试用户**：建议使用 `创建权限层级用户.sql` 构造不同层级用户
- **验证维度**：
  - UI：菜单/按钮是否可见、是否可点
  - 接口：绕过 UI 直接调用接口也必须失败
  - 资源隔离：同一用户在不同项目/工作空间下的权限差异是否符合预期

### 8.1 造数脚本（创建测试用户）注意事项

- **用户能否登录的关键字段**：`user.source` 必须是 `LOCAL`
  - 常见问题：新建用户 `source = NULL` 会导致登录校验失败
- **默认密码与加密**：脚本使用 `MD5('metersphere')` 作为默认密码（以脚本为准）
- **系统级能力与 super_group**：
  - 例如“创建全局系统字段”这类系统级能力通常要求用户属于 `super_group`
  - `admin`（系统管理员）与 `super_group`（超级管理员）不是一回事，前者不一定具备后者的系统级能力

### 8.2 常见踩坑（开发/二开时高频出现）

- **不要用不存在的方法**：`SessionUtils` 中没有 `isAdmin()` / `isSystemAdmin()` 之类方法
- **系统级判断建议**：用 `SessionUtils.getUser().getGroups()` 判断是否包含 `UserGroupConstants.SUPER_GROUP`
- **接口鉴权不等于资源隔离**：有 `@RequiresPermissions` 但没做资源归属校验，仍可能出现“跨项目/跨空间”访问漏洞

---

## 7. 最终结论（拿来就用）

- 权限管理最擅长：
  - **模块/菜单/操作入口控制**（看得见、点得动）
  - **操作能力控制**（CRUD、导入导出、运行、批量）
  - **资源范围控制**（system/workspace/project）
- 权限管理不擅长（需二开/其他机制补齐）：
  - **字段级 + 页面级展示控制**
  - **复杂业务规则**（状态机/字段只读等）
