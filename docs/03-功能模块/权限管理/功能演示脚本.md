## 演示要点

### 1. 强调纯前端实现

- **无需修改后端SQL**：所有过滤逻辑都在前端完成
- **复用现有机制**：利用高级搜索的 `filters` 参数
- **改动面小**：只修改了 `IssueList.vue` 一个文件的逻辑

### 2. 强调性能优化

- **懒加载**：只在页面激活时执行1次
- **纯内存操作**：权限过滤逻辑 < 1ms
- **无额外请求**：翻页时不重复获取用户组

### 3. 强调用户体验

- **透明过滤**：用户无感知，自动施加权限过滤
- **搜索自由**：搜索时能看到所有符合条件的缺陷
- **翻页保持**：翻页时自动保持权限过滤

### 4. 强调可维护性

- **代码清晰**：详细的中文注释
- **边界清晰**：集中在少数文件
- **可回滚**：升级时能快速对照、快速搬运

## 常见问题解答

### Q1: 为什么不在后端实现权限过滤？

**A**: 我们评估了多种方案，最终选择纯前端实现的原因是：
1. **改动面小**：只修改前端一个文件，不需要修改后端SQL
2. **复用现有机制**：利用高级搜索的 `filters` 参数，不需要新增后端逻辑
3. **易于维护**：升级时能快速对照、快速搬运
4. **性能无影响**：纯内存操作，执行时间 < 1ms

### Q2: 如果用户不属于任何用户组会怎样？

**A**: 如果用户不属于 developer 或 tester 用户组，不会施加权限过滤，用户可以看到所有缺陷。

### Q3: 搜索时为什么要清除权限过滤？

**A**: 因为用户进行搜索/筛选时，应该能看到所有符合条件的缺陷，而不是只看到自己的缺陷。这样才能满足实际业务需求。

### Q4: 翻页时如何保持权限过滤？

**A**: 翻页时不清除 `page.condition.filters`，后端会自动应用这些过滤条件。我们通过设置 `pageRefresh = true` 来告诉 MsTable 组件不要清空跨页勾选。

### Q5: 如果项目有多个"处理人"字段怎么办？

**A**: 当前实现假设项目绑定的缺陷模板中只绑定了一个"处理人"字段。如果有多个，需要修改代码逻辑，选择正确的字段。
