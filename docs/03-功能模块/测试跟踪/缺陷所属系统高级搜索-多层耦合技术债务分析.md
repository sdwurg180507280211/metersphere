## 一、问题现象

### 1.1 用户反馈
- 缺陷管理高级搜索中，**单选所属系统**字段筛选返回 0 条数据
- 手动创建的缺陷可以正常筛选
- Excel 导入的缺陷无法筛选

### 1.2 影响范围
- 217 条缺陷记录的所属系统字段无法被高级搜索匹配
- 占该字段总记录数的 **95%+**

## 二、技术债务根因分析

### 2.1 问题本质：三重不匹配

这是一个典型的**多层耦合技术债务**，涉及前端、后端、数据库三层，任何一层的不匹配都会导致功能失效。

```
┌─────────────────────────────────────────────────────────────────┐
│                        问题链路图                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  [前端] generateColumnKey()                                     │
│     │                                                           │
│     │ associatedSystem → "custom-{id}" ❌                       │
│     │ (应该是 "custom_single-{id}")                             │
│     ▼                                                           │
│  [后端] ExtIssuesMapper.xml                                     │
│     │                                                           │
│     │ 只处理 custom_single- 和 custom_multiple-                 │
│     │ 不处理 custom- 前缀 ❌                                    │
│     ▼                                                           │
│  [数据库] custom_field_issues.value                             │
│     │                                                           │
│     │ Excel导入存储: "瑞博士智能陪练平台" (名称) ❌              │
│     │ 手动创建存储: "0b2f1fd0-..." (ID) ✅                      │
│     │ 高级搜索传递: "0b2f1fd0-..." (ID)                         │
│     ▼                                                           │
│  [结果] 名称 ≠ ID → 永远匹配不上 → 返回 0 条                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 第一层：前端 Key 生成不匹配

**文件位置**：`framework/sdk-parent/frontend/src/components/search/custom-component.js`

**问题代码**（修复前）：
```javascript
export function generateColumnKey(field) {
  if (field.type === 'select' || field.type === 'radio' || field.type === 'member') {
    return 'custom_single-' + field.id;  // ✅ 被 SQL 处理
  } else if (field.type === 'multipleSelect' || field.type === 'checkbox' || field.type === 'multipleMember') {
    return 'custom_multiple-' + field.id;  // ✅ 被 SQL 处理
  } else {
    return 'custom-' + field.id;  // ❌ associatedSystem 走这里，但 SQL 不处理！
  }
}
```

**问题分析**：
- `associatedSystem` 类型走 else 分支，生成 `custom-{id}`
- 但后端 SQL 只处理 `custom_single-` 和 `custom_multiple-` 前缀
- 导致 `associatedSystem` 类型的高级搜索条件被完全忽略

### 2.3 第二层：后端 SQL 缺少处理分支

**文件位置**：`test-track/backend/src/main/java/io/metersphere/base/mapper/ext/ExtIssuesMapper.xml`

**问题代码**：
```xml
<when test="key.startsWith('custom_single')">
    and issues.id in (
        select resource_id from custom_field_issues
        where concat('custom_single-',field_id) = #{key}
        and trim(both '"' from value) in (...)
    )
</when>
<when test="key.startsWith('custom_multiple')">
    and issues.id in (
        select resource_id from custom_field_issues
        where concat('custom_multiple-',field_id) = #{key}
        and JSON_CONTAINS(value, #{value})
    )
</when>
<!-- 没有 custom- 的处理分支！ -->
```

**问题分析**：
- SQL 只处理 `custom_single-` 和 `custom_multiple-` 两种前缀
- `custom-` 前缀的条件被完全忽略，不会生成任何 WHERE 子句

### 2.4 第三层：数据存储格式不一致

**文件位置**：`test-track/backend/src/main/java/io/metersphere/excel/listener/IssueExcelListener.java`

**问题代码**（修复前）：
```java
// buildFields 方法中的 else 分支
} else {
    customFieldItemDTO.setValue(v.toString());
    customFieldResourceDTO.setValue("\"" + v + "\"");  // 直接存储 Excel 中的值
}
```

**问题分析**：
- Excel 导入时，`associatedSystem` 类型走 else 分支
- 直接存储用户在 Excel 中填写的**系统名称**（如 `"瑞博士智能陪练平台"`）
- 而手动创建时存储的是**系统 ID**（如 `"0b2f1fd0-020b-407e-bdab-7831155c3dee"`）
- 高级搜索下拉框选择后传递的是**系统 ID**
- 导致：搜索条件（ID）永远无法匹配数据库中的值（名称）

### 3.2 名称与ID对应关系


**结果**：

**结论**：209 条记录存储的是名称，但高级搜索传的是 ID，永远匹配不上。

## 四、修复方案

### 4.1 前端修复

**文件**：`framework/sdk-parent/frontend/src/components/search/custom-component.js`

```javascript
export function generateColumnKey(field) {
  // 单选类型字段：生成 custom_single-{id} 格式的 key
  if (field.type === 'select' || field.type === 'radio' || field.type === 'member'
      || field.type === 'associatedSystem') {  // 新增
    return 'custom_single-' + field.id;
  } else if (field.type === 'multipleSelect' || field.type === 'checkbox' || field.type === 'multipleMember'
      || field.type === 'multipleAssociatedSystem') {  // 新增
    return 'custom_multiple-' + field.id;
  } else {
    return 'custom-' + field.id;
  }
}
```

### 4.2 后端修复

**文件**：`test-track/backend/src/main/java/io/metersphere/excel/listener/IssueExcelListener.java`

**修改内容**：
1. 新增所属系统映射（名称→ID、简称→ID、ID→ID）
2. 在 `buildFields` 方法中添加 `ASSOCIATED_SYSTEM` 类型处理
3. 新增 `parseAssociatedSystemValue` 方法

```java
// 新增成员变量
private Map<String, String> associatedSystemNameMap = new HashMap<>();
private Map<String, String> associatedSystemDescMap = new HashMap<>();
private Map<String, String> associatedSystemIdMap = new HashMap<>();

// 构造函数中初始化映射
private void initAssociatedSystemMap() {
    BaseAssociatedSystemService service = CommonBeanFactory.getBean(BaseAssociatedSystemService.class);
    List<AssociatedSystem> systems = service.getAllAssociatedSystems(workspaceId);
    systems.forEach(system -> {
        associatedSystemNameMap.put(system.getName().toLowerCase(), system.getId());
        associatedSystemDescMap.put(system.getDescription().toLowerCase(), system.getId());
        associatedSystemIdMap.put(system.getId().toLowerCase(), system.getId());
    });
}

// buildFields 方法中添加处理分支
} else if (StringUtils.equalsAnyIgnoreCase(type, CustomFieldType.ASSOCIATED_SYSTEM.getValue())) {
    String systemId = parseAssociatedSystemValue(v.toString());
    customFieldItemDTO.setValue(systemId);
    customFieldResourceDTO.setValue(systemId);  // 不加引号
}

// 新增解析方法
private String parseAssociatedSystemValue(String value) {
    String valueLower = value.toLowerCase().trim();
    // 优先匹配ID → 简称 → 名称
    if (associatedSystemIdMap.containsKey(valueLower)) return associatedSystemIdMap.get(valueLower);
    if (associatedSystemDescMap.containsKey(valueLower)) return associatedSystemDescMap.get(valueLower);
    if (associatedSystemNameMap.containsKey(valueLower)) return associatedSystemNameMap.get(valueLower);
    return value;
}
```

### 4.3 数据库修复

```sql
-- 1. 备份
CREATE TABLE custom_field_issues_backup_assoc_fix 
AS SELECT * FROM custom_field_issues 
WHERE field_id IN (SELECT id FROM custom_field WHERE type = 'associatedSystem');

-- 2. 将名称转换为ID
UPDATE custom_field_issues cfi
JOIN custom_field cf ON cfi.field_id = cf.id
JOIN associated_system asys ON TRIM(BOTH '"' FROM cfi.value) = asys.name
SET cfi.value = asys.id
WHERE cf.type = 'associatedSystem'
AND cfi.value NOT LIKE '%-%-%-%-%';

-- 3. 去除引号（统一格式）
UPDATE custom_field_issues cfi
JOIN custom_field cf ON cfi.field_id = cf.id
SET cfi.value = TRIM(BOTH '"' FROM cfi.value)
WHERE cf.type = 'associatedSystem'
AND cfi.value LIKE '"%"';
```

## 五、技术债务特征总结

### 5.1 多层耦合特征

| 层级 | 组件 | 职责 | 问题 |
|------|------|------|------|
| 前端 | `generateColumnKey()` | 生成搜索条件 key | 类型判断不完整 |
| 后端 SQL | `ExtIssuesMapper.xml` | 解析搜索条件 | 只处理部分前缀 |
| 后端 Java | `IssueExcelListener.java` | 数据导入 | 未转换名称为ID |
| 数据库 | `custom_field_issues` | 数据存储 | 格式不一致 |

### 5.2 债务形成原因

1. **缺乏统一规范**：没有定义自定义字段值的存储格式标准
2. **类型扩展不完整**：新增 `associatedSystem` 类型时，未同步修改所有相关代码
3. **测试覆盖不足**：未测试 Excel 导入后的高级搜索功能
4. **代码审查缺失**：多处代码使用硬编码的类型判断，未统一管理

### 5.3 债务影响

- **功能失效**：高级搜索核心功能无法使用
- **用户体验**：用户无法按所属系统筛选缺陷
- **数据一致性**：同一字段存储格式不统一
- **维护成本**：修复需要同时修改前端、后端、数据库三层

## 六、经验教训

### 6.1 开发规范

1. **新增字段类型时**，必须检查所有相关代码：
   - 前端：`generateColumnKey()`、`getComponentName()`、`getComponentOptions()`
   - 后端：`ExtIssuesMapper.xml`、`IssueExcelListener.java`、`CustomFieldValidatorFactory.java`
   
2. **数据存储格式**必须统一：
   - 选择类型字段统一存储 value（ID），不存储 text（名称）
   - 是否带引号需要统一规范

3. **测试用例**必须覆盖：
   - 手动创建 → 高级搜索
   - Excel 导入 → 高级搜索
   - 导出 → 再导入 → 高级搜索

### 6.2 代码审查要点

- [ ] 新增字段类型是否同步修改了所有相关文件？
- [ ] 数据存储格式是否与现有类型保持一致？
- [ ] 高级搜索功能是否正常工作？
- [ ] Excel 导入/导出是否正常工作？
