# 测试跟踪-功能用例编辑交互优化方案（交互设计+最小改动落点）

> 目标：优化“编辑按钮 -> 只读页面 -> 再次编辑按钮”的割裂体验；统一各入口的打开方式与返回机制；尽量避免新标签页打开；改动边界清晰、可回滚、可迁移。

---

## 1. 背景与问题陈述

### 1.1 用户真实痛点

- **入口不一致**
  - 同一个“功能用例编辑能力”，在不同模块/入口可能以“新标签页 / 同页路由 / 抽屉 / 弹窗”等多种形态出现。
- **两段式编辑打断心智**
  - 用户点击“编辑”后进入页面，却先看到只读展示，还需要再次点击“编辑”才进入可编辑态。
- **返回机制缺失或不可预期**
  - 当编辑页以“同页打开”承载更多入口时，如果没有明确的来源上下文（例如 `redirect`），会出现“无返回按钮/返回不回来源”的体验问题。
- **新标签页打开的副作用**
  - 虽然能避免丢失来源上下文，但会造成“标签页堆积、上下文分散、操作路径变长”的不适。

---

## 2. 现状梳理（代码层事实）

### 2.1 功能用例列表（测试跟踪-用例-功能用例）

#### 2.1.1 列表行点击/查看入口（`TestCaseList.vue`）

- `ms-table` 配置：
  - `@handleRowClick="handleEdit"`
- `handleEdit(testCase)` 实现：
  - 调用 `openCaseEdit({ caseId: testCase.id }, this)`

#### 2.1.2 `openCaseEdit` 当前行为（`test-track/frontend/src/business/case/test-case.js`）

- 通过 `v.$router.resolve({ path, query })` 生成 URL
- 使用 `window.open(TestCaseData.href, '_blank')`
- **结论**：列表行点击/编辑按钮会以**新标签页**打开 `/track/case/edit/:caseId`

#### 2.1.3 “预览”入口不是“只读->再次编辑”的来源（`TestCasePreview.vue`）

- 列表页中“预览”按钮会打开 `TestCasePreview`（抽屉）
- `TestCasePreview.vue` 内容为步骤/前置条件等只读展示
- **没有**出现“编辑/再次编辑按钮”逻辑
- **结论**：你感知到的“只读页还能点编辑按钮”，主要发生在 `TestCaseEdit.vue` 这个编辑页内部，而非 `TestCasePreview` 抽屉。

---

### 2.2 编辑页自身的“只读->可编辑”机制（`TestCaseEdit.vue`）

#### 2.2.1 两段式编辑的根因

- `editableState`：默认 `false`
- `editable`（计算属性）逻辑：
  - `editable = isAdd || editableState`
- `toEdit()`：
  - `this.editableState = true`

#### 2.2.2 由此形成的用户链路

- 从列表进入 `/track/case/edit/:id` 后
  - 页面默认处于“展示态/只读态”
- 用户再次点击页头“编辑”按钮
  - 才切换为“可编辑态”

## 3. 交互设计目标（建议作为产品规范）

- **入口一致性**：同类入口触发同类行为
  - 行点击偏“查看”，编辑按钮偏“立即编辑”
- **状态清晰**：让用户知道自己处于 `view` 还是 `edit`
- **返回可预测**：从哪里来，回哪里去（或至少回列表）
- **尽量不新标签**：但保留必要场景（例如执行上下文不应丢失）
- **二开原则**：改动面小、边界清晰、可回滚、可迁移

---

## 4. 推荐方案（统一到“view/edit 模式 + redirect 返回”）

### 4.1 核心规则（建议落成统一规范）

- **规则 A：列表“行点击”默认进入查看（view）模式**
  - 心智：点行=看详情
- **规则 B：列表“编辑按钮”直接进入编辑（edit）模式**
  - 心智：点编辑=立刻可编辑（避免二次点击编辑）
- **规则 C：查看页（view）保留“编辑”按钮**
  - 点击后切换到 edit（同页切换，不新开页）
- **规则 D：同页打开必须带来源返回信息**
  - 通过 `redirect`（来源 fullPath）实现“返回来源”

---

### 4.2 路由与参数约定（建议）

- **查看模式**：
  - `/track/case/edit/:id?mode=view&redirect=<来源fullPath>`
- **编辑模式**：
  - `/track/case/edit/:id?mode=edit&redirect=<来源fullPath>`

> 说明：
> - `mode` 用于初始化 `editableState`，并让“只读->编辑”成为显式状态切换。
> - `redirect` 用于实现明确返回；没有 `redirect` 时再 fallback 到 `router.back()` 或列表。

---

### 4.3 编辑页行为约定（建议）

- **初始化**：根据 `mode` 决定进入 view 还是 edit
- **view -> edit**：点击“编辑”
  - 不新增 history：建议使用 `router.replace` 将 `mode` 改为 `edit`
- **edit -> view**：点击“取消编辑”
  - 建议 `router.replace` 改回 `mode=view`
  - 是否需要“未保存确认弹窗”可二期考虑
- **返回按钮**：
  - 优先按 `redirect` 返回
  - 无 `redirect` 再 fallback：`router.back()` 或 `/track/case/all`

---

## 5. 入口对照表（现状 vs 建议）

| 入口 | 现状打开方式 | 建议打开方式 | 备注 |
|---|---|---|---|
| 功能用例列表 行点击 | 新标签页打开编辑页 | 同页打开 view 模式 | 点行=查看更符合心智 |
| 功能用例列表 编辑按钮 | 新标签页打开编辑页 | 同页打开 edit 模式 | 点编辑=立即编辑，避免二次编辑 |
| 编辑页内“再次编辑” | `toEdit()` 切 editableState | view->edit（replace） | 将“再次编辑”显式化 |
| 列表预览抽屉（TestCasePreview） | 抽屉只读展示 | 保持不变 | 它不是两段式编辑的根因 |
| 测试计划-功能用例抽屉（FunctionalTestCaseEdit） | 抽屉内编辑/执行上下文 | 建议保持抽屉 | 避免丢上下文，改动过大会牵扯执行链路 |
| 公共用例库（PublicTestCaseShow） | 抽屉内嵌 `TestCaseEdit` | 保持抽屉 | 当前已能在抽屉内切换编辑/只读 |

## 6. 最小改动落点（少文件、可回滚、可迁移）

### 6.1 建议修改范围（集中在 2-3 个文件）

1. **`TestCaseList.vue`**
   - 行点击：改为 `router.push` 打开 `mode=view`（带 `redirect`）
   - 编辑按钮：改为 `router.push` 打开 `mode=edit`（带 `redirect`）

2. **`TestCaseEdit.vue`**
   - 读取 `this.$route.query.mode` 初始化 `editableState`
   - `toEdit()`：除 `editableState=true` 外，再 `router.replace` 将 `mode=edit`
   - “取消编辑”：`router.replace` 回 `mode=view`
   - 顶部补“返回”按钮：按 `redirect` 返回

> 说明：不建议全局改造 `openCaseEdit`，否则会影响公共库/测试计划/评审等已有依赖，违反“改动面小、边界清晰”的二开原则。

---

## 7. 定稿项：返回策略（已选择选项 2）

### 7.1 返回策略：严格回来源（redirect 精确返回）

- **定稿选择**：采用“选项 2”，统一通过 `redirect=<来源fullPath>` 精确返回来源页（列表/测试计划/公共库/搜索结果等）。
- **符合操作习惯的理由**
  - 用户心智：进入编辑是一段“临时工作台”，完成后应回到原来的工作上下文（筛选条件、分页、模块树定位等）。
  - 多入口场景下，相比 `router.back()` 更可控；相比“强制回列表”更符合用户预期。
- **实现约束与注意事项（避免踩坑）**
  - `redirect` **只允许站内路由**（建议：以 `/` 开头，且不包含 `http://` / `https://`），避免 open redirect 风险；不合法则 fallback 回 `/track/case/all` 或 `router.back()`。
  - `redirect` 建议直接传 `this.$route.fullPath`（路由会自动进行 URL 编码），编辑页读取后直接 `this.$router.push(redirect)`。
  - `mode` 的切换（`view <-> edit`）建议用 `router.replace`，避免历史栈被多次“模式切换”污染。
  - 对“抽屉/弹窗来源”的现实约束：`redirect` 只能恢复到**来源路由页面**，无法自动恢复抽屉打开状态；如确有强需求，可二期扩展额外参数（例如 `from=plan&drawerCaseId=...`）用于恢复 UI 状态。

## 8. 当前结论摘要（给评审/沟通用）

- 你感受到的“编辑按钮 -> 只读页 -> 再次编辑”主要来自 `TestCaseEdit.vue` 的 `editableState` 设计，而不是列表的 `TestCasePreview` 抽屉。
- 若要“尽量不新标签”，核心不是简单把 `window.open` 改成 `router.push`，而是：
  - 统一入口语义（行点击=查看、编辑按钮=立即编辑）
  - 引入明确的 `mode=view/edit` 状态
  - 为同页打开补齐 `redirect` 返回机制
- 最小改动建议集中在 `TestCaseList.vue` + `TestCaseEdit.vue`，不要全局动 `openCaseEdit`；返回策略定稿为“选项 2：redirect 精确返回来源”。

## 9. 已落地实现（代码已完成）

### 9.1 改动文件清单（可回滚边界）

我在做：将本次改动控制在 2 个核心文件内，并明确记录文件清单。  
目的是：保证改动边界清晰、升级可对照、回滚成本低。  
如果不这样做,就无法实现：出现问题时快速定位与回退，且容易误伤官方核心链路。

- `test-track/frontend/src/business/case/components/TestCaseEdit.vue`
- `test-track/frontend/src/business/case/components/TestCaseList.vue`

### 9.2 路由参数约定（已落地）

我在做：把“查看/编辑模式”与“来源返回”落成显式参数。  
目的是：让入口一致、状态可追溯、返回可预测。  
如果不这样做,就无法实现：同页打开时精确返回来源页，也无法避免“二段式编辑”割裂。

- **查看模式**：`/track/case/edit/:id?mode=view&redirect=<来源fullPath>`
- **编辑模式**：`/track/case/edit/:id?mode=edit&redirect=<来源fullPath>`

### 9.3 `TestCaseList.vue`（功能用例列表）入口行为变化（已落地）

我在做：把“行点击/操作列按钮”统一改为同页路由跳转，并按语义区分 view/edit。  
目的是：入口一致性（行点击偏查看、编辑按钮偏立即编辑）并携带 redirect 精确返回。  
如果不这样做,就无法实现：同页链路与精确返回策略（仍会走新标签打开）。

- **行点击（`@handleRowClick`）**：
  - 现在：跳转 `/track/case/edit/:id`，`mode=view`，`redirect=当前列表 fullPath`
- **操作列新增“编辑”按钮**：
  - 权限：`PROJECT_TRACK_CASE:READ+EDIT`
  - 现在：跳转 `/track/case/edit/:id`，`mode=edit`，`redirect=当前列表 fullPath`

实现要点（便于检索）：

- 新增 `routerToCaseEdit(caseId, mode)` 统一跳转
- 新增 `getEditRedirectQuery(mode)` 统一组装 `mode/redirect`

### 9.4 `TestCaseEdit.vue`（编辑页）模式/返回/安全（已落地）

我在做：让编辑页基于 `mode` 显式进入 view/edit，并提供“返回来源页”按钮，同时防止 open redirect。  
目的是：消除“只读页再次点击编辑”的割裂，并把返回体验固定为“严格回来源”。  
如果不这样做,就无法实现：跨入口一致体验；且 redirect 可能带来站外跳转风险。

- **初始化 editableState**：
  - 根据 `this.$route.query.mode` 设置 `editableState`
  - `mode=edit` 且非只读权限时进入编辑态；其他情况进入查看态
- **view -> edit**（点击页头“编辑”）：
  - 使用 `router.replace` 将 `mode` 切为 `edit`
  - 同时设置 `editableState=true`
- **edit -> view**（点击“取消”）：
  - 复用现有 `handleCommand(4)`
  - 新增：先 `router.replace` 将 `mode` 改回 `view`
- **保存后回 view**：
  - 仍保持现有“保存后 editableState=false”行为
  - 新增：保存成功时同步 `router.replace(mode=view)`，让 URL 与界面一致
- **返回按钮**（新增）：
  - 优先按 `redirect` 返回
  - `redirect` 不存在/不合法时 fallback：`/track/case/all?projectId=<当前项目>`

安全实现（open redirect 防护）：

- 仅允许 `redirect` 为站内路径（以 `/` 开头且不以 `//` 开头）
- 明确拒绝包含协议片段的字符串（例如 `http:` / `https:`）
