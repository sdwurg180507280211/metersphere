# 测试计划-功能用例-只读编辑页-附件置灰设计原理（结合实际实现）

## 1. 现象与边界

在【测试计划-功能用例-只读编辑页】中，附件区的“写操作入口”会表现为置灰/不可操作，典型包括：

- **本地上传入口**：不可用
- **关联文件入口**：不可用
- **删除 / 取消关联**：不可用
- **预览 / 下载**：可用（只读场景允许读取，不允许写入）

注意：这里讨论的“附件”是**用例资产的一部分**（用例维度的附件/关联文件），并非测试计划执行过程的临时证据。

## 2. 为什么要置灰：设计语义与实际场景

### 2.1 测试计划功能用例页的职责定位

测试计划功能用例页更接近“执行/查看”场景，而不是“维护用例资产”场景：

- **执行场景**：围绕用例在测试计划中的执行，记录执行信息
- **资产隔离**：执行过程中不应修改用例本体资产（包括附件），避免影响其它计划/引用
- **语义一致**：页面呈现为只读时，应保证所有会改变资产的数据入口都不可用

如果允许在测试计划执行页修改用例附件，会引发：

- **基线污染**：执行过程把“过程材料”写进“用例资产”，用例资产在执行中被动变化
- **追溯困难**：难区分“用例固有附件”与“本次执行/本次讨论产生的材料”
- **体验不一致**：文本/步骤不可改，但附件可改，用户会感知语义冲突

### 2.2 过程证据应该沉淀在哪里（结合实际页面）

测试计划场景中，过程证据更适合沉淀到“评论/执行记录”里，而不是用例附件。

从实现看，测试计划页右侧存在评论组件（`ReviewComment`），并设置 `default-type="PLAN"`，用于承载计划执行过程的沟通与证据。

## 3. 实际实现：置灰不是一句“readOnly”这么简单

### 3.1 只读传递链路（真实代码）

在测试计划功能用例页（`FunctionalTestCaseEdit.vue`）里，“其他信息”组件被**强制只读**并标记为测试计划场景：

```vue
<test-case-edit-other-info
  :read-only="true"
  :is-test-plan="true"
  :form="testCase"
  :case-id="testCase.caseId"
/>
```

这意味着：附件 tab 的入口在该页面上天然处于只读语义下。

### 3.2 上传/关联入口：主要由 `readOnly` 控制

`TestCaseEditOtherInfo.vue` 的附件 tab 中，“上传/关联”的禁用条件是：

```vue
<el-upload :disabled="readOnly || isCopy">
  <el-button :disabled="readOnly || isCopy" type="text">本地上传</el-button>
</el-upload>

<el-button type="text" :disabled="readOnly || isCopy" @click="associationFile">
  关联文件
</el-button>
```

- **readOnly=true**：禁用上传与关联
- **isCopy=true**：复制场景也会禁用（避免复制过程中产生非预期写入）

### 3.3 删除/取消关联入口：由 `readOnly` + `isDelete` 双因子控制（关键）

附件列表组件 `TestCaseAttachment.vue` 的删除/取消关联按钮禁用条件并非只有 `readOnly`：

```vue
<el-button
  :disabled="readOnly || !isDelete"
  v-if="..."
  icon="el-icon-delete"
/>

<el-button
  :disabled="readOnly || !isDelete"
  v-if="..."
  icon="el-icon-unlock"
/>
```

而 `isDelete` 的来源在 `TestCaseEditOtherInfo.vue`：

```vue
<test-case-attachment
  :read-only="readOnly"
  :is-delete="!isTestPlan"
/>
```

因为测试计划场景 `isTestPlan=true`，所以：

- `isDelete = !isTestPlan = false`
- **即使未来某个场景把 `readOnly` 放开，测试计划场景仍然不会开放删除/取消关联**（这是“场景策略”，不是单纯只读）。

### 3.4 关键参数表（按真实行为解释）

| 参数 | 作用 | 测试计划场景典型值 | 直接影响 |
|------|------|-------------------|----------|
| `readOnly` | 只读语义开关 | `true`（在测试计划页被写死） | 禁用上传/关联；也会参与禁用删除/取消关联 |
| `isTestPlan` | 场景标识 | `true` | 导致 `isDelete=false`，从而关闭删除/取消关联 |
| `isDelete` | 是否允许删除类写操作 | `false`（由 `!isTestPlan` 推导） | 控制删除/取消关联是否可用 |
| `isCopy` | 复制场景保护 | 视场景而定 | 禁用上传/关联，避免复制过程中写入 |

## 4. 与“用例评审只读页”的相同点与不同点

### 4.1 相同点：过程场景不修改用例资产

- 两者都体现“过程数据与用例资产分层”的思想
- 只读语义下：允许查看（预览/下载），禁止写入（上传/关联/删除）

### 4.2 不同点：证据沉淀模块不同（但理念一致）

- **用例评审场景**：证据倾向沉淀到“评审评论/评审过程”体系
- **测试计划场景**：证据倾向沉淀到 `ReviewComment(PLAN)` 或执行记录

即：两者都不把过程证据写入“用例附件资产”，但“承载证据的模块”不同。

## 5. 用户体验层面的落地

- **操作一致性**：只读页所有会改动资产的入口都不可用
- **可用操作保留**：预览/下载不影响资产变更，保留可用
- **替代路径明确**：通过评论/执行记录沉淀过程证据，而不是改动用例附件

## 6. 为什么必须用评论系统承载执行证据（充分论证）

### 6.1 数据归属的根本差异

#### 用例附件 vs 执行证据的本质区别
- **用例附件**（TestCaseAttachment）：
  - **归属对象**：用例本体（`AttachmentType.TEST_CASE.type()`）
  - **生命周期**：与用例同生命周期，长期存在
  - **影响范围**：所有引用该用例的测试计划、评审、报告等
  - **变更影响**：属于"用例资产变更"，需要审计和通知

- **执行证据**（评论附件）：
  - **归属对象**：执行过程（`TestCaseComment.type = "PLAN"`）
  - **生命周期**：与本次执行同生命周期，过程性数据
  - **影响范围**：仅影响当前测试计划的执行记录
  - **变更影响**：属于"执行过程记录"，不影响用例本体

#### 实际代码证据
```java
// 用例附件直接挂载到用例
@PostMapping("/testcase/upload")
public void uploadTestCaseAttachment(@RequestParam String belongId, MultipartFile file) {
    request.setBelongType(AttachmentType.TEST_CASE.type()); // 直接绑定用例
}

// 评论附件挂载到评论，评论绑定到执行过程
public TestCaseComment saveComment(SaveCommentRequest request) {
    request.setType(TestCaseCommentType.CASE.name()); // 评论类型
    request.setBelongId(planId); // 绑定到测试计划
}
```

### 6.2 架构设计的必然选择

#### 分层架构的完整性
```
┌─────────────────────────────────────────────────────────┐
│                    用例资产层                              │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐       │
│  │   用例文本   │  │   用例步骤   │  │   用例附件   │       │
│  │   (长期)    │  │   (长期)    │  │   (长期)    │       │
│  └─────────────┘  └─────────────┘  └─────────────┘       │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                   执行记录层                               │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐       │
│  │   执行状态   │  │   执行结果   │  │   执行评论   │       │
│  │  (单次执行)  │  │  (单次执行)  │  │  (单次执行)  │       │
│  │             │  │             │  │  └─ 附件证据  │       │
│  └─────────────┘  └─────────────┘  └─────────────┘       │
└─────────────────────────────────────────────────────────┘
```

如果允许在执行层直接修改资产层，会破坏整个分层架构：
- **数据边界模糊**：无法区分"用例固有"vs"执行过程产生"
- **审计链路断裂**：用例变更审计会包含执行过程的临时材料
- **权限模型混乱**：执行权限和资产权限混在一起

### 6.3 业务语义的明确性

#### 执行过程的本质
测试计划执行页面的核心语义是"执行和记录"，不是"维护用例"：

- **执行场景**：按照既定用例执行，记录执行结果和过程
- **过程证据**：执行过程中产生的截图、日志、临时文件等
- **用例维护**：应该在专门的用例编辑页面进行

#### 用户体验的一致性
用户在只读页面有明确的预期：
- **查看**：可以看到用例的所有信息（包括已有附件）
- **记录**：可以记录执行过程（通过评论、状态等）
- **不能修改**：不能修改用例本体（包括附件）

如果允许上传附件，用户会困惑：
- 为什么文本不能改，但附件可以改？
- 上传的附件会影响其他测试计划吗？
- 这个附件是用例的一部分还是本次执行的证据？

### 6.4 技术实现的合理性

#### 评论系统的设计优势
```java
// TestCaseComment 实体设计
@Data
public class TestCaseComment {
    private String id;
    private String caseId;        // 关联用例
    private String author;        // 执行人
    private Long createTime;      // 执行时间
    private String type;          // CASE/REVIEW/PLAN
    private String belongId;      // 归属对象（测试计划ID）
    private String description;   // 评论内容
}
```

评论系统的优势：
- **时间维度**：天然记录执行时间、执行人
- **空间维度**：通过 `belongId` 明确归属（测试计划）
- **类型维度**：通过 `type` 区分场景（CASE/REVIEW/PLAN）
- **内容维度**：支持文本+附件的复合内容

#### 实际页面实现
```vue
<!-- FunctionalTestCaseEdit.vue -->
<review-comment
  default-type="PLAN"        <!-- 明确标识为计划执行场景 -->
  :case-id="testCase.caseId"
  ref="comment"
/>
```

这个设计确保：
- **场景明确**：`default-type="PLAN"` 表明这是计划执行评论
- **归属清晰**：评论属于测试计划，不属于用例
- **权限合理**：执行权限控制评论，不影响用例权限

### 6.5 替代方案的不可行性分析

#### 方案1：直接放开用例附件上传
**问题**：
- 数据归属错误：执行材料写入用例资产
- 影响范围失控：影响所有引用该用例的场景
- 审计混乱：用例变更审计包含执行过程材料

#### 方案2：新增"执行附件"类型
**问题**：
- 架构复杂化：需要新增存储模型、权限模型、审计模型
- 维护成本高：需要处理附件同步、版本管理、生命周期管理
- 语义模糊：用户难以区分"用例附件"vs"执行附件"

#### 方案3：临时文件机制
**问题**：
- 用户体验差：需要额外的"保存到用例"操作
- 数据一致性：临时文件的生命周期管理复杂
- 技术复杂度：需要设计临时存储、清理机制

### 6.6 评论系统的必然性结论

基于以上分析，评论系统是承载执行证据的**唯一合理选择**：

1. **数据归属正确**：执行证据归属于执行过程，不影响用例资产
2. **架构一致性**：保持分层架构的完整性，不破坏设计原则
3. **业务语义清晰**：执行记录就是执行记录，用例维护就是用例维护
4. **技术实现简单**：复用现有评论系统，无需新增复杂机制
5. **用户体验一致**：只读页面的行为符合用户预期
6. **维护成本低**：无需新增复杂的权限、审计、生命周期管理

**结论**：使用评论系统承载执行证据不是"可选方案"，而是"必然选择"。任何试图绕过评论系统的方案都会带来数据归属、架构设计、业务语义、技术实现等方面的问题。

## 7. 总结（对研发/产品的可执行结论）

- 测试计划功能用例只读页附件置灰是合理且与实现一致的。
- 实现层面：
  - 上传/关联主要由 `readOnly` 控制
  - 删除/取消关联由 `readOnly` + `isDelete` 双因子控制，测试计划场景会强制 `isDelete=false`
- 如果未来需要在测试计划页开放某些“写操作”，必须同时评估：
  - 是否要解除 `readOnly` 的硬编码
  - 是否要调整 `isDelete = !isTestPlan` 这一场景策略
