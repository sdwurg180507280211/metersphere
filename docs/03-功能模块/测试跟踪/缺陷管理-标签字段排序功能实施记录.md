## 技术实现细节

### 标签显示逻辑

- **多标签显示**：当标签数量 > 1 时，显示第一个标签 + "+N" 的形式
- **单标签显示**：直接显示完整标签内容
- **Tooltip提示**：鼠标悬停时显示所有标签的完整内容

### 工具函数说明

**来源：** `test-track/frontend/src/business/case/test-case.js`

1. **getTagToolTips(tags)**
   - 功能：将标签数组转换为逗号分隔的字符串，用于 tooltip 显示
   - 返回：`"标签1,标签2,标签3"`

2. **parseColumnTag(tags)**
   - 功能：解析标签数组，用于列表显示
   - 返回：
     - 多标签：`["标签1", "+2"]`
     - 单标签：`["标签1"]`

## 排序功能不生效的问题排查与修复

### 问题现象
缺陷管理--关联测试用例弹窗的标签字段排序不起作用

### 深度排查过程

#### 1. 前端排查
- ✅ 前端已添加 `@order="initTableData"` 事件监听
- ✅ 前端已添加 `sortable="custom"` 属性
- ✅ 前端代码无问题

#### 2. 后端API排查
- 前端调用：`getTestCaseRelateIssue(pageNum, pageSize, param)`
- 后端接口：`POST /test/case/relate/issue/{goPage}/{pageSize}`
- Controller方法：`TestCaseController.getTestCaseIssueRelateList()`
- Service方法：`TestCaseService.getTestCaseIssueRelateList()`
- Mapper方法：`ExtTestCaseMapper.getTestCaseByNotInIssue()`

#### 3. SQL排查（发现根本原因）

**对比测试计划关联用例弹窗的SQL：**

```xml
<!-- 测试计划关联用例弹窗 - getTestCaseByNotInPlan -->
<select id="getTestCaseByNotInPlan" resultType="io.metersphere.dto.TestCaseDTO">
    select test_case.id, test_case.name, test_case.priority, test_case.type, test_case.review_status,
    test_case.num, test_case.custom_num, test_case.priority, test_case.tags, test_case.create_time,
    test_case.last_execute_result,
    test_case.update_time,
    project_version.name versionName, test_case.version_id versionId
    from test_case
    left join project_version on project_version.project_id = test_case.project_id and project_version.id =
    test_case.version_id
    <if test="!request.repeatCase">
        left join test_plan_test_case on test_case.id = test_plan_test_case.case_id and test_plan_test_case.plan_id = #{request.planId}
    </if>
    <include refid="notInQueryWhereCondition"/>
    <if test="!request.repeatCase">
        and test_plan_test_case.case_id is null
    </if>
    <include refid="io.metersphere.base.mapper.ext.ExtBaseMapper.orders"/>  <!-- ✅ 有排序 -->
</select>
```

**缺陷管理关联用例弹窗的SQL（修复前）：**

```xml
<!-- 缺陷管理关联用例弹窗 - getTestCaseByNotInIssue -->
<select id="getTestCaseByNotInIssue" resultType="io.metersphere.dto.TestCaseDTO">
    select
    <include refid="io.metersphere.base.mapper.TestCaseMapper.Base_Column_List"/>
    from test_case
    <include refid="notInQueryWhereCondition"></include>
    <!-- ❌ 缺少排序语句！ -->
</select>
```

### 根本原因

**`getTestCaseByNotInIssue` 的SQL查询缺少排序语句：**
```xml
<include refid="io.metersphere.base.mapper.ext.ExtBaseMapper.orders"/>
```

这导致：
- 前端点击排序按钮，确实触发了后端查询
- 后端接收到了排序参数（`request.getOrders()`）
- 但SQL查询没有使用这些排序参数
- 结果：数据库返回的数据没有排序

### 修复方案

**文件：** `test-track/backend/src/main/java/io/metersphere/base/mapper/ext/ExtTestCaseMapper.xml`

**修改：** 在 `getTestCaseByNotInIssue` 的SQL查询末尾添加排序语句

```xml
<select id="getTestCaseByNotInIssue" resultType="io.metersphere.dto.TestCaseDTO">
    select
    <include refid="io.metersphere.base.mapper.TestCaseMapper.Base_Column_List"/>
    from test_case
    <include refid="notInQueryWhereCondition"></include>
    <include refid="io.metersphere.base.mapper.ext.ExtBaseMapper.orders"/>  <!-- ✅ 添加排序 -->
</select>
```

### 修复后的效果
- ✅ 点击ID列排序按钮，正常按ID升序/降序排列
- ✅ 点击标签列排序按钮，正常按标签升序/降序排列
- ✅ 点击优先级列排序按钮，正常按优先级升序/降序排列
- ✅ 与测试计划关联用例弹窗的排序行为保持一致

### 技术说明

**`ExtBaseMapper.orders` 的作用：**
- 这是MyBatis的动态SQL片段
- 根据前端传递的 `request.orders` 参数动态生成 `ORDER BY` 子句
- 支持多字段排序、升序/降序

**示例：**
- 前端传递：`orders: [{name: 'tags', type: 'asc'}]`
- 生成SQL：`ORDER BY test_case.tags ASC`

### 为什么之前没发现这个问题？

1. **前端代码看起来正常**：有 `@order` 事件、有 `sortable="custom"` 属性
2. **后端Service层有排序逻辑**：`request.setOrders(ServiceUtils.getDefaultOrder(request.getOrders()))`
3. **但SQL层缺少排序**：MyBatis XML中没有使用排序参数

这是一个**前端正常、Service层正常、但SQL层缺失**的典型案例。
