# 评审评论支持普通文件附件-实现方案

## 1. 需求背景

在用例评审过程中，除了已有的 markdown 图片上传能力外，评审人员还需要上传其他类型的文件作为评审证据，例如：
- ZIP 压缩包（包含多个截图或日志）
- PDF 文档（需求文档、设计文档）
- LOG 文件（执行日志、错误日志）
- 其他业务相关文件

这些文件应该：
- **归属于评论事件**：作为评审过程的证据，不污染用例资产
- **不影响用例附件**：不写入用例的附件体系
- **不污染项目文件列表**：不在项目文件管理中显示
- **支持完整生命周期**：上传、展示、下载、删除、评论删除时自动清理

## 2. 设计原则

### 2.1 复用现有文件体系
- 使用 `file_metadata` 表存储文件元数据
- 使用 `file_association` 表建立关联关系
- 新增 `FILE_ASSOCIATION_TYPE = "REVIEW_COMMENT_FILE"` 类型

### 2.2 数据归属清晰
- `file_association.source_id = commentId`：文件归属于具体评论
- `file_association.type = "REVIEW_COMMENT_FILE"`：区分于 markdown 图片（MD_IMAGE）
- `file_association.project_id = testCase.projectId`：继承用例的项目权限

### 2.3 避免冲突与污染
- **文件名冲突**：使用 UUID 作为存储文件名，原始文件名存 `attachInfo`
- **项目文件列表污染**：设置 `file_metadata.latest = false` 避免出现在项目文件列表
- **权限隔离**：只有评论作者可以删除文件

#### 3.3.1 文件上传逻辑
```java
public FileMetadataWithBLOBs upload(String commentId, MultipartFile file) {
    TestCaseComment comment = getAndCheckCommentOwner(commentId);
    TestCaseWithBLOBs testCase = getAndCheckTestCaseInCurrentProject(comment.getCaseId());

    // 避免文件名冲突：UUID存储 + 原始名存attachInfo
    String originalName = file.getOriginalFilename();
    String suffix = originalName.substring(originalName.lastIndexOf('.'));
    String storeName = UUID.randomUUID().toString().replace("-", "") + suffix;

    FileMetadataWithBLOBs fileMetadata = new FileMetadataWithBLOBs();
    fileMetadata.setName(storeName);
    Map<String, Object> attachInfo = new HashMap<>();
    attachInfo.put("originalName", originalName);
    fileMetadata.setAttachInfo(JSON.toJSONString(attachInfo));
    fileMetadataService.saveFile(file, fileMetadata);

    // 避免出现在项目文件列表中
    FileMetadata update = new FileMetadata();
    update.setId(fileMetadata.getId());
    update.setLatest(false);
    fileMetadataMapper.updateByPrimaryKeySelective(update);

    // 建立关联关系
    FileAssociation association = new FileAssociation();
    association.setType(FILE_ASSOCIATION_TYPE);
    association.setSourceId(commentId);
    association.setFileMetadataId(fileMetadata.getId());
    association.setProjectId(testCase.getProjectId());
    fileAssociationMapper.insert(association);

    return fileMetadata;
}
```

#### 3.3.2 权限校验
```java
private TestCaseComment getAndCheckCommentOwner(String commentId) {
    TestCaseComment comment = testCaseCommentMapper.selectByPrimaryKey(commentId);
    if (comment == null) {
        MSException.throwException(Translator.get("resource_not_exist"));
    }
    if (!StringUtils.equals(comment.getAuthor(), SessionUtils.getUser().getId())) {
        MSException.throwException(Translator.get("check_owner_comment"));
    }
    return comment;
}

private TestCaseWithBLOBs getAndCheckTestCaseInCurrentProject(String caseId) {
    TestCaseWithBLOBs testCase = testCaseMapper.selectByPrimaryKey(caseId);
    if (testCase == null || !StringUtils.equals(testCase.getProjectId(), SessionUtils.getCurrentProjectId())) {
        MSException.throwException(Translator.get("check_owner_comment"));
    }
    return testCase;
}
```

#### 3.3.3 删除策略
```java
public void delete(String commentId, String fileId) {
    TestCaseComment comment = getAndCheckCommentOwner(commentId);
    getAndCheckTestCaseInCurrentProject(comment.getCaseId());

    // 删除关联关系
    FileAssociationExample example = new FileAssociationExample();
    example.createCriteria()
            .andSourceIdEqualTo(commentId)
            .andTypeEqualTo(FILE_ASSOCIATION_TYPE)
            .andFileMetadataIdEqualTo(fileId);
    fileAssociationMapper.deleteByExample(example);

    // 如果文件不被任何资源引用，则删除文件
    if (CollectionUtils.isEmpty(fileAssociationService.getByFileId(fileId))) {
        fileMetadataService.deleteFile(fileId);
    }
}
```

public void deleteCaseComment(String caseId) {
    List<String> commentIds = getCaseComments(caseId).stream().map(TestCaseCommentDTO::getId).toList();
    
    // 批量清理评论文件

### 4.4 评论历史展示文件列表

<template>
  <div class="main">
    <!-- ... 原有内容 ... -->

    <div class="comment-files" v-if="files && files.length > 0">

import {deleteTestCaseCommentFile, downloadTestCaseCommentFile, listTestCaseCommentFiles, uploadTestCaseCommentFile} from "@/api/test-case-comment-file";

    return {
      files: [],  // 新增文件列表

  },

      this.getFiles();

      if (!this.comment || !this.comment.id) {
        this.files = [];
        return;

        this.files = res.data || [];

      if (file.attachInfo) {
        try {
          let info = JSON.parse(file.attachInfo);
          if (info && info.originalName) {
            return info.originalName;
          }
        } catch (e) {
          // ignore
        }
      }
      return file.name;

      downloadTestCaseCommentFile(file.id, this.getFileName(file));

        this.$warning(this.$t('test_track.comment.cannot_delete'));
        return;

        this.$success(this.$t('commons.delete_success'));
        this.getFiles();
      });

        return;

## 5. 关键技术决策

### 5.1 为什么使用 `file_association` 而不是新建表？
- **复用现有能力**：文件存储、权限、下载等逻辑已经完备
- **保持一致性**：与 markdown 图片（MD_IMAGE）使用相同的关联机制
- **降低维护成本**：不需要新增存储、清理、权限等逻辑

### 5.2 为什么设置 `latest = false`？
- **避免污染项目文件列表**：项目文件管理页面的 SQL 过滤条件包含 `latest = true`
- **保持语义清晰**：评论文件是过程数据，不是项目资产
- **避免用户困惑**：用户不会在项目文件中看到评论上传的临时文件

### 5.3 为什么用 UUID 作为存储文件名？
- **避免同名冲突**：`file_metadata.name` 在同一项目下要求唯一
- **支持多语言**：避免中文文件名在不同系统下的兼容问题
- **安全性考虑**：避免文件名猜测和路径遍历攻击

### 5.4 为什么删除评论时要清理文件？
- **避免垃圾文件**：防止评论删除后文件仍占用存储空间
- **数据一致性**：确保删除操作是完整的，不留下孤立数据
- **权限一致性**：只有评论作者能删除文件，删除评论时应该一并清理

## 6. 使用场景

### 6.1 评审过程中上传证据
1. 评审人员打开用例评审页面
2. 点击评论按钮，打开评论编辑弹窗
3. 输入评论内容，选择相关文件（ZIP、PDF、LOG等）
4. 点击发送，评论保存成功后自动上传文件
5. 其他评审人员可以看到评论内容和附件，支持下载

### 6.2 编辑评论时添加文件
1. 评审人员点击已有评论的编辑按钮
2. 在编辑弹窗中添加新的文件
3. 保存评论后，新文件自动上传并显示

### 6.3 删除文件
1. 只有评论作者可以看到删除按钮
2. 点击删除后，文件关联关系被移除
3. 如果文件不被任何其他资源引用，文件本身也被删除

## 7. 权限控制

### 7.1 上传权限
- **评论作者**：可以上传文件到自己的评论
- **其他用户**：不能上传文件到别人的评论

### 7.2 下载权限
- **项目成员**：可以下载评论文件（继承用例的查看权限）
- **非项目成员**：无权访问

### 7.3 删除权限
- **评论作者**：可以删除自己评论的文件
- **其他用户**：不能删除（按钮禁用）

## 8. 错误处理

### 8.1 上传失败
- **单个文件失败**：显示错误提示，继续上传其他文件
- **网络异常**：显示错误提示，用户可以重试
- **权限不足**：显示权限错误提示

### 8.2 下载失败
- **文件不存在**：返回 404 错误
- **权限不足**：返回 403 错误

### 8.3 删除失败
- **权限不足**：显示权限错误提示
- **文件已被删除**：静默处理，刷新列表

## 9. 性能考虑

### 9.1 文件大小限制
- **继承系统限制**：使用 `@MsFileLimit` 注解
- **前端校验**：可以在上传前进行文件大小预检

### 9.2 并发上传
- **串行上传**：避免并发压力，逐个上传文件
- **错误隔离**：单个文件失败不影响其他文件

### 9.3 列表加载
- **按需加载**：只在评论展开时加载文件列表
- **缓存机制**：可以在评论会话期间缓存文件列表

## 10. 扩展性

### 10.1 支持其他评论类型
当前实现支持评审评论（REVIEW），但架构设计可以轻松扩展到：
- 测试计划评论（PLAN）
- 用例评论（CASE）
- 缺陷评论（ISSUE）

### 10.2 文件预览
未来可以扩展文件预览功能：
- PDF 在线预览
- 图片在线预览
- 文本文件在线预览

### 10.3 文件版本管理
如果需要文件版本管理，可以：
- 在 `file_metadata` 中增加版本字段
- 保留历史版本文件
- 提供版本回退功能

## 11. 总结

本实现方案通过复用现有的 `file_metadata` 和 `file_association` 体系，在不破坏原有架构的前提下，为评审评论增加了普通文件上传能力。关键特点：

1. **数据归属清晰**：文件归属于评论，不影响用例资产
2. **用户体验友好**：支持多文件上传、下载、删除
3. **权限控制完善**：只有作者可以删除，项目成员可以下载
4. **技术实现简洁**：复用现有文件体系，最小化代码变更
5. **扩展性良好**：可以轻松扩展到其他评论类型

该方案已经在 MeterSphere 项目中实现并测试通过，为评审过程提供了完整的文件证据支持。
