# 缺陷管理：关闭后可重新打开与复测次数实现总结

## 一、需求背景

- **需求1**：缺陷状态为“关闭”后，允许创建人（测试人员）重新打开（流转到 `reopened`）。
- **需求2**：新增“复测次数”字段，**仅**在 `closed -> reopened` 时自动 +1（懒初始化），并记录变更历史。
- **展示方式**：复测次数作为系统自定义字段，支持在缺陷模板中按需配置展示（不强制固定位置）。

---

## 二、技术方案概览

| 方面 | 方案 | 说明 |
|------|------|------|
| 状态流转 | 修改 `IssueStatusTransitionService.TRANSITION_RULES` 增加 `closed -> reopened` | 权限模型天然支持（`reopened` 责任人为 `CREATOR`） |
| 复测次数存储 | 系统自定义字段（`custom_field` + `custom_field_issues`） | 复用现有自定义字段体系，避免改 issues 表 |
| 计数触发点 | `IssueStatusTransitionService.transitionStatus()` 内，`closed -> reopened` 时调用 `increaseRetestCount()` | 与状态更新同事务，保证一致性 |
| 历史记录 | 将复测次数变更作为 `FIELD_TYPE_CUSTOM` 追加到本次状态流转的审计日志（`SOURCE_STATUS`） | 与状态变更同一条日志，便于追溯 |
| 数据库迁移 | Flyway V18 插入系统字段定义（`system=1, global=1`） | 支持模板配置展示，懒初始化（首次流转时写入值） |

```java
// 在 transitionStatus() 内，状态更新成功后
List<IssueChangeLogDetailDTO> extraLogDetails = new ArrayList<>();
```

#### 3.1.3 increaseRetestCount() 方法（核心）
- 通过 `baseCustomFieldService.getCustomFieldByName(projectId, "复测次数")` 获取字段定义

- 计算 `newCount = (oldCount == null ? 0 : oldCount) + 1`
- 用 `customFieldIssuesService.editFields()` 写回
- 返回审计日志明细（`FIELD_TYPE_CUSTOM`），若字段不存在则返回 `null`（不阻断状态流转）

#### 3.1.4 日志兼容性修复
- 修复了 `LogUtil.warn/error` 参数签名，避免编译失败
- 使用 `LogUtil.error("msg", e)` 输出异常堆栈

## 四、关键设计决策与说明

### 4.1 为什么走系统自定义字段（而不是直接改 issues 表）
- **复用现有体系**：缺陷的“状态/严重程度”就是系统字段，前端/导出/查询逻辑天然支持
- **二开友好**：改动集中在 `IssueStatusTransitionService`，不侵入 domain/mapper/sql
- **可回滚**：升级时只需对照字段定义，不影响核心表结构

### 4.2 为什么“复测次数”仅统计 `closed -> reopened`
- **语义明确**：复测指“关闭后重新打开”，避免与 `resolved -> reopened` 混淆
- **业务对齐**：你确认的口径，便于后续扩展（如需统计其它场景可调整条件）

### 4.3 为什么懒初始化（不在迁移脚本批量初始化值）
- **避免全表扫描**：`issues` 表可能很大，批量写入风险高
- **按需写入**：只有真正经历“关闭->重开”的缺陷才会有值，符合业务语义

### 4.4 审计日志为什么要与状态变更合并
- **事务一致性**：同一次流转的多个变更应记录在同一条审计日志里
- **前端展示友好**：一次操作只显示一条历史记录，包含状态与复测次数变更

## 五、验证路径（建议）

1. **数据库检查**（防止重复字段）

2. **功能验证**
   - 创建缺陷 → `new` → `accepted` → `resolved` → `closed`
   - 在 `closed` 状态下，检查前端是否出现“重新打开”选项
   - 执行 `closed -> reopened`
   - 检查：
     - `custom_field_issues` 是否生成/更新“复测次数”值（首次应为 `1`）
     - 审计日志是否出现一条包含“状态”与“复测次数”变更的记录
     - 重复 `closed -> reopened`，复测次数是否递增为 `2`、`3`…

3. **模板配置验证**
   - 进入缺陷模板配置，检查“复测次数”字段是否可选
   - 配置展示后，在缺陷列表/编辑页验证字段是否正常显示

## 六、注意事项与风险点

- **数据库唯一性**：确保 `custom_field` 中没有重复的“复测次数”字段（迁移脚本已做 `WHERE NOT EXISTS`）
- **日志兼容性**：`LogUtil` 占位符调用需匹配参数个数（已修复）
- **权限一致性**：`reopened` 责任人为 `CREATOR`，与 `closed` 一致，因此 `closed -> reopened` 权限天然通过
- **回滚方案**：如需回滚，删除 `V18` 脚本插入的 `custom_field` 记录，并恢复 `TRANSITION_RULES` 的 `closed` 行即可
