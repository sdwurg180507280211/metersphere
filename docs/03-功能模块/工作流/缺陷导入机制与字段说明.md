# 缺陷导入机制与字段说明

## 一、背景与目的

- 我在做：整理 MeterSphere 缺陷导入（Excel）相关知识点：覆盖/不覆盖机制、支持哪些列（字段）以及常见注意事项。
- 目的是：让测试/研发在导入缺陷时“按规则填写”，避免导入后出现状态流转、字段解析、数据覆盖误操作等问题。
- 如果不这样做，就无法实现：团队统一导入规范，历史数据修复也容易走弯路（比如处理人填姓名导致权限异常）。

## 二、导入入口与关键参数

### 2.1 前端入口

- 我在做：明确前端导入弹窗与参数含义。
- 目的是：理解“覆盖/不覆盖”到底传给后端什么参数。
- 如果不这样做，就无法实现：准确解释覆盖机制，容易误解成“覆盖全部”。

前端组件：
- `test-track/frontend/src/business/issue/components/import/IssueImport.vue`

关键参数：
- `importType`：
  - `Update`：覆盖模式
  - `Create`：不覆盖模式

请求：
- `POST /issues/import`
- 表单参数（request part）：`IssueImportRequest`（包含 `projectId/workspaceId/userId/importType`）

后端入口：
- `IssuesController.issueImport()` → `IssuesService.issueImport()`

## 三、覆盖/不覆盖机制（核心）

### 3.1 匹配依据：Excel 的“ID列”

- 我在做：说明系统如何判断“这条缺陷是否已存在”。
- 目的是：解释为什么覆盖模式必须带正确的 ID。
- 如果不这样做，就无法实现：可控的更新，导入会变成“新建一堆重复缺陷”。

后端逻辑在：
- `IssueExcelListener.invoke()`

匹配规则：
- 取 Excel 的 `ID` 列（缺陷编号） → 映射到 `IssueExcelData.num`
- 通过 `issuesService.checkIssueExist(num, projectId)` 在同项目内查找是否存在

匹配结果：
- `num` 为空：直接视为新增
- `num` 非空但数据库不存在：新增
- `num` 非空且数据库存在：根据 `importType` 决定更新或跳过

### 3.2 覆盖模式（Update）

- 我在做：总结 Update 的行为边界。
- 目的是：让你确认该功能“可用且可预期”。
- 如果不这样做，就无法实现：用覆盖模式安全修复历史数据。

行为：
- 若 Excel 的 ID 在库中存在：更新该缺陷
- 若 Excel 的 ID 在库中不存在：新增该缺陷

代码要点（`IssueExcelListener.invoke()`）：
- `request.importType == Update` 时：
  - 将 `issueExcelData.id` 设置为已存在缺陷的真实 `issues.id`
  - 进入 `updateList`
  - 最终走 `issuesService.updateImportData()` → `updateIssues()`

### 3.3 不覆盖模式（Create）

- 我在做：总结 Create 的行为边界。
- 目的是：避免误覆盖线上数据。
- 如果不这样做，就无法实现：用“不覆盖”做增量导入。

行为：
- 若 Excel 的 ID 在库中存在：跳过更新（不会覆盖）
- 若 Excel 的 ID 在库中不存在：新增

代码要点：
- `request.importType != Update`（即 Create）时：
  - 仍会把该行加入 `updateList`，但 `updateFlag = false`
  - `saveData()` 时会 `filter(IssueExcelData::getUpdateFlag)`，因此不会执行更新

### 3.4 覆盖/不覆盖对“历史数据”的建议用法

- 我在做：给出最推荐的修复路径。
- 目的是：在不丢失历史记录的情况下修复字段（例如处理人格式）。
- 如果不这样做，就无法实现：保留评论/附件/流转历史，只修复字段值。

推荐：
- 历史缺陷字段有误（如处理人格式、人名/ID混用） → **优先使用覆盖模式（Update）**
- 原因：覆盖模式会更新字段，但不会删除缺陷，也不会丢失缺陷关联数据（评论/附件/日志等通常仍保留）

## 四、缺陷导入支持哪些列（字段）

> 结论：缺陷导入模板 = **基础列（系统内置） + 自定义字段列（模板配置/插件字段）**。

### 4.1 基础列（模板固定列）

- 我在做：从代码枚举与解析逻辑抽取“模板固定列”。
- 目的是：明确 Excel 必须/可以填写哪些基础信息。
- 如果不这样做，就无法实现：给团队一个可执行的导入表头规范。

基础列来自：
- `IssueImportField`（当 `request == null` 且非第三方模板时用于生成基础列头）
- `IssueExportHeadField`（用于解析/匹配部分标准列，如 ID、平台状态等）

当前代码中（`IssueImportField`）启用的基础列只有：
- `缺陷标题`（TITLE，必填）
- `缺陷描述`（DESCRIPTION，必填）

说明：
- `IssueImportField` 里曾存在 `状态/处理人/严重程度` 的定义，但目前是注释掉的（不再作为“固定基础列”输出）。

另外，导入模板依然会包含：
- `ID` 列（用于覆盖/不覆盖匹配）
  - 由模板生成逻辑中的表头组合决定（见 4.3）

### 4.2 自定义字段列（推荐重点关注）

- 我在做：解释为什么导入列“看起来很多”，但本质是项目自定义字段。
- 目的是：让你知道哪些列是“随项目模板变化”的。
- 如果不这样做，就无法实现：跨项目复用 Excel 模板，容易导入失败。

来源：
- `IssuesService.issueImportTemplate()` 会读取项目缺陷模板的自定义字段：
  - `IssueTemplateDao.issueTemplate.getCustomFields()`
  - 以及插件字段 `getPluginCustomFields(projectId)`
  - TAPD 还会额外追加处理人字段（`buildTapdUserCustomField`）

模板表头：
- `IssueExcelData.getHead()` 在 `request == null` 时，会在基础列后追加每个 `customFields` 的 `name` 作为表头列名。

因此：
- 你看到的“处理人/状态/严重程度/优先级/模块/版本/标签……”等，都可能是自定义字段列。
- **同一套 Excel 在不同项目可能列不一致**（因为项目缺陷模板不同）。

### 4.3 模板表头生成机制（为什么模板里会有 ID/标题/描述 + 自定义字段）

- 我在做：把模板生成链路写清楚。
- 目的是：解释“模板到底长什么样”为何由后端决定。
- 如果不这样做，就无法实现：精准定位“支持列”来源。

核心代码：
- `IssuesService.issueImportTemplate(projectId, response)`
  - 拉取项目成员列表（用于 MEMBER 字段提示）
  - 拉取自定义字段/插件字段
  - 调用：`new IssueExcelDataFactory().getIssueExcelDataLocal().getHead(isThirdTemplate, customFields, null)` 生成 `heads`
  - 通过 `IssueTemplateHeadWriteHandler` 写入表头批注（必填、格式、可选项等提示）

## 五、字段解析与格式要求（导入时怎么把单元格变成值）

### 5.1 标准列解析

- 我在做：说明导入时识别标题/描述/ID的依据。
- 目的是：避免表头改名导致解析失败。
- 如果不这样做，就无法实现：稳定导入。

解析逻辑在 `IssueExcelListener.parseDataToModel()`：
- `ID`：根据表头名等于 `IssueExportHeadField.ID.getName()`（显示名）来识别，然后 `Integer.parseInt`。
- `标题`：根据 `IssueExportHeadField.TITLE.getId()`（`title`）来识别。
- `描述`：根据 `IssueExportHeadField.DESCRIPTION.getId()`（`description`）来识别。

### 5.2 自定义字段解析

- 我在做：说明自定义字段怎么解析（分隔符、数组、JSON）。
- 目的是：让业务填写 Excel 时知道如何填多选/多成员。
- 如果不这样做，就无法实现：减少导入错误。

规则：
- 单元格包含 `,` 或 `;`：会拆分为列表，并把每项包装成 `"xxx"`，形成类似数组的结构。
- 否则按字符串处理。

在构建自定义字段请求时（`buildFields`）：
- 多选/复选框、级联等，会把单值转为类似 `[...]` 的形式后再解析。

## 六、重点注意事项（强制）

### 6.1 “处理人”务必填写用户ID

- 我在做：强调最容易踩坑的字段。
- 目的是：避免导入后状态流转权限异常。
- 如果不这样做，就无法实现：导入缺陷像新建缺陷一样正常流转。

原因：
- 状态流转权限判断依赖处理人字段的用户ID（系统期望 JSON 数组形式 `['用户ID']`）。
- 历史问题就是因为导入时处理人写了姓名，导致权限校验失败。

实践建议：
- 导入模板里处理人列：填写用户ID
- 若已导入错误：使用“覆盖模式（Update）”导入修复后的 Excel 更新字段

### 6.2 覆盖模式的风险提示

- 我在做：说明覆盖的风险。
- 目的是：避免误覆盖用户在页面上做过的修改。
- 如果不这样做，就无法实现：安全上线使用。

风险：
- 覆盖模式会更新 Excel 中出现的字段（包括自定义字段）。
- 若 Excel 是旧数据导出的，可能把线上最新修改覆盖回旧值。

建议：
- 覆盖前先导出最新数据作为基线
- 小批量先试（例如10条）
