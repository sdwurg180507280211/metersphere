# 复测次数字段设计方案对比分析

## 问题背景

缺陷有一个"复测次数"固定属性，随着状态流转 `closed -> reopened`，复测次数自动 +1。

**核心约束**：
- 无论通过什么手段新增缺陷（UI、API、导入），复测次数初始值都必须是 0
- 即使导入时指定了复测次数不为 0，也应该强制重置为 0
- 只有通过状态流转 `closed -> reopened` 才能增加

**设计选择**：
1. 方案A：写死在 `issues` 表中（新增 `retest_count` 字段）
2. 方案B：作为全局系统模版字段（现有实现）

---

## 方案对比

### 方案A：写死在 issues 表中

#### 实现方式
```sql
-- 数据库迁移
ALTER TABLE issues ADD COLUMN retest_count INT DEFAULT 0 NOT NULL COMMENT '复测次数';
```

```java
// Issues.java 实体类
@Data

    private String id;
    private String title;
    // ... 其他字段
    private Integer retestCount;  // 新增
}
```

#### 优点

| 优点 | 说明 | 重要性 |
|------|------|--------|
| **性能最优** | 直接查询，无需 JOIN `custom_field_issues` 表 | ⭐⭐⭐⭐⭐ |
| **查询简单** | `SELECT retest_count FROM issues WHERE id=?` | ⭐⭐⭐⭐⭐ |
| **索引友好** | 可直接在 `retest_count` 上建索引，支持高效排序和筛选 | ⭐⭐⭐⭐⭐ |
| **类型安全** | 数据库层面强制 INT 类型，不会出现类型转换问题 | ⭐⭐⭐⭐ |
| **默认值保证** | `DEFAULT 0 NOT NULL` 确保新增缺陷自动为 0 | ⭐⭐⭐⭐⭐ |
| **导入控制简单** | 在 `insertIssues()` 中强制设置 `issues.setRetestCount(0)` | ⭐⭐⭐⭐ |
| **统计方便** | `SELECT AVG(retest_count), MAX(retest_count) FROM issues` | ⭐⭐⭐⭐ |
| **导出简单** | 直接映射到 Excel 列，无需特殊处理 | ⭐⭐⭐ |

#### 缺点

| 缺点 | 说明 | 影响 |
|------|------|------|
| **改动核心表** | 修改 `issues` 表结构，影响所有查询 | ⭐⭐⭐⭐ |
| **二开不友好** | 官方升级时需要手动迁移该字段 | ⭐⭐⭐⭐⭐ |
| **不可配置** | 无法通过模版控制是否显示 | ⭐⭐ |
| **前端改动大** | 需要修改所有涉及 Issues 的前端组件 | ⭐⭐⭐ |
| **回滚困难** | 一旦上线，回滚需要删除列（风险高） | ⭐⭐⭐⭐ |

### 方案B：全局系统模版字段（现有实现）

#### 实现方式
```sql
-- 数据库迁移
INSERT INTO custom_field (id, name, scene, type, remark, system, global, create_time, update_time, project_id, third_part)
VALUES (UUID(), '复测次数', 'ISSUE', 'int', '缺陷关闭后重新打开的次数', 1, 1, UNIX_TIMESTAMP() * 1000, UNIX_TIMESTAMP() * 1000, NULL, 0)
WHERE NOT EXISTS (SELECT 1 FROM custom_field WHERE name = '复测次数' AND scene = 'ISSUE' AND system = 1);
```

#### 优点

| 优点 | 说明 | 重要性 |
|------|------|--------|
| **二开友好** | 不改 `issues` 表，升级时只需对照字段定义 | ⭐⭐⭐⭐⭐ |
| **改动面小** | 只修改 `IssueStatusTransitionService`，不侵入核心链路 | ⭐⭐⭐⭐⭐ |
| **可回滚** | 删除 `custom_field` 记录即可回滚 | ⭐⭐⭐⭐⭐ |
| **复用现有体系** | 自定义字段的查询、导出、显示逻辑天然支持 | ⭐⭐⭐⭐ |
| **模版可配置** | 可以在模版中控制是否显示、是否必填 | ⭐⭐⭐ |
| **前端改动小** | 前端已有自定义字段渲染逻辑，无需额外开发 | ⭐⭐⭐⭐ |
| **扩展性好** | 未来可以增加其他系统字段，不影响核心表 | ⭐⭐⭐⭐ |

#### 缺点

| 缺点 | 说明 | 影响 |
|------|------|------|
| **性能较差** | 需要 JOIN `custom_field_issues` 表 | ⭐⭐⭐⭐ |
| **查询复杂** | 需要通过 `custom_field_issues` 关联查询 | ⭐⭐⭐ |
| **索引不友好** | 无法直接在 `value` 字段上建索引（TEXT 类型） | ⭐⭐⭐⭐ |
| **类型转换** | 存储为 JSON 字符串，需要手动转换为 INT | ⭐⭐⭐ |
| **默认值控制复杂** | 需要在多个地方确保初始值为 0 | ⭐⭐⭐⭐ |
| **导入控制复杂** | 需要在 `buildFields()` 中特殊处理，忽略用户输入 | ⭐⭐⭐⭐⭐ |
| **统计困难** | `SELECT AVG(CAST(value AS SIGNED)) FROM custom_field_issues` | ⭐⭐⭐ |

## 核心问题：如何确保初始值为0

### 方案A（表字段）

```sql
-- ✅ 数据库层面保证
ALTER TABLE issues ADD COLUMN retest_count INT DEFAULT 0 NOT NULL;
```

**优点**：
- 数据库层面强制约束，任何插入都会自动设置为 0
- 无需在代码中多处检查

**缺点**：
- 无法阻止用户通过 SQL 直接修改

### 方案B（自定义字段）

**需要在多个入口确保初始值为0**：

| 入口 | 处理方式 | 风险 |
|------|----------|------|
| UI新增 | `IssuesService.addIssues()` 中调用 `ensureRetestCountField()` | ⭐⭐ |
| API新增 | 同上，需要在 `addIssues()` 中强制检查 | ⭐⭐⭐ |
| Excel导入 | `IssueExcelListener.buildFields()` 中特殊处理 | ⭐⭐⭐⭐⭐ |
| 第三方平台同步 | `AbstractIssuePlatform.addIssue()` 中需要检查 | ⭐⭐⭐ |
| 数据库直接插入 | 无法阻止（需要依赖应用层逻辑） | ⭐⭐⭐⭐⭐ |

**问题**：
- **导入时最容易出问题**：用户可能在Excel中填写复测次数，需要强制忽略
- **API调用风险**：恶意用户可能通过API传入非0值
- **第三方同步风险**：从Jira等平台同步时，可能带入错误的值

---

## 推荐方案

### 结论：**方案B（全局系统模版字段）更优**

虽然方案A在性能和简单性上有优势，但考虑到：

1. **二开原则**：MeterSphere是二次开发项目，改动核心表会导致升级困难
2. **可维护性**：方案B改动集中，易于回滚
3. **扩展性**：未来可能有更多系统字段（如"修复次数"、"验证次数"等）

**但需要解决方案B的核心问题：确保初始值为0**

## 方案B的完善措施

### 1. 导入时强制忽略用户输入（最重要）

```java
// IssueExcelListener.buildFields()
private void buildFields(IssueExcelData issueExcelData, IssuesUpdateRequest issuesUpdateRequest) {
    Boolean addFlag = issueExcelData.getAddFlag();
    List<CustomFieldResourceDTO> addFields = new ArrayList<>();
    List<CustomFieldResourceDTO> editFields = new ArrayList<>();
    List<CustomFieldItemDTO> requestFields = new ArrayList<>();
    
    Map<String, List<CustomFieldDao>> customFieldMap = customFields.stream()
        .collect(Collectors.groupingBy(CustomFieldDao::getName));
    
    // ✅ 遍历模版的所有字段
    for (CustomFieldDao customFieldDao : customFields) {
        String fieldName = customFieldDao.getName();
        String type = customFieldDao.getType();
        
        // ✅ 特殊处理：复测次数强制为0，完全忽略Excel输入
        if (StringUtils.equals(fieldName, SystemCustomField.ISSUE_RETEST_COUNT)) {
            CustomFieldResourceDTO resource = new CustomFieldResourceDTO();
            resource.setFieldId(customFieldDao.getId());
            resource.setResourceId(issuesUpdateRequest.getId());
            resource.setValue("0");  // 强制为0
            
            CustomFieldItemDTO itemDTO = new CustomFieldItemDTO();
            BeanUtils.copyBean(itemDTO, customFieldDao);
            itemDTO.setValue(0);
            
            if (addFlag) {
                addFields.add(resource);
            } else {
                // 更新时也强制为0（防止覆盖模式修改）
                editFields.add(resource);
            }
            requestFields.add(itemDTO);
            continue;  // ✅ 跳过，不从Excel读取
        }
        
        // ... 其他字段正常处理
    }
    
    if (addFlag) {
        issuesUpdateRequest.setAddFields(addFields);
    } else {
        issuesUpdateRequest.setEditFields(editFields);
    }
    issuesUpdateRequest.setRequestFields(requestFields);
}
```

### 2. UI/API新增时强制初始化

```java
// IssuesService.addIssues()
public IssuesWithBLOBs addIssues(IssuesUpdateRequest issuesRequest, List<MultipartFile> files) {
    // ... 现有逻辑
    
    // ✅ 在保存自定义字段前，强制确保复测次数为0
    ensureRetestCountFieldIsZero(issuesRequest);
    
    customFieldIssuesService.addFields(issuesId, issuesRequest.getAddFields());
    // ...
}

/**
 * 确保复测次数字段存在且为0
 * 目的：防止通过API传入非0值
 */
private void ensureRetestCountFieldIsZero(IssuesUpdateRequest request) {
    CustomField retestCountField = baseCustomFieldService.getCustomFieldByName(
        request.getProjectId(), SystemCustomField.ISSUE_RETEST_COUNT);
    
    if (retestCountField == null) {
        return;  // 字段不存在，跳过
    }
    
    String fieldId = retestCountField.getId();
    
    // 检查 addFields 中是否已有复测次数
    Optional<CustomFieldResourceDTO> existing = request.getAddFields().stream()
        .filter(f -> f.getFieldId().equals(fieldId))
        .findFirst();
    
    if (existing.isPresent()) {
        // 存在则强制重置为0
        existing.get().setValue("0");
    } else {
        // 不存在则添加，初始值为0
        CustomFieldResourceDTO resource = new CustomFieldResourceDTO();
        resource.setFieldId(fieldId);
        resource.setValue("0");
        request.getAddFields().add(resource);
    }
}
```

### 3. 更新时禁止修改复测次数

```java
// IssuesService.updateIssues()
public IssuesWithBLOBs updateIssues(IssuesUpdateRequest issuesRequest) {
    // ... 现有逻辑
    
    // ✅ 更新时移除复测次数字段（防止用户通过API修改）
    removeRetestCountFromEditFields(issuesRequest);
    
    customFieldIssuesService.editFields(issuesRequest.getId(), issuesRequest.getEditFields());
    // ...
}

/**
 * 从编辑字段中移除复测次数
 * 目的：复测次数只能通过状态流转自动增加，不允许手动修改
 */
private void removeRetestCountFromEditFields(IssuesUpdateRequest request) {
    CustomField retestCountField = baseCustomFieldService.getCustomFieldByName(
        request.getProjectId(), SystemCustomField.ISSUE_RETEST_COUNT);
    
    if (retestCountField != null) {
        String fieldId = retestCountField.getId();
        // 移除复测次数字段
        request.getEditFields().removeIf(f -> f.getFieldId().equals(fieldId));
        request.getRequestFields().removeIf(f -> f.getId().equals(fieldId));
    }
}
```

## 性能优化建议（方案B）

虽然选择了方案B，但可以通过以下方式优化性能：

### 1. 添加虚拟列（MySQL 5.7+）

```sql
-- 在 custom_field_issues 表中添加虚拟列
ALTER TABLE custom_field_issues 
ADD COLUMN value_int INT GENERATED ALWAYS AS (
    CASE 
        WHEN value REGEXP '^[0-9]+$' THEN CAST(value AS SIGNED)
        ELSE NULL 
    END
) STORED;

-- 在虚拟列上建索引
CREATE INDEX idx_value_int ON custom_field_issues(value_int);
```

**优点**：
- 可以在整数值上建索引，支持高效排序和筛选
- 不改变现有数据结构

### 2. 缓存复测次数

```java
// 在 Issues 实体中添加瞬态字段
@Data
public class Issues implements Serializable {
    // ... 现有字段
    
    @Transient  // 不映射到数据库
    private Integer retestCount;  // 从 custom_field_issues 查询后缓存
}
```

### 3. 批量查询优化

```java
// ExtIssuesMapper.xml
<select id="getIssuesWithRetestCount" resultType="IssuesDao">
    SELECT 
        i.*,
        cfi.value AS retest_count
    FROM issues i
    LEFT JOIN custom_field_issues cfi 
        ON i.id = cfi.resource_id 
        AND cfi.field_id = (
            SELECT id FROM custom_field 
            WHERE name = '复测次数' AND scene = 'ISSUE' AND system = 1
            LIMIT 1
        )
    WHERE i.project_id = #{projectId}
</select>
```

**最终推荐：方案B（全局系统模版字段）**

**理由**：
1. 符合二次开发原则（改动面小、边界清晰、可回滚）
2. 复用现有自定义字段体系，前端无需额外开发
3. 未来扩展性好（可以增加更多系统字段）
4. 虽然性能略差，但可以通过虚拟列、缓存等方式优化

**关键措施**：
1. ✅ 导入时强制忽略用户输入（最重要）
2. ✅ UI/API新增时强制初始化为0
3. ✅ 更新时禁止修改复测次数
4. ✅ 前端禁用复测次数编辑
5. ✅ 只允许通过状态流转 `closed -> reopened` 自动增加
