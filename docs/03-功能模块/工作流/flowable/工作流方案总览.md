## 🎯 核心目标

在 MeterSphere 中引入 Flowable 工作流框架，实现：
- ✅ 测试计划审批流程
- ✅ 缺陷处理流程管理
- ✅ 可视化流程设计
- ✅ 灵活的审批规则配置

---

## 💡 为什么需要工作流？

### 当前痛点

```
❌ 测试计划创建后直接执行，缺少审批控制
❌ 缺陷状态变更是简单状态机，无法适应复杂规则
❌ 测试用例评审缺少正式流程
❌ 接口变更没有审批机制
```

### 引入工作流后

```
✅ 标准化的审批流程
✅ 可配置的审批规则
✅ 自动任务分配
✅ 完整的审批历史
✅ 超时自动提醒
```

**业务场景举例：**

```
测试计划审批流程：
─────────────────
创建测试计划 
  → 提交审批 
  → 技术评审（自动分配给技术主管）
  → 批准/驳回 
  → 执行测试

优势：
- 关键操作有审批控制
- 自动通知相关人员
- 完整的历史记录
- 支持多级审批
```

## 🔑 核心设计决策

### 决策 1: 独立部署工作流服务

**为什么？**
- ✅ 松耦合：不影响现有服务
- ✅ 可复用：所有模块都可以使用
- ✅ 独立演进：可以独立升级 Flowable
- ✅ 资源隔离：独立的 JVM

**对比：**
```
❌ 嵌入式：在每个服务中集成 Flowable
   问题：重复集成、耦合度高、难以维护

✅ 独立服务：workflow-service
   优势：单一职责、松耦合、易维护
```

### 决策 2: 独立数据库

**架构：**
```
metersphere_db         # 业务数据库
  ├── test_plan
  ├── test_case
  └── ...

metersphere_workflow   # 工作流数据库（独立）
  ├── act_*            # Flowable 表（30+ 张）
  └── ms_workflow_*    # 扩展表
```

**好处：**
- ✅ 清晰边界
- ✅ 独立备份和优化
- ✅ 安全隔离

### 决策 3: REST API + Kafka 混合通信

**场景分类：**
```
查询操作 → REST API（同步）
  - 查询待办任务
  - 查询流程状态

状态变更 → Kafka 消息（异步）
  - 流程完成通知
  - 任务分配通知
```

### 决策 4: 选择 Flowable 而非 Activiti

| 对比项 | Flowable | Activiti |
|--------|----------|----------|
| Spring Boot 3 支持 | ✅ 完美 | ❌ 不支持 |
| 功能完整性 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| 社区活跃度 | 活跃 | 一般 |
| 文档质量 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |

### 决策 5: 选择 bpmn-js 作为流程编辑器

**为什么？**
- ✅ 符合 BPMN 2.0 标准
- ✅ Flowable 官方推荐
- ✅ 可扩展性强
- ✅ 社区活跃

**对比自研：**
```
自研编辑器：
  ❌ 开发成本高（3-6 个月）
  ❌ 功能不完整
  ❌ 难以与 Flowable 集成

bpmn-js：
  ✅ 开箱即用（1-2 周集成）
  ✅ 标准化
  ✅ 可定制
```

## ⚠️ 主要风险

### 风险 1: 性能问题
```
问题：Flowable 表数据量大，查询慢
缓解：缓存 + 归档 + 索引优化
```

### 风险 2: 用户接受度
```
问题：审批流程增加操作步骤
缓解：渐进引入 + 快捷审批 + 充分培训
```

### 风险 3: 开发周期
```
问题：11 周可能延期
缓解：敏捷迭代 + MVP 优先 + 20% 缓冲
```
