## 一、业务场景分析

### 1.1 为什么需要引入工作流？

**当前 MeterSphere 的业务流程痛点：**

#### 场景 1: 测试审批流程
```
现状：测试计划的创建、执行、发布都是即时的
问题：
├── 缺乏审批控制（任何人都能直接执行测试）
├── 无法实现分级审批（如：技术主管 → 项目经理 → 质量部门）
└── 版本发布前缺少必要的卡点

引入工作流后：
测试计划创建 → 提交审批 → 技术评审 → 批准/驳回 → 执行测试
              ↓
         自动通知相关人员
         记录审批历史
         支持加签、转办
```

#### 场景 2: 缺陷处理流程
```
现状：缺陷状态变更是简单的状态机
问题：
├── 缺少复杂的流转规则（如：严重缺陷必须技术主管审批）
├── 无法自动分配处理人
├── 缺少超时提醒
└── 流程固化，难以适应不同项目需求

引入工作流后：
发现缺陷 → 自动分类 → 分配开发 → 修复 → 测试验证 → 关闭
                    ↓             ↓          ↓
                 可配置规则    超时提醒   自动通知
```

#### 场景 3: 测试用例评审流程
```
现状：用例评审通过简单的状态标记
问题：
├── 缺少正式的评审流程
├── 无法追溯评审意见
├── 评审结果无归档
└── 缺少多轮评审支持

引入工作流后：
用例编写 → 提交评审 → 专家评审 → 修改 → 再评审 → 通过
                    ↓                      ↓
                并行评审               版本管理
                意见汇总              历史追溯
```

#### 场景 4: 接口变更审批
```
现状：接口定义变更后直接生效
问题：
├── 缺少变更影响分析
├── 没有审批控制
├── 无法回滚
└── 上下游团队不知情

引入工作流后：
变更申请 → 影响分析 → 多方会签 → 通知相关方 → 执行变更
              ↓
         自动检测影响范围
         相关测试计划自动调整
```

### 1.2 业务价值

**1. 合规性管理**
- 满足企业级质量管理体系要求（如 ISO 9001、CMMI）
- 关键操作留痕，可追溯
- 支持审计

**2. 流程标准化**
- 统一的审批流程
- 减少人为错误
- 提高团队协作效率

**3. 灵活性**
- 不同项目可定制不同流程
- 流程可视化配置
- 支持流程优化迭代

**4. 自动化**
- 自动任务分配
- 超时自动提醒
- 流程状态自动通知

## 二、技术方案设计

### 2.1 Flowable vs Activiti vs Camunda

**对比分析：**

| 维度 | Flowable | Activiti | Camunda |
|------|----------|----------|---------|
| **开源协议** | Apache 2.0 | Apache 2.0 | Apache 2.0 |
| **活跃度** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| **Spring Boot 3支持** | ✅ 完美支持 | ❌ 不支持 | ⚠️ 部分支持 |
| **BPMN 2.0** | ✅ 完整 | ✅ 完整 | ✅ 完整 |
| **DMN决策引擎** | ✅ | ❌ | ✅ |
| **CMMN案例管理** | ✅ | ❌ | ✅ |
| **UI组件** | ✅ 完整 | ⚠️ 基础 | ✅ 完整 |
| **社区支持** | 活跃 | 一般 | 活跃 |
| **文档质量** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |

**推荐：Flowable**

**理由：**
1. **完美兼容 Spring Boot 3.2.12**（MeterSphere 当前版本）
2. **功能最完整**：BPMN + DMN + CMMN 全支持
3. **社区活跃**：持续更新，问题响应快
4. **丰富的 UI 组件**：提供现成的流程设计器
5. **企业级特性**：任务分配、超时处理、事件监听等

### 2.2 Flowable 核心概念

```
Flowable 架构
├── ProcessEngine（流程引擎）
│   ├── RepositoryService（流程定义管理）
│   ├── RuntimeService（流程实例管理）
│   ├── TaskService（任务管理）
│   ├── HistoryService（历史数据）
│   ├── ManagementService（引擎管理）
│   └── IdentityService（用户组管理）
│
├── BPMN 2.0（业务流程建模）
│   ├── 事件（开始、结束、中间事件）
│   ├── 任务（用户任务、服务任务）
│   ├── 网关（排他、并行、包容网关）
│   └── 子流程
│
├── DMN（决策管理）
│   └── 决策表（规则引擎）
│
└── 表单引擎
    └── 动态表单渲染
```

**关键概念映射到 MeterSphere：**

```
Flowable 概念          →  MeterSphere 业务
────────────────────────────────────────────
ProcessDefinition      →  测试审批流程模板
ProcessInstance        →  具体的测试计划审批实例
Task                   →  审批任务、评审任务
User                   →  MeterSphere 用户
Group                  →  MeterSphere 用户组/角色
Variable               →  流程变量（测试计划ID、缺陷信息等）
```

### 3.3 服务间通信设计

**方式 1: REST API（同步调用）**
```
场景：工作流服务需要查询测试计划信息

workflow-service → HTTP → test-track-service
                    ↓
              获取测试计划详情
              验证权限
              返回数据
```

**方式 2: Kafka 消息（异步解耦）**
```
场景：流程状态变更需要通知相关模块

workflow-service
    ↓ 发送消息
Kafka Topic: workflow.status.changed
    ↓ 订阅消息
test-track-service
    └→ 更新测试计划状态
workstation-service
    └→ 更新用户待办任务
```

**方式 3: Dubbo RPC（高性能调用）**
```
场景：高频调用的核心接口

workflow-service
    ↓ Dubbo RPC
user-service.getUserInfo()
    ↓
返回用户信息（用于任务分配）
```

**推荐方案：**
- **查询操作**: REST API（简单、易调试）
- **状态变更**: Kafka 消息（解耦、异步）
- **高频调用**: Dubbo RPC（性能优）

### 3.4 数据库设计策略

**选项 1: 共享数据库（不推荐）**
```
问题：
├── 耦合度高
├── Flowable 表和业务表混在一起
├── 升级困难
└── 事务边界模糊
```

**选项 2: 独立数据库（推荐）**
```
架构：
metersphere_db         # MeterSphere 业务数据库
  ├── test_plan
  ├── test_case
  └── ...

metersphere_workflow   # Flowable 工作流数据库
  ├── ACT_*            # Flowable 核心表（30+ 张）
  └── ms_workflow_*    # 扩展表

优势：
✅ 松耦合
✅ 独立升级
✅ 清晰的边界
✅ 更好的性能优化
```

**扩展表设计：**
```sql
-- 流程模板表（业务流程与 Flowable 流程定义的映射）
CREATE TABLE ms_workflow_template (
    id VARCHAR(50) PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    business_type VARCHAR(50) NOT NULL,  -- TEST_PLAN, DEFECT, API_CHANGE
    process_definition_key VARCHAR(100),
    version INT,
    enabled BOOLEAN DEFAULT TRUE,
    project_id VARCHAR(50),
    created_by VARCHAR(50),
    created_at TIMESTAMP
);

-- 流程实例业务关联表
CREATE TABLE ms_workflow_instance (
    id VARCHAR(50) PRIMARY KEY,
    process_instance_id VARCHAR(64),     -- Flowable 流程实例ID
    business_id VARCHAR(50) NOT NULL,    -- 业务ID（如测试计划ID）
    business_type VARCHAR(50) NOT NULL,
    current_task_id VARCHAR(64),
    current_assignee VARCHAR(50),
    status VARCHAR(20),                  -- RUNNING, COMPLETED, CANCELLED
    started_at TIMESTAMP,
    completed_at TIMESTAMP
);

-- 任务扩展信息表
CREATE TABLE ms_workflow_task_ext (
    id VARCHAR(50) PRIMARY KEY,
    task_id VARCHAR(64) NOT NULL,       -- Flowable 任务ID
    business_data JSON,                  -- 业务数据快照
    approval_comment TEXT,
    approval_time TIMESTAMP,
    attachments JSON                     -- 附件信息
);
```

## 四、前端流程编辑器方案

### 4.1 流程编辑器选型

**对比分析：**

| 编辑器 | 优势 | 劣势 | 推荐度 |
|--------|------|------|--------|
| **bpmn-js** | - Flowable 官方推荐<br>- BPMN 2.0 标准<br>- 开源免费<br>- 功能完整 | - 学习曲线陡<br>- 样式定制复杂 | ⭐⭐⭐⭐⭐ |
| **Vue Flow** | - Vue 生态<br>- 简单易用<br>- 可视化友好 | - 非 BPMN 标准<br>- 需要转换 | ⭐⭐⭐ |
| **Flowable Modeler** | - Flowable 官方UI<br>- 开箱即用 | - 不易定制<br>- 与 Vue 集成复杂 | ⭐⭐⭐⭐ |

**推荐方案：bpmn-js + 定制化封装**

**理由：**
1. **标准化**：完全符合 BPMN 2.0 标准
2. **可扩展**：支持自定义元素、属性面板
3. **社区支持**：文档完善，案例丰富
4. **与 Flowable 无缝集成**：生成的 XML 可直接部署

### 5.2 状态同步机制

**问题：** 业务状态 vs 流程状态

```
场景：测试计划的状态
├── 业务状态（MeterSphere）：
│   ├── DRAFT（草稿）
│   ├── PREPARED（准备中）
│   ├── RUNNING（执行中）
│   └── COMPLETED（已完成）
│
└── 流程状态（Flowable）：
    ├── RUNNING（流程进行中）
    ├── COMPLETED（流程已完成）
    └── CANCELLED（流程已取消）
```

**同步策略：**

```
方案 1: 流程驱动业务状态（推荐）
───────────────────────────────
流程状态变更 → 触发监听器 → 更新业务状态

优势：
✅ 流程是真相源
✅ 状态一致性强
✅ 便于审计

实现：
1. 在流程节点配置监听器
2. 监听器通过 REST API 更新业务状态
3. 业务状态变更记录到流程变量

方案 2: 双向同步（不推荐）
─────────────────────────
业务状态 ←→ 流程状态

问题：
❌ 容易产生冲突
❌ 复杂度高
❌ 难以维护
```

#### 风险 1: 性能问题
```
问题：
├── Flowable 表数据量大（30+ 张表）

✅ 异步处理非关键路径
```

```
问题：
工作流事务 vs 业务事务
├── 流程启动成功，业务操作失败
└── 业务操作成功，流程启动失败

解决方案：
方案 1: Saga 模式（推荐）
  └── 补偿机制，保证最终一致性

方案 2: 两阶段提交
  └── 强一致性，但性能差

方案 3: 消息队列
  └── 异步补偿
```

#### 风险 3: 版本兼容性
```
问题：
├── Spring Boot 3 与 Flowable 版本兼容
├── Vue 2 vs Vue 3 的 bpmn-js 集成
└── 与现有微服务的兼容

缓解措施：
✅ 使用最新稳定版 Flowable (7.0+)
✅ 充分的兼容性测试
✅ 预留适配层
```

```java
public void completeTask(String taskId, String userId) {

        .taskId(taskId)

        .singleResult();

    taskService.complete(taskId);
}
```

```xml
<!-- 推荐的依赖版本 -->
<properties>
    <flowable.version>7.0.1</flowable.version>
    <spring-boot.version>3.2.12</spring-boot.version>
    <mybatis.version>3.0.3</mybatis.version>
</properties>

<dependencies>
    <!-- Flowable 核心 -->
    <dependency>
        <groupId>org.flowable</groupId>
        <artifactId>flowable-spring-boot-starter-process</artifactId>
        <version>${flowable.version}</version>
    </dependency>
    
    <!-- Flowable UI（可选）-->
    <dependency>
        <groupId>org.flowable</groupId>
        <artifactId>flowable-spring-boot-starter-ui-modeler</artifactId>
        <version>${flowable.version}</version>
    </dependency>
</dependencies>
```
