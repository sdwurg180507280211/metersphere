# 缺陷管理状态流转完整文档

## 一、状态定义

系统支持8种缺陷状态：

1. **新建 (new)** - 创建缺陷时的初始状态
2. **已接收 (accepted)** - 处理人接收缺陷
3. **已解决 (resolved)** - 处理人解决缺陷
4. **关闭 (closed)** - 创建人验证关闭
5. **重新打开 (reopened)** - 创建人重新打开缺陷
6. **无效 (invalid)** - 处理人判断无效
7. **已取消 (cancelled)** - 创建人取消缺陷（终止状态）
8. **挂起 (on_hold)** - 处理人挂起缺陷

## 二、状态流转规则

状态流转遵循以下规则（定义在 `IssueStatusTransitionService.TRANSITION_RULES`）：

| 当前状态 | 可流转到的状态 |
|---------|--------------|
| 新建 (new) | 已接收 (accepted) |
| 已接收 (accepted) | 已解决 (resolved)、无效 (invalid)、挂起 (on_hold) |
| 已解决 (resolved) | 关闭 (closed)、重新打开 (reopened) |
| 无效 (invalid) | 已取消 (cancelled)、重新打开 (reopened) |
| 挂起 (on_hold) | 已解决 (resolved) |
| 重新打开 (reopened) | 已接收 (accepted) |
| 关闭 (closed) | 重新打开 (reopened) |
| 已取消 (cancelled) | 无流转（终止状态） |

**状态流转路径总结**：
- 正常流程：new → accepted → resolved → closed
- 无效流程：new → accepted → invalid → cancelled
- 挂起流程：new → accepted → on_hold → resolved → closed
- 重新打开流程：resolved/invalid → reopened → accepted

## 二点一、复测次数

复测次数是缺陷的系统自定义字段（名称：**复测次数**），用于统计缺陷被测试人员复测的次数。

### 统计口径

- 仅当状态从 **关闭 (closed)** 流转到 **重新打开 (reopened)** 时，复测次数 +1。
- 其它场景（例如 resolved -> reopened、invalid -> reopened）不计入复测次数。

### 展示方式

- 复测次数作为系统自定义字段落库，可在缺陷模板中按需配置是否展示（走模板配置）。

## 三、状态责任人分配

每个状态都有明确的责任人（定义在 `IssueStatusTransitionService.STATUS_OWNER`）：

### 创建人负责的状态（测试人员）
- **新建 (new)**: 创建缺陷时的初始状态
- **关闭 (closed)**: 测试人员验证关闭
- **重新打开 (reopened)**: 测试人员重新打开
- **已取消 (cancelled)**: 测试人员取消

### 处理人负责的状态（开发人员）
- **已接收 (accepted)**: 开发人员接收缺陷
- **已解决 (resolved)**: 开发人员解决缺陷
- **无效 (invalid)**: 开发人员判断无效
- **挂起 (on_hold)**: 开发人员挂起

## 四、权限判断逻辑

### 核心原则

**权限判断基于目标状态的责任人，而非当前状态的责任人。**

系统检查当前状态可以流转到的所有目标状态，如果所有目标状态都由同一个人负责，那么应该由那个人来编辑。

### 判断流程

1. 获取当前状态可流转到的目标状态列表
2. 检查所有目标状态的责任人类型
3. 如果所有目标状态都由同一责任人类型负责，则只有该责任人可以编辑
4. 如果是终止状态（无可流转状态），则检查当前状态的责任人

### 示例说明

**示例1：已解决状态**
- 当前状态：已解决（处理人负责）
- 可流转到：关闭、重新打开（都由创建人负责）
- 权限判断：所有目标状态都由创建人负责 → **创建人可以编辑，处理人不能编辑**

**示例2：新建状态**
- 当前状态：新建（创建人负责）
- 可流转到：已接收（处理人负责）
- 权限判断：所有目标状态都由处理人负责 → **处理人可以编辑，创建人不能编辑**

**示例3：无效状态**
- 当前状态：无效（处理人负责）
- 可流转到：已取消、重新打开（都由创建人负责）
- 权限判断：所有目标状态都由创建人负责 → **创建人可以编辑，处理人不能编辑**

**示例4：已接收状态**
- 当前状态：已接收（处理人负责）
- 可流转到：已解决、无效、挂起（都由处理人负责）
- 权限判断：所有目标状态都由处理人负责 → **处理人可以编辑，创建人不能编辑**

**示例5：关闭状态**
- 当前状态：关闭（创建人负责）
- 可流转到：重新打开（创建人负责）
- 权限判断：目标状态责任人均为创建人 → **创建人可以编辑并重新打开**

## 五、核心业务流程

### 5.1 正常流程

1. **创建缺陷**: 创建人创建缺陷 → 状态为"新建"
2. **接收缺陷**: 处理人将"新建"流转到"已接收"
3. **处理缺陷**: 处理人将"已接收"流转到"已解决"、"无效"或"挂起"
4. **验证关闭**:
    - 如果状态为"已解决" → 创建人验证，流转到"关闭"或"重新打开"
    - 如果状态为"无效" → 创建人决定，流转到"已取消"或"重新打开"
5. **重新处理**: 如果重新打开，流转回"已接收"，由处理人继续处理

### 5.2 挂起流程

1. 处理人将"已接收"流转到"挂起"
2. 处理人将"挂起"流转到"已解决"
3. 后续流程同正常流程

## 六、状态流转代码原理

### 6.1 核心架构

状态流转功能的核心实现在 `IssueStatusTransitionService` 类中，采用**规则引擎模式**，通过静态配置定义流转规则和责任人，实现灵活的状态流转控制。

### 6.2 核心数据结构

#### 6.2.1 状态流转规则（TRANSITION_RULES）

```java
// 定义在 IssueStatusTransitionService 中
private static final Map<String, List<String>> TRANSITION_RULES = new HashMap<String, List<String>>() {{
    put("new", Arrays.asList("accepted"));  // 新建只能流转到已接收
    put("accepted", Arrays.asList("resolved", "invalid", "on_hold"));  // 已接收可以流转到已解决、无效、挂起
    put("resolved", Arrays.asList("closed", "reopened"));  // 已解决可以流转到关闭、重新打开
    put("invalid", Arrays.asList("cancelled", "reopened"));  // 无效可以流转到已取消、重新打开
    put("on_hold", Arrays.asList("resolved"));  // 挂起只能流转到已解决
    put("reopened", Arrays.asList("accepted"));  // 重新打开只能流转到已接收
    put("closed", Arrays.asList("reopened"));  // 关闭可以由创建人重新打开
    // cancelled 是终止状态，无流转
}};
```

**设计原理**：
- 使用 `Map<当前状态, List<可流转状态>>` 结构，清晰定义状态流转关系
- 终止状态（cancelled）不在规则中，表示无后续流转
- 规则定义在类加载时初始化，性能高效

#### 6.2.2 状态责任人定义（STATUS_OWNER）

```java
// 定义在 IssueStatusTransitionService 中
private static final String CREATOR = "CREATOR";    // 创建人类型
private static final String PROCESSOR = "PROCESSOR"; // 处理人类型

private static final Map<String, String> STATUS_OWNER = new HashMap<String, String>() {{
    put("new", CREATOR);           // 创建人负责：测试人员创建缺陷
    put("closed", CREATOR);        // 创建人负责：测试人员验证关闭
    put("reopened", CREATOR);      // 创建人负责：测试人员重新打开
    put("cancelled", CREATOR);     // 创建人负责：测试人员取消
    put("accepted", PROCESSOR);    // 处理人负责：开发人员接收缺陷
    put("resolved", PROCESSOR);    // 处理人负责：开发人员解决缺陷
    put("invalid", PROCESSOR);     // 处理人负责：开发人员判断无效
    put("on_hold", PROCESSOR);     // 处理人负责：开发人员挂起
}};
```

**设计原理**：
- 将责任人类型抽象为常量（CREATOR、PROCESSOR），便于扩展
- 每个状态明确指定责任人类型，实现细粒度权限控制
- 与流转规则分离，职责清晰

### 6.3 核心方法实现

#### 6.3.1 获取可流转状态列表（getAvailableTransitions）

```java
public List<String> getAvailableTransitions(String issueId, String projectId) {
    // 1. 参数校验
    if (StringUtils.isBlank(issueId)) {
        return new ArrayList<>();
    }
    
    // 2. 获取缺陷信息
    IssuesWithBLOBs issue = issuesMapper.selectByPrimaryKey(issueId);
    if (issue == null) {
        return new ArrayList<>();
    }
    
    // 3. 权限检查：检查当前用户是否是当前状态的责任人
    if (!canEditStatus(issueId)) {
        return new ArrayList<>();  // 无权限则返回空列表
    }
    
    // 4. 获取当前状态
    String status = getCurrentStatus(issue);
    if (StringUtils.isBlank(status)) {
        status = "new";  // 默认状态
    }
    
    // 5. 从流转规则中获取可流转状态列表
    return TRANSITION_RULES.getOrDefault(status, new ArrayList<>());
}
```

**执行流程**：
1. **参数校验**：检查缺陷ID是否有效
2. **获取缺陷**：从数据库加载缺陷信息
3. **权限检查**：调用 `canEditStatus()` 检查用户是否有权限编辑
4. **获取当前状态**：从 `custom_field_issues` 表读取状态值
5. **查询规则**：从 `TRANSITION_RULES` 中查找当前状态对应的可流转状态列表

**设计要点**：
- 先检查权限，再返回可流转状态，确保安全性
- 如果无权限，返回空列表，前端据此隐藏状态选择器
- 使用 `getOrDefault()` 处理未定义状态，返回空列表

#### 6.3.2 权限判断（canEditStatus）

```java
private boolean canEditStatus(String issueId) {
    // 1. 获取缺陷和当前状态
    IssuesWithBLOBs issue = issuesMapper.selectByPrimaryKey(issueId);
    if (issue == null) {
        return false;
    }
    
    String currentStatus = getCurrentStatus(issue);
    if (StringUtils.isBlank(currentStatus)) {
        currentStatus = "new";
    }
    
    // 2. 获取当前状态可流转到的目标状态列表
    List<String> targetStatuses = TRANSITION_RULES.get(currentStatus);
    
    // 3. 如果是终止状态（无可流转状态）
    if (CollectionUtils.isEmpty(targetStatuses)) {
        // 检查当前状态的责任人
        String ownerType = STATUS_OWNER.get(currentStatus);
        if (StringUtils.isBlank(ownerType)) {
            return false;
        }
        return checkUserIsOwner(issue, issueId, ownerType);
    }
    
    // 4. 检查所有目标状态的责任人是否一致
    Set<String> ownerTypes = new HashSet<>();
    for (String targetStatus : targetStatuses) {
        String ownerType = STATUS_OWNER.get(targetStatus);
        if (StringUtils.isNotBlank(ownerType)) {
            ownerTypes.add(ownerType);
        }
    }
    
    // 5. 如果所有目标状态都由同一个人负责，那么应该由那个人来编辑
    if (ownerTypes.size() == 1) {
        String targetOwnerType = ownerTypes.iterator().next();
        return checkUserIsOwner(issue, issueId, targetOwnerType);
    }
    
    // 6. 如果目标状态由不同的人负责，返回false
    return false;
}
```

**核心逻辑**：
1. **终止状态处理**：如果当前状态无可流转状态（终止状态），检查当前状态的责任人
2. **目标状态分析**：收集所有目标状态的责任人类型到Set中
3. **一致性检查**：如果所有目标状态的责任人类型相同（Set.size() == 1），则只有该责任人可以编辑
4. **用户验证**：调用 `checkUserIsOwner()` 检查当前用户是否是指定的责任人

**设计亮点**：
- **基于目标状态的责任人**：这是核心设计理念，确保权限判断的准确性
- **使用Set去重**：自动处理目标状态责任人类型相同的情况
- **终止状态特殊处理**：终止状态无法流转，检查当前状态责任人

#### 6.3.3 用户身份验证（checkUserIsOwner）

```java
private boolean checkUserIsOwner(Issues issue, String issueId, String ownerType) {
    SessionUser user = SessionUtils.getUser();
    if (user == null) {
        return false;
    }
    
    if (CREATOR.equals(ownerType)) {
        // 检查是否是创建人
        String creatorId = getCreator(issue);
        return StringUtils.equals(creatorId, user.getId());
    } else if (PROCESSOR.equals(ownerType)) {
        // 检查是否是处理人
        String processorId = getProcessor(issueId);
        if (StringUtils.isBlank(processorId)) {
            return false;  // 如果没有处理人，返回false
        }
        return StringUtils.equals(processorId, user.getId());
    }
    
    return false;
}
```

**实现细节**：
- **创建人获取**：优先从 `issues.creator` 字段获取，如果为空则从自定义字段获取
- **处理人获取**：从 `custom_field_issues` 表获取处理人字段值（member类型，JSON数组格式）
- **身份匹配**：比较获取到的用户ID与当前登录用户ID

#### 6.3.4 执行状态流转（transitionStatus）

```java
public void transitionStatus(String issueId, String toStatus, String comment) {
    // 1. 参数校验
    if (StringUtils.isBlank(issueId) || StringUtils.isBlank(toStatus)) {
        MSException.throwException(Translator.get("issue_status_transition_param_error"));
    }
    
    // 2. 获取缺陷信息
    IssuesWithBLOBs issue = issuesMapper.selectByPrimaryKey(issueId);
    if (issue == null) {
        MSException.throwException(Translator.get("issue_not_found"));
    }
    
    // 3. 权限检查
    if (!canEditStatus(issueId)) {
        MSException.throwException("只有当前状态的责任人可以编辑状态");
    }
    
    // 4. 获取当前状态
    String currentStatus = getCurrentStatus(issue);
    
    // 5. 检查流转规则是否允许
    if (!canTransition(currentStatus, toStatus)) {
        MSException.throwException(Translator.get("issue_status_transition_not_allowed"));
    }
    
    // 6. 更新缺陷状态（通过自定义字段服务）
    CustomField customField = baseCustomFieldService.getCustomFieldByName(
        issue.getProjectId(), SystemCustomField.ISSUE_STATUS);
    if (customField != null) {
        String fieldId = customField.getId();
        CustomFieldResourceDTO resource = new CustomFieldResourceDTO();
        resource.setFieldId(fieldId);
        resource.setResourceId(issue.getId());
        resource.setValue(JSON.toJSONString(toStatus));  // 状态值需要JSON序列化
        customFieldIssuesService.editFields(issue.getId(), Collections.singletonList(resource));
    }
    
    // 7. 保存流转记录
    saveTransitionRecord(issueId, currentStatus, toStatus, comment, issue.getProjectId());
}
```

**执行流程**：
1. **参数校验**：检查缺陷ID和目标状态是否有效
2. **获取缺陷**：从数据库加载缺陷信息
3. **权限检查**：再次确认用户有权限执行流转
4. **检查流转规则是否允许**：检查流转规则是否允许从当前状态流转到目标状态
5. **更新状态**：通过 `customFieldIssuesService` 更新 `custom_field_issues` 表中的状态值
6. **记录历史**：调用 `saveTransitionRecord()` 保存流转记录到审计日志（`IssueChangeLogService`），用于追溯

**关键点**：
- **双重验证**：既检查权限，又检查规则，提高安全性
- **状态存储统一**：使用自定义字段机制存储状态，与系统架构一致
- **历史记录**：每次流转都记录到审计日志，用于追溯

### 6.4 状态存储机制

#### 6.4.1 状态存储位置

状态值存储在 `custom_field_issues` 表中，作为自定义字段值：

```sql
-- custom_field_issues 表结构
CREATE TABLE custom_field_issues (
    id VARCHAR(50) PRIMARY KEY,
    field_id VARCHAR(50),      -- 状态字段ID: beb57501-19c8-4ca3-8dfb-2cef7c0ea087
    resource_id VARCHAR(50),   -- 缺陷ID
    value TEXT                 -- 状态值（JSON格式字符串，如: "new"）
);
```

#### 6.4.2 状态获取（getCurrentStatus）

```java
public String getCurrentStatus(Issues issue) {
    // 1. 获取状态自定义字段
    CustomField customField = baseCustomFieldService.getCustomFieldByName(
        issue.getProjectId(), SystemCustomField.ISSUE_STATUS);
    if (customField == null) {
        return "new";  // 默认状态
    }
    
    // 2. 从 custom_field_issues 表查询状态值
    CustomFieldIssuesExample example = new CustomFieldIssuesExample();
    example.createCriteria()
            .andFieldIdEqualTo(customField.getId())
            .andResourceIdEqualTo(issue.getId());
    List<CustomFieldIssues> customFieldIssues = customFieldIssuesMapper.selectByExample(example);
    
    // 3. 解析状态值（移除JSON引号）
    if (CollectionUtils.isEmpty(customFieldIssues)) {
        return "new";  // 默认状态
    }
    
    String status = customFieldIssues.get(0).getValue();
    if (StringUtils.isNotBlank(status)) {
        return status.replaceAll("\"", StringUtils.EMPTY);  // 移除JSON字符串的引号
    }
    return "new";
}
```

**设计说明**：
- 状态值以JSON字符串格式存储（如 `"new"`），读取时需要移除引号
- 如果查询不到状态值，返回默认状态 `"new"`
- 使用MyBatis的Example查询，代码简洁

public void saveTransitionRecord(String issueId, String fromStatus, String toStatus, 
                                 String comment, String projectId) {
    SessionUser user = SessionUtils.getUser();
    if (user == null) {
        return;
    }

    IssueChangeLogDetailDTO d = new IssueChangeLogDetailDTO();
    d.setFieldType(IssueChangeLogService.FIELD_TYPE_SYSTEM);
    d.setFieldKey("status");
    d.setFieldName("状态");
    d.setOldValue(fromStatus);
    d.setNewValue(toStatus);

}

### 6.7 前端实现原理

#### 6.7.1 权限控制

前端通过调用 `getAvailableTransitions` API获取可流转状态列表：
- **如果列表为空**：说明当前用户无权限编辑，隐藏状态选择器
- **如果列表有值**：显示为下拉菜单，只显示可流转的状态

#### 6.7.2 状态字段渲染

在列表页和编辑页都支持状态流转：
- **列表页**：点击状态字段，显示可流转状态下拉菜单
- **编辑页**：状态字段下拉框只显示可流转状态

**实现示例**（IssueList.vue）：
```javascript
getAvailableTransitionsForIssue(issue) {
    // 异步获取可流转的状态值列表（从后端）
    getAvailableTransitions(issue.id).then((response) => {
        const availableStatusValues = response.data || [];
        // 过滤出可流转的状态选项
        const availableOptions = statusField.options.filter(option => 
            availableStatusValues.includes(option.value)
        );
        // 更新缓存，显示在下拉菜单中
        this.$set(this.availableTransitionsCache, issue.id, availableOptions);
    });
}
```

### 6.8 设计模式总结

#### 6.8.1 规则引擎模式

- **规则定义**：使用静态Map定义流转规则和责任人，配置与代码分离
- **规则查询**：通过Map查找快速获取规则，性能高效
- **规则扩展**：新增状态只需修改Map配置，无需修改业务逻辑

#### 6.8.2 策略模式

- **权限策略**：基于目标状态责任人的权限判断策略
- **状态策略**：不同状态有不同的流转规则和责任人对策

#### 6.8.3 责任链模式

- **权限检查链**：参数校验 → 权限检查 → 规则验证 → 状态更新 → 历史记录

### 6.9 关键设计决策

1. **基于目标状态的责任人**：确保权限判断的准确性，避免状态混乱
2. **规则与代码分离**：使用Map配置规则，便于维护和扩展
3. **双重验证机制**：既检查权限，又检查规则，提高安全性
4. **历史记录完整**：每次流转都记录，便于追溯和审计
5. **状态存储统一**：使用自定义字段机制存储状态，与系统架构一致
