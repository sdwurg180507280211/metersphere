# PaiSmart 知识库检索 → MeterSphere analytics-stat 迁移评估

> 评估日期：2026-02-26
> 评估范围：PaiSmart 知识库检索功能迁移到 MeterSphere analytics-stat 模块的技术可行性

---

## 一、结论

**技术上完全可行，但需要引入新的基础设施依赖（Elasticsearch）和外部 API（Embedding 服务）。**

迁移的核心挑战不在代码层面，而在于：
1. MeterSphere 现有部署环境没有 Elasticsearch，需要新增容器
2. 向量生成依赖外部 Embedding API（通义千问），需要网络和 API Key
3. analytics-stat 是轻量模块，引入 ES + Kafka 文件处理链路会显著增加复杂度

推荐方案：**分两期实施，第一期只做"检索"（复用已有 ES 数据），第二期再做"入库"（文件上传 + 解析 + 向量化）。**

## 二、PaiSmart 知识库检索架构分析

### 2.1 整体数据流

```
文件上传 → MinIO 存储 → Kafka 消息 → 文件解析(Tika) → 文本分块 → MySQL 存储
                                                                    ↓
                                                          Embedding API(通义千问)
                                                                    ↓
                                                          向量生成(2048维)
                                                                    ↓
                                                          Elasticsearch 索引
                                                                    ↓
                                                          混合检索(KNN + BM25)
```

### 2.2 核心组件清单

| 组件 | 技术 | 作用 | MeterSphere 是否已有 |
|------|------|------|---------------------|
| 文件存储 | MinIO | 存储上传的原始文件 | ✅ 已有 |
| 消息队列 | Kafka | 异步处理文件解析任务 | ✅ 已有 (3.6.1) |
| 文件解析 | Apache Tika 2.9.1 | 提取 PDF/Word/Excel 等文件的文本 | ❌ 需新增 |
| 文本分块 | 自研（语义分块） | 将长文本按语义切分为 512 字符的块 | ❌ 需新增 |
| 向量生成 | 通义千问 text-embedding-v4 | 将文本转为 2048 维向量 | ❌ 需新增（外部 API） |
| 向量存储 | Elasticsearch 8.10 + IK 分词 | 存储向量和文本，支持 KNN + BM25 | ❌ 需新增容器 |
| 混合检索 | ES KNN + BM25 Rescore | 向量相似度 + 关键词匹配的混合排序 | ❌ 需新增 |
| 数据库 | MySQL (JPA) | 存储文件元数据、分块信息 | ✅ 已有（但 ORM 不同） |
| 缓存 | Redis | Token 缓存、组织标签缓存 | ✅ 已有 (Redisson) |

### 2.3 检索算法详解

PaiSmart 的 `HybridSearchService` 实现了两阶段混合检索：

**第一阶段 - KNN 向量召回**：
- 将用户查询文本通过 Embedding API 转为 2048 维向量
- 在 ES 中用 cosine 相似度做 KNN 搜索，召回 topK × 30 条候选

**第二阶段 - BM25 Rescore**：
- 对召回的候选集用 BM25 文本匹配重新打分
- 权重分配：KNN 分数 × 0.2 + BM25 分数 × 1.0（BM25 主导）

**权限过滤**（filter 层，不影响打分）：
- 用户自己上传的文档
- 标记为公开的文档
- 用户所属组织标签匹配的文档

**降级策略**：向量生成失败时，自动降级为纯 BM25 文本检索。

### 2.4 ES 索引结构

```json
{
  "fileMd5": "keyword",        // 文件指纹
  "chunkId": "integer",        // 分块序号
  "textContent": "text",       // 文本内容（IK 分词）
  "vector": "dense_vector",    // 2048 维向量（cosine 相似度）
  "modelVersion": "keyword",   // 模型版本
  "userId": "keyword",         // 上传用户
  "orgTag": "keyword",         // 组织标签
  "isPublic": "boolean"        // 是否公开
}
```

### 2.5 MySQL 表结构（PaiSmart 侧）

| 表名 | 作用 | 关键字段 |
|------|------|---------|
| file_upload | 文件上传记录 | id, file_md5, file_name, user_id, org_tag, is_public, status |
| document_vectors | 文本分块存储 | vector_id, file_md5, chunk_id, text_content, user_id, org_tag |
| chunk_info | 文件分块元数据 | id, file_md5, chunk_index, chunk_md5, storage_path |

## 三、MeterSphere analytics-stat 模块现状

### 3.1 模块定位

analytics-stat 是 MeterSphere 的分析统计微服务（端口 8009），目前是一个相对轻量的模块。

- 后端：Spring Boot 3.2.12，依赖 MeterSphere SDK，使用 MyBatis 做数据访问
- 前端：已迁移到 Vue 3.4 + Vite 5 + TypeScript + Element Plus（微前端子应用）
- ORM：MyBatis（非 JPA），这是与 PaiSmart 的关键差异
- 认证：不独立处理，由 Gateway 统一认证后透传用户信息

## 四、技术差异与适配方案

### 4.1 ORM 差异（JPA → MyBatis）

PaiSmart 使用 JPA 的 `@Entity` + `Repository` 模式，MeterSphere 使用 MyBatis 的 `Mapper` + XML 模式。

**适配方案**：
- 新建 Flyway 迁移脚本创建 `kb_file_upload`、`kb_document_vectors`、`kb_chunk_info` 三张表（加 `kb_` 前缀避免冲突）
- 用 MyBatis Mapper 接口替代 JPA Repository
- 实体类去掉 JPA 注解，改为普通 POJO + Lombok

**工作量**：约 1 天，涉及 3 个 Mapper 接口 + 3 个 XML 映射文件

### 4.2 认证体系差异

PaiSmart 使用 Spring Security + JWT 自行管理认证，MeterSphere 由 Gateway 统一认证。

**适配方案**：
- 不需要迁移 PaiSmart 的 SecurityConfig、JwtAuthenticationFilter
- analytics-stat 的 Controller 通过 SDK 的 `SessionUtils.getUserId()` 获取当前用户
- 权限过滤中的 `userId` 和 `orgTag` 映射到 MeterSphere 的用户 ID 和工作空间 ID

**工作量**：约 0.5 天，主要是权限模型的映射

### 4.3 Elasticsearch 集成

MeterSphere 现有部署没有 ES，这是最大的新增依赖。

**适配方案**：
- Docker 部署新增 ES 8.10 容器（含 IK 分词插件），内存限制 2G
- 后端 pom.xml 新增 `elasticsearch-java` 8.10.0 依赖
- 复用 PaiSmart 的 `EsConfig`、`EsIndexInitializer`、`ElasticsearchService`，改包名为 `io.metersphere.knowledge`
- ES 连接信息通过 `application.properties` 配置

**工作量**：约 1 天（含 Docker 配置和连接调试）

**资源影响**：ES 容器需要 2G 内存，宿主机总内存需求从 ~10G 增加到 ~12G

### 4.4 Embedding API 集成

**适配方案**：
- 复用 PaiSmart 的 `EmbeddingClient`（基于 WebClient 调用通义千问 API）
- analytics-stat 后端 pom.xml 已通过 SDK 间接依赖 `spring-boot-starter-webflux`，无需额外引入
- API Key 通过 `application.properties` 或 MeterSphere 的 `system_parameter` 表配置
- 保留降级策略：Embedding API 不可用时自动降级为纯 BM25 文本检索

**工作量**：约 0.5 天

**注意**：通义千问 text-embedding-v4 API 有调用频率限制，批量入库时需控制并发

### 4.5 前端适配

PaiSmart 前端使用 Naive UI，analytics-stat 使用 Element Plus，组件库不同但都是 Vue 3。

**适配方案**：
- 知识库检索对话框（`search-dialog.vue`）从 Naive UI 组件改为 Element Plus 组件
  - `NModal` → `el-dialog`
  - `NForm` / `NInput` / `NInputNumber` → `el-form` / `el-input` / `el-input-number`
  - `NCard` / `NTag` / `NHighlight` → `el-card` / `el-tag` + 自定义高亮
  - `NScrollbar` / `NSpin` / `NEmpty` → `el-scrollbar` / `v-loading` / `el-empty`
- 文件上传对话框（`upload-dialog.vue`）同理改写
- API 调用从 PaiSmart 的 `@sa/axios` 改为 analytics-stat 现有的 axios 实例
- 新增路由：`/analytics/knowledge` 挂载到现有路由树

**工作量**：约 2 天（两个对话框 + 知识库管理列表页）

### 4.6 Kafka 复用

MeterSphere 已有 Kafka 3.6.1，可直接复用。

**适配方案**：
- 新建 Topic：`knowledge-file-processing`（避免与现有 Topic 冲突）
- 复用 PaiSmart 的 `FileProcessingConsumer` 消费逻辑，改包名
- Producer 端在文件上传 Controller 中发送消息

**工作量**：约 0.5 天

### 4.7 MinIO 复用

MeterSphere 已有 MinIO，可直接复用。

**适配方案**：
- 新建 Bucket：`knowledge-base`（与现有 `uploads` 隔离）
- 复用 SDK 中已有的 MinIO 工具类，或直接引入 MinIO SDK

**工作量**：约 0.5 天

## 六、基础设施变更

### 6.1 Elasticsearch 部署（已完成 ✅）

> **部署方式**：与 MeterSphere 现有中间件（MySQL、Redis、Kafka、MinIO）保持一致，采用 Docker Compose 模块化部署，配置文件位于 `/opt/metersphere/` 目录。

#### 已创建的配置文件

**1. `/opt/metersphere/docker-compose-elasticsearch.yml`**

```yaml
services:
  elasticsearch:
    image: ${MS_IMAGE_PREFIX}/elasticsearch:8.10.4
    container_name: elasticsearch
    healthcheck:
      test: ["CMD-SHELL", "curl -s -u elastic:${MS_ES_PASSWORD} http://localhost:9200/_cluster/health | grep -q '\"status\":\"green\"\\|\"status\":\"yellow\"'"]
      interval: 30s
      timeout: 10s
      retries: 50
    environment:
      TZ: "Asia/Shanghai"
      ELASTIC_PASSWORD: ${MS_ES_PASSWORD}
      discovery.type: single-node
      xpack.security.enabled: "true"
      xpack.security.http.ssl.enabled: "false"
      ES_JAVA_OPTS: "-Xms${MS_ES_HEAP_SIZE:-2g} -Xmx${MS_ES_HEAP_SIZE:-2g}"
    ports:
      - ${MS_ES_PORT}:9200
    volumes:
      - ms-es-data:/usr/share/elasticsearch/data
    networks:
      - ms-network
    restart: always
    deploy:
      resources:
        limits:
          memory: ${MS_ES_MEM_LIMIT:-2g}

volumes:
  ms-es-data:
```

## 十、推荐实施路径

**第一期（检索优先）**：先部署 ES，手动导入测试数据到 ES 索引，实现检索 API 和前端对话框。这样可以最快验证检索效果，不依赖完整的入库链路。

**第二期（入库补全）**：在检索验证通过后，补全文件上传 → 解析 → 分块 → 向量化 → ES 索引的完整链路。

**可选优化**：

- 支持从 MeterSphere 现有测试用例、接口文档中自动提取文本入库
- 在 AI 对话场景中集成 RAG（检索增强生成），用知识库结果辅助 AI 回答
- 支持多种 Embedding 模型切换（通义千问 / OpenAI / 本地模型）
