# 全局性能优化方案：切换项目卡顿问题

> 日期：2026-02-25
> 范围：所有业务模块（非单一页面问题）
> 目标：优化 `framework/sdk-parent/frontend` 中的共享组件，使所有页面受益

---

## 一、问题现象

切换项目后，所有业务模块页面（缺陷管理、用例管理、接口测试等）均出现明显卡顿，正式环境实测数据：

| 指标 | 数值 | 说明 |
|------|------|------|
| 页面总耗时 | 14.1s | 从切换项目到页面可交互 |
| Scripting | 7054ms (50%) | JS 执行占一半时间 |
| Recalculate Style | 2734ms (24.2% Self Time) | DOM 节点过多导致样式重算慢 |
| Total Blocking Time | 9594ms | 主线程阻塞近 10 秒，用户无法操作 |

---

## 二、根因分析（聚焦共享组件）

### 根因 1：el-table fixed 列导致 DOM 三倍膨胀

**涉及组件**：`MsTable.vue` + `MsTableColumn.vue`

Element UI 的 el-table 在有 `fixed="left"` 和 `fixed="right"` 列时，会将整个表格 DOM 复制 3 份（主体 + 左固定 + 右固定）。

实测数据（缺陷管理页，仅 8 行数据）：
- 22 个可见列 × 3 份 = 76 个 `<th>` 元素
- 表格区域 DOM 节点：3479 个（占页面总 DOM 的 76%）
- 正式环境 50 行时，DOM 节点轻松超过 2 万

这是 Recalculate Style 耗时 2734ms 的首要原因。列数越多，问题越严重。

### 根因 2：MsTable.doLayout() 执行 3 次

**涉及组件**：`MsTable.vue`

```javascript
// framework/sdk-parent/frontend/src/components/table/MsTable.vue
doLayout() {
  if (this.$refs.table) {
    // 表格错位问题，执行三次
    for (let i = 1; i <= 3; i++) {
      setTimeout(this.$refs.table.doLayout, 300 * i);
    }
  }
}
```

每次数据变化（`data` watcher）都会触发此方法，即 3 次 `el-table.doLayout()`，每次都对 3 份 DOM 做完整的布局计算。在 DOM 节点 2 万+ 的情况下，3 次重排的开销非常大。

### 根因 3：MsTableColumn 的 render-header 函数

**涉及组件**：`MsTableColumn.vue`

每个 MsTableColumn 都绑定了 `renderHeader` 函数（判断文字长度 > 7 则加 tooltip）。22 列 × 3 份 = 66 次 render-header 调用，产生 38 个 `[Element Warn][TableColumn] render-header` 控制台警告。

Element UI 官方推荐使用 scoped-slot `#header` 替代 `render-header` 属性。

### 根因 4：项目切换触发全页面强制重载

**涉及组件**：`ProjectSearchList.vue`

```
切换项目流程：
1. switchProject() API 调用 → 更新后端 session
2. $EventBus.$emit('projectChange') → 通知当前应用
3. broadcastEvent({ type: 'projectChange' }) → 通知所有微前端子应用
4. reloadPage() → this.reload() → 强制销毁并重建整个页面组件树
```

`this.reload()` 是通过 `inject` 注入的方法，实际效果是将路由组件的 `v-if` 置为 false 再置为 true，强制整个页面从零开始重建。这意味着：
- 所有组件的 `created`/`mounted`/`activated` 重新执行
- 所有 API 请求重新发起
- 所有 DOM 重新创建和渲染

### 根因 5：stopFullScreenLoading 硬编码 2 秒延迟

**涉及文件**：`framework/sdk-parent/frontend/src/utils/index.js`

```javascript
// framework/sdk-parent/frontend/src/utils/index.js
export function stopFullScreenLoading(loading, timeout) {
  timeout = timeout ? timeout : 2000;  // 默认 2 秒延迟
  setTimeout(() => {
    loading.close();
  }, timeout);
}
```

切换项目时，`ProjectSearchList.vue` 的 `change()` 方法调用 `fullScreenLoading()` 显示"资源切换中..."遮罩。当 `switchProject` API 返回且 `reloadPage()` 执行完毕后，调用 `stopFullScreenLoading(loading)` 关闭遮罩。但这个函数内部有一个硬编码的 2 秒 `setTimeout`，意味着：

- 即使所有数据已经加载完成，用户还要额外等 2 秒才能操作页面
- 这 2 秒是纯粹的人为等待，没有任何实际作用

同样的问题也存在于工作空间切换（`HeaderWs.vue`）和语言切换（`LanguageSwitch.vue`）中。

### 根因 6：各业务页面 activated() 中的串行 API 调用

虽然这不是共享组件的问题，但它是一个普遍模式。以缺陷管理页为例：

```
串行链：getUserGroupProject → getProjectMember → getIssuePartTemplateWithProject 
        → applyUserGroupFilter → getIssues
并行：getPlatformStatus + getThirdPartTemplate + getProjectMember
总计：21 个 API 请求
```

其他模块（用例管理、接口测试等）也有类似的串行初始化模式。

## 三、优化方案

### 方案 1：减少默认显示列数（优先级最高）

**改动位置**：各业务模块的自定义列配置

**思路**：
- 将默认显示列从 22 个减少到 8-10 个
- 用户可通过自定义列设置手动开启更多列
- 列数从 22 降到 10，DOM 节点减少约 55%（10 × 3 = 30 个 `<th>` vs 22 × 3 = 66 个）
- 这是投入产出比最高的优化，不改任何组件代码

**预期效果**：Recalculate Style 从 2734ms 降到 ~1200ms

### 方案 2：doLayout() 合并为单次调用（优先级高）

**改动位置**：`framework/sdk-parent/frontend/src/components/table/MsTable.vue`

**思路**：
- 将 3 次 setTimeout 改为 1 次 requestAnimationFrame + 防抖
- 如果仍有错位问题，最多保留 2 次（立即 + 延迟 300ms），不需要 3 次

```
改动前：3 次 setTimeout（300ms、600ms、900ms）
改动后：1 次 requestAnimationFrame（~16ms）+ 可选 1 次 300ms 延迟兜底
```

**预期效果**：减少 2/3 的重排计算，Total Blocking Time 降低约 2-3 秒

**风险**：可能在某些极端情况下出现表格错位，需要在各模块页面验证

### 方案 3：render-header 改为 scoped-slot（优先级中）

**改动位置**：`framework/sdk-parent/frontend/src/components/table/MsTableColumn.vue`

**思路**：
- 将 `:render-header="renderHeader"` 改为 `<template #header="{ column }">`
- 消除 Element UI 的 render-header 警告
- 减少 VNode 创建开销

**预期效果**：消除 38 个控制台警告，减少约 10-15% 的表头渲染开销

**风险**：低。scoped-slot 是 Element UI 推荐的方式，功能完全等价

### 方案 4：各模块 API 调用并行化（优先级中）

**改动位置**：各业务模块的 `activated()` 方法

**思路**：
- 将无依赖关系的 API 调用用 `Promise.all` 并行发起
- 例如 `getProjectMember`、`getPlatformStatus`、`getThirdPartTemplate` 可以并行
- 只有真正有依赖的调用才保持串行

```
改动前（串行）：A → B → C → D → E  总耗时 = A+B+C+D+E
改动后（并行）：[A, B, C] → [D, E]  总耗时 = max(A,B,C) + max(D,E)
```

**预期效果**：API 等待时间从 ~5s 降到 ~2s

**风险**：需要仔细分析每个 API 之间的依赖关系，不能盲目并行

### 方案 5：消除 stopFullScreenLoading 的 2 秒人为延迟（优先级高）

**改动位置**：`framework/sdk-parent/frontend/src/utils/index.js`

**思路**：
- 将 `stopFullScreenLoading` 的默认 timeout 从 2000ms 改为 0（或直接调用 `loading.close()`）
- 当前逻辑：数据加载完成 → 等 2 秒 → 关闭遮罩。这 2 秒是纯粹的人为等待
- 如果担心闪烁（loading 一闪而过），可以保留 200-300ms 的最小显示时间，但不需要 2 秒

```
改动前：stopFullScreenLoading(loading) → setTimeout(close, 2000)
改动后：stopFullScreenLoading(loading) → setTimeout(close, 300)  // 或直接 close()
```

**影响范围**：所有调用 `stopFullScreenLoading` 的地方都会受益：
- `ProjectSearchList.vue` — 切换项目
- `HeaderWs.vue` — 切换工作空间
- `LanguageSwitch.vue` — 切换语言
- `MsProject.vue` / `MsAssociatedSystem.vue` — 项目列表跳转
- `WorkstationDetail.vue` — 工作台切换项目

**预期效果**：用户感知的等待时间直接减少 ~1.7s（2000ms → 300ms）

**风险**：极低。这只是一个 loading 遮罩的关闭时机，不影响任何业务逻辑

### 方案 6：缓存不变数据（优先级低）

**改动位置**：各业务模块 + 可考虑在 SDK 层提供缓存工具

**思路**：
- 以下数据在项目切换后不需要每次都请求：
  - 所属系统列表（`getSystemList`）
  - 平台状态映射（`getPlatformStatus`）
  - 项目成员列表（`getProjectMember`）
- 使用 sessionStorage 或 Pinia store 缓存，设置合理的过期时间（如 5 分钟）
- 切换项目时清除缓存，下次请求时重新获取

**预期效果**：减少 3-5 个 API 请求，节省 ~1s

**风险**：缓存一致性问题，需要在数据变更时主动清除缓存

## 四、实施优先级与改动范围

| 优先级 | 方案 | 改动范围 | 预期收益 | 风险 |
|--------|------|----------|----------|------|
| P0 | 减少默认列数 | 各模块列配置 | DOM 减少 55% | 无 |
| P1 | doLayout 合并 | MsTable.vue（1 个文件） | TBT 降低 2-3s | 低 |
| P1 | 消除 loading 2 秒延迟 | utils/index.js（1 个文件） | 感知等待减少 1.7s | 极低 |
| P2 | render-header → slot | MsTableColumn.vue（1 个文件） | 消除警告，减少渲染开销 | 低 |
| P3 | API 并行化 | 各模块 activated() | API 等待减少 3s | 中 |
| P4 | 缓存不变数据 | 各模块 + SDK 工具 | 减少 3-5 个请求 | 中 |

---

## 五、二次开发注意事项

1. **改动面小**：P1 和 P2 只改 `framework/sdk-parent/frontend` 下的 2 个文件，所有模块自动受益
2. **边界清晰**：P0 在各模块自己的列配置中改，不影响共享组件
3. **可回滚**：每个方案独立，可以单独上线和回滚
4. **不引入新依赖**：所有优化都基于现有技术栈，不需要额外的 npm 包
5. **不做虚拟滚动**：每页最多 50 行数据，虚拟滚动的复杂度不值得引入

## 六、验证方法

优化前后对比，使用 Chrome DevTools Performance 面板录制：

1. 打开目标页面，切换到一个有数据的项目
2. 开始录制 → 切换到另一个项目 → 停止录制
3. 对比以下指标：
   - Total Blocking Time（目标：< 3s）
   - Recalculate Style Self Time（目标：< 500ms）
   - DOM 节点数（目标：< 5000）
   - 控制台 Element Warn 数量（目标：0）

也可以用 JS 快速检查 DOM 节点数：
```javascript
// 页面总 DOM 节点
document.querySelectorAll('*').length
// 表格区域 DOM 节点
document.querySelectorAll('.el-table *').length
// th 元素数量（应该是可见列数的 3 倍以内）
document.querySelectorAll('.el-table th').length
```
