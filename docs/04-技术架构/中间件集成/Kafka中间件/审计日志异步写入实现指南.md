# 审计日志异步写入实现指南

## 一、功能概述

基于 Kafka 的审计日志异步写入功能，实现了用户操作的完整记录和追踪。通过消息队列解耦审计日志写入和业务操作，提升系统性能的同时确保审计日志的完整性。

### 1.1 核心特性

- **异步写入**: 通过 Kafka 异步处理，不影响主业务流程性能
- **数据完整性**: 支持操作前后数据对比，完整记录数据变更
- **多种使用方式**: 支持注解和编程两种方式记录审计日志
- **自动信息填充**: 自动获取用户、项目、请求等上下文信息
- **异常处理**: 审计日志记录失败不影响主业务流程
- **批量处理**: 支持批量操作的审计日志记录

## 三、使用方式

### 3.1 注解方式（推荐）

#### 3.1.1 基础用法

```java
@Service
public class UserService {
    
    @AuditLog(
        module = "USER_MANAGEMENT",
        content = "创建用户: #{#userRequest.name}",
        sourceId = "#{#result.id}",
        modifiedValue = "#{#result}"
    )
    public UserDTO createUser(UserRequest userRequest) {
        // 业务逻辑
        return userService.insert(userRequest);
    }
}
```

#### 3.1.2 记录数据变更

```java
@AuditLog(
    module = "USER_MANAGEMENT",
    content = "更新用户: #{#userRequest.name}",
    sourceId = "#{#userRequest.id}",
    recordOriginal = true,
    originalValue = "#{#originalUser}",
    modifiedValue = "#{#result}",
    history = true
)
public UserDTO updateUser(UserRequest userRequest) {
    // 获取原始数据
    User originalUser = userMapper.selectByPrimaryKey(userRequest.getId());
    
    // 执行更新
    UserDTO result = userService.update(userRequest);
    
    return result;
}
```

#### 3.1.3 注解参数说明

| 参数 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `module` | String | 是 | 操作模块 | "USER_MANAGEMENT" |
| `content` | String | 是 | 操作内容描述，支持 SpEL | "创建用户: #{#user.name}" |
| `type` | String | 否 | 操作类型，不填自动推断 | "ADD", "UPDATE", "DELETE" |
| `sourceId` | String | 否 | 资源ID，支持 SpEL | "#{#user.id}" |
| `history` | boolean | 否 | 是否保存历史记录，默认 true | true |
| `recordOriginal` | boolean | 否 | 是否记录原始数据，默认 false | true |
| `originalValue` | String | 否 | 原始数据 SpEL 表达式 | "#{#originalUser}" |
| `modifiedValue` | String | 否 | 变更后数据 SpEL 表达式 | "#{#result}" |

### 3.2 编程方式

#### 3.2.1 简单审计日志

```java
@Service
public class TestCaseService {
    
    @Resource
    private AuditLogService auditLogService;
    
    public void deleteTestCase(String testCaseId) {
        try {
            // 执行删除逻辑
            testCaseMapper.deleteByPrimaryKey(testCaseId);
            
            // 记录审计日志
            auditLogService.sendAuditLog(
                "DELETE",
                "TEST_CASE_MANAGEMENT",
                "删除测试用例: " + testCaseId,
                testCaseId
            );
            
        } catch (Exception e) {
            // 记录失败日志
            auditLogService.sendSystemOperationLog(
                "DELETE",
                "TEST_CASE_MANAGEMENT",
                "删除测试用例失败: " + e.getMessage(),
                testCaseId
            );
            throw e;
        }
    }
}
```

#### 3.2.2 完整审计日志

```java
public void updateTestCase(TestCase testCase) {
    // 获取原始数据
    TestCase originalTestCase = testCaseMapper.selectByPrimaryKey(testCase.getId());
    
    // 执行更新
    testCaseMapper.updateByPrimaryKey(testCase);
    
    // 记录审计日志
    auditLogService.sendUserOperationLog(
        "UPDATE",
        "TEST_CASE_MANAGEMENT",
        "更新测试用例: " + testCase.getName(),
        testCase.getId(),
        originalTestCase,  // 原始数据
        testCase          // 变更后数据
    );
}
```

#### 3.2.3 批量操作审计日志

```java
public void batchDeleteTestCases(List<String> testCaseIds) {
    for (String testCaseId : testCaseIds) {
        TestCase testCase = testCaseMapper.selectByPrimaryKey(testCaseId);
        testCaseMapper.deleteByPrimaryKey(testCaseId);
        
        auditLogService.sendUserOperationLog(
            "DELETE",
            "TEST_CASE_MANAGEMENT",
            "批量删除测试用例: " + testCase.getName(),
            testCaseId,
            JSON.toJSONString(testCase),
            null
        );
    }
}
```

在 `api-test` 模块的 `KafkaConfig` 中添加主题创建：

## 五、SpEL 表达式使用

### 5.1 可用变量

| 变量 | 说明 | 示例 |
|------|------|------|
| `#参数名` | 方法参数 | `#{#userRequest.name}` |
| `#result` | 方法返回值 | `#{#result.id}` |
| `#exception` | 异常信息（异常时） | `#{#exception.message}` |
| `#methodName` | 方法名 | `#{#methodName}` |
| `#className` | 类名 | `#{#className}` |

### 5.2 常用表达式

```java
// 获取对象属性
content = "创建用户: #{#user.name} (#{#user.email})"

// 条件表达式
content = "#{#user.enabled ? '启用' : '禁用'}用户: #{#user.name}"

// 集合操作
content = "批量操作用户，数量: #{#userIds.size()}"

// 字符串操作
sourceId = "#{#user.id != null ? #user.id : 'unknown'}"

// 复杂表达式
content = "#{#operation.toUpperCase()}用户: #{#user.name} 在项目 #{#project.name} 中"
```

## 六、最佳实践

### 6.1 注解使用建议

1. **优先使用注解方式**: 代码简洁，维护方便
2. **合理使用 SpEL**: 避免过于复杂的表达式
3. **敏感信息处理**: 密码等敏感信息不要记录到审计日志
4. **异常处理**: 确保审计日志记录失败不影响主业务

### 6.2 性能优化建议

1. **异步处理**: 审计日志通过 Kafka 异步处理，不阻塞主流程
2. **批量发送**: Kafka 配置合理的批量大小和延迟时间
3. **线程池配置**: 合理配置消费者线程池大小
4. **数据压缩**: 启用 Kafka 消息压缩

### 6.3 数据安全建议

1. **数据脱敏**: 敏感数据在记录前进行脱敏处理
2. **访问控制**: 限制审计日志的查看权限
3. **数据保留**: 设置合理的日志保留期限
4. **备份策略**: 重要审计日志需要定期备份

### 7.2 常见问题排查

#### 7.2.1 审计日志丢失

**可能原因**:
- Kafka 服务不可用
- 消费者处理异常
- 网络问题

**排查步骤**:
1. 检查 Kafka 服务状态
2. 查看消费者日志
3. 检查网络连接
4. 验证主题配置

#### 7.2.2 审计日志延迟

**可能原因**:
- 消费者处理速度慢
- 数据库写入性能问题
- 线程池配置不合理

**解决方案**:
1. 增加消费者实例
2. 优化数据库写入
3. 调整线程池配置
4. 启用批量写入

### 7.3 日志清理策略

```sql

DELETE FROM operation_log WHERE create_time < DATE_SUB(NOW(), INTERVAL 90 DAY);

-- 清理 365 天前的历史记录
```
