# MeterSphere UI实现方式对比分析

> 对比旧版本(metersphere)和新版本(metersphere-3.6)在前端交互模式上的差异

---

## 一、核心差异概述

### 1.1 设计理念转变

**旧版本：页面跳转为主**
- 重度依赖浏览器标签页管理
- 编辑操作打开新窗口
- 上下文频繁切换

**新版本：抽屉(Drawer)为主**
- 单页应用内完成所有操作
- 多层抽屉叠加设计
- 保持上下文连续性

### 1.2 用户体验对比

| 场景 | 旧版本体验 | 新版本体验 |
|-----|----------|-----------|
| 编辑用例 | 新标签页打开 → 编辑 → 关闭标签 → 手动刷新列表 | 点击编辑 → 抽屉滑出 → 编辑 → 关闭抽屉(自动刷新) |
| 测试计划执行 | 全屏抽屉 → 上下翻页 → 无法看到其他用例 | 左侧列表 + 右侧详情 → 点击切换 → 一览全局 |
| 查找特定用例 | 需要记住位置 → 盲目翻页 | 左侧搜索/筛选 → 直接定位 |
| 多任务处理 | 多个浏览器标签页 → 容易混乱 | 抽屉叠加 → 层次清晰 |

## 二、测试用例编辑功能对比

### 2.1 旧版本实现思路

**交互流程：**
```
测试用例列表 
  ↓ (点击编辑按钮)
window.open() 新窗口
  ↓
完整的编辑页面 (独立路由)
  ↓ (保存后)
关闭窗口 → 回到列表页
  ↓
手动刷新列表
```

**核心机制：**
- 历史实现曾使用 `window.open(...)` 打开新标签页（已在旧项目中收敛为同页路由跳转，见 2.3）
- 路由配置了独立的编辑页面路径 `/track/case/edit/:caseId`
- 编辑页面是完整的 Vue 页面组件 (2000+ 行代码)
- 编辑完成后需手动刷新原页面

**优点：**
- 实现简单，逻辑清晰
- 各个页面独立，耦合度低
- 可以同时打开多个编辑窗口

**缺点：**
- 用户需要管理多个浏览器标签页
- 编辑后需要手动回到列表页刷新
- 上下文丢失（不记得从哪里来的）
- 浏览器历史记录混乱

### 2.2 新版本实现思路

**交互流程：**
```
测试用例列表 
  ↓ (点击编辑按钮)
显示抽屉 (showDetailDrawer = true)
  ↓
抽屉内嵌套详情页面
  ↓ (点击编辑按钮)
二级抽屉 (EditCaseDetailDrawer)
  ↓ (保存)
自动关闭抽屉 + 自动刷新列表
```

**核心机制：**
- 使用自定义 `MsDrawer` 组件作为容器
- 抽屉内加载 `CaseDetailDrawer` 详情组件
- 支持二级抽屉嵌套（详情抽屉内再打开编辑抽屉）
- 通过事件通信自动刷新父组件数据

**优点：**
- 不离开当前页面，上下文完整
- 保存后自动刷新，操作流畅
- 支持抽屉内切换上下用例
- 可以调整抽屉宽度甚至全屏

**缺点：**
- 组件嵌套层次较深
- 需要处理抽屉间的事件传递
- 实现复杂度稍高

### 2.3 旧项目实践：功能用例编辑页收敛（方案一落地）

#### 2.3.1 目标与边界

- **目标**
  - 将功能用例的“查看/编辑/复制/新建”等入口统一收敛到同一个编辑页面：`/track/case/edit/:caseId`（创建仍走 `/track/case/create`，但同样在当前标签页内打开）。
  - 编辑页顶部补齐页面级导航：**面包屑 + 返回列表**（方案一：独立页面 + 面包屑导航）。
  - 消除新标签页带来的上下文割裂与“编辑后需手动回到原页刷新”的问题。
- **边界（明确不做什么）**
  - 不引入多层抽屉/二级抽屉来替代编辑页（抽屉能力保留给其他更适合的场景）。
  - 不改造编辑页内部的业务表单结构（避免大范围回归风险）。

#### 2.3.2 设计决策（为什么选方案一）

- **编辑天然需要“大空间 + 专注”**
  - 用例编辑页本身内容密度高、操作链路长（步骤、富文本、附件、版本等），抽屉模式容易产生“空间不足/交互层级叠加”的复杂度。
- **旧项目已有稳定的编辑页路由与组件**
  - 旧项目已经存在 `TestCaseEdit.vue` 且路由已配置 `case/edit/:caseId`，因此更适合做“入口收敛 + 导航补齐”，以低风险获得高收益。

#### 2.3.3 落地改动点（轻代码）

- **改动 1：移除新标签页打开**
  - 将 `openCaseEdit/openCaseCreate` 从 `window.open` 改为 `router.push`，实现“在当前标签页打开编辑页/创建页”。
- **改动 2：列表页入口统一**
  - 功能用例列表的“查看/行点击”统一跳转到 `/track/case/edit/:caseId`，并透传 `projectId` 等必要参数。
- **改动 3：编辑页补齐面包屑导航**
  - 在 `TestCaseEdit.vue` 顶部加入“功能用例 > 创建/编辑”的面包屑，并提供返回列表入口（返回到 `/track/case/all`）。

#### 2.3.4 回归检查清单

- **入口一致性**
  - 列表页点击“查看/行点击”是否仍会打开新标签页（应否）。
  - “复制用例/从公共用例进入/从评审进入/从计划进入”等入口是否仍会打开新标签页（应否）。
- **路由与权限**
  - `projectId` 透传与权限校验是否仍正常（无权限是否会被拦截/跳转）。
- **返回体验**
  - 编辑页面包屑点击“功能用例”是否能稳定回到列表。
  - 浏览器后退是否不会触发异常关闭逻辑/死循环（关注编辑页内部的 `listenGoBack` 行为）。
- **保存链路**
  - 保存/保存并新建/保存并加入公共库/取消等操作是否与原逻辑一致。

## 五、方案一 vs 方案二：选型指南（重思路）

### 5.1 两种方案的本质区别

- **方案一：独立页面 + 面包屑导航**
  - 通过路由承载“详情/编辑”上下文，强调 URL 可分享、浏览器历史可用、页面专注。
  - 典型形态：列表 → 路由跳转详情/编辑页 → 面包屑/返回回到列表。
- **方案二：抽屉（Detail Drawer / 多层抽屉）**
  - 在列表页内完成“查看/轻量编辑”，强调上下文不丢失、可快速切换、多任务并行的层次管理。
  - 典型形态：列表 → 抽屉展示详情 → 抽屉内切换上一条/下一条 → 操作后自动刷新列表。

### 5.2 适用场景建议

| 维度 | 方案一（独立页面）更适合 | 方案二（抽屉）更适合 |
|---|---|---|
| 操作复杂度 | 高：多区域编辑、富文本、附件、版本等 | 中低：以查看为主，少量字段编辑 |
| 页面空间需求 | 强依赖大空间、需要专注 | 希望保留列表上下文 |
| 导航诉求 | 需要面包屑/URL 分享/浏览器前进后退 | 需要快速预览、连续翻阅、不中断列表 |
| 回归风险 | 较低（复用既有页面） | 较高（涉及抽屉状态/事件/刷新联动） |

### 5.3 落地时的统一原则

- **入口收敛优先于 UI 形态**
  - 先统一“从哪里进、进到哪里、怎么返回”，再决定是页面还是抽屉。
- **数据刷新要自动化**
  - 方案一：保存成功后应明确回到列表并触发刷新（或列表页支持自动刷新）。
  - 方案二：关闭抽屉/保存后应自动刷新列表，减少手动刷新。
- **避免重复造轮子**
  - 旧项目已有 `MsDrawer/MsDetailDrawer` 等能力时，优先复用组件与既有交互约定。

## 三、测试计划功能用例执行对比

### 3.1 旧版本实现思路

**布局方式：全屏抽屉 + 单列展示**

```
┌─────────────────────────────────────┐
│ [返回] [<上一个] [下一个>]           │ 顶部操作栏
├─────────────────────────────────────┤
│                                     │
│         当前用例详情内容             │ 主体区域
│         - 基本信息                   │ (单用例展示)
│         - 执行步骤                   │
│         - 关联信息                   │
│         - ...                       │
│                                     │
└─────────────────────────────────────┘
```

**交互特点：**
- 使用 Element UI 的 `el-drawer` 全屏模式 (`size="100%"`)
- 只显示当前单个用例的详情
- 通过顶部的"上一个/下一个"按钮切换用例
- 需要关闭抽屉才能回到用例列表
- 无法快速跳转到其他用例

**痛点：**
1. **视野受限**: 只能看到当前用例，不知道还有哪些用例待执行
2. **切换困难**: 需要连续点击翻页按钮才能找到目标用例
3. **效率低下**: 无法跳过某些用例，必须顺序浏览
4. **缺少筛选**: 无法按执行状态筛选用例

### 3.2 新版本实现思路

**布局方式：左右分栏 + 列表详情模式**

```
┌──────────┬──────────────────────────┐
│ 搜索框   │  用例详情标题              │ 顶部
├──────────┼──────────────────────────┤
│ [筛选器] │  [基本信息] [详情] [缺陷] │ Tab切换
├──────────┼──────────────────────────┤
│ 用例001  │                          │
│ 用例002★ │     详情内容区域          │ 主体区域
│ 用例003  │     - 用例步骤            │
│ 用例004  │     - 预期结果            │ 左侧:列表
│ ...      │     - 执行信息            │ 右侧:详情
│          │                          │
├──────────┼──────────────────────────┤
│ [分页器] │  [执行提交区域]           │ 底部
└──────────┴──────────────────────────┘

★ 表示当前激活用例
```

**交互特点：**
- 左侧固定 318px 宽度显示用例列表
- 右侧弹性宽度显示用例详情
- 左侧列表支持：
  - 搜索（关键词）
  - 筛选（按执行结果）
  - 分页浏览
  - 点击切换（激活状态高亮）
- 右侧详情支持：
  - 多Tab切换（基本信息/详情/缺陷/历史）
  - 直接编辑（二级抽屉）
  - 执行提交（底部固定）

**优势：**
1. **全局视野**: 一屏看到所有用例概览
2. **快速切换**: 点击列表项直接跳转
3. **精准筛选**: 按状态筛选，快速定位
4. **灵活搜索**: 关键词搜索目标用例
5. **效率提升**: 可以跳过已执行用例，专注未完成部分

## 四、核心UI组件架构对比

### 4.1 旧版本组件架构

**组件层次：**
```
页面级组件 (TestCaseList)
  ↓ 路由跳转
独立页面组件 (TestCaseEdit)
  ↓ 包含
各种业务组件 (表单、步骤、附件等)
```

**特点：**
- 扁平化架构
- 组件之间通过路由连接
- 使用 Element UI 原生组件 (`el-drawer`, `el-dialog`)
- 组件间通信主要通过路由参数

### 4.2 新版本组件架构

**组件层次：**
```
页面级组件 (CaseTable)
  ↓ 打开抽屉
通用抽屉容器 (MsDetailDrawer)
  ↓ 内嵌
业务详情组件 (CaseDetailDrawer)
  ↓ 再打开抽屉
二级编辑抽屉 (EditCaseDetailDrawer)
  ↓ 包含
复用的业务组件 (CaseTemplateDetail)
```

**特点：**
- 分层架构，职责清晰
- 容器组件 + 业务组件分离
- 自定义增强型组件 (`MsDrawer`, `MsDetailDrawer`)
- 组件间通信通过 `emit` 事件和 `v-model`

## 五、关键技术实现差异

### 5.1 抽屉组件封装

**旧版本：直接使用 Element UI**
- 功能：基础的显示/隐藏、宽度设置
- 局限：功能单一，需要自己实现很多逻辑

**新版本：自定义增强型抽屉**

`MsDrawer` 增强功能：
- ✅ **全屏切换**: 一键切换全屏/普通模式
- ✅ **宽度拖拽**: 鼠标拖拽边缘调整宽度
- ✅ **无遮罩模式**: 可以同时操作背景内容
- ✅ **保存并继续**: 保存后不关闭，继续创建
- ✅ **插槽丰富**: 标题左右、页脚左右多个插槽

`MsDetailDrawer` 业务增强：
- ✅ **上下翻页**: 集成翻页按钮组件
- ✅ **自动加载**: 自动调用详情接口
- ✅ **表格联动**: 与父页面表格数据同步
- ✅ **加载状态**: 统一管理 loading 状态

### 5.2 用例切换机制

**旧版本：翻页按钮模式**
- 实现：`ms-previous-next-button` 组件
- 逻辑：基于当前索引 +1 或 -1
- 缺点：只能顺序切换，无法跳跃

**新版本：列表点击模式**
- 实现：左侧渲染用例列表
- 逻辑：点击列表项触发 `changeActiveCase(item)`
- 优点：任意跳转 + 状态筛选 + 关键词搜索

### 5.3 数据刷新策略

**旧版本：手动刷新**
```javascript
// 编辑完成后
window.close() // 关闭编辑窗口
// 用户需要手动点击列表页的刷新按钮
```

**新版本：自动刷新**
```javascript
// 编辑完成后
emit('success')  // 触发父组件事件
  ↓
父组件监听 @success
  ↓
自动调用 initData() 刷新列表
  ↓
关闭抽屉
```

## 六、左右分栏布局设计思路

### 6.1 布局核心思想

**Master-Detail 模式**
- **Master (列表)**: 提供数据概览和导航
- **Detail (详情)**: 展示完整信息和操作

**设计原则：**
1. **固定宽度列表**: 左侧 318px 固定，保证列表项可读
2. **弹性宽度详情**: 右侧自适应，充分利用空间
3. **独立滚动区域**: 左右各自滚动，互不影响
4. **视觉分隔明确**: 使用边框或阴影区分两侧

### 6.2 左侧列表设计要点

**信息密度平衡：**
- 显示关键信息：编号、名称、状态
- 隐藏次要信息：详细描述、创建人等
- 使用图标/标签：执行结果用颜色标识

**交互状态设计：**
```
正常状态: 白色背景 + 灰色边框
悬停状态: 边框变为主题色
激活状态: 背景色 + 主题色边框
```

**搜索和筛选：**
- 顶部搜索框：实时搜索用例名称/编号
- 筛选下拉：按执行结果多选筛选
- 筛选后重新分页，保持列表连贯

### 6.3 右侧详情设计要点

**分 Tab 组织信息：**
- 基本信息 Tab：模块、标签、评审状态、自定义字段
- 详情 Tab：步骤描述、预期结果、实际结果
- 缺陷列表 Tab：关联的 Bug 列表
- 执行历史 Tab：历次执行记录

**底部操作区固定：**
- 使用 `position: sticky` 或 `fixed`
- 始终可见执行提交按钮
- 避免滚动后找不到提交入口

## 七、抽屉嵌套设计思路

### 7.1 为什么需要嵌套抽屉

**场景举例：**
```
用户在查看用例详情时
  ↓
发现需要修改某些字段
  ↓
不想关闭详情抽屉（会丢失上下文）
  ↓
在详情抽屉上再打开编辑抽屉
  ↓
编辑完成后关闭编辑抽屉
  ↓
详情抽屉自动刷新显示最新数据
```

### 7.2 嵌套层级控制

**层级规划：**
```
第一层: 列表详情抽屉 (只读查看)
  宽度: 860px
  作用: 展示用例完整信息
  
第二层: 编辑抽屉 (表单编辑)
  宽度: 1200px (更宽，容纳更多表单)
  作用: 修改用例字段
  
第三层: 辅助弹窗 (如附件预览、关联选择)
  宽度: 根据内容自适应
  作用: 临时操作
```

**建议：**
- ✅ 最多三层嵌套，更多层次用户会迷失
- ✅ 每层宽度递增，视觉上有层次感
- ✅ 深层抽屉应该有明确的标题表明所在位置

### 7.3 事件通信机制

**父子组件通信：**
```
父组件 (列表页)
  ↓ props 传递
子组件 (详情抽屉)
  visible: boolean
  detailId: string
  
子组件
  ↓ emit 回传
父组件
  @success="refreshList"
  @loaded="handleLoaded"
```

**跨层级通信（避免）：**
- ❌ 不建议：孙组件直接 emit 到祖先组件
- ✅ 推荐：逐层传递事件，或使用全局状态管理

## 八、改造旧项目的实施思路

### 8.1 渐进式改造策略

**阶段一：封装通用抽屉组件**
- 创建 `MsDrawer` 组件（增强功能）
- 创建 `MsDetailDrawer` 组件（业务封装）
- 不影响现有功能，先准备好基础设施

**阶段二：改造用例编辑功能**
- 创建 `CaseDetailDrawer` 组件
- 修改列表页的编辑按钮逻辑
- 从 `openCaseEdit()` 改为 `showDetailDrawer = true`
- 保留原路由作为备用（兼容旧链接）

**阶段三：改造测试计划执行页**
- 重构页面布局为左右分栏
- 左侧添加用例列表组件
- 右侧复用现有详情内容
- 调整样式和交互逻辑

**阶段四：优化和统一**
- 统一所有列表页的编辑方式
- 添加更多增强功能（搜索、筛选）
- 性能优化（虚拟滚动、懒加载）

### 8.2 技术风险和应对

**风险1：组件复杂度增加**
- 应对：做好组件文档，明确 props 和 events
- 应对：编写示例代码，降低使用门槛

**风险2：兼容性问题**
- 应对：保留旧路由，通过 URL 参数判断使用新旧模式
- 应对：灰度发布，部分用户先试用新版

**风险3：性能问题**
- 应对：抽屉内容懒加载，打开时才请求数据
- 应对：列表虚拟滚动，处理大量用例场景

**风险4：移动端适配**
- 应对：小屏设备自动降级为全屏抽屉
- 应对：左右分栏在移动端改为上下滑动

## 十、最佳实践建议

### 10.1 抽屉使用原则

**什么时候用抽屉：**
- ✅ 快速查看详情，不需要完全专注
- ✅ 需要保持原页面上下文
- ✅ 临时性操作，完成后返回原位置
- ✅ 需要频繁在列表和详情间切换

**什么时候用独立页面：**
- ✅ 复杂的多步骤流程（向导式）
- ✅ 需要专注的长时间操作（写文档、编程）
- ✅ 需要分享链接给他人打开
- ✅ 可能需要浏览器标签页管理

### 10.2 左右分栏使用原则

**什么时候用左右分栏：**
- ✅ 有明确的列表和详情关系
- ✅ 用户需要频繁切换查看
- ✅ 屏幕宽度足够（≥1280px）
- ✅ 列表项数量适中（不要太多层级）

**什么时候避免左右分栏：**
- ❌ 移动端或小屏设备
- ❌ 列表和详情都需要很大空间
- ❌ 交互逻辑过于复杂
- ❌ 用户只关注单个项目

### 10.3 性能优化建议

**抽屉懒加载：**
```javascript
// 打开抽屉时才加载详情数据
watch(() => showDrawer.value, (visible) => {
  if (visible) {
    loadDetailData()
  }
})
```

**列表虚拟滚动：**
- 当列表数据 > 100 条时，使用虚拟滚动
- 只渲染可视区域的 DOM 节点
- 推荐库：`vue-virtual-scroller`

**分页优化：**
- 左侧列表使用小分页 (每页10-20条)
- 提供快速跳页功能
- 记住用户的分页偏好

## 十一、总结

### 核心观点

1. **抽屉模式 > 新窗口模式**：在现代Web应用中，抽屉能更好地保持用户上下文，减少认知负担。

2. **左右分栏 > 全屏单列**：对于列表详情场景，分栏布局能显著提升切换效率和全局掌控感。

3. **自动化 > 手动操作**：能自动完成的操作（如刷新、关闭）就不要让用户手动点击。

4. **渐进式改造 > 推倒重来**：通过封装通用组件，逐步替换旧逻辑，降低风险。

### 改造价值

- **用户体验**：操作更流畅，学习成本更低
- **开发效率**：通用组件复用，后续开发更快
- **产品竞争力**：现代化UI提升产品形象
- **维护成本**：组件化架构更易维护
