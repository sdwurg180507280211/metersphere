# develop 与 master 分支选择性合并方法论

> 最后更新：2026-06-01

本文从 `docs/INDEX.md` 迁移并补充，用于指导 `develop-v2.10.26` 与 `master` 之间的选择性代码同步。

当前仓库默认远程已调整为 GitHub 仓库 `https://github.com/sdwurg180507280211/metersphere.git`，本地 `master` 跟踪 `origin/master`，本地 `develop-v2.10.26` 跟踪 `origin/develop-v2.10.26`。两个分支应保持同名分支推送，不通过整分支 PR 互相覆盖。

当前保护清单已结合近期 micro-app 相关提交核对：`6b34cb9daa`、`4fb5d84e6e`、`088065d002`、`4cbcd13dc9`、`67945c5b8b`、`b85cfb6a06`、`ec495520b9`、`21c40a9b34`。当前差异分类另见 `docs/04-技术架构/微前端架构/develop-master差异总结.md`。

## 1. 适用场景

当前仓库中，`master` 已包含微前端相关改造，并且包含 `develop-v2.10.26` 没有的独立分析统计服务 `analytics-stat`。`develop-v2.10.26` 仍持续承载业务代码演进。直接执行普通 `git merge` 容易把以下内容混在一起：

- 文档目录、脚本、配置文件；
- 构建产物；
- `master` 独有的 micro-app 微前端代码；
- `master` 独有的 `analytics-stat` 分析统计微服务；
- `develop` 新增或修改的业务代码。

因此推荐使用“选择性同步”策略：**不做全量 merge，而是按规则从来源分支检出需要同步的代码文件，再恢复保护文件并验证暂存区。**

## 2. 核心原则

1. **只同步代码，不同步文档和构建产物。**
2. **保护 `master` 独有的 micro-app 文件和 `analytics-stat` 独立微服务，避免被 develop 旧代码覆盖或删除。**
3. **所有同步结果以暂存区为准，提交前必须检查 `git diff --cached`。**
4. **删除、重命名、SQL migration 必须单独审核，不能只按文件后缀自动处理。**
5. **该流程不是 Git 语义上的 merge，提交信息中必须记录来源分支和来源 commit。**

## 3. 文件分类规则

| 类别 | 文件类型 | 默认处理方式 |
|------|----------|--------------|
| 后端代码 | `.java`、MyBatis mapper `.xml`、业务 `.properties` | 从来源分支同步 |
| 前端代码 | `.vue`、`.js`、`.ts`、源码目录下的 `.css`/`.scss`/`.less` | 从来源分支同步 |
| 前端依赖 | `package.json`、lock 文件 | 如代码引入新依赖，必须同步并验证构建 |
| Flyway SQL | 新增版本化 migration | 可以同步，但必须按第 9 节规则检查 |
| 文档 | `.md`、`.html`、`.docx`、`docs/` 下文件 | 默认保留目标分支 |
| 脚本/配置 | `.sh`、`.yml`、`.example`、`Makefile` | 默认保留目标分支，确需同步时人工审核 |
| 构建产物 | `dist/`、`static/js/`、`target/`、`node_modules/` | 不同步，各分支自行构建 |

注意：后缀规则只是候选筛选，最终仍要结合路径和业务语义判断。例如 `.xml` 可能是 MyBatis mapper，也可能是配置；`.css` 可能是源码，也可能是构建产物。

## 4. develop -> master 推荐流程

### 4.1 前置检查

操作前必须确保工作区干净：

```bash
git status --porcelain
```

如果有输出，先提交、stash，或使用独立 worktree 操作。不要在已有本地改动的 `master` 上直接执行同步。

### 4.2 更新分支并固定来源

推荐以远端分支为来源，避免本地 develop 不是最新：

```bash
git fetch origin

git checkout master
git pull --ff-only

SOURCE=origin/develop-v2.10.26
MASTER_COMMIT=$(git rev-parse HEAD)
BASE=$(git merge-base HEAD "$SOURCE")
```

其中：

- `SOURCE`：本次同步来源；
- `MASTER_COMMIT`：同步前的 master 版本，用于恢复保护文件；
- `BASE`：master 与 develop 的共同祖先，用于查看 develop 相对基线的变化。

### 4.3 先看状态差异，不要只看文件名

```bash
git diff --name-status --find-renames "$BASE" "$SOURCE"
```

需要特别关注：

- `A`：新增文件；
- `M`：修改文件；
- `D`：删除文件；
- `R`：重命名文件。

不要只使用 `git diff --name-only`，否则容易漏掉删除和重命名语义。

### 4.4 生成代码候选清单

```bash
git diff --name-only --diff-filter=ACMRT "$BASE" "$SOURCE" -- \
  | grep -E '(\.java|\.vue|\.js|\.ts|\.xml|\.properties|\.json|\.css|\.scss|\.less|\.sql)$' \
  | grep -Ev '(^docs/|/dist/|/static/js/|/target/|/node_modules/)' \
  > /tmp/develop-code-files.txt
```

然后人工审核候选清单：

```bash
less /tmp/develop-code-files.txt
```

重点检查：

- 是否误包含文档、构建产物、环境配置；
- 是否漏掉 `package.json`、lock 文件等依赖变更；
- `.xml` 是否真的是业务代码或 mapper；
- `.sql` 是否符合 Flyway 规则。

### 4.5 批量同步代码文件

确认候选清单无误后执行：

```bash
if [ -s /tmp/develop-code-files.txt ]; then
  git restore --source="$SOURCE" --staged --worktree --pathspec-from-file=/tmp/develop-code-files.txt
fi
```

相比逐个执行 `git checkout develop-v2.10.26 -- <file>`，该方式更可重复，也更不容易漏文件。

### 4.6 单独处理删除和重命名

查看删除、重命名文件：

```bash
git diff --name-status --diff-filter=DR --find-renames "$BASE" "$SOURCE"
```

处理原则：

- develop 删除的代码文件，如果确认为业务删除，应在 master 中 `git rm <file>`；
- develop 重命名的代码文件，应删除旧路径并同步新路径；
- 对于配置、文档、构建产物的删除或重命名，默认不跟随 develop。

其中 `analytics-stat/` 是整目录保护项。若 develop 同步后出现 `analytics-stat` 下大量 `D` 删除状态，执行上述恢复命令会把该目录恢复为同步前的 master 版本。恢复后仍需通过第 8 节检查确认没有误带入 `dist/`、`static/` 等构建产物。

## 7. 检查保护文件中是否存在 develop 业务改动

整文件恢复保护文件可能会回退 develop 在这些文件中的业务改动，因此必须检查 develop 相对 `BASE` 是否修改过保护文件：

```bash
while IFS= read -r f; do
  git diff "$BASE" "$SOURCE" -- "$f"
done < /tmp/merge-protect-files.txt
```

判断规则：

- 如果差异仅是 develop 缺少 master 的 micro-app 代码，可以保持 master 版本；
- 如果差异包含 develop 新增的业务路由、接口、字段、文案等，不能简单整文件恢复，必须人工把业务改动合入 master；
- 人工合并后，再检查该文件的暂存区差异。

## 8. 提交前验证

### 8.1 查看暂存区文件类型

```bash
git diff --cached --name-status
git diff --cached --stat
```

确认暂存区只包含本次需要同步的代码文件。

### 8.2 确认保护文件没有被误覆盖

```bash
while IFS= read -r f; do
  git diff --cached -- "$f"
done < /tmp/merge-protect-files.txt
```

如果某个保护文件确实需要包含 develop 的业务改动，应人工确认后保留；否则这里不应有输出。

### 8.3 检查是否误同步文档或构建产物

```bash
git diff --cached --name-only | grep -E '(^docs/|/dist/|/static/js/|/target/|/node_modules/)'
```

如果有输出，通常应从暂存区和工作区恢复这些文件。

## 9. Flyway SQL 规则

Flyway migration 不能简单按 `.sql` 后缀同步，必须区分新增、修改和删除。

### 9.1 可以同步

新增版本化 migration，例如：

```text
V150__add_xxx_table.sql
```

前提是版本号没有和 master 已有版本冲突。

### 9.2 默认禁止直接同步

以下场景需要人工处理：

- 修改已存在的 `Vxxx__xxx.sql`；
- 删除已存在的 `Vxxx__xxx.sql`；
- develop 与 master 使用相同版本号但文件名或内容不同；
- 修改已经在环境中执行过的 migration。

原因是 Flyway 会校验已执行 migration 的 checksum。修改历史 SQL 可能导致服务启动时报 checksum mismatch。正确做法通常是新增一个后续版本的修正 migration。

### 9.3 提交前单独检查 SQL

```bash
git diff --cached --name-status -- '*.sql'
```

对每个 SQL 文件确认：

- 是新增还是修改；
- 版本号是否冲突；
- 是否会影响已执行环境；
- 是否需要 DBA 或后端负责人复核。

## 10. 构建与测试

提交前至少执行与本次变更相关的构建或测试。根据变更范围选择：

- 后端 Java 变更：执行对应 Maven 模块测试或编译；
- 前端变更：执行对应模块的依赖安装检查和构建；
- SQL 变更：执行 Flyway 校验或启动相关服务验证 migration。

不要在未验证构建的情况下提交涉及依赖、路由、入口文件或 SQL migration 的同步。

## 11. 提交信息建议

由于该流程不是 Git merge，Git 历史不会记录 develop 已经被语义合并。因此提交信息应记录来源信息：

```text
chore: sync develop-v2.10.26 code changes to master

Source: origin/develop-v2.10.26@<source_commit>
Base: <merge_base_commit>
Scope: code files only, excluding docs/build artifacts/config
Protected: master-only micro-app files restored or manually merged
```

这样后续排查时可以知道本次同步来自哪个 develop commit。

## 12. 普通业务改动双分支同步：优先 cherry-pick

如果某次修改不涉及以下专项内容：

- `master` 独有的 micro-app 迁移代码；
- `master` 独有的 AI/分析统计模块；
- workflow 独立方向；
- 文档、构建产物、运行日志、本地环境配置；

而只是普通业务代码修复或功能调整，推荐使用 `cherry-pick` 在两个分支之间同步对应提交，不要执行整分支 merge，也不要创建 `master -> develop-v2.10.26` 或 `develop-v2.10.26 -> master` 的整分支 PR。

### 12.1 从 master 同步普通业务提交到 develop

适用场景：先在 `master` 完成并提交普通业务改动，希望同步到 `develop-v2.10.26`。

```bash
# 在 master 上完成提交后记录提交号
git switch master
git status --porcelain
COMMIT=$(git rev-parse HEAD)

# 推送 master 同名分支
git push origin master

# 切到 develop 同名分支并保持最新
git switch develop-v2.10.26
git pull --ff-only

# 只同步这个业务提交
git cherry-pick -x "$COMMIT"

# 验证后推送 develop 同名分支
git push origin develop-v2.10.26
```

`-x` 会在提交信息中追加来源提交号，便于后续追溯：

```text
(cherry picked from commit <commit>)
```

如果有多个连续提交：

```bash
git cherry-pick -x <first_commit>^..<last_commit>
```

如果是多个不连续提交：

```bash
git cherry-pick -x <commit1> <commit2> <commit3>
```

### 12.2 从 develop 同步普通业务提交到 master

适用场景：先在 `develop-v2.10.26` 完成并提交普通业务改动，希望同步到 `master`。

```bash
git switch develop-v2.10.26
git status --porcelain
COMMIT=$(git rev-parse HEAD)

git push origin develop-v2.10.26

git switch master
git pull --ff-only

git cherry-pick -x "$COMMIT"

git push origin master
```

### 12.3 cherry-pick 冲突处理

如果 cherry-pick 发生冲突：

```bash
git status
```

只解决当前业务提交涉及的冲突。不要借冲突处理顺手同步 micro-app、AI/分析统计、workflow、文档或构建产物。

解决后继续：

```bash
git add <resolved-files>
git cherry-pick --continue
```

如果发现该提交实际包含专项内容，不适合直接同步，应中止并重新拆分提交：

```bash
git cherry-pick --abort
```

### 12.4 分支与远程约定

当前默认远程为 GitHub：

```text
origin = https://github.com/sdwurg180507280211/metersphere.git
```

分支应保持同名对应：

```text
master              -> origin/master
develop-v2.10.26    -> origin/develop-v2.10.26
```

普通同步完成后分别推送同名分支：

```bash
git push origin master
git push origin develop-v2.10.26
```

不要用整分支 PR 表达普通业务同步。GitHub 上如果出现 `develop-v2.10.26 -> master` 或 `master -> develop-v2.10.26` 的整分支 PR，应先确认是否只包含目标业务提交；否则应关闭或改用 cherry-pick 后的同名分支推送。

## 13. master -> develop 选择性同步兜底方案

当历史上已经在 `master` 累积了多处普通业务改动，且无法明确拆成单个提交时，可以使用选择性文件同步兜底。该方案风险高于 cherry-pick，只适合人工确认过的少量文件。

```bash
git status --porcelain

git fetch origin

git checkout develop-v2.10.26
git pull --ff-only

SOURCE=origin/master
DEVELOP_COMMIT=$(git rev-parse HEAD)
BASE=$(git merge-base HEAD "$SOURCE")

git diff --name-status --find-renames "$BASE" "$SOURCE"
```

然后人工挑选普通业务代码文件，避免包含 micro-app、AI/分析统计、workflow、文档、构建产物和环境配置。

```bash
git diff --name-only --diff-filter=ACMRT "$BASE" "$SOURCE" -- \
  | grep -E '(\.java|\.vue|\.js|\.ts|\.xml|\.properties|\.json|\.css|\.scss|\.less|\.sql)$' \
  | grep -Ev '(^docs/|/dist/|/static/js/|/target/|/node_modules/|^analytics-stat/)' \
  > /tmp/master-code-files.txt
```

人工审核后再同步：

```bash
if [ -s /tmp/master-code-files.txt ]; then
  git restore --source="$SOURCE" --staged --worktree --pathspec-from-file=/tmp/master-code-files.txt
fi
```

注意：

- 该方案不是默认方案；默认优先 cherry-pick。
- 不建议把 micro-app 相关文件同步到 develop，除非明确要推进 develop 也接收 micro-app 架构。
- 不建议把 AI/分析统计模块同步到 develop，除非明确要让 develop 接收该模块。
- 同步后必须检查 `git diff --cached --name-status`，确认暂存区只包含本次目标业务文件。

## 14. 常见风险清单

| 风险 | 后果 | 建议 |
|------|------|------|
| 工作区不干净就开始同步 | 本地改动混入合并提交或被覆盖 | 操作前必须 `git status --porcelain` 为空 |
| 保护清单过期 | master-only micro-app 代码被 develop 覆盖 | 维护 `.merge-protect`，并定期核对 |
| 只看 `name-only` | 漏掉删除、重命名 | 使用 `git diff --name-status --find-renames` |
| 整文件恢复保护文件 | 回退 develop 业务改动 | 检查 `git diff "$BASE" "$SOURCE" -- <protected_file>` |
| 修改历史 Flyway SQL | 启动时报 checksum mismatch | 新增修正 migration，不改已执行 migration |
| 不同步依赖文件 | 前端构建失败 | 涉及新依赖时同步 `package.json` 和 lock 文件 |
| 把构建产物提交进来 | 污染分支，制造无意义冲突 | 排除 `dist/`、`static/js/`、`target/` |
| 提交信息不记录来源 | 后续无法追溯同步范围 | 记录 source commit、merge-base、保护策略 |
