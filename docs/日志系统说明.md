# MeterSphere 日志系统说明

本文总结当前项目中后端服务的日志配置方式，重点说明本地运行时 `/opt/metersphere/logs` 目录下日志的来源、结构和排查方法。

## 1. 日志系统总体设计

MeterSphere 后端服务使用 Spring Boot 默认日志体系，具体日志实现为 Logback。各服务在自己的 `src/main/resources/logback.xml` 中定义日志输出规则，并通过 `application.properties` 中的 `logging.file.path` 指定日志落盘目录。

日志配置有几个共同特点：

- 每个微服务独立写入 `/opt/metersphere/logs/<服务目录>`。
- 主要日志按级别拆分为 `info.log`、`warn.log`、`error.log`、`debug.log`。
- 日志采用 Logback `RollingFileAppender` 按日期和大小滚动。
- 历史日志统一进入各服务目录下的 `history/` 子目录。
- 默认保留 30 天历史日志，可通过外部配置 `logger.max.history` 覆盖。
- 多数文件 Appender 外层包了一层 `AsyncAppender`，队列大小为 10000，减少业务线程直接写文件的阻塞。
- 日志级别默认取 `${logger.level:INFO}`，即外部未配置时为 `INFO`。

## 2. 服务日志目录映射

各服务的日志目录由 `application.properties` 指定：

| 服务 | 配置文件 | 日志目录 |
| --- | --- | --- |
| gateway | `framework/gateway/src/main/resources/application.properties` | `/opt/metersphere/logs/gateway` |
| eureka | `framework/eureka/src/main/resources/application.properties` | `/opt/metersphere/logs/${spring.application.name}`，默认 `/opt/metersphere/logs/eureka` |
| api-test | `api-test/backend/src/main/resources/application.properties` | `/opt/metersphere/logs/api-test` |
| performance-test | `performance-test/backend/src/main/resources/application.properties` | `/opt/metersphere/logs/performance-test` |
| project-management | `project-management/backend/src/main/resources/application.properties` | `/opt/metersphere/logs/project-management` |
| report-stat | `report-stat/backend/src/main/resources/application.properties` | `/opt/metersphere/logs/report-stat` |
| system-setting | `system-setting/backend/src/main/resources/application.properties` | `/opt/metersphere/logs/system-setting` |
| test-track | `test-track/backend/src/main/resources/application.properties` | `/opt/metersphere/logs/test-track` |
| workstation | `workstation/backend/src/main/resources/application.properties` | `/opt/metersphere/logs/workstation` |

因此本地 `/opt/metersphere/logs` 的典型结构如下：

```text
/opt/metersphere/logs
├── gateway
├── eureka
├── api-test
├── performance-test
├── project-management
├── report-stat
├── system-setting
├── test-track
└── workstation
```

## 3. 单个服务目录结构

以 `/opt/metersphere/logs/api-test` 为例，常见结构如下：

```text
/opt/metersphere/logs/api-test
├── info.log                 # INFO 及以上主日志
├── warn.log                 # WARN 日志
├── error.log                # ERROR 日志
├── debug.log                # DEBUG 日志
├── api
│   └── info.log             # Controller 调用日志
├── eureka
│   └── info.log             # Netflix/Eureka 相关日志
├── ms-jmeter-run.log        # api-test 模块中的 JMeter 执行过程日志
└── history
    ├── info.20260511-0.log
    ├── warn.20260511-0.log
    ├── error.20260511-0.log
    ├── debug.20260511-0.log
    ├── api
    │   └── info.20260511-0.log
    └── eureka
        └── info.20260511-0.log
```

其他业务服务大体类似，但会根据模块功能多出特定日志，例如性能测试、项目管理、报表、系统设置、工作台等模块中可能存在：

```text
ms-jmeter-run-log.log
history/ms-jmeter-run-log.<日期>-<序号>.log
testPlan/info.log
history/testPlan/info.<日期>-<序号>.log
```

`test-track` 模块没有 JMeter 执行日志配置，但包含测试计划执行日志 `testPlan/info.log`。

## 4. 主要日志文件含义

### 4.1 `info.log`

服务主日志。大部分业务日志、启动日志、框架日志都会进入这里。

对应 Logback 配置：

```xml
<File>${logging.file.path}/info.log</File>
<filter class="ch.qos.logback.classic.filter.ThresholdFilter">
    <level>INFO</level>
</filter>
```

注意：这里使用的是 `ThresholdFilter`，所以 `info.log` 不只包含 INFO，也可能包含 WARN、ERROR。

### 4.2 `warn.log`

只收集 WARN 级别日志。

对应配置使用 `LevelFilter`：

```xml
<level>WARN</level>
<onMatch>ACCEPT</onMatch>
<onMismatch>DENY</onMismatch>
```

### 4.3 `error.log`

只收集 ERROR 级别日志，排查异常优先看这个文件。

对应配置同样使用 `LevelFilter`，只接收 ERROR。

### 4.4 `debug.log`

只收集 DEBUG 级别日志。默认服务日志级别是 INFO，所以如果没有把 `logger.level` 调整为 DEBUG，通常不会有太多内容。

### 4.5 `api/info.log`

Controller 接口调用日志。

来源是公共 SDK 中的切面：

```text
framework/sdk-parent/sdk/src/main/java/io/metersphere/controller/handler/WebLogAspect.java
```

该切面拦截：

```java
execution(public * io.metersphere..*.*Controller.*(..))
```

输出内容包括：

- HTTP 方法
- 请求 URL
- 接口耗时
- DEBUG 级别下还会输出参数和返回值

普通 INFO 级别下示例格式大致为：

```text
GET: http://localhost:8004/api/xxx TIME: 35ms
```

### 4.6 `eureka/info.log`

Netflix Eureka 相关日志会单独进入该文件，主要来自 logger 名称 `com.netflix`。

### 4.7 `ms-jmeter-run.log` / `ms-jmeter-run-log.log`

JMeter 执行过程日志，logger 名称是：

```text
ms-jmeter-run-log
```

公共工具类位置：

```text
framework/sdk-parent/jmeter/src/main/java/io/metersphere/utils/LoggerUtil.java
```

该工具通过：

```java
LoggerFactory.getLogger("ms-jmeter-run-log")
```

写入 JMeter 执行过程日志。

不同模块文件名略有差异：

| 模块 | JMeter 执行日志文件 |
| --- | --- |
| api-test | `ms-jmeter-run.log` |
| performance-test | `ms-jmeter-run-log.log` |
| project-management | `ms-jmeter-run-log.log` |
| report-stat | `ms-jmeter-run-log.log` |
| system-setting | `ms-jmeter-run-log.log` |
| workstation | `ms-jmeter-run-log.log` |

### 4.8 `testPlan/info.log`

测试计划执行日志，logger 名称是：

```text
testPlanExecuteLog
```

在 `test-track`、`performance-test`、`project-management`、`report-stat`、`system-setting`、`workstation` 等模块的 `logback.xml` 中都有相关配置。

## 5. 日志滚动与保留策略

各文件日志基本使用相同的滚动策略：

```xml
<rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
    <FileNamePattern>${logging.file.path}/history/info.%d{yyyyMMdd}-%i.log</FileNamePattern>
    <maxHistory>${logger.max.history:-30}</maxHistory>
    <TimeBasedFileNamingAndTriggeringPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedFNATP">
        <maxFileSize>50MB</maxFileSize>
    </TimeBasedFileNamingAndTriggeringPolicy>
</rollingPolicy>
```

含义：

- 按天滚动，文件名带 `yyyyMMdd`。
- 单个归档文件超过 `50MB` 后按序号继续切分。
- 历史日志默认保留 `30` 天。
- 可以通过 `/opt/metersphere/conf/metersphere.properties` 中的 `logger.max.history` 覆盖保留天数。

例如：

```properties
logger.max.history=7
```

表示历史日志保留 7 天。

## 6. 外部配置覆盖

多数 `logback.xml` 都会加载外部配置文件：

```xml
<property file="/opt/metersphere/conf/metersphere.properties"/>
```

部分业务模块还会加载公共配置：

```xml
<property resource="commons.properties"/>
```

因此日志相关参数可以通过 `/opt/metersphere/conf/metersphere.properties` 覆盖，例如：

```properties
logger.level=DEBUG
logger.sql.level=DEBUG
logger.max.history=15
```

常见参数含义：

| 参数 | 作用 | 默认值 |
| --- | --- | --- |
| `logger.level` | `io.metersphere` 等业务 logger 的日志级别 | `INFO` |
| `logger.sql.level` | MyBatis mapper 日志级别 | 无默认值，需外部配置时生效 |
| `logger.max.history` | 历史日志保留天数 | `30` |

注意：`logger.sql.level` 主要绑定到：

```xml
<logger name="io.metersphere.base.mapper" level="${logger.sql.level}">
    <appender-ref ref="console"/>
</logger>
```

SQL 日志默认输出到 console，不一定进入业务日志文件。

## 7. Console 日志与文件日志的关系

各服务同时配置了：

```xml
<appender name="console" class="ch.qos.logback.core.ConsoleAppender">
```

`root` logger 通常包含：

```xml
<root level="INFO">
    <appender-ref ref="infoAsyncAppender"/>
    <appender-ref ref="console"/>
</root>
```

所以日志会同时输出到：

- 服务标准输出，即容器环境中的 `docker logs`。
- `/opt/metersphere/logs/<服务>/info.log` 等文件。

如果是 Docker 部署，`docker logs <container>` 看到的是标准输出；`/opt/metersphere/logs` 下看到的是 Logback 文件 Appender 写出的落盘日志。

## 8. 常用排查路径

### 8.1 服务启动失败

优先看对应服务主日志和错误日志：

```bash
tail -f /opt/metersphere/logs/<服务>/info.log
tail -f /opt/metersphere/logs/<服务>/error.log
```

例如接口测试服务：

```bash
tail -f /opt/metersphere/logs/api-test/info.log
tail -f /opt/metersphere/logs/api-test/error.log
```

### 8.2 接口请求慢或接口异常

先看接口调用日志：

```bash
tail -f /opt/metersphere/logs/<服务>/api/info.log
```

这里能看到 Controller 请求耗时。然后再结合：

```bash
tail -f /opt/metersphere/logs/<服务>/info.log
tail -f /opt/metersphere/logs/<服务>/error.log
```

### 8.3 JMeter / 接口自动化 / 性能测试执行异常

优先看执行过程日志：

```bash
tail -f /opt/metersphere/logs/api-test/ms-jmeter-run.log
tail -f /opt/metersphere/logs/performance-test/ms-jmeter-run-log.log
```

再结合服务主日志：

```bash
tail -f /opt/metersphere/logs/api-test/error.log
tail -f /opt/metersphere/logs/performance-test/error.log
```

### 8.4 测试计划执行问题

查看测试计划执行日志：

```bash
tail -f /opt/metersphere/logs/test-track/testPlan/info.log
```

如果问题涉及性能、项目、报表等模块，也可查看对应模块下的：

```bash
tail -f /opt/metersphere/logs/<服务>/testPlan/info.log
```

### 8.5 服务注册、服务发现问题

查看 Eureka 独立服务日志：

```bash
tail -f /opt/metersphere/logs/eureka/info.log
tail -f /opt/metersphere/logs/eureka/error.log
```

也可查看业务服务内单独收集的 Netflix/Eureka 日志：

```bash
tail -f /opt/metersphere/logs/<服务>/eureka/info.log
```

### 8.6 网关转发问题

查看 gateway：

```bash
tail -f /opt/metersphere/logs/gateway/info.log
tail -f /opt/metersphere/logs/gateway/error.log
```

如果请求已经转发到后端，还需要继续查看目标业务服务的 `api/info.log`、`info.log` 和 `error.log`。

## 10. 总结

`/opt/metersphere/logs` 是 MeterSphere 后端微服务的统一日志根目录。每个服务按照自己的服务名写入独立子目录，并在子目录内按日志用途和级别拆分文件。日常排查时可以按以下顺序定位：

1. 先确定问题属于哪个服务。
2. 看 `/opt/metersphere/logs/<服务>/error.log` 是否有异常。
3. 看 `/opt/metersphere/logs/<服务>/info.log` 了解上下文。
4. 接口问题看 `/opt/metersphere/logs/<服务>/api/info.log`。
5. JMeter 执行问题看 `ms-jmeter-run*.log`。
6. 测试计划问题看 `testPlan/info.log`。
7. 网关和注册发现问题分别看 `gateway`、`eureka` 目录。
