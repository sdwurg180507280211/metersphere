## 1. 节点定义（为什么要记）

我在做：统一“节点”口径，避免下次记录失真。
目的是：让每次升级记录都可横向对比（知道“哪个点是基线、哪个点是搬运结果”）。
如果不这样做,就无法实现：可持续维护（不同人记录的节点无法互相对照）。

- **Node-B（Base）**：官方纯净基线点（通常是官方 tag 或基于 tag 建的 `base/*` 分支）。
- **Node-Biz（Business）**：承载二开差异的业务分支结果点（例如 `feature/workflow-on-v2.10.26` 的 HEAD）。
- **Node-Entry（Team Entry）**：团队入口分支点（例如 `develop-v2.10.26` 的 HEAD）。
- **Node-FP（Forward-Port Window）**：旧入口分支（旧基线）最近增量，需要向新入口分支搬运的时间窗口边界与搬运结果点。

## 6. 下次升级（v2.10.26-lts → v2.10.xx-lts）你直接复用的“节点式流程”

我在做：把升级步骤写成“按节点推进”的清单。
目的是：你下次只要替换版本号与时间窗口，就能重复执行。
如果不这样做,就无法实现：升级流程的标准化（每次升级都像新项目一样重来）。

### 6.1 Node-B：创建新官方基线（纯官方）

```bash
# 我在做：拉取官方最新 tag（只更新引用，不动工作区）
# 目的是：拿到目标版本点 v2.10.xx-lts
# 如果不这样做,就无法实现：创建正确的基线节点
git fetch upstream --prune --tags

# 我在做：用 tag 创建纯官方基线分支
# 目的是：形成 Node-B@new（纯官方）
# 如果不这样做,就无法实现：二开差异与官方差异边界清晰
git checkout -b base/v2.10.xx-lts v2.10.xx-lts
```

### 6.2 Node-Biz：把“二开差异层”搬运到新基线

> 推荐以 diff/patch 驱动（而不是依赖 commit message）。

```bash
# 我在做：导出上一轮业务差异补丁（示例：从 v2.10.26-lts 到你维护的业务入口分支）
# 目的是：把二开差异变成可迁移输入
# 如果不这样做,就无法实现：在新基线复用你的二开代码
git diff --binary v2.10.26-lts..develop-v2.10.26 > /tmp/ms-diff-from-v2.10.26.patch

# 我在做：在新业务分支上应用补丁（3way 提升自动合并率）
# 目的是：快速形成 Node-Biz@new
# 如果不这样做,就无法实现：迁移效率（冲突会大量人工化）
git checkout -b feature/workflow-on-v2.10.xx base/v2.10.xx-lts
git apply --3way /tmp/ms-diff-from-v2.10.26.patch
```

### 6.3 Node-Entry：形成团队入口分支

```bash
# 我在做：从业务分支创建团队入口分支
# 目的是：让团队只面向一个入口开发
# 如果不这样做,就无法实现：入口统一与权限/保护策略
git checkout -b develop-v2.10.xx
```

### 6.4 Node-FP：旧入口分支的增量 forward-port

我在做：继续坚持“时间窗口 + 净差异”的 forward-port 策略。
目的是：避免 revert/重复提交导致的大规模冲突与噪音搬运。
如果不这样做,就无法实现：可控增量迁移（会陷入 cherry-pick 冲突地狱）。

- 建议每次升级前先在本文件新增：
