# 2025-12-30 二次开发提交合并：squash 并强推（Metersphere）

## 1. 我在做：把“最近 N 个二开提交”合并成 1 个提交
目的是：解决二开过程中多次提交导致 `git log` 很杂乱的问题，让团队能一眼看清“二开到底改了哪些文件”。
如果不这样做,就无法实现：历史记录里充满零散提交，后续升级/回溯时难以快速定位二开差异。

> 适用背景：
>
>- 基于官方特定版本（如 `v2.10.26-lts`）做二次开发
>- 开发过程中产生多次小提交（WIP/调试/修复）
>- 希望在最终交付/里程碑时，把这些提交压缩为 1 个“可读的业务提交”

---

## 2. 我在做：先说明这件事的本质（会改写历史）
目的是：让你在执行前就知道风险边界。
如果不这样做,就无法实现：在共享分支强推后，其他人会遇到 pull/push 失败甚至产生重复提交。

- `squash`（压缩提交）会导致：
  - 原来的多个 commit hash 全部变化（历史被重写）
- 如果这些提交已经推送到远端分支（例如 `origin/develop-v2.10.26`）：
  - 需要 **强推**（建议 `--force-with-lease`）才能把改写后的历史推回远端

结论：
- **能做**，但必须明确：这是“改写历史”的操作。

## 4. 我在做：本次实际执行的“非交互式 squash”方案（推荐记录）
目的是：避免 `git rebase -i` 依赖编辑器导致卡住，同时保证可回滚。
如果不这样做,就无法实现：交互式 rebase 可能因编辑器/冲突处理导致操作成本更高。

> 说明：本次目标是“只合并最近 6 个提交”为 1 个提交，并强推到 `origin/develop-v2.10.26`。

### 4.1 Step 0：确认当前分支与提交范围
我在做：确认 HEAD 与远端分支指向，以及最近提交列表。
目的是：确保合并范围准确（只合并最近 6 个提交，不多不少）。
如果不这样做,就无法实现：可能 squash 错范围，导致不该合并的提交也被重写。

命令（只读）：
- `git status -sb`
- `git log --oneline --decorate -n 10`

### 4.2 Step 1：确保工作区干净（二选一）
我在做：把未提交改动先提交或暂存（stash）。
目的是：保证 `reset/rebase` 在干净工作区执行。
如果不这样做,就无法实现：Git 可能拒绝执行，或把本地改动混入历史改写流程。

二选一：
- 方案 A（更稳）：
  - `git add -A`
  - `git commit -m "wip: before squash"`
- 方案 B（更快）：
  - `git stash -u`

### 4.3 Step 2：创建备份分支（强推前强烈建议）
我在做：创建一个备份分支指向当前 HEAD。
目的是：提供一键回滚的“安全点”。
如果不这样做,就无法实现：强推后想恢复原历史只能依赖 reflog，风险更高。

命令：
- `git branch backup/before-squash-last6-<timestamp>`

> 本次实际创建示例：`backup/before-squash-last6-20251230-163950`

### 4.4 Step 3：把最近 6 个提交合并为 1 个提交（核心）
我在做：对最近 6 个提交做 `soft reset`，把它们变成“暂存区的一次性变更”，然后重新提交。
目的是：最终生成 1 个干净提交，包含原 6 个提交的全部改动。
如果不这样做,就无法实现：把多提交压缩成单提交的目标。

命令：
- `git reset --soft HEAD~6` 或者
- `git reset --soft 745a8c99b1`
- `git commit -m "<最终聚合提交信息>"`

> 解释：
>
>- `--soft` 是语法糖：只移动 HEAD，不动工作区文件内容，并把差异放到 index（暂存区）。
>- 所以你可以直接 `git commit` 生成一个新提交，包含原来 6 个提交的全部修改。

### 4.5 Step 4：强推到远端（必须使用 --force-with-lease）
我在做：把改写后的历史推送回 `origin/develop-v2.10.26`。
目的是：让远端分支历史也变成“单提交版本”。
如果不这样做,就无法实现：普通 push 会被拒绝（non-fast-forward）。

命令：
- `git push --force-with-lease origin develop-v2.10.26`

> 解释：
>
>- `--force-with-lease` 是语法糖：只在“远端没有被别人推进新提交”的情况下才会强推。
>- 相比 `--force`，它能降低覆盖他人提交的风险。

### 4.6 Step 5：如果之前 stash 了，恢复改动
我在做：把 stash 的改动恢复回来。
目的是：继续开发或按新的提交历史补充提交。
如果不这样做,就无法实现：本地被 stash 的改动无法回到工作区。

命令：
- `git stash list`
- `git stash pop`

## 5. 我在做：给另一位开发者的同步指令（强推后必须通知）
目的是：让对方把本地旧历史对齐到你强推后的新历史。
如果不这样做,就无法实现：对方会持续遇到 pull/push 失败、产生重复提交或冲突。

对方执行（最快方式，会丢弃对方本地未备份提交）：
```bash
git fetch origin
git checkout develop-v2.10.26
git reset --hard origin/develop-v2.10.26
```

如果对方本地有提交需要保留：
- 先让对方把 `git log --oneline --decorate -n 30` 发出来，再决定 `rebase/cherry-pick` 搬运。

## 6. 我在做：回答“后续我每次二开提交后，还要合并吗？”
目的是：给出长期可执行的工作流，避免反复强推。
如果不这样做,就无法实现：要么历史一直很乱，要么频繁强推影响协作。

### 6.1 推荐结论
- **不建议每次提交后都 squash 并强推**。
- 推荐把 squash 作为：
  - **里程碑动作**（例如：当天收工、提测前、合并前、发布前）
  - 或者 **PR/MR 合并策略**（在 PR 上 squash merge）

原因：
- 频繁改写远端历史会让协作者反复 `reset --hard`，成本很高。
- 二开场景更重要的是：
  - **差异边界清晰**
  - **升级可搬运**
  - **回滚可控**

### 6.2 更适合二开的分支策略（建议）
我在做：把“官方基线”与“二开差异”分离。
目的是：升级时能快速对照、快速搬运。
如果不这样做,就无法实现：升级时二开差异扩散到多分支、多提交，冲突难控制。

建议：
- **基线分支**：只跟随官方 tag/官方分支（永远不放二开）
  - 例如：`base/v2.10.26-lts`
- **二开分支**：只承载业务差异
  - 例如：`develop-v2.10.26` / `feature/xxx-on-v2.10.26`

在这种策略下：
- 日常开发：可以多次小提交（便于回滚/定位）
- 提测/发布：再把这一段提交 squash 为 1 个（或少量几个）里程碑提交

## 7. 我在做：回答“把自己的二开提交合并到一个提交里，方便查看差异，这样做可以吗？”
目的是：给出是否可行的结论与注意事项。
如果不这样做,就无法实现：你会担心这种方式是否违背最佳实践。

### 7.1 可以做（并且在二开场景很常见）
- 你要的效果是：
  - **一个提交 = 一份二开差异快照**
- 这对以下事情很有帮助：
  - 版本升级对比
  - 交付验收
  - 代码审计

### 7.2 必须注意的边界
- 如果分支是共享分支：
  - squash + 强推前必须与协作者约定好窗口，并通知同步方式
- 不要把“日常开发阶段”提交全部压成 1 个长期存在：
  - 否则定位问题时缺少中间过程信息
- 更推荐：
  - **里程碑压缩**（阶段性压缩）
  - 或 **PR squash merge**（合并到主分支时压缩）

## 8. 我在做：本次操作的回滚方案
目的是：即使强推后发现问题，也能快速恢复。
如果不这样做,就无法实现：远端历史恢复会非常困难。

### 8.1 本地回滚（最稳）
- 找到备份分支（例如 `backup/before-squash-last6-20251230-163950`）
- 直接 reset：
  - `git reset --hard backup/before-squash-last6-20251230-163950`

### 8.2 远端回滚（需要再次强推）
- 把远端分支强推回旧 commit：
  - `git push --force-with-lease origin backup/before-squash-last6-20251230-163950:develop-v2.10.26`
