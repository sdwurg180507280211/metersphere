我在做：把“二次开发差异”固化成可查、可更新、可用于冲突定位的清单。
目的是：后续每月升级（`v2.10.xx-lts`）出现冲突时，能快速判断“冲突文件属于哪个二开功能点/该怎么回归”。

- **差异口径**：`v2.10.23-lts..develop`（即：从旧基线到当前二开分支之间的所有变更）

## 3. 如何自动生成/更新差异清单（每月必做）

我在做：给出一套只读、可复制的命令，让你每月都能自动刷新清单。
目的是：把“维护成本”降到最低（否则你很难坚持每月更新）。
如果不这样做,就无法实现：长期维护（靠手工整理会越来越难、越来越不准）。

### 3.1 依赖链检查

```bash
# 我在做：检查 git 是否存在
# 目的是：确认你能执行 git 操作
# 如果不这样做,就无法实现：后续生成差异清单
which git

# 我在做：确认 git 版本
# 目的是：排除不同 git 版本导致的输出差异
# 如果不这样做,就无法实现：多人协作时口径一致
git --version
```

### 3.2 生成“全量文件清单”（含新增/修改/删除）

```bash
# 我在做：生成二开分支相对旧基线的文件变更清单（name-status）
# 目的是：得到“改了哪些文件”的权威事实来源
# 如果不这样做,就无法实现：冲突时快速定位文件归属
git -c core.quotePath=false diff --name-status v2.10.23-lts..develop > /tmp/ms-diff-v2.10.23-to-develop.name-status.txt

# 我在做：确认清单行数（用于判断本月变更规模）
# 目的是：让你一眼知道本月差异是“很小/中等/很大”
# 如果不这样做,就无法实现：升级风险评估
wc -l /tmp/ms-diff-v2.10.23-to-develop.name-status.txt
```

### 3.3 生成“业务代码清单”（排除 docs/scripts/IDE 配置等噪音）

```bash
# 我在做：过滤出更可能影响功能与升级冲突的“业务代码文件”
# 目的是：把你真正需要关注的冲突点集中出来
# 如果不这样做,就无法实现：这份文档用于冲突定位（会被大量文档变更淹没）
awk -F'\t' '{print $1"\t"$2}' /tmp/ms-diff-v2.10.23-to-develop.name-status.txt \
  | egrep -v '^(A|M|D)\t(docs/|scripts/|\.cursor/|\.idea/|\.mvn/|README|LICENSE|Jenkinsfile|\.gitignore|\.cursorignore)' \
  > /tmp/ms-business-diff.name-status.txt

wc -l /tmp/ms-business-diff.name-status.txt
```
