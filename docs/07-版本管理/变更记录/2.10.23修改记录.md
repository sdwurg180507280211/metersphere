### 1. 测试用例导入功能优化

**业务需求：**
- 用户使用的Excel表头命名不统一（如"用例描述"、"操作步骤"、"模块"等），导致导入失败
- 不同模板项目导入时，编辑模式（步骤/文本）设置不正确

**解决方案：**
- 支持常见表头别名的自动映射
- 根据项目绑定的模板自动设置编辑模式

**涉及类：**
- `TestCaseNoModelDataListener`（导入监听器）
- `TestCaseTemplateDao`（模板数据对象）
- `TrackTestCaseTemplateService`（模板服务）

**核心修改：**

```java
// 1. 通过模板处理器获取表头映射（支持别名）
// 模板2处理器：TestCaseTemplate2Handler.java
protected Map<String, String> getSpecificHeadMapping() {
    Map<String, String> mapping = new HashMap<>();
    mapping.put("用例ID", "ID");           // 别名 → 标准表头（固定字段：ID）
    mapping.put("用例描述", "用例名称");    // 别名 → 标准表头（固定字段：NAME）
    mapping.put("操作步骤", "步骤描述");    // 别名 → 标准表头（固定字段：STEP_DESC）
    mapping.put("模块", "所属模块");        // 别名 → 标准表头（固定字段：MODULE）
    return mapping;
}

// 2. 根据模板对象获取默认编辑模式（配置驱动）
private String getDefaultStepModelByTemplate() {
    if (currentTemplate != null && StringUtils.isNotBlank(currentTemplate.getStepModel())) {
        return currentTemplate.getStepModel();  // 从模板配置读取
    }
    return TestCaseConstants.StepModel.STEP.name();
}

// 3. 在解析数据时设置默认编辑模式
if (StringUtils.isBlank(data.getStepModel())) {
    data.setStepModel(getDefaultStepModelByTemplate());
}
```

**效果：**
- 支持"用例描述"、"操作步骤"等别名表头
- 模板2项目导入时自动使用文本模式（TEXT）
- 其他项目导入时使用步骤模式（STEP）

### 2. 脑图编辑功能优化

**业务需求：**
- 在脑图中创建测试用例时，之前固定使用步骤模式（STEP），使用模板2的项目需要手动改为文本模式

**涉及类：**

```java

if (StringUtils.isBlank(request.getStepModel())) {
```

**效果：适用于所有模版**

### 4. 用例模板全局化功能

**业务需求：**
- 将"用例模板1"、"用例模板2"、"用例模板3"改为全局系统模板，使其在所有项目中可见
- 修复编辑项目时下拉列表只显示一个默认模板的问题，应显示所有全局系统模板

**解决方案：**
- 通过数据库迁移脚本将三个模板改为全局系统模板（`system=1, global=1, project_id=NULL`）
- 修改 `TestCaseTemplateService.getOption()` 方法，使其与 `IssueTemplateService.getOption()` 保持一致
- 添加 `getSystemTemplates()` 方法，返回所有系统模板（包括全局系统模板）

**涉及类：**
- `TestCaseTemplateService`（用例模板服务）
- `V144__convert_case_templates_to_global.sql`（数据库迁移脚本）

**核心修改：**

```java
// 1. 新增 getSystemTemplates() 方法
public List<TestCaseTemplate> getSystemTemplates(String projectId) {
    TestCaseTemplateExample example = new TestCaseTemplateExample();
    example.createCriteria().andProjectIdEqualTo(projectId)
            .andSystemEqualTo(true);
    example.or(example.createCriteria().andGlobalEqualTo(true));
    List<TestCaseTemplate> testCaseTemplates = testCaseTemplateMapper.selectByExample(example);
    Iterator<TestCaseTemplate> iterator = testCaseTemplates.iterator();
    while (iterator.hasNext()) {
        TestCaseTemplate next = iterator.next();
        for (TestCaseTemplate item : testCaseTemplates) {
            if (next.getGlobal() && !item.getGlobal() && StringUtils.equals(item.getName(), next.getName())) {
                // 如果有项目级的模板则过滤掉全局模板
                iterator.remove();
                break;
            }
        }
    }
    return testCaseTemplates;
}

// 2. 修改 getOption() 方法
public List<TestCaseTemplate> getOption(String projectId) {
    // ...
    // 之前：testCaseTemplates.add(getDefaultTemplate(projectId));  // 只返回一个默认模板
    // 现在：testCaseTemplates.addAll(getSystemTemplates(projectId)); // 返回所有系统模板
    testCaseTemplates.addAll(getSystemTemplates(projectId));
    return testCaseTemplates;
}
```

**数据库迁移脚本：**

```sql
-- V144__convert_case_templates_to_global.sql
UPDATE test_case_template
SET `system` = 1,
    `global` = 1,
    `project_id` = NULL,
    `update_time` = unix_timestamp() * 1000
WHERE name IN ('用例模板1', '用例模板2', '用例模板3')
  AND (`system` != 1 OR `global` != 1 OR `project_id` IS NOT NULL);
```

**效果：**
- "用例模板1"、"用例模板2"、"用例模板3"成为全局系统模板
- 编辑项目时，下拉列表显示所有全局系统模板（包括这三个模板）
- 新建项目可以选择这些全局系统模板
- 实现与缺陷模板（IssueTemplate）一致的行为

## 二、设计思想

### 1. 配置驱动 vs 硬编码

**重构前（硬编码）：**
```java
if (templateFileName.contains("template-2")) {
    return "TEXT";  // ❌ 新增模板需要改代码
}
```

**重构后（配置驱动）：**
```java
return currentTemplate.getStepModel();  // ✅ 从数据库配置读取
```

**优势：**
- 新增模板只需配置数据库，无需修改代码（开闭原则）
- 配置集中管理，易于维护（单一职责）

---

### 2. 单一数据源原则

**重构前：**
```java
private String currentTemplateFileName;  // 只保存文件名
// 需要通过文件名判断模板类型，逻辑分散
```

**重构后：**
```java
private TestCaseTemplateDao currentTemplate;  // 保存完整对象
// 所有模板配置从对象获取，单一数据源
```

**优势：**
- 避免数据分散
- 降低查询次数
- 提高内聚性

---

### 3. 职责分离

**前端职责：** 展示和收集用户输入
```javascript
stepModel: ""  // 不再设置默认值，交由后端处理
```

**后端职责：** 业务逻辑和默认值设置
```java
// 根据模板配置设置默认值
request.setStepModel(template.getStepModel());
```

**优势：**
- 前后端职责清晰
- 配置变更时只需改后端
- 降低耦合度

---

### 4. 策略模式思想

不同模板有不同的配置策略，通过模板对象的字段来驱动策略：

```java
// 策略接口（隐式）
template.getStepModel()  → "STEP" 或 "TEXT"
template.getExcelTemplateFile()  → 文件路径或null

// 具体策略由数据库配置决定
模板1: stepModel="STEP", excelTemplateFile="testcase-template-1.xlsx"
模板2: stepModel="TEXT", excelTemplateFile="testcase-template-2.xlsx"
```

**优势：**
- 策略可配置化
- 易于扩展新策略
- 符合开闭原则

## 三、核心业务流程

### 导入用例流程

```
1. 读取Excel表头
   ↓
2. 表头映射转换
   - "用例描述" → "用例名称"
   - "操作步骤" → "步骤描述"
   ↓
3. 解析数据行
   ↓
4. 设置默认编辑模式
   - 获取项目模板 → template.getStepModel()
   - 模板2 → "TEXT"
   - 其他 → "STEP"
   ↓
5. 保存用例
```

### 脑图创建用例流程

```
前端：
1. 用户在脑图中创建用例
   ↓
2. 构建用例对象（stepModel = ""）
   ↓
3. 提交到后端

后端：
4. 判断是新建还是编辑
   ↓
5. 新建：调用 setCustomDefault()
   - stepModel为空 → 从模板获取 → 设置为"TEXT"或"STEP"
   ↓
6. 编辑：在 minderEdit() 中处理
   - stepModel为空 → 优先保留数据库原值 → 原值也为空才使用模板
   ↓
7. 保存用例
```

## 五、设计模式和思想

### 1. 配置驱动设计

**核心思想：** 将业务逻辑从代码转移到配置

```
硬编码判断              配置驱动
文件名匹配  ────→  数据库字段
if-else树  ────→  template.getStepModel()
```

**优势：** 新增模板无需改代码

---

### 2. 单一数据源原则

**核心思想：** 模板相关配置统一从模板对象获取

```
之前：分散的数据源
- 文件名字符串
- 硬编码的映射规则
- 分散的配置判断

现在：单一数据源
- TestCaseTemplateDao对象
  ├─ stepModel（编辑模式）
  ├─ excelTemplateFile（模板文件）
  └─ customFields（自定义字段）
```

**优势：** 高内聚，易于维护

---

### 3. 开闭原则

**对扩展开放，对修改关闭**

```
扩展新模板的方式：
1. 在数据库插入模板记录（配置 stepModel）
2. 放置Excel模板文件到 resources/excel-templates/
3. 配置 excel_template_file 字段

无需修改代码！
```

---

**表现层（前端）：**

- 处理业务逻辑

**数据层（数据库）：**

**优势：** 低耦合，易于测试和维护

- 内部字段名对应 `TestCaseExcelData` 对象的属性名

### 2. 映射规则

**映射目的：** 将用户输入的别名表头映射为系统识别的标准表头名称

**映射判断标准：**
- 判断条件：`customHeadMapping.containsKey(Excel表头名称)`
- 匹配方式：**精确匹配**（区分大小写）
- 映射来源：由模板处理器的 `getHeadMapping()` 方法提供

**映射流程：**
```
Excel表头（用户输入）
    ↓ 第一层映射：customHeadMapping（模板处理器提供）
标准表头（系统识别）
    ↓ 第二层映射：excelHeadToFieldNameDic（注解映射）
内部字段名（数据库字段）
```

### 3. 需要映射的字段

**需要映射的都是固定字段的别名**，映射规则由各模板处理器定义：

#### 模板2的映射规则（示例）

| 别名表头 | 映射为标准表头 | 对应固定字段 | 说明 |
|---------|--------------|------------|------|
| 用例ID | ID | ID | 用户可能使用"用例ID"而不是"ID" |
| 用例描述 | 用例名称 | NAME | 用户可能使用"用例描述"而不是"用例名称" |
| 操作步骤 | 步骤描述 | STEP_DESC | 用户可能使用"操作步骤"而不是"步骤描述" |
| 模块 | 所属模块 | MODULE | 用户可能使用"模块"而不是"所属模块" |

#### 模板1的映射规则（示例）

| 别名表头 | 映射为标准表头 | 对应固定字段 | 说明 |
|---------|--------------|------------|------|
| 序号 | ID | ID | 用户可能使用"序号"作为ID字段 |
| 菜单路径/功能点 | ID | ID | 特殊场景下的ID别名 |
| 测试场景/用例/步骤 | ID | ID | 特殊场景下的ID别名 |


**映射生效时机：**
- 在 `formatHeadMap()` 方法中执行
- 读取Excel表头时自动应用映射规则
- 映射后的表头名称用于后续的数据解析

### 5. 映射规则总结

| 规则项 | 说明 |
|--------|------|
| **映射对象** | 固定字段的别名表头 |
| **映射目标** | 固定字段的标准表头名称 |
| **判断标准** | `customHeadMapping.containsKey(表头名称)` |
| **匹配方式** | 精确匹配（区分大小写） |
| **配置位置** | 各模板处理器的 `getSpecificHeadMapping()` 方法 |
| **生效时机** | Excel表头读取时自动应用 |

**重要原则：**
- ✅ 需要映射的都是固定字段的别名
- ✅ 映射将别名转换为标准表头名称
- ✅ 不同模板可以定义不同的映射规则
- ✅ 映射规则可扩展，新增模板时只需配置映射关系

## 十、用例模板全局化详细说明

### 问题背景

在编辑项目时，用例模板下拉列表只显示一个默认模板，即使将"用例模板1"、"用例模板2"、"用例模板3"改为全局系统模板后，它们也不会出现在下拉列表中。

### 根本原因

`TestCaseTemplateService.getOption()` 方法的实现与 `IssueTemplateService.getOption()` 不一致：

**之前的实现：**
```java
// 只返回一个默认模板
testCaseTemplates.add(getDefaultTemplate(projectId));
```

**问题：**
- `getDefaultTemplate()` 优先返回项目级系统模板
- 如果没有项目级系统模板，才返回一个全局模板
- 结果：即使有多个全局系统模板，下拉列表也只显示一个

### 解决方案

参考 `IssueTemplateService` 的实现，添加 `getSystemTemplates()` 方法，返回所有系统模板：

**修改后的实现：**
```java
// 返回所有系统模板（项目级+全局）
testCaseTemplates.addAll(getSystemTemplates(projectId));
```

**`getSystemTemplates()` 方法逻辑：**
1. 查询项目级系统模板（`project_id=projectId AND system=1`）
2. 查询全局系统模板（`global=1`）
3. 如果存在项目级同名模板，则过滤掉全局模板（避免重复）

### 数据库变更

**迁移脚本：** `V144__convert_case_templates_to_global.sql`

```sql
UPDATE test_case_template
SET `system` = 1,
    `global` = 1,
    `project_id` = NULL,
    `update_time` = unix_timestamp() * 1000
WHERE name IN ('用例模板1', '用例模板2', '用例模板3')
  AND (`system` != 1 OR `global` != 1 OR `project_id` IS NOT NULL);
```

**关键字段说明：**
- `system=1`：标记为系统模板，不可在UI中删除
- `global=1`：标记为全局模板，所有项目可见
- `project_id=NULL`：全局模板必须为NULL（关键！）

### 效果验证

**修改前：**
- 编辑项目时，下拉列表只显示一个默认模板
- 全局系统模板不显示

**修改后：**
- 编辑项目时，下拉列表显示：
  1. 项目级非系统模板
  2. 所有系统模板（项目级系统模板 + 全局系统模板）
- "用例模板1"、"用例模板2"、"用例模板3"会出现在所有项目的下拉列表中

### 代码一致性

修改后的实现与 `IssueTemplateService` 保持一致，确保：
- 用例模板和缺陷模板的行为一致
- 全局系统模板在所有项目中可见
- 代码逻辑统一，易于维护
