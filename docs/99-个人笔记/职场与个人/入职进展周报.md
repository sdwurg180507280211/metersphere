## 第2周（2025-12-02 ~ 2025-12-06）

### 技术收获
**CI/CD部署模式：**

1. **Push模式（推送模式）**
    - 机制：CI工具主动推送部署到目标环境
    - 代表：Jenkins、GitLab CI、GitHub Actions
    - 流程：代码提交 → CI触发 → 构建镜像 → 推送到K8s/服务器
    - 优点：实时性好，反馈快
    - 缺点：CI需要集群访问权限，安全性相对较低

2. **Pull模式（拉取模式）**
    - 机制：K8s集群主动拉取配置变更并部署
    - 代表：ArgoCD、Flux、GitOps工具链
    - 流程：代码提交 → 更新Git配置 → 集群自动检测变更 → 拉取部署
    - 优点：更安全（集群不暴露给CI）、天然支持GitOps
    - 缺点：有一定延迟（轮询间隔）

3. **选型建议：**
    - 小项目/快速迭代：Push模式（Jenkins/GitLab CI）
    - 生产环境/多集群：Pull模式（ArgoCD）
    - 混合模式：CI构建镜像（Push），K8s部署（Pull）

### 项目进展
- [ ] 了解公司当前CI/CD方案
- [ ] 评估MeterSphere适合哪种部署模式
- [ ] 研究ArgoCD在测试平台的应用场景

### 本周思考
CI/CD不只是"自动化部署"，更是一种研发文化：
- Push模式体现"快速交付"
- Pull模式体现"安全可控"
- GitOps将基础设施配置化，是未来趋势

对于MeterSphere项目，可能采用混合方案：
- 开发环境用Push快速验证
- 生产环境用Pull保证稳定

#### 2️⃣ 微前端架构：独立 vs 集成
**技术背景：**
- MeterSphere采用qiankun微前端框架
- 工作流服务需要前端展示层

**三种方案对比：**

**方案A：独立前端应用（推荐：同栈 Vue2.7 子应用）**
```
workflow-service（后端）
    ↓ REST API
workflow-frontend（Vue2.7 + ElementUI，同 MeterSphere 技术栈）
    ↓ qiankun子应用
MeterSphere主应用
```
✅ 优点：子应用隔离、可独立发布；同时保持与主项目同栈，降低维护成本  
❌ 缺点：需要新增/维护一个前端工程与部署流程

**方案B：集成到主应用**
```
workflow-service（后端）
    ↓ REST API
MeterSphere主应用（直接调用API）
```
✅ 优点：无需额外前端工程  
❌ 缺点：耦合度高、升级困难、代码混乱

**方案C：iframe嵌入（仅作为临时验证备选）**
```
workflow-service（内嵌管理页面）
    ↓ iframe
MeterSphere主应用
```
✅ 优点：最简单实现  
❌ 缺点：跨域问题、性能差、用户体验割裂

**技术决策建议：**
- **短期（第5周）**：继续在 `system-setting` 内以页面形态推进（快速迭代、降低拆分成本）
- **中期（评审通过后）**：方案A（qiankun 子应用，保持 Vue2.7 同栈）
- 理由：先用同仓 PoC 验证闭环与需求，再在不影响主工程的前提下抽离为子应用

#### 3️⃣ 权限体系对接：解耦 vs 耦合
**核心矛盾：**
- MeterSphere有完整的RBAC权限模型
- Flowable也有自己的权限体系
- 如何避免双重维护？

**对接方案：**

**方案1：MeterSphere 权限为唯一来源（推荐）**
✅ 单一权限源（MeterSphere RBAC）  
✅ Flowable 仅作为流程执行引擎（不单独维护用户/权限）  
❌ 需要补齐权限点定义与接口鉴权

**方案2：同步权限数据**
- 定时同步MS用户→Flowable用户表
- ❌ 数据一致性问题
- ❌ 维护成本高

**技术实现关键点（下一步落地项）：**
1. 新增系统级权限点（如 `SYSTEM_WORKFLOW:READ`/`SYSTEM_WORKFLOW:EDIT` 等），并在权限管理页面可授权
2. 前端路由 `meta.permissions` 与后端权限点对齐（控制菜单/页面可见性）
3. workflow-service 接口层统一做鉴权（基于 MeterSphere 登录态/Token/网关鉴权链路）
4. 任务查询/完成等接口按当前用户做数据过滤（assignee/candidateUser/candidateGroup）

**代码示例：**
```java
// 流程启动时绑定用户
runtimeService.startProcessInstanceByKey("defect-workflow",
                                         Map.of("assignee", currentUser.getId()));

// 任务查询时过滤权限
        taskService.createTaskQuery()
    .taskAssignee(currentUser.getId())
        .or()
    .taskCandidateUser(currentUser.getId())
        .taskCandidateGroupIn(currentUser.getRoles())
        .list();
```

### 技术风险预警
**风险1：微前端技术债务**
- qiankun学习曲线：需要1-2周熟悉
- 沙箱隔离问题：CSS样式冲突
- **缓解措施**：优先保持 Vue2.7 同栈；抽离为 qiankun 子应用时提前约束样式作用域与全局事件（快捷键/滚动/弹窗）

**风险2：Flowable版本兼容性**
- Flowable 7 + Spring Boot 3：相对新的组合
- 可能遇到依赖冲突、文档缺失
- **缓解措施**：加入Flowable社区，提前准备降级方案（Flowable 6.8）

**风险4：依赖版本漂移导致构建不稳定**
- BPMN 设计器相关依赖（bpmn-js 等）如果用 `^` 易发生版本漂移，出现“本机可用/他人不可用”
- **缓解措施**：关键依赖锁定精确版本 + 统一 lockfile 管理

### 本周踩坑/经验沉淀
1. **依赖管理**：BPMN 设计器依赖对版本敏感，锁定精确版本可显著降低构建/运行不确定性
2. **前端导航**：系统设置菜单不宜被子页面撑爆，采用「单入口 + Tabs 承载」更利于扩展
3. **接口闭环优先**：先跑通 deploy→start→task→complete 的最小闭环，再逐步产品化（权限、审计、异常码、分页与索引）

**风险3：性能问题**
- 工作流实例过多时查询慢
- **预防措施**：提前设计索引策略、分页查询、历史数据归档

### 原理讲解（为什么这么做）

#### 1️⃣ 为什么要做“workflow-service 微服务”，而不是直接塞进 MeterSphere 主服务？
**核心原则：隔离变化，降低耦合。**

- **MeterSphere 主工程**的生命周期、升级节奏由社区版本决定；
- **工作流引擎（Flowable）**属于“强侵入依赖”（数据库表、作业线程、事件机制、权限与任务查询等），一旦和主工程耦合，会带来：
    - 升级冲突（依赖树/版本冲突）
    - 数据库污染（引擎表数量多、结构固定）
    - 运行风险外溢（异步 job、定时器、历史清理影响主系统稳定性）

因此把 Flowable 放进 `workflow-service`：
- **对外只暴露 REST API**
- **数据库独立**（`metersphere_workflow`）
- **注册/网关统一接入**（Eureka + Gateway）

这样设计的收益：
- 改动面小（主工程少改）
- 可回滚（出问题直接下线该微服务，不影响主系统核心链路）
- 可迁移（未来可独立演进、甚至独立部署）

#### 2️⃣ 为什么要做“模型库（版本化保存）”，而不是只做“上传 BPMN 直接部署”？
**核心原则：部署是发布行为，模型是资产。**

直接部署 BPMN（PoC）能证明引擎可用，但无法支撑长期使用：
- 没有版本：改一次流程就覆盖，无法回滚
- 没有审计：谁改的、为什么改的不可追溯
- 没有发布链路：编辑态与发布态无法区分

模型库的产品化结构：
- **模型（Model）**：保存“可编辑源数据”（XML/SVG/元信息）
- **部署记录（Deploy）**：保存“发布结果”（deploymentId、processDefinitionId、发布时间、发布版本）

这样前端才能做出 shanxian 那种“模型列表 + 发布状态”的体验。

#### 3️⃣ 为什么会有 camunda:assignee / flowable:assignee 兼容问题？怎么解决？
**核心原理：BPMN 2.0 是标准，但厂商扩展属性不是标准。**

- BPMN 标准定义了流程结构（任务、网关、事件）
- 但“任务分配人/候选人/候选组”等属于引擎扩展：
    - Camunda 常用 `camunda:assignee`
    - Flowable 常用 `flowable:assignee`

如果前端设计器输出的是 Camunda 扩展，而后端使用 Flowable 引擎：
- **部署可能成功**，但运行时解析不到 assignee，导致任务分配异常
- 或者设计器无法加载/显示扩展字段

本周采取的工程化方案：
- **模型库保存“原始 XML（camunda 兼容）”**，保证设计器可编辑
- **部署时转换 XML（camunda → flowable）**，保证 Flowable 运行正确

这本质是“编辑态与运行态解耦”的策略。

#### 4️⃣ 为什么 multipart 能解决 XML 放在 JSON 里的 400/转义问题？
**核心原理：JSON 对字符串有严格转义规则，XML 内含大量引号与特殊字符。**

把整段 XML 放 JSON：
- 必须把 `"`、换行、反斜杠等转义正确
- 任何一个地方转义失败都可能导致 400 Bad Request

multipart（文件上传）的优点：
- XML 作为文件内容传输，不需要 JSON 字符串转义
- 前端/工具（Apifox）操作更自然
- 接口更稳定，可读性更好

#### 5️⃣ 为什么前端入口要收敛为“模型管理单入口”？
**核心原则：用户路径要短，入口要少，状态要清晰。**

一开始的 PoC 页面会有很多“功能页”（部署、启动、查任务、完成任务），但这些更像调试台。

对标 shanxian 的产品体验：
- 用户主路径只有一条：**模型列表 → 编辑 → 保存版本 → 部署**
- 其它能力（启动/任务）应当是“运维/调试能力”，而不是主入口

因此本周前端做了两件关键事：
- 入口 Tab 精简/收敛，避免“空页面”
- 设计器抽屉工具条固定化（主操作不随表单滚动），减少操作迷路

**1️⃣ 缺陷描述字段懒加载优化**

    - 利用 Element UI Popover 的 `@show` 事件，鼠标悬停到"预览"按钮时才触发内容加载
    - 首次加载后设置 `contentLoaded = true`，保持渲染状态避免重复加载
- **效果**：
    - 首屏加载时间减少 50-70%
    - 网络请求数减少 80-90%（不再加载所有描述中的图片）
    - 内存占用减少 40-60%
- **技术点**：Vue 组件懒加载、Popover 生命周期事件、v-if vs v-show 选择

**2️⃣ 用户组数据权限过滤（纯前端方案）**
- **需求**：开发人员和测试人员默认只能看到与自己相关的缺陷
- **实现方案**：
    - 在 `IssueList.vue` 中添加用户组判断逻辑
    - 调用 `/user/group/list/{userId}` 接口获取当前用户的用户组
    - 根据用户组动态添加过滤条件到 `page.condition.filters`
- **权限规则**：
    - 开发人员组（developer）：默认只能看到**处理人是自己**的缺陷
    - 测试人员组（tester）：默认只能看到**创建人是自己**的缺陷
    - 其他用户组：没有限制
- **交互逻辑**：
    - **初始化加载**时施加权限过滤（默认只看自己相关的缺陷）
    - **点击高级搜索/重置/筛选**时清除权限过滤（可查看当前项目下所有缺陷）
    - **翻页**时保持当前过滤状态

**技术原理：**
```
核心机制：复用 MeterSphere 现有的高级搜索过滤机制

1. 数据流：
   页面初始化
       ↓
   getUserGroupProject() 获取用户组ID（developer/tester/其他）
       ↓
   applyUserGroupFilter() 根据用户组设置过滤条件
       ↓
   page.condition.filters = { 'creator': [userId] }  // tester
   page.condition.filters = { 'custom_single-xxx': [userId] }  // developer
       ↓
   getIssues() 发送请求，后端根据 filters 拼接 SQL WHERE 条件
       ↓
   返回过滤后的缺陷列表

2. 过滤条件的 key 生成：
   - 创建人（系统字段）：直接使用 'creator'
   - 处理人（自定义字段）：使用 generateColumnKey(field) 生成
     格式：'custom_single-{fieldId}' 或 'custom_multiple-{fieldId}'
   - 这个 key 与后端 MyBatis 动态 SQL 中的字段映射一致

3. 为什么是"纯前端"方案：
   - 不修改后端 SQL 查询逻辑
   - 不新增后端接口
   - 只在前端设置 filters 参数，复用现有的高级搜索机制
   - 后端已有的 filters 处理逻辑会自动生效
```

**扩展方向：**
```
1. 新增用户组规则：
   - 在 applyUserGroupFilter() 中添加新的 else if 分支
   - 例如：QA Lead 只看本团队成员创建的缺陷
   
2. 多条件组合：
   - 当前是单条件过滤（创建人 OR 处理人）
   - 可扩展为：(创建人=自己 OR 处理人=自己) AND 状态!=已关闭
   
3. 配置化：
   - 将用户组与过滤规则的映射关系存入数据库
   - 管理员可在系统设置中配置不同用户组的默认过滤规则
   
4. 后端强化（安全性提升）：
   - 当前方案依赖前端设置 filters，用户可通过 DevTools 绕过
   - 如需强制限制，需在后端 Service 层根据用户组追加 SQL 条件
```

- **技术点**：前端权限过滤、用户组 API 调用、条件动态拼接、generateColumnKey 函数

**3️⃣ 微前端架构专项：qiankun → micro-app 迁移**
- **背景**：MeterSphere 原有微前端方案基于 qiankun，子应用需要 UMD 打包、导出 `bootstrap/mount/unmount/update` 生命周期，并依赖 qiankun 注入的 publicPath 和 props 通信。随着模块增多，静态资源路径、子应用路由、重复 Layout、菜单高亮和模块接入成本逐渐暴露，需要切换到更轻量、组件化的 micro-app 方案。
- **迁移目标**：
    - 将主应用从 qiankun 注册/激活模式切换为 `<micro-app>` 组件化加载模式。
    - 将子应用从 qiankun 生命周期导出迁移到 micro-app 的 `window.mount(data)` / `window.unmount()` 生命周期模式。
    - 降低子应用构建要求，移除 qiankun 相关 UMD library 配置，统一 public path、路由透传和跨应用通信方式。
- **完成内容**：
    - **主应用改造**：移除 qiankun 相关入口，新增 micro-app 初始化、子应用配置、运行时策略和 `<micro-app>` 容器加载逻辑。
    - **子应用接入**：完成 api-test、performance-test、test-track、workstation 等子应用的 micro-app 接入改造。
    - **生命周期迁移**：将原 qiankun `bootstrap/mount/unmount/update` 模式调整为 micro-app UMD 生命周期模式，通过 `window.mount(data)` 接收主应用传入数据。
    - **资源路径修复**：处理 micro-app 模式下静态资源不可见问题，调整子应用 `public-path.js`，适配 `__MICRO_APP_PUBLIC_PATH__`。
    - **路由协同修复**：修复 performance、api-test 子应用 `main.js` 中路由选择逻辑，micro-app 环境下使用完整 router，避免 `data.defaultPath` 路由不匹配导致页面空白。
    - **Layout 重复治理**：在 test-track、workstation 等子应用中使用透传组件替换 Layout，避免子应用在主应用内重复渲染侧边栏。
    - **构建配置调整**：移除 qiankun 所需的 UMD library 配置，保留 micro-app 运行时生命周期模式，降低子应用构建耦合。
- **关键问题与解决**：
    - **静态资源 404/不可见**：通过 micro-app public path 适配解决。
    - **子应用页面空白**：修正主应用传入路径与子应用 router 的匹配关系。
    - **重复侧边栏**：通过 `PassThrough`/透传布局替代子应用内 Layout。
    - **通信方式变化**：从 qiankun props 传递逐步切换为 micro-app data 与 EventBus 适配。
- **技术点**：qiankun 迁移、micro-app、WebComponent 式微前端、子应用生命周期、public path、主子应用路由协同、EventBus 适配、Layout 透传、Vue2 子应用改造。
- **可沉淀到简历的表达**：
