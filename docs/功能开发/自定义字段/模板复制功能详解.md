## 一、功能概述

### 1.1 功能描述

**MeterSphere支持将一个项目的缺陷模板（包括模板中的自定义字段）复制到其他项目。**

```
源项目A的缺陷模板
├── 模板基本信息（名称、描述、内容）
└── 自定义字段列表
    ├── 需求号（文本框）
    ├── 测试环境（下拉框）
    └── 优先级（下拉框）
              ↓ 复制
目标项目B、C、D...
├── 新增模板副本
└── 新增/更新自定义字段
```

### 1.2 核心特点

| 特点 | 说明 |
|------|------|
| **批量操作** | ✅ 可以同时复制到多个项目 |
| **包含字段** | ✅ 模板的所有自定义字段一起复制 |
| **两种模式** | ✅ 追加模式 / 覆盖模式 |
| **工作空间级** | ⚠️ 仅限当前工作空间内的项目 |
| **权限控制** | ⚠️ 需要目标项目的自定义字段权限 |
| **⚠️ 不会自动绑定** | ❌ **复制后需要手动设置为默认模板** |

### 1.3 ⚠️ 重要限制

**复制后的模板不会自动成为项目的默认模板！**

```
复制操作完成后：
目标项目B
├── 原有模板（仍然是默认模板）     ← 当前在用 ✅
└── 新复制的模板_copy_xxx         ← 只是新增，未绑定 ❌

❌ 创建缺陷时，仍然使用原有模板
❌ 新复制的字段不会显示

✅ 需要手动操作：
   项目设置 → 项目信息 → 缺陷模板 → 选择新复制的模板 → 保存
```

**这意味着**：
- ❌ **不能一键完成**：复制到100个项目后，还需要在100个项目中逐个手动绑定
- ❌ **容易遗漏**：可能忘记某些项目的绑定操作
- ❌ **操作繁琐**：每个项目都要进入项目设置手动选择

## 二、操作步骤

### 2.1 完整操作流程

#### 阶段1：复制模板

```
1. 登录 MeterSphere
   ↓
2. 选择源项目（包含你想要的字段和模板）
   ↓
3. 项目设置 → 模版管理 → 缺陷模版
   ↓
4. 找到目标模板，点击右侧"操作"列的"复制"按钮
   ↓
5. 弹出对话框：
   ┌────────────────────────────────────────┐
   │ 复制缺陷模板                            │
   ├────────────────────────────────────────┤
   │ 注意：复制的模板会在目标项目中新增；      │
   │ 此操作有可能会影响目标项目的历史缺陷，    │
   │ 建议仅在新项目中使用。                   │
   │ ⚠️ 请谨慎操作，此操作不可逆！            │
   ├────────────────────────────────────────┤
   │ 复制模式：                              │
   │ （系统字段选项值和自定义字段及字段选项值的设置）│
   │ ☑ 追加                                  │
   │ ☐ 覆盖                                  │
   ├────────────────────────────────────────┤
   │ 目标项目：                              │
   │ ☐ 项目A                                │
   │ ☑ 项目B ✅                             │
   │ ☑ 项目C（当前项目）                     │
   │ ☐ 项目D（没有自定义字段权限） ❌        │
   ├────────────────────────────────────────┤
   │          [取消]      [确定]             │
   └────────────────────────────────────────┘
   ↓
6. 选择复制模式（追加/覆盖）
   ↓
7. 勾选目标项目
   ↓
8. 点击"确定"
   ↓
9. 模板复制完成 ⚠️ 但尚未绑定！
```

#### ⚠️ 阶段2：逐个项目绑定模板（必须手动操作）

**关键步骤**：复制完成后，**必须在每个目标项目中手动设置新模板为默认模板**！

```
针对每个目标项目（重复N次）：

1. 切换到目标项目B
   ↓
2. 项目设置 → 项目信息（Project Info）
   ↓
3. 找到"缺陷模板"（Issue Template）下拉框
   ↓
4. 当前值：default（原有模板）
   ↓
5. 选择：default_copy_xxx（新复制的模板）  ← ⚠️ 关键！
   ↓
6. 点击"保存"
   ↓
7. 完成该项目的绑定 ✅
   ↓
重复步骤1-7，直到所有目标项目都完成绑定
```

**示例**：复制到10个项目

```
复制操作：1次（勾选10个项目）             ← 批量
绑定操作：10次（每个项目单独设置）        ← ⚠️ 逐个手动操作
```

## 三、追加模式 vs 覆盖模式

### 3.1 两种模式对比

| 场景 | 追加模式 | 覆盖模式 |
|------|---------|---------|
| **目标项目没有同名字段** | ✅ 新增字段 | ✅ 新增字段 |
| **目标项目有同名系统字段** | ✅ 追加选项值 | ⚠️ 覆盖选项值和类型 |
| **目标项目有同名自定义字段** | ⚠️ 创建 `字段名_copy_xxx` | ⚠️ 保持原字段，更新选项 |
| **模板名称冲突** | ✅ 创建 `模板名_copy_xxx` | ✅ 创建 `模板名_copy_xxx` |

### 3.2 详细说明

#### 场景1：目标项目**没有**同名字段

**两种模式效果相同**：直接创建新字段

```sql
-- 在目标项目中新增字段
INSERT INTO custom_field (id, name, scene, type, ..., project_id)
VALUES (uuid(), '需求号', 'ISSUE', 'input', ..., '目标项目ID');

-- 关联到新模板
INSERT INTO custom_field_template (field_id, template_id, ...)
VALUES ('新字段ID', '新模板ID', ...);
```

---

#### 场景2：目标项目**有同名系统字段**

##### ✅ 追加模式

**代码逻辑**（第422-440行）：

```java
if (sourceCustomField.getSystem()) {
    // 系统字段
    if (StringUtils.equals("select", sourceCustomField.getType())) {
        // 下拉框选项 - 追加新选项
        List<CustomFieldOptionDTO> options = JSON.parseArray(sourceCustomField.getOptions(), CustomFieldOptionDTO.class);
        options.removeIf(sourceOption -> 
            StringUtils.contains(tarCustomField.getOptions(), sourceOption.getText().toString())
        );
        if (CollectionUtils.isNotEmpty(options)) {
            // 合并选项：原有的 + 新增的
            optionsStr = StringUtils.replace(tarCustomField.getOptions(), "]", ",") 
                       + StringUtils.replace(optionsStr, "[", StringUtils.EMPTY);
        }
    }
    tarCustomField.setOptions(optionsStr);
    customFieldMapper.updateByPrimaryKeyWithBLOBs(tarCustomField);
}
```

**示例**：

```
源项目A："严重程度" 字段
├── 类型：select（下拉框）
└── 选项：[P0, P1, P2, P3, Critical]

目标项目B："严重程度" 字段（已存在）
├── 类型：select
└── 选项：[P0, P1, P2, P3]

🎯 追加模式执行后：
目标项目B："严重程度" 字段
├── 类型：select
└── 选项：[P0, P1, P2, P3, Critical] ← 追加了 Critical
```

##### ⚠️ 覆盖模式

**代码逻辑**（第413-419行）：

```java
if (StringUtils.equals("1", request.getCopyModel())) {
    // 覆盖模式
    if (!StringUtils.equals(sourceCustomField.getType(), tarCustomField.getType())) {
        tarCustomField.setType(sourceCustomField.getType());  // 覆盖类型
    }
    tarCustomField.setOptions(sourceCustomField.getOptions());  // 覆盖选项
    customFieldMapper.updateByPrimaryKeyWithBLOBs(tarCustomField);
}
```

**示例**：

```
源项目A："严重程度" 字段
├── 类型：select
└── 选项：[Critical, Major, Minor]

目标项目B："严重程度" 字段（已存在）
├── 类型：select
└── 选项：[P0, P1, P2, P3]

🎯 覆盖模式执行后：
目标项目B："严重程度" 字段
├── 类型：select
└── 选项：[Critical, Major, Minor] ← 完全覆盖，原有P0-P3丢失！
```

#### 场景3：目标项目**有同名自定义字段**（非系统字段）

##### ✅ 追加模式

**代码逻辑**（第441-447行）：

```java
else {
```

```
└── 选项：[开发环境, 测试环境, 生产环境]
```

**效果**：保持原字段，仅更新类型和选项

```
目标项目B：
└── "测试环境" 字段
    ├── 类型：可能被更新
    └── 选项：[开发环境, 测试环境, 生产环境] ← 覆盖
```

## 四、应用场景

### 4.1 ✅ 适用场景

#### 场景1：新项目快速初始化

```
问题：新建了10个项目，想让所有项目都有相同的缺陷模板和字段
解决：
1. 在项目A配置好模板（包括"需求号"等自定义字段）
2. 使用"复制"功能，选择9个新项目
3. 选择"追加模式"
4. 复制完成
5. ⚠️ 逐个进入9个项目，手动设置新模板为默认模板
6. 完成配置 ✅
```

**优点**：相对快速、统一
**缺点**：⚠️ 需要逐个项目手动绑定模板

#### 场景2：标准模板推广

```
问题：制定了新的缺陷管理规范，需要在多个项目推广
解决：
1. 在模板项目创建标准模板
2. 复制到需要的项目
3. 各项目根据实际情况微调
```

**优点**：标准化、可定制

#### 场景3：系统字段选项补充

```
问题："严重程度"字段只有P0-P3，想在多个项目中增加"Critical"选项
解决：
1. 在项目A添加 Critical 选项
2. 复制到其他项目
3. 选择"追加模式" ← 关键！
4. 其他项目的"严重程度"选项自动追加 Critical ✅
```

**优点**：避免手动逐个修改

---

### 4.2 ⚠️ 不适用场景

#### 场景1：全局系统字段（推荐数据库脚本）

```
❌ 不推荐：使用复制功能添加"需求号"到所有项目
问题：
1. 每个项目的"需求号"是独立副本，不是真正的全局字段
2. 后续修改不同步
3. ⚠️ 复制后需要逐个项目手动绑定模板（工作量大）
4. 无法确保100%覆盖（可能漏掉某些项目或忘记绑定）
5. 新建项目不会自动包含

✅ 推荐：使用数据库迁移脚本创建全局系统字段
优点：
1. 真正的全局共享
2. 新项目自动包含
3. 系统级保护（不可删除）
4. 100%覆盖率
```

#### 场景2：历史数据较多的项目

```
⚠️ 风险：复制到有大量历史缺陷的项目
问题：
1. 可能影响历史缺陷的显示
2. 字段关联关系可能出现问题
3. 操作不可逆

建议：仅在新项目或测试项目中使用
```

## 五、技术实现

### 5.1 核心代码流程

```java
public List<IssueTemplate> copy(CopyIssueTemplateRequest request) {
    // 1. 获取源模板和字段
    IssueTemplate sourceTemplate = issueTemplateMapper.selectByPrimaryKey(request.getId());
    List<CustomFieldTemplate> sourceCustomFieldTemplates = customFieldTemplateService.getCustomFields(request.getId());
    List<CustomField> sourceCustomFields = customFieldService.getFieldByIds(fields);
    
    // 2. 遍历目标项目
    request.getTargetProjectIds().forEach(targetProjectId -> {
        // 3. 创建新模板（名称可能加_copy后缀）
        IssueTemplate newTemplate = createNewTemplate(sourceTemplate, targetProjectId);
        
        // 4. 遍历源字段
        sourceCustomFieldTemplates.forEach(sourceFieldTemplate -> {
            CustomField targetField;
            
            // 5. 检查目标项目是否有同名字段
            if (目标项目没有同名字段) {
                // 直接创建新字段
                targetField = createNewField(sourceField, targetProjectId);
            } else {
                // 已存在同名字段
                targetField = getExistingField(sourceField.getName(), targetProjectId);
                
                if (request.getCopyModel().equals("1")) {
                    // 覆盖模式：更新类型和选项
                    targetField.setType(sourceField.getType());
                    targetField.setOptions(sourceField.getOptions());
                    customFieldMapper.update(targetField);
                } else {
                    // 追加模式
                    if (sourceField.getSystem()) {
                        // 系统字段：追加选项
                        String mergedOptions = mergeOptions(targetField.getOptions(), sourceField.getOptions());
                        targetField.setOptions(mergedOptions);
                        customFieldMapper.update(targetField);
                    } else {
                        // 自定义字段：创建_copy副本
                        targetField = createNewField(sourceField, targetProjectId);
                        targetField.setName(sourceField.getName() + "_copy_" + randomString);
                    }
                }
            }
            
            // 6. 创建字段-模板关联
            createFieldTemplateRelation(targetField.getId(), newTemplate.getId());
        });
    });
    
    // 7. 批量插入数据库
    batchInsert(newTemplates, newFields, newFieldTemplateRelations);
}
```

### 5.2 数据库变更

#### 复制1个模板到1个项目，涉及的表：

| 表名 | 操作 | 说明 |
|------|------|------|
| `issue_template` | INSERT 1行 | 新模板记录 |
| `custom_field` | INSERT N行 | 新字段（如果不存在） |
| `custom_field` | UPDATE M行 | 更新已存在字段（覆盖/追加模式） |
| `custom_field_template` | INSERT N行 | 字段-模板关联 |

**示例**：复制包含5个字段的模板到3个项目

```
issue_template:           +3 行（3个新模板）
custom_field:             +0~15 行（取决于目标项目是否已有同名字段）
custom_field_template:    +15 行（3个模板 × 5个字段）
```

## 六、与全局系统字段的对比

### 6.1 功能对比表

| 特性 | 模板复制 | 全局系统字段 |
|------|---------|-------------|
| **创建方式** | ✅ UI界面操作 | ❌ 数据库脚本 |
| **技术门槛** | ✅ 低（点击即可） | ⚠️ 高（需要SQL） |
| **覆盖范围** | ⚠️ 手动选择项目 | ✅ 自动覆盖所有项目 |
| **新项目** | ❌ 不会自动包含 | ✅ 自动包含 |
| **字段性质** | 项目级独立副本 | 全局共享引用 |
| **后续修改** | ❌ 不同步 | ✅ 全局同步 |
| **可删除性** | ✅ 可以删除 | ❌ 不可删除 |
| **适用场景** | 少量项目、个性化需求 | 全局统一、核心字段 |

### 6.2 决策建议

#### 使用模板复制的情况：

```
✅ 适合：
1. 项目数量极少（<5个） ← ⚠️ 需要逐个绑定
2. 各项目可能有个性化需求
3. 测试/临时使用
4. 不要求新项目自动包含

❌ 不适合：
1. 项目数量多（>10个） ← 绑定工作量太大
2. 需要统一管理的核心字段
3. 经常有新项目
```

#### 使用全局系统字段的情况：

```
✅ 适合：
1. 公司/部门级统一规范字段
2. 核心业务字段（如"需求号"）
3. 需要100%覆盖率
4. 新项目必须包含
5. 后续需要全局统一管理
6. 不允许被删除
```

### 6.3 示例场景

#### 场景：为所有项目添加"需求号"字段

##### 方案1：模板复制 ❌ 不推荐

```
步骤：
1. 在项目A创建"需求号"字段
2. 编辑缺陷模板，添加该字段
3. 点击"复制"，选择所有项目（假设50个）
4. 复制完成
5. ⚠️ 逐个进入50个项目，手动设置新模板为默认模板
6. 完成

工作量：
复制：1次
绑定：50次 ← ⚠️ 巨大工作量！
时间：≈ 30-50分钟

问题：
❌ 需要逐个项目手动绑定（工作量与项目数成正比）
❌ 容易遗漏某些项目
❌ 新建项目不会自动包含
❌ 各项目的"需求号"是独立的（修改不同步）
❌ 可能被项目管理员误删除
```

##### 方案2：全局系统字段 ✅

```
步骤：
```

## 七、常见问题 FAQ

### Q1: 复制模式选"追加"还是"覆盖"？

**A:** 根据需求选择

| 需求 | 推荐模式 |
|------|---------|
| 在系统字段中**增加**选项 | ✅ 追加模式 |
| 完全**替换**系统字段选项 | ⚠️ 覆盖模式（慎用） |
| 目标项目可能有同名自定义字段 | ✅ 追加模式（创建副本） |
| 首次复制到空项目 | ✅ 两者都可以 |

### Q2: 能否撤销复制操作？

**A:** ❌ **不能！操作不可逆！**

**建议**：
- 先在测试项目验证
- 确认无误后再复制到生产项目

### Q3: 为什么有些项目无法选择？

**A:** 权限不足

```
可能原因：
1. 没有目标项目的"自定义字段"权限
2. 不在当前工作空间
3. 项目已被删除/禁用
```

**解决**：联系项目管理员分配权限

### Q4: 复制后能否修改字段？

**A:** ✅ 可以

```
复制后的字段是独立副本，可以：
✅ 修改字段名称
✅ 修改选项
✅ 删除字段
✅ 调整顺序

⚠️ 注意：修改后不影响其他项目
```

### Q5: 复制包含几百个项目，操作慢吗？

**A:** 复制操作本身不慢，但后续绑定工作量巨大！

**时间估算**：

| 操作 | 时间 |
|------|------|
| 复制到100个项目 | ≈ 30秒（一次性） |
| ⚠️ 逐个绑定100个项目 | ≈ 10-15分钟（每个项目1次） |

**工作量对比**：

```
复制10个项目：
├── 复制操作：1次（勾选10个项目）
└── 绑定操作：10次 ⚠️

复制100个项目：
├── 复制操作：1次（勾选100个项目）
└── 绑定操作：100次 ⚠️⚠️⚠️
```

**结论**：项目越多，手动绑定的工作量越大！

## 八、总结

### 核心要点

1. **模板复制是项目级批量配置工具**，适合快速初始化
2. **追加/覆盖模式影响字段处理**，谨慎选择
3. **操作不可逆**，务必先测试
4. **不等同于全局字段**，是独立副本

### 最佳实践

```
✅ 推荐用法：
- 新项目批量初始化
- 标准模板推广
- 系统字段选项补充（追加模式）
- 测试环境快速配置

❌ 不推荐用法：
- 替代全局系统字段
- 历史项目大规模变更
- 需要持续同步的场景
```

### 与全局字段互补

```
模板复制：灵活、快速、项目级独立
    ↕
全局系统字段：统一、持久、全局共享
```

**根据需求选择合适的方案！** 🎉
