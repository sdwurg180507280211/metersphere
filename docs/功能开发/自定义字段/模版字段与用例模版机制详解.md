# 模版字段与用例模版机制详解

## 一、核心概念

### 1.1 系统属性 vs 全局属性

#### 系统属性（system=1）
- **定义**：系统内置的字段或模板，不可删除
- **特点**：
  - `system=1` 表示系统级资源
  - 通常与 `global=1` 配合使用，形成"全局系统字段"或"全局系统模板"
  - 在UI中通常有删除保护（按钮禁用）

#### 全局属性（global=1）
- **定义**：全局共享的字段或模板，所有项目可见
- **特点**：
  - `global=1` 表示全局级资源
  - `project_id=NULL`（全局资源不归属于特定项目）
  - 可以被所有项目引用

#### 组合情况

| system | global | 说明 | 示例 |
|--------|--------|------|------|
| 1 | 1 | 全局系统字段/模板 | 状态、严重程度、default模板 |
| 0 | 1 | 全局自定义字段/模板 | 通过数据库脚本创建的全局自定义字段 |
| 1 | 0 | 项目系统字段/模板 | 项目级系统模板（较少见） |
| 0 | 0 | 项目自定义字段/模板 | 项目内创建的自定义字段 |

### 1.2 全局模版和全局字段

#### 全局模版（global=1）
- **用例模版**：`test_case_template` 表中 `global=1` 的记录
- **缺陷模版**：`issue_template` 表中 `global=1` 的记录
- **特点**：
  - 系统初始化时创建（如 default、Jira、禅道、TAPD 默认模板）
  - 所有项目共享
  - 作为项目模板的"模板"

#### 全局字段（global=1）
- **定义**：`custom_field` 表中 `global=1` 的记录
- **特点**：
  - 系统初始化时创建（如用例状态、责任人、用例等级）
  - 所有项目共享
  - 通过 `custom_field_template` 表关联到模板

## 二、项目模版使用机制

### 2.1 核心思路

**设计理念**：延迟创建 + 动态查询
- 项目创建时**不会**自动创建模版副本
- 使用模版时通过**动态查询**机制获取
- 只有在**编辑全局字段**时才批量创建项目级模版副本

### 2.2 项目创建时的行为

**实际流程**：
1. 创建项目记录（`project`表）
2. `case_template_id` 和 `issue_template_id` 字段为 **NULL**
3. **不会**创建任何模版副本
4. **不会**设置模版关联

**代码位置**：`SystemProjectService.addProject()` → `saveProject()`

**关键点**：
- ❌ 不调用任何模版创建方法
- ❌ 不设置 `case_template_id`
- ✅ 只发送 Kafka 消息创建默认节点（用例模块、评审模块等）

### 2.3 模版副本创建的触发时机

**唯一触发条件**：在某个项目中**编辑全局字段**时

**触发流程**：
1. 用户在项目A中编辑全局字段（如修改"用例状态"的选项）
2. 后端调用 `CustomFieldService.update()`
3. 检测到 `customField.getGlobal() == true`
4. 创建项目级字段副本
5. 调用 `TestCaseTemplateService.handleSystemFieldCreate()`
6. **为该项目创建所有全局模版的副本**

**代码位置**：`CustomFieldService.update()` → `handleSystemFieldCreate()`

**关键特点**：
- ✅ 一次性创建**所有**全局模版的副本（不只是当前使用的）
- ✅ 遍历所有 `global=1` 的模版
- ✅ 为每个全局模版创建项目级副本（如果不存在同名的）
- ✅ 更新项目的 `case_template_id` 指向项目级模版

### 2.4 模版副本创建的详细流程

#### 步骤1：查询所有全局模版和项目模版

```java
// TestCaseTemplateService.handleSystemFieldCreate()
// 1. 查询全局模板和项目模板
List<TestCaseTemplateWithBLOBs> testCaseTemplates = 
    testCaseTemplateMapper.selectByExample(example);

// 2. 按global属性分组
Map<Boolean, List<TestCaseTemplateWithBLOBs>> templatesMap = 
    testCaseTemplates.stream()
        .collect(Collectors.groupingBy(TestCaseTemplateWithBLOBs::getGlobal));

List<TestCaseTemplateWithBLOBs> globalTemplates = templatesMap.get(true);
List<TestCaseTemplateWithBLOBs> projectTemplates = templatesMap.get(false);
```

#### 步骤2：为每个全局模版创建项目级副本

```java
globalTemplates.forEach(global -> {
    // 检查项目是否有同名模板
    List<TestCaseTemplateWithBLOBs> projectTemplate = 
        projectTemplates.stream()
            .filter(i -> i.getName().equals(global.getName()))
            .collect(Collectors.toList());
    
    // 如果没有项目级别的模板就创建
    if (CollectionUtils.isEmpty(projectTemplate)) {
        // 创建模板副本
        TestCaseTemplateWithBLOBs template = new TestCaseTemplateWithBLOBs();
        BeanUtils.copyBean(template, global);  // 复制全局模板属性
        template.setId(UUID.randomUUID().toString());
        template.setGlobal(false);              // 设置为项目级
        template.setProjectId(customField.getProjectId());
        testCaseTemplateMapper.insert(template);
        
        // 更新项目的模板关联
        projectService.updateCaseTemplate(global.getId(), template.getId(), 
                                         customField.getProjectId());
    }
});
```

**关键点**：
- ✅ **遍历所有全局模版**：不只是当前使用的模版
- ✅ **模板会创建副本**：从全局模板复制，生成新的项目级模板记录
- ✅ **模板ID不同**：项目级模板有独立的ID
- ✅ **global属性改变**：从 `global=1` 变为 `global=0`
- ✅ **project_id设置**：设置为具体项目ID
- ✅ **更新项目关联**：`project.case_template_id` 指向项目级模版ID

#### 步骤3：处理字段关联关系

```java
// 获取全局模板的字段关联关系
List<CustomFieldTemplate> customFieldTemplate =
    customFieldTemplateService.getSystemFieldCreateTemplate(customField, global.getId());

// 创建项目级模板的字段关联
customFieldTemplateService.create(customFieldTemplate, template.getId(),
                                 TemplateConstants.FieldTemplateScene.TEST_CASE.name());
```

**关键方法**：`getSystemFieldCreateTemplate()`

```java
public List<CustomFieldTemplate> getSystemFieldCreateTemplate(
        CustomFieldDao customField, String templateId) {
    // 1. 获取全局模板的所有字段关联关系
    CustomFieldTemplateExample example = new CustomFieldTemplateExample();
    example.createCriteria().andTemplateIdEqualTo(templateId);
    List<CustomFieldTemplate> fieldTemplates = 
        customFieldTemplateMapper.selectByExample(example);
    
    // 2. 如果字段有项目级副本，替换field_id
    for (int i = 0; i < fieldTemplates.size(); i++) {
        if (StringUtils.equals(fieldTemplates.get(i).getFieldId(), 
                              customField.getOriginGlobalId())) {
            // 将全局字段ID替换成项目级字段ID
            fieldTemplates.get(i).setFieldId(customField.getId());
            break;
        }
    }
    return fieldTemplates;
}
```

## 三、项目如何使用模版：动态查询机制

### 3.1 核心思路

**问题**：项目创建时 `case_template_id` 为 NULL，那项目如何使用模版？

**答案**：通过 `getTemplate()` 方法的**动态查询机制**

### 3.2 动态查询流程

**代码位置**：`TestCaseTemplateService.getTemplate(projectId)`

**查询逻辑**：
```
1. 获取项目的 case_template_id
   ↓
2. 如果 case_template_id 不为空
   → 查询该模版
   → 如果查到，返回该模版
   → 如果没查到，调用 getDefaultTemplate()
   ↓
3. 如果 case_template_id 为空
   → 直接调用 getDefaultTemplate()
```

### 3.3 getDefaultTemplate() 的逻辑

**代码位置**：`TestCaseTemplateService.getDefaultTemplate(projectId)`

**查询逻辑**：
```
1. 优先查找项目级系统模版
   → WHERE project_id = 项目ID AND system = 1
   → 如果找到，返回第一个
   ↓
2. 如果没找到项目级系统模版
   → 查找全局模版
   → WHERE global = 1
   → 返回第一个（通常是 default 模版）
```

### 3.4 完整的模版使用流程

**场景1：新创建的项目（未编辑过全局字段）**
```
项目创建
  ↓
case_template_id = NULL
  ↓
用户创建用例，需要获取模版
  ↓
调用 getTemplate(projectId)
  ↓
case_template_id 为 NULL
  ↓
调用 getDefaultTemplate(projectId)
  ↓
查找项目级系统模版 → 没找到
  ↓
查找全局模版 → 找到 default (global=1)
  ↓
返回全局 default 模版
  ↓
项目使用全局模版
```

**场景2：编辑过全局字段的项目**
```
用户在项目中编辑全局字段
  ↓
触发 handleSystemFieldCreate()
  ↓
创建所有全局模版的项目级副本
  ↓
更新 case_template_id = 项目级模版ID
  ↓
用户创建用例，需要获取模版
  ↓
调用 getTemplate(projectId)
  ↓
case_template_id 不为空
  ↓
查询该模版 → 找到项目级模版
  ↓
返回项目级模版
  ↓
项目使用项目级模版
```

### 3.5 设计优势

**延迟创建的好处**：
- ✅ **节省存储**：大部分项目不需要定制模版，无需创建副本
- ✅ **统一管理**：未创建副本的项目共享全局模版，修改全局模版立即生效
- ✅ **按需定制**：只有需要定制的项目才创建副本
- ✅ **性能优化**：减少数据库记录，提高查询效率

**动态查询的好处**：
- ✅ **灵活切换**：可以在全局模版和项目级模版之间无缝切换
- ✅ **向后兼容**：老项目（case_template_id=NULL）也能正常使用
- ✅ **降级机制**：如果项目级模版被删除，自动降级到全局模版

## 四、关键问题：全局模版字段会创建副本吗？

### 4.1 答案：**不会创建字段副本，但会创建字段关联关系的副本**

#### 字段本身（custom_field表）

**全局字段不会创建副本**：
- 全局字段（`global=1`）在整个系统中只有一份
- 项目级模板直接引用全局字段的ID
- 不会在 `custom_field` 表中创建新的字段记录

**验证**：
```sql
-- 查询：全局字段和项目级模板的关联
SELECT cft.id, cft.field_id, cft.template_id, 
       cf.name as field_name, cf.global as field_global,
       tct.name as template_name, tct.global as template_global
FROM custom_field_template cft
JOIN custom_field cf ON cft.field_id = cf.id
JOIN test_case_template tct ON cft.template_id = tct.id
WHERE cft.scene='TEST_CASE' AND tct.global=0;

-- 结果：项目级模板关联的字段，field_global=1（全局字段）
-- 说明：项目级模板直接引用全局字段，没有创建字段副本
```

#### 字段关联关系（custom_field_template表）

**字段关联关系会创建副本**：
- 全局模板的字段关联关系会被复制到项目级模板
- 在 `custom_field_template` 表中创建新的关联记录
- 新的关联记录：
  - `template_id`：指向项目级模板ID
  - `field_id`：指向全局字段ID（或项目级字段ID，如果有副本）

**示例**：
```
全局模板（global=1）：
├── custom_field_template (id=1, template_id=global_template_id, field_id=global_field_id)
└── custom_field_template (id=2, template_id=global_template_id, field_id=global_field_id2)

创建项目级模板副本后：
├── 项目级模板（global=0）
│   ├── custom_field_template (id=3, template_id=project_template_id, field_id=global_field_id)  ← 新创建
│   └── custom_field_template (id=4, template_id=project_template_id, field_id=global_field_id2) ← 新创建
```

### 4.2 特殊情况：字段有项目级副本时

如果全局字段有对应的项目级副本（通过 `originGlobalId` 关联），则：

1. **字段关联关系创建时**：使用项目级字段ID，而不是全局字段ID
2. **目的**：允许项目自定义字段的选项或类型，而不影响全局字段

**代码逻辑**：
```java
// getSystemFieldCreateTemplate() 方法中
if (StringUtils.equals(fieldTemplates.get(i).getFieldId(), 
                      customField.getOriginGlobalId())) {
    // 替换为项目级字段ID
    fieldTemplates.get(i).setFieldId(customField.getId());
}
```

## 五、完整机制流程图

```
【全局层】
├── 全局用例模板（global=1）
│   ├── id: global_template_id
│   ├── name: "default"
│   └── global: 1
│
└── 全局字段（global=1）
    ├── 字段A（id: global_field_a_id, global=1）
    ├── 字段B（id: global_field_b_id, global=1）
    └── 字段C（id: global_field_c_id, global=1）
        │
        └── 字段关联（custom_field_template）
            ├── (template_id=global_template_id, field_id=global_field_a_id)
            ├── (template_id=global_template_id, field_id=global_field_b_id)
            └── (template_id=global_template_id, field_id=global_field_c_id)

【触发：创建全局系统字段或项目绑定全局模板】
        ↓

【项目层】
├── 项目级用例模板（global=0）← 从全局模板复制创建
│   ├── id: project_template_id（新ID）
│   ├── name: "default"（相同名称）
│   ├── global: 0（改为项目级）
│   └── project_id: project_id（设置项目ID）
│
└── 字段关联（custom_field_template）← 从全局模板复制创建
    ├── (id=新ID1, template_id=project_template_id, field_id=global_field_a_id) ← 新创建
    ├── (id=新ID2, template_id=project_template_id, field_id=global_field_b_id) ← 新创建
    └── (id=新ID3, template_id=project_template_id, field_id=global_field_c_id) ← 新创建

【关键点】
✅ 模板：创建了副本（新记录，新ID）
✅ 字段：没有创建副本（直接引用全局字段ID）
✅ 字段关联：创建了副本（新记录，新ID，但field_id指向全局字段）
```

## 六、数据库表结构说明

### 6.1 test_case_template（用例模板表）

| 字段 | 说明 | 全局模板 | 项目模板 |

| id | 主键 | 全局唯一ID | 项目级唯一ID |
| name | 字段名称 | "用例状态" | "用例状态"（相同） |
| global | 是否全局 | 1 | 0 |
| project_id | 所属项目 | NULL | 具体项目ID |
| system | 是否系统 | 1 | 0或1 |

### 6.3 custom_field_template（字段-模板关联表）

| 字段 | 说明 | 全局模板关联 | 项目模板关联 |

## 八、总结

### 8.1 核心机制总结

#### 1. 项目创建时的行为

**关键点**：
- ❌ **不创建模版副本**：项目创建时不会自动创建任何模版副本
- ❌ **不设置模版ID**：`case_template_id` 和 `issue_template_id` 为 NULL
- ✅ **延迟创建策略**：只在需要时才创建副本

#### 2. 模版使用机制

**动态查询流程**：
```
getTemplate(projectId)
  ↓
检查 case_template_id
  ↓
如果为 NULL → getDefaultTemplate()
  ↓
优先查找项目级系统模版 → 没找到
  ↓
返回全局模版（default）
```

**关键点**：
- ✅ **动态查询**：不依赖预先创建的副本
- ✅ **优雅降级**：项目级模版 → 全局模版
- ✅ **向后兼容**：老项目（case_template_id=NULL）也能正常使用

#### 3. 模版副本创建时机

**唯一触发条件**：在项目中**编辑全局字段**时

**创建范围**：
- ✅ **批量创建**：一次性创建**所有**全局模版的副本（不只是当前使用的）
- ✅ **模版副本**：从全局模版复制，生成新的项目级模版记录
- ✅ **字段关联副本**：创建新的 `custom_field_template` 记录
- ❌ **字段本身不复制**：项目级模版直接引用全局字段ID

#### 4. 副本创建的详细规则

**模版副本**：✅ 创建
- 从全局模版复制，生成新的项目级模版记录
- 新ID、新记录，但内容相同
- `global` 从 1 变为 0
- `project_id` 设置为具体项目ID

**字段副本**：❌ 不创建
- 全局字段在整个系统中只有一份
- 项目级模版直接引用全局字段ID
- 除非通过编辑全局字段触发字段副本创建

**字段关联副本**：✅ 创建
- 从全局模版的字段关联关系复制
- 创建新的 `custom_field_template` 记录
- `template_id` 指向项目级模版ID
- `field_id` 指向全局字段ID（除非有项目级字段副本）

### 8.2 设计思路与优势

#### 设计理念

**延迟创建 + 动态查询 + 按需定制**

1. **延迟创建**：
   - 项目创建时不预先创建模版副本
   - 只在编辑全局字段时才批量创建
   - 避免为不需要定制的项目创建大量冗余数据

2. **动态查询**：
   - 使用时通过 `getTemplate()` 动态查询
   - 优先使用项目级模版，没有则使用全局模版
   - 提供灵活的降级机制

3. **按需定制**：
   - 只有需要定制的项目才创建副本
   - 一次创建所有模版副本，避免多次触发
   - 保证项目有完整的模版体系

#### 设计优势

1. **存储优化**：
   - 大部分项目不需要定制模版，无需创建副本
   - 减少数据库记录数量，节省存储空间
   - 全局字段只有一份，避免重复存储

2. **统一管理**：
   - 未创建副本的项目共享全局模版
   - 修改全局模版或字段，所有项目立即生效
   - 便于统一规范和管理

3. **灵活定制**：
   - 项目可以创建自己的模版副本
   - 可以自定义字段顺序、必填性等
   - 不影响其他项目和全局模版

4. **性能优化**：
   - 动态查询机制高效
   - 减少项目创建时的初始化开销
   - 提高系统整体性能

### 8.3 重要注意事项

#### 1. 全局资源的影响范围

**修改全局字段**：
- ⚠️ 会影响**所有**直接使用全局字段的项目
- ⚠️ 包括未创建模版副本的项目
- ⚠️ 修改后立即生效，无法回滚
- ✅ 建议：在测试环境验证后再应用到生产环境

**修改全局模版**：
- ⚠️ 只能在数据库中直接修改
- ⚠️ 会影响所有未创建副本的项目
- ⚠️ 已创建副本的项目不受影响
- ✅ 建议：谨慎修改，提前通知所有项目团队

#### 2. 项目级副本的创建时机

**何时创建**：
- ✅ 在项目中编辑全局字段时自动触发
- ✅ 一次性创建所有全局模版的副本
- ❌ 项目创建时不会自动创建
- ❌ 不能通过UI手动触发（只能通过编辑全局字段）

**创建后的影响**：
- ✅ 项目的 `case_template_id` 更新为项目级模版ID
- ✅ 项目开始使用项目级模版，不再受全局模版影响
- ✅ 但仍然引用全局字段（除非创建了字段副本）

#### 3. 字段关联关系的独立性

**项目级模版的字段关联**：
- ✅ 是独立的 `custom_field_template` 记录
- ✅ 可以自定义字段顺序
- ✅ 可以自定义必填性
- ✅ 可以自定义默认值
- ❌ 但字段本身（`custom_field`）仍然是全局的

#### 4. 数据一致性考虑

**全局字段的一致性**：
- ✅ 全局字段只有一份，保证数据一致性
- ⚠️ 修改全局字段会影响所有项目
- ✅ 如需项目独立定制，需创建项目级字段副本（通过 `originGlobalId` 关联）

## 十、常见问题解答

### 10.1 系统字段编辑时也是新建一个副本吗？

**答案：是的，系统字段（global=1）编辑时也会创建项目级字段副本。**

#### 后端代码逻辑

```java
// CustomFieldService.update()
public void update(CustomField customField) {
    if (customField.getGlobal() != null && customField.getGlobal()) {
        CustomFieldDao customFieldDao = new CustomFieldDao();
        BeanUtils.copyBean(customFieldDao, customField);
        customFieldDao.setOriginGlobalId(customField.getId());  // 保存原始全局字段ID
        // 如果是全局字段，则创建对应项目字段
        add(customFieldDao);  // 创建项目级字段副本
        if (StringUtils.equals(customField.getScene(), TemplateConstants.FieldTemplateScene.TEST_CASE.name())) {
            testCaseTemplateService.handleSystemFieldCreate(customFieldDao);
        } else if (StringUtils.equals(customField.getScene(), TemplateConstants.FieldTemplateScene.API.name())) {
            apiTemplateService.handleSystemFieldCreate(customFieldDao);
        } else {
            issueTemplateService.handleSystemFieldCreate(customFieldDao);
        }
    } else {
        // 非全局字段，直接更新
        checkExist(customField);
        customField.setUpdateTime(System.currentTimeMillis());
        customFieldMapper.updateByPrimaryKeySelective(customField);
    }
}
```

#### 关键逻辑

1. **检测全局字段**：当 `customField.getGlobal() == true` 时
2. **创建项目级副本**：
   - 调用 `add()` 方法创建新的项目级字段（`global=false`）
   - 设置 `originGlobalId` 指向原始全局字段ID
   - 新字段的 `project_id` 设置为当前项目ID
3. **处理模板关联**：
   - 调用 `handleSystemFieldCreate()` 方法
   - 更新项目级模板的字段关联，从全局字段ID替换为项目级字段ID

#### 与模板编辑的对比

| 操作 | 全局模板编辑 | 全局字段编辑 |
|------|------------|------------|
| 前端是否允许 | ✅ 允许（没有禁用） | ✅ 允许（部分字段禁用） |
| 后端检测 | `request.getGlobal() == true` | `customField.getGlobal() == true` |
| 实际行为 | 创建项目级模板副本 | 创建项目级字段副本 |
| 是否修改全局资源 | ❌ 不修改 | ❌ 不修改 |
| 关联关系处理 | 复制字段关联关系 | 更新模板的字段关联 |

#### 总结

- ✅ **系统字段编辑**：会创建项目级字段副本，不会修改全局字段本身
- ✅ **系统模板编辑**：会创建项目级模板副本，不会修改全局模板本身
- ✅ **设计一致性**：字段和模板都采用相同的保护机制
- ✅ **真正修改全局资源**：只能在数据库中直接修改

### 10.2 全局模板只能在数据库中修改吗？

**答案：是的，全局模板（global=1）只能在数据库中直接修改。**

#### 为什么页面上可以"编辑"系统模板？

**重要发现**：虽然缺陷模板是系统模板（`system=1, global=1`），但前端**没有禁用编辑按钮**！

**前端代码分析**：
```javascript
// IssuesTemplateList.vue
operators: [
  {
    tip: this.$t('commons.edit'), icon: "el-icon-edit",
    exec: this.handleEdit
    // ⚠️ 注意：没有 isDisable 属性！
  }, {
    tip: this.$t('commons.copy'), icon: "el-icon-copy-document", type: "success",
    exec: this.handleCopy
    // ⚠️ 注意：没有 isDisable 属性！
  }, {
    tip: this.$t('commons.delete'), icon: "el-icon-delete", type: "danger",
    exec: this.handleDelete
    // ⚠️ 注意：没有 isDisable 属性！
  }
]
```

**对比用例模板**（有禁用逻辑）：
```javascript
// TestCaseTemplateList.vue
operators: [
  {
    tip: this.$t('commons.edit'), icon: "el-icon-edit",
    exec: this.handleEdit
    // ⚠️ 编辑按钮也没有禁用
  }, {
    tip: this.$t('commons.copy'), icon: "el-icon-copy-document", type: "success",
    exec: this.handleCopy,
    isDisable: this.systemDisable  // ✅ 复制按钮有禁用
  }, {
    tip: this.$t('commons.delete'), icon: "el-icon-delete", type: "danger",
    exec: this.handleDelete,
    isDisable: this.systemDisable  // ✅ 删除按钮有禁用
  }
]
```

**结论**：
- ❌ **缺陷模板列表**：编辑、复制、删除按钮都**没有禁用**系统模板的操作
- ✅ **用例模板列表**：复制、删除按钮**有禁用**系统模板的操作，但编辑按钮也没有禁用
- ⚠️ **这是前端的一个不一致之处**：系统模板理论上不应该允许编辑，但前端没有完全禁用

#### 后端保护机制

虽然前端允许编辑，但**后端有保护机制**：

**后端代码逻辑**：
```java
// IssueTemplateService.update()
public void update(UpdateIssueTemplateRequest request) {
    if (request.getGlobal() != null && request.getGlobal()) {
        String originId = request.getId();
        // 如果是全局模板，则创建对应项目模板副本
        String id = add(request);
        projectService.updateIssueTemplate(originId, id, request.getProjectId());
    } else {
        // 非全局模板，直接更新
        // ...
    }
}
```

**关键逻辑**：
- 当 `request.getGlobal() == true` 时，**不会直接更新全局模板**
- 而是调用 `add()` 方法创建一个**项目级模板副本**
- 然后更新项目的模板关联，指向新创建的项目级模板

**前端行为**：
- 前端在保存模板时，会设置 `param.projectId = getCurrentProjectID()`
- 即使编辑的是全局模板，也会传入项目ID
- 后端检测到 `global=true` 时，会创建项目级副本

**实际效果**：
- ✅ **UI中"编辑"全局模板**：实际上会创建项目级模板副本，**不会修改全局模板本身**
- ✅ **真正修改全局模板**：只能在数据库中直接修改 `issue_template` 或 `test_case_template` 表
- ✅ **后端保护**：即使前端允许编辑，后端也不会真正修改全局模板，而是创建副本

#### 总结

| 操作 | 前端是否允许 | 后端实际行为 | 是否修改全局模板 |
|------|------------|------------|----------------|
| 编辑系统模板 | ✅ 允许（没有禁用） | 创建项目级副本 | ❌ 不修改 |
| 复制系统模板 | ⚠️ 缺陷模板允许，用例模板禁用 | 创建项目级副本 | ❌ 不修改 |
| 删除系统模板 | ⚠️ 缺陷模板允许，用例模板禁用 | 后端会检查，不允许删除 | ❌ 不删除 |
| 数据库直接修改 | N/A | 直接修改 | ✅ 会修改 |

#### 修改全局模板的SQL示例

```sql

SET content = '新的模板内容',
    update_time = unix_timestamp() * 1000
WHERE id = '5d7c87d2-f405-4ec1-9a3d-71b514cdfda3' 
  AND `global` = 1;
```

### 10.3 缺陷模板有全局字段，但没有系统字段，是吗？

**答案：不对。缺陷模板既有全局字段，也有系统字段，它们都是"全局系统字段"（global=1, system=1）。**

#### 数据库验证结果

**缺陷模板**：
```sql
SELECT id, name, `global`, `system`, project_id 
FROM issue_template 
WHERE `global`=1;

-- 结果：
-- id: 5d7c87d2-f405-4ec1-9a3d-71b514cdfda3
-- name: default
-- global: 1
-- system: 1  ← 系统模板
-- project_id: global
```

**缺陷模板关联的字段**：
```sql
SELECT cf.id, cf.name, cf.scene, cf.`global`, cf.`system`, cf.project_id 
FROM custom_field cf 
JOIN custom_field_template cft ON cf.id = cft.field_id 
JOIN issue_template it ON cft.template_id = it.id 
WHERE it.`global`=1 AND cft.scene='ISSUE' 
GROUP BY cf.id;

-- 结果示例：
-- 状态：global=1, system=1  ← 全局系统字段
-- 严重程度：global=1, system=1  ← 全局系统字段
-- 处理人：global=1, system=1  ← 全局系统字段
-- 需求号：global=1, system=1  ← 全局系统字段
```

#### 结论

- ✅ **缺陷模板**：都是 `global=1, system=1`（全局系统模板）
- ✅ **缺陷字段**：都是 `global=1, system=1`（全局系统字段）
- ❌ **不存在**：只有全局属性但没有系统属性的缺陷模板或字段

**说明**：
- 系统初始化时创建的缺陷模板和字段都是"全局系统"类型
- 这样设计是为了：
  - 全局共享（所有项目可见）
  - 系统保护（不可删除）

### 10.4 全局的只能是系统字段吗？

**答案：不是。理论上可以存在全局自定义字段（global=1, system=0），但实际使用中几乎都是全局系统字段。**

#### 数据库验证结果

```sql
-- 查询所有全局字段
SELECT id, name, scene, `global`, `system`, project_id 
FROM custom_field 
WHERE `global`=1 
ORDER BY `system`;

-- 实际结果：所有全局字段的 system 都是 1
-- 说明：当前系统中没有 global=1, system=0 的全局自定义字段
```

#### 理论上的组合

根据系统设计，字段可以有四种组合：

| system | global | 说明 | 创建方式 | 实际使用 |
|--------|--------|------|---------|---------|
| 1 | 1 | 全局系统字段 | 数据库脚本 | ✅ 大量使用（状态、严重程度、需求号等） |
| 0 | 1 | 全局自定义字段 | 数据库脚本 | ❌ 几乎不使用 |
| 1 | 0 | 项目系统字段 | 数据库脚本 | ⚠️ 较少使用 |
| 0 | 0 | 项目自定义字段 | UI界面 | ✅ 大量使用 |

#### 为什么全局字段几乎都是系统字段？

**1. UI无法创建全局字段**

```java
// CustomFieldService.add()
public String add(CustomField customField) {
    // ...
    customField.setGlobal(false);  // ⭐ 强制设置为 false
    // ...
}
```

**结论**：
- UI只能创建项目级字段（global=0）
- 全局字段（global=1）只能通过数据库脚本创建

**2. 系统初始化时都是系统字段**

系统初始化时创建的全局字段都是系统字段：
- 用例状态、用例等级、责任人（TEST_CASE场景）
- 状态、严重程度、处理人、需求号（ISSUE场景）

**3. 全局自定义字段的局限性**

即使通过数据库脚本创建 `global=1, system=0` 的字段：
- ⚠️ **可能被删除**：因为 system=0，有权限的用户可以删除
- ⚠️ **可能被改名**：缺少系统保护
- ⚠️ **缺少保护**：没有系统级保护机制

**4. 实际使用建议**

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 核心业务字段（如"需求号"） | `system=1, global=1` | 需要系统保护，不可删除 |
| 公司级通用字段 | `system=1, global=1` | 需要全局共享，需要保护 |
| 可选通用字段 | `system=0, global=1` | 理论上可以，但不推荐 |
| 项目特定字段 | `system=0, global=0` | 项目内创建，灵活管理 |

#### 总结

- ✅ **理论上**：可以存在 `global=1, system=0` 的全局自定义字段
- ❌ **实际中**：几乎不使用，因为缺少系统保护
- ✅ **推荐做法**：全局字段都使用 `system=1, global=1`（全局系统字段）
- ✅ **原因**：需要系统级保护，防止误删和误改

**结论**：虽然技术上可以创建全局自定义字段，但实际使用中，全局字段几乎都是系统字段，因为需要系统级保护。

### 10.5 如果要修改全局缺陷模板，只能在数据库中修改吗？

**答案：是的，只能通过数据库直接修改。**

#### 修改方式

**方式1：直接修改全局模板表**
```sql
-- 修改全局缺陷模板的基本信息
UPDATE issue_template 
SET title = '新的标题',
    content = '新的内容',
    description = '新的描述',
    update_time = unix_timestamp() * 1000
WHERE id = '5d7c87d2-f405-4ec1-9a3d-71b514cdfda3' 
  AND `global` = 1;
```

**方式2：修改全局模板的字段关联关系**
```sql
-- 添加字段关联
INSERT INTO custom_field_template (id, field_id, template_id, scene, required, `order`, default_value)
VALUES (uuid(), 'field_id', 'template_id', 'ISSUE', 0, 10, '');

-- 删除字段关联
DELETE FROM custom_field_template 
WHERE template_id = 'template_id' 
  AND field_id = 'field_id';
```

**方式3：修改全局字段的选项**
```sql
-- 修改全局系统字段的选项（会影响所有项目）
UPDATE custom_field 
SET options = '[{"value":"P0","text":"P0"},{"value":"P1","text":"P1"}]',
    update_time = unix_timestamp() * 1000
WHERE id = 'd392af07-fdfe-4475-a459-87d59f0b1626' 
  AND `global` = 1;
```

#### 注意事项

⚠️ **修改全局模板的影响**：
- 会影响所有使用该全局模板的项目
- 但已经创建的项目级模板副本不会自动更新
- 只有新建项目或重新绑定全局模板时，才会使用新的全局模板内容

⚠️ **修改全局字段的影响**：
- 会影响所有使用该全局字段的项目
- 因为项目级模板直接引用全局字段ID
- 修改后立即生效，所有项目都会看到变化

⚠️ **建议**：
- 修改前先备份数据库
- 在测试环境验证后再应用到生产环境
- 通知所有项目团队，避免影响正在进行的测试
