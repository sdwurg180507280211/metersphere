# MeterSphere 国际化字段命名规则

## ❓ 问题：`issue_phone` 这个字符串是随意创建的吗？

**答案：不是！** 它遵循了项目的命名规范。

---

## 📐 命名规则详解

### 1. 命名结构分为两部分

#### 第一部分：`table-constants.js` 中的映射路径
```javascript
export const SYSTEM_FIELD_NAME_MAP = {
  // 格式：中文字段名: 'i18n路径'
  手机: 'test_track.issue_phone',
}
```

#### 第二部分：i18n 翻译文件中的 key
```javascript
// zh-CN.js / en-US.js / zh-TW.js
export default {
  test_track: {
    issue: {
      issue_phone: "手机" / "Phone" / "手機"
    }
  }
}
```

---

## 🎯 两种命名约定

### 约定1：`custom_field.*` - 用于系统内置字段

**适用场景**：MeterSphere 原有的系统字段

```javascript
// table-constants.js
创建人: 'custom_field.issue_creator',
处理人: 'custom_field.issue_processor',
状态: 'custom_field.issue_status',
严重程度: 'custom_field.issue_severity',
```

**对应的翻译**：
```javascript
// zh-CN.js
custom_field: {
  issue_creator: '创建人',
  issue_processor: '处理人',
  issue_status: '状态',
  issue_severity: '严重程度',
}
```

**特点**：
- ✅ 系统核心字段
- ✅ 所有模块通用
- ✅ 翻译定义在根级 `custom_field` 下

---

### 约定2：`test_track.issue.*` - 用于新增字段

**适用场景**：后续新增的业务字段

```javascript
// table-constants.js
需求号: 'test_track.issue.requirement_number',
手机: 'test_track.issue_phone',
```

**对应的翻译**：
```javascript
// zh-CN.js
test_track: {
  issue: {
    requirement_number: '需求号',
    issue_phone: '手机',
  }
}
```

**特点**：
- ✅ 业务扩展字段
- ✅ 翻译定义在 `test_track.issue` 命名空间下
- ✅ 与缺陷管理模块相关

---

## 🔍 为什么有两种约定？

### 历史原因

1. **早期设计**（`custom_field.*`）
   - MeterSphere 最初定义的系统字段
   - 放在根级 `custom_field` 命名空间
   - 所有模块共享

2. **后期扩展**（`test_track.issue.*`）
   - 后续新增的业务字段
   - 为了避免污染根级命名空间
   - 按模块组织，更清晰

### 对比

| 特性 | `custom_field.issue_*` | `test_track.issue.*` |
|------|----------------------|---------------------|
| **用途** | 系统核心字段 | 业务扩展字段 |
| **位置** | 根级 `custom_field` | `test_track.issue` |
| **示例** | 创建人、处理人、状态 | 需求号、手机 |
| **灵活性** | 较固定 | 较灵活 |
| **推荐** | 不建议新增 | ✅ 推荐 |

## ✅ 命名规则总结

### 规则1：路径命名（`table-constants.js`）

```javascript
字段中文名: 'namespace.子namespace.字段key'
```

**约定**：
- 新增缺陷字段使用：`test_track.issue.*`
- 新增用例字段使用：`test_track.case.*`
- 全局通用字段使用：`custom_field.*`

### 规则2：Key 命名（i18n 文件）

```javascript
字段key: "翻译值"
```

**命名风格**：
- ✅ 使用 **snake_case**（下划线命名）
- ✅ 使用英文描述
- ✅ 见名知意
- ❌ 不使用拼音
- ❌ 不使用驼峰命名

**示例**：
```javascript
// ✅ 好的命名
requirement_number: '需求号',
issue_phone: '手机',
phone_number: '手机号码',
contact_email: '联系邮箱',

// ❌ 不好的命名
xuqiuhao: '需求号',          // ❌ 使用拼音
issuePhone: '手机',          // ❌ 驼峰命名
phone: '手机',               // ⚠️ 太简短，不明确
```

## 🏗️ 国际化多层架构

### 📋 前端国际化层级结构

#### **Framework 层**（`framework/sdk-parent/frontend/src/i18n/`）
- **用途**：全局共享的国际化配置
- **范围**：所有模块共用（test-track、api-test、performance-test 等）
- **文件结构**：
  ```
  framework/sdk-parent/frontend/src/i18n/lang/
  ├── zh-CN.js          # 全局中文
  ├── en-US.js          # 全局英文  
  ├── zh-TW.js          # 全局繁中
  └── track/            # 测试跟踪专用
      ├── zh-CN.js      # 测试跟踪中文
      ├── en-US.js      # 测试跟踪英文
      └── zh-TW.js      # 测试跟踪繁中
  ```

#### **Module 层**（`test-track/frontend/src/i18n/`）
- **用途**：test-track 模块专属的国际化配置
- **范围**：仅 test-track 模块使用
- **文件结构**：
  ```
  test-track/frontend/src/i18n/lang/
  ├── zh-CN.js          # test-track 中文
  ├── en-US.js          # test-track 英文
  └── zh-TW.js          # test-track 繁中
  ```

#### **优先级机制**
```
Module 层 > Framework 层的 track/ > Framework 层全局
```

- **test-track 模块**会先加载自己的国际化文件
- 然后加载 `framework/sdk-parent/frontend/src/i18n/lang/track/` 的配置
- 最后加载 `framework/sdk-parent/frontend/src/i18n/lang/` 的全局配置

## 🔧 后端国际化操作

### 🎯 后端国际化的主要用途

#### 1. **API 响应中的文本**
```java
// 后端返回的错误消息、提示信息等

```

#### 2. **日志和审计记录**
```java
// 审计日志中记录的字段名称

```

#### 3. **第三方平台同步**
```java
// 同步到 Jira、Tapd 等平台时的字段名称

```

#### 4. **邮件通知**
```java
// 邮件模板中的字段显示
emailService.sendNotification("缺陷 '复测次数' 已更新");
```

### 🔄 `Translator.get()` 内部逻辑

#### **核心方法**
```java
public static String get(String key) {
    return messageSource.getMessage(key, null, "Not Support Key: " + key, LocaleContextHolder.getLocale());
}
```

#### **执行流程**
1. **获取当前语言**：从 `LocaleContextHolder.getLocale()` 获取
2. **查找消息**：`messageSource.getMessage(key, null, defaultMessage, locale)`
3. **文件匹配**：按语言优先级查找 properties 文件
4. **返回结果**：找到则返回翻译，未找到返回默认消息

#### **配置文件**
```properties
# commons.properties
spring.messages.basename=i18n/messages,i18n/commons
```

#### **文件结构**
```
src/main/resources/
├── i18n/
│   ├── messages_zh_CN.properties   # 简体中文
│   ├── messages_en_US.properties   # 英文
│   └── messages_zh_TW.properties   # 繁体中文
```

### 🎯 实际应用场景

#### **API 错误消息**
```java
if (retestCount < 0) {
    throw new MSException(Translator.get("issue_retest_count_invalid"));
}
```

#### **审计日志记录**
```java
IssueChangeLogDetailDTO detail = new IssueChangeLogDetailDTO();
detail.setFieldName(Translator.get("issue_retest_count"));
detail.setOldValue("0");
detail.setNewValue("1");
```

#### **第三方平台同步**
```java
String displayName = Translator.get("issue_" + fieldName);
jiraApi.updateCustomField(displayName, value);
```

## 📝 总结

MeterSphere 的国际化系统采用**多层架构**，前后端分工明确：

- **前端**：负责 UI 显示，使用嵌套命名空间
- **后端**：负责系统消息，使用 Spring MessageSource
- **配置**：前后端独立配置，通过不同机制生效

理解这套机制，才能正确实现字段的国际化功能。
