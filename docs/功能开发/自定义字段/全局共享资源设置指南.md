## 一、核心概念

### 1.1 资源隔离层级

MeterSphere 采用**三层隔离架构**：

```
全局层 (Global)
├── system=1, global=1
├── 所有工作空间和项目共享
└── 不可在项目中删除或修改核心属性

项目层 (Project)
├── global=0, project_id=具体ID
├── 仅当前项目可见
└── 可在项目设置中修改/删除

注意：MeterSphere 的 custom_field 表没有 workspace_id 字段，只有 project_id 字段
```

### 1.2 字段的三种类型

| 类型 | system | global | project_id | 说明 |
|------|--------|--------|------------|------|
| **全局系统字段** | 1 | 1 | NULL | 系统预定义，全局共享，不可删除 |
| **全局自定义字段** | 0 | 1 | NULL | 工作空间管理员创建，全局共享 |
| **项目自定义字段** | 0 | 0 | 具体ID | 项目内创建，仅本项目可见 |

### 2.2 完整操作步骤（6步法）

#### ✅ **步骤1：数据库 - 创建全局字段**

```sql
-- 在 system-setting/backend/src/main/resources/db/migration/ 下创建新的迁移脚本
-- 例如：V143__add_requirement_number_field.sql

INSERT INTO custom_field (
    id, 
    name, 
    scene, 
    type, 
    remark, 
    options, 
    `system`, 
    `global`, 
    create_time, 
    update_time,
    create_user,
    project_id,
    third_part
)
SELECT 
    'req-num-field-id-2024-12-01-001',  -- ⭐ 唯一ID
    '需求号',                             -- ⭐ 字段名称
    'ISSUE',                             -- ⭐ 场景：ISSUE/TEST_CASE/API
    'input',                             -- ⭐ 类型：input/select/date等
    '',                                  -- remark
    '[]',                                -- options
    1,                                   -- ⭐ system=1（系统字段）
    1,                                   -- ⭐ global=1（全局字段）
    unix_timestamp() * 1000,            -- create_time
    unix_timestamp() * 1000,            -- update_time
    NULL,                                -- create_user（全局系统字段可为NULL）
    NULL,                                -- ⭐ project_id=NULL（全局字段）
    0                                    -- third_part=0（非第三方字段）
WHERE NOT EXISTS (
    SELECT 1 FROM custom_field 
    WHERE name = '需求号' AND scene = 'ISSUE' AND `system` = 1 AND `global` = 1
);
```

**关键参数说明**：
- `system=1`：标记为系统字段，不可在UI中删除
- `global=1`：标记为全局字段，所有项目可见
- `project_id=NULL`：全局字段的 `project_id` 必须为 NULL（项目级字段才有具体的 project_id）
- `third_part=0`：非第三方平台字段

#### ✅ **步骤3：数据库 - 关联到所有项目模板**

```sql
-- ⭐ 关键步骤！确保现有项目也能看到新字段
INSERT INTO custom_field_template (id, field_id, template_id, scene, required, `order`, default_value)
SELECT 
    uuid() as id,
    (SELECT id FROM custom_field WHERE name = '需求号' AND scene = 'ISSUE' AND `system` = 1 AND global = 1 LIMIT 1) as field_id,
    it.id as template_id,
    'ISSUE' as scene,
    0 as required,
    2 as `order`,
    '' as default_value
FROM issue_template it
WHERE it.global = 0  -- ⭐ 只处理项目级别的模板
  AND NOT EXISTS (
      SELECT 1 
      FROM custom_field_template cft 
      WHERE cft.template_id = it.id 
        AND cft.field_id = (SELECT id FROM custom_field WHERE name = '需求号' AND scene = 'ISSUE' AND `system` = 1 AND global = 1 LIMIT 1)
  );
```

**为什么必须这样做？**
- 全局字段 **≠** 自动显示在所有项目
- 字段显示基于 `custom_field_template` 关联
- 必须为每个项目模板创建关联记录

## 五、常见问题 FAQ

### Q1: 为什么设置了 global=1，项目中还看不到？

**A:** global=1 只表示字段是全局的，但**不代表自动显示**。必须：
1. 创建 `custom_field_template` 关联记录
2. 关联到项目使用的模板ID

### Q2: 如果只关联了全局模板，现有项目能看到吗？

**A:** ❌ **不能！** 必须执行**步骤3**，批量关联所有项目级模板。

**原因**：
- 项目使用的是**项目级模板**（global=0）
- 项目级模板是基于全局模板创建的副本
- 后续添加的字段不会自动同步到已存在的项目模板

**真实案例**：
```
❌ 错误情况：
  - 只关联了4个全局模板
  - 结果：只有默认项目看到字段，其他项目看不到

✅ 正确做法：
  - 关联4个全局模板 + 所有项目级模板
  - 结果：所有项目都能看到字段
```

### Q3: system=1 和 global=1 的区别？

| 属性 | 含义 | 效果 |
|------|------|------|
| `system=1` | 系统字段 | 不可在UI中删除/修改核心属性 |
| `global=1` | 全局字段 | 作用域为全局（需配合关联表） |

**最佳实践**：系统预定义的字段应该 `system=1 AND global=1`

### Q4: 能通过UI界面创建全局自定义字段（system=0, global=1）吗？

**A:** ❌ **不能！**

**代码分析**：

```java
// CustomFieldService.java - add() 方法
public String add(CustomField customField) {
    // ...
    customField.setGlobal(false);  // ⭐ 强制设置为 false
    customField.setThirdPart(false);
    customFieldMapper.insert(customField);
    return customField.getId();
}
```

**结论**：
- ✅ UI只能创建**项目级自定义字段**（global=0, project_id=当前项目）
- ✅ **全局自定义字段**（global=1）只能通过**数据库迁移脚本**创建
- ✅ **全局系统字段**（system=1, global=1）只能通过**数据库迁移脚本**创建

### Q4.1: UI创建的字段是什么级别？

**A:** 项目级自定义字段（global=0）

**操作路径**：
```
1. 选择项目
2. 项目设置 → 模版管理 → 自定义字段
3. 点击"创建"按钮
4. 填写字段信息（名称、场景、类型、选项等）
5. 保存
```

**效果**：
- ✅ 仅当前项目可见
- ❌ 其他项目不可见
- ✅ 可以在项目设置中修改/删除

### Q4.2: 想要全局共享，必须怎么做？

**A:** 只能通过数据库迁移脚本！

| 方式 | 创建位置 | system | global | 效果 |
|------|----------|--------|--------|------|
| **UI界面** | 项目设置 → 模版管理 → 自定义字段 | 0 | 0 | ❌ 仅当前项目 |
| **数据库脚本** | V143__xxx.sql | 0 | 1 | ✅ 全局共享（需关联模板） |
| **数据库脚本** | V143__xxx.sql | 1 | 1 | ✅ 全局系统字段 |

### Q5: 如果是工作空间级共享，怎么做？

**A:** 暂不支持。

**系统设计**：
- 全局层（global=1）：整个系统共享
- 项目层（global=0）：单个项目独立

**没有工作空间层的字段隔离机制**。

### Q6: 新建项目会自动包含"需求号"字段吗？

**A:** 会的！

**原因**：
- 新建项目时，会基于全局模板创建项目级模板
- 创建过程会复制 `custom_field_template` 关联关系
- 由于全局模板已关联"需求号"，新项目会自动包含

### Q7: ⚠️ 为什么数据库中出现了多个"需求号"字段？

**A:** 可能是在UI中手动创建了项目级的"需求号"字段，导致重复。

**问题表现**：
```sql
-- 查询结果显示多个"需求号"
SELECT id, name, `system`, `global`, project_id 
FROM custom_field 
WHERE name = '需求号' AND scene = 'ISSUE';

-- 结果示例：
-- req-num-field-id-xxx  | 需求号 | 1 | 1 | NULL      ← 全局系统字段 ✅
-- abc-123-def           | 需求号 | 0 | 0 | project-1 ← 项目级字段 ❌
-- xyz-456-ghi           | 需求号 | 0 | 0 | project-2 ← 项目级字段 ❌
```

**修复方案**：
```sql
-- 1. 删除项目级字段的关联
DELETE cft FROM custom_field_template cft
JOIN custom_field cf ON cft.field_id = cf.id
WHERE cf.name = '需求号' 
  AND cf.`system` = 0 
  AND cf.`global` = 0;

-- 2. 删除项目级字段本身
DELETE FROM custom_field 
WHERE name = '需求号' 
  AND `system` = 0 
  AND `global` = 0;

-- 3. 为所有项目模板关联全局系统字段
INSERT INTO custom_field_template (id, field_id, template_id, scene, required, `order`, default_value)
SELECT 
    uuid() as id,
    (SELECT id FROM custom_field WHERE name = '需求号' AND scene = 'ISSUE' AND `system` = 1 AND `global` = 1 LIMIT 1) as field_id,
    it.id as template_id,
    'ISSUE' as scene,
    0 as required,
    2 as `order`,
    '' as default_value
FROM issue_template it
WHERE it.global = 0
  AND NOT EXISTS (
      SELECT 1 FROM custom_field_template cft 
      WHERE cft.template_id = it.id 
        AND cft.field_id = (SELECT id FROM custom_field WHERE name = '需求号' AND scene = 'ISSUE' AND `system` = 1 AND `global` = 1 LIMIT 1)
  );
```

**预防措施**：
- ✅ 使用 `system=1` 标记系统字段，防止UI删除
- ✅ 在SQL脚本中使用唯一的固定ID（如 `req-num-field-id-2024-12-01-001`）
- ✅ 使用 `WHERE NOT EXISTS` 防止重复插入

### 7.3 为什么UI不能创建全局字段？

#### 🔐 **设计原因**

**1. 权限控制**
```
项目管理员权限：
├── ✅ 管理本项目的自定义字段
├── ✅ 创建/修改/删除项目字段
└── ❌ 不能影响其他项目
```

**2. 数据安全**
- 全局字段影响**所有项目**
- 需要更高权限（系统管理员）
- UI操作风险大，容易误操作

**3. 代码证据**

```java
// CustomFieldService.java - add() 方法
public String add(CustomField customField) {
    checkExist(customField);
    customField.setId(UUID.randomUUID().toString());
    customField.setCreateTime(System.currentTimeMillis());
    customField.setUpdateTime(System.currentTimeMillis());
    customField.setGlobal(false);  // ⭐ 硬编码强制设置为 false
    customField.setThirdPart(false);
    customField.setCreateUser(SessionUtils.getUserId());
    customFieldMapper.insert(customField);
    return customField.getId();
}
```

**结论**：代码层面**强制**所有UI创建的字段都是项目级（global=0）

### 7.4 如果想要项目共享字段怎么办？

#### 方案1️⃣：模板复制功能（推荐用于少量项目）

**操作步骤**：

```
1. 在项目A创建自定义字段
   ↓
2. 编辑缺陷模板，添加该字段
   ↓
3. 项目设置 → 模版管理 → 缺陷模版
   ↓
4. 找到模板，点击"复制"按钮
   ↓
5. 选择复制模式：
   • 追加模式：在系统字段中追加选项，自定义字段创建副本
   • 覆盖模式：覆盖系统字段选项和类型
   ↓
6. 勾选目标项目
   ↓
7. 确定 ✅
```

**复制模式说明**：

| 场景 | 追加模式 | 覆盖模式 |
|------|---------|---------|
| 目标项目没有同名字段 | ✅ 新增字段 | ✅ 新增字段 |
| 目标项目有同名系统字段 | ✅ 追加选项 | ⚠️ 覆盖选项 |
| 目标项目有同名自定义字段 | ✅ 创建"字段名_copy" | ⚠️ 更新字段 |

**示例**：将"需求号"字段复制到10个项目

```
源项目A：
└── 缺陷模板
    └── 需求号（input）

执行复制后：
目标项目B、C、D...J：
└── 缺陷模板_copy_xxx
    └── 需求号（input） ← 独立副本
```

**优点**：
- ✅ 无需数据库操作
- ✅ UI界面操作
- ✅ 可以批量选择多个项目
- ✅ 支持追加/覆盖两种模式

**缺点**：
- ❌ **复制后需要逐个项目手动绑定模板**（⚠️ 关键限制！）
- ❌ 工作量与项目数成正比（10个项目 = 10次绑定操作）
- ❌ 每个项目都是独立副本，后续修改不同步
- ❌ 新建项目不会自动包含
- ❌ 容易遗漏某些项目的绑定
- ❌ 字段可能被项目管理员删除

**工作量估算**：

| 项目数 | 复制操作 | 绑定操作 | 总时间 |
|--------|---------|---------|--------|
| 5个 | 1次 | 5次 | ≈ 5分钟 |
| 10个 | 1次 | 10次 | ≈ 10分钟 |
| 50个 | 1次 | 50次 | ≈ 30-50分钟 ⚠️ |

**适用场景**：
- ✅ 项目数量极少（<5个）
- ✅ 各项目可能有个性化需求
- ✅ 测试/临时使用
- ❌ 不适合项目数量多的情况
- ❌ 不适合全局统一规范字段

**详细说明**：参见 [模板复制功能详解.md](./模板复制功能详解.md)

### 8.2 回答用户问题

#### ❓ 问题：`global=1, system=0` 的"全局自定义字段"能达到同样效果吗？

**答**：✅ **理论上可以**，但有限制！

**限制**：
1. ❌ **UI无法创建**：代码强制设置 `global=false`
2. ✅ **只能通过数据库脚本**：创建 global=1 字段
3. ✅ **必须关联模板**：否则不会显示
4. ⚠️ **可能被删除**：因为 system=0，有权限的用户可以删除

**对比**：

```sql
-- 方式1：全局系统字段（推荐）
INSERT INTO custom_field (id, name, scene, type, system, global, ...)
VALUES (..., '需求号', 'ISSUE', 'input', 1, 1, ...);
-- ✅ 不可删除
-- ✅ 不可修改名称
-- ✅ 系统级保护

-- 方式2：全局自定义字段（不推荐用于核心字段）
INSERT INTO custom_field (id, name, scene, type, system, global, ...)
VALUES (..., '需求号', 'ISSUE', 'input', 0, 1, ...);
-- ⚠️ 可能被误删
-- ⚠️ 可能被改名
-- ⚠️ 缺少保护
```

**结论**：
- 对于**核心业务字段**（如"需求号"），应使用 `system=1, global=1`
- 对于**可选通用字段**，可以使用 `system=0, global=1`

## 十、删除系统字段完整流程

### 10.1 删除顺序说明 ⚠️

删除系统字段时，必须按照以下顺序操作，避免外键约束和数据残留问题：

```
步骤1：删除字段值数据
  ├── custom_field_issues（缺陷字段值）
  ├── custom_field_test_case（用例字段值）
  └── custom_field_api（API字段值）

步骤2：删除模板关联
  ├── 删除所有项目模板关联（custom_field_template，global=0）
  └── 删除全局模板关联（custom_field_template，global=1）

步骤3：删除字段本身
  └── 删除 custom_field 表中的字段记录
```

**为什么必须按这个顺序？**

1. **字段值表没有外键约束**
   - `custom_field_issues`、`custom_field_test_case`、`custom_field_api` 表没有设置外键
   - 删除字段时不会自动级联删除值数据
   - 必须先手动删除，避免产生"孤儿数据"

2. **模板关联表有逻辑依赖**
   - `custom_field_template` 表通过 `field_id` 关联到 `custom_field`
   - 虽然可能没有外键约束，但逻辑上必须先删除关联
   - 避免删除字段后关联记录变成无效引用

3. **最后删除字段本身**
   - 确保所有依赖都已清理
   - 避免删除过程中的数据不一致

### 10.4 通过Java代码删除（系统字段限制）

**⚠️ 注意**：系统字段（`system=1`）在代码层面**不允许删除**！

```java
// CustomFieldService.java - delete() 方法
public void delete(String id) {
    // 检查是否是系统字段
    CustomField field = customFieldMapper.selectByPrimaryKey(id);
    if (field != null && field.getSystem() != null && field.getSystem()) {
        MSException.throwException("系统字段不能删除");  // ⚠️ 直接抛出异常
    }
    
    // 删除字段本身
    customFieldMapper.deleteByPrimaryKey(id);
    
    // 删除模板关联（自动处理）
    customFieldTemplateService.deleteByFieldId(id);
    
    // ⚠️ 注意：代码中没有删除字段值数据的逻辑！
    // 需要手动执行SQL删除 custom_field_issues/test_case/api 中的数据
}
```

**结论**：
- ✅ **项目级字段**（system=0）：可以通过UI或代码删除
- ❌ **系统字段**（system=1）：**只能通过数据库脚本删除**

### 10.7 常见问题 FAQ

#### Q1: 为什么删除字段后，UI中还能看到？

**A:** 可能的原因：
1. ❌ **浏览器缓存未清除**：清除缓存后刷新
2. ❌ **前端未重新编译**：重新编译前端代码
3. ❌ **服务未重启**：重启后端服务
4. ❌ **删除不完整**：检查是否还有关联记录

#### Q2: 删除字段后，历史数据会丢失吗？

**A:** ⚠️ **会丢失！**
- 删除 `custom_field_issues/test_case/api` 中的数据后，历史字段值无法恢复
- **建议**：删除前先导出数据备份

#### Q3: 能否只删除字段，保留历史数据？

**A:** ❌ **不建议！**
- 保留历史数据会导致：
  - 数据库中存在"孤儿数据"（字段已删除，但值还在）
  - 查询时可能出现异常
  - 数据不一致

**如果必须保留**：
- 可以考虑将字段改为 `system=0`（取消系统字段标记）
- 或者将字段标记为"已废弃"，而不是删除

#### Q4: 删除系统字段需要什么权限？

**A:** 
- **数据库脚本删除**：需要数据库管理员权限
- **代码删除**：系统字段无法通过代码删除（代码层面已限制）

#### Q5: 删除字段会影响其他项目吗？

**A:** ✅ **会！**
- 全局系统字段（global=1）影响所有项目
- 删除后，所有项目都无法再使用该字段
- 删除前务必通知所有项目团队

### 🎯 核心要点

1. **设置全局字段三要素**：
   ```sql
   system = 1        -- 系统字段（推荐，防止误删）
   global = 1        -- 全局字段（必须）
   project_id = NULL -- 全局字段必须为NULL
   ```

2. **关联三步走**：
   - ✅ 关联全局模板（4个平台）
   - ✅ 关联所有项目模板（批量）⭐ **最容易遗漏！**
   - ✅ 前端国际化配置

3. **验证三检查**：
   - ✅ 数据库记录是否完整
   - ✅ UI是否正常显示
   - ✅ 跨项目是否生效

4. **常见陷阱**：
   - ❌ 只关联了全局模板，忘记关联项目模板
   - ❌ 数据库正确但前端未重新编译
   - ❌ 前端已编译但浏览器缓存未清除
   - ❌ UI创建了重复的项目级字段

### 📦 快速参考

| 需求 | system | global | project_id | 关联范围 | 新项目 |
|------|--------|--------|------------|----------|--------|
| **全局系统字段** | 1 | 1 | NULL | 全局+所有项目模板 | ✅ 自动包含 |
| **全局自定义字段** | 0 | 1 | NULL | 全局+所有项目模板 | ✅ 自动包含 |
| **项目字段** | 0 | 0 | 具体项目ID | 单个项目模板 | ❌ 不包含 |

### ⚠️ 血泪教训

1. **global=1 不等于自动显示**
   - 必须在 `custom_field_template` 中创建关联
   - 必须关联到**所有项目级模板**，不能只关联全局模板

2. **项目级模板是副本，不是引用**
   - 新增全局字段后，老项目看不到
   - 必须手动为老项目模板添加关联

3. **前端三连：编译→重启→清缓存**
   - 修改国际化后，必须重新编译前端
   - 必须重启服务
   - 必须清除浏览器缓存

4. **防止重复字段**
   - 使用固定ID（如 `req-num-field-id-xxx`）
   - 使用 `WHERE NOT EXISTS` 防止重复插入
   - 定期检查是否有重复字段
