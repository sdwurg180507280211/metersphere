# 用例评审：只读页附件置灰与“评论承载证据”的设计原理

## 1. 背景

在 **测试跟踪 -> 用例评审 -> 功能用例** 场景中，存在两类“信息载体”：

- **用例附件（资产层）**：挂在用例本体上，属于用例长期资产。
- **评审评论（过程层）**：挂在评审过程上，属于评审执行过程产生的沟通/证据。

现象是：

- 在“只读编辑/查看”页面里，用例附件上传入口会置灰（不可上传/关联/删除）。
- 评审过程中产生的证据，主要通过“评审评论”沉淀（评论支持 markdown 图片上传并归档）。

## 2. 为什么附件会置灰（设计理念）

### 2.1 只读语义一致：只读=不改变用例资产

- 只读页的核心语义是“只看，不改”。
- 用例附件在数据模型上属于用例内容的一部分，上传/删除/关联都会改变用例资产集合。

因此在只读页：

- **允许查看**（预览/下载）：不改变数据。
- **禁止写入**（上传/关联/删除/取消关联）：避免用例资产在评审态发生变更。

如果只读页允许上传附件，会导致语义冲突：

- 文本/步骤不能改，但附件能改；用户会感知系统行为不一致。

### 2.2 证据归属应属于“评审事件”，而非“用例本体”

评审过程中产生的截图、讨论结论、执行现象等，本质是：

- **过程数据**（属于某次评审、某条评论、某次动作的上下文）
- **沟通证据**（需要可追溯：谁、何时、针对哪个评审点提供了什么材料）

如果把过程证据直接沉淀到“用例附件（资产层）”，会带来：

- **基线污染**：评审尚未结束，用例资产已被修改。
- **来源不清**：难区分“用例固有附件”与“评审临时材料”。
- **审计粒度差**：附件变化难天然落到“某条评审记录/某次评审动作”上。

因此采用分层设计：

- 用例附件：沉淀长期资产。
- 评审评论：沉淀过程证据与讨论记录。

### 2.3 权限边界更清晰

评审参与者未必拥有“用例编辑权限”。

- 如果只读页放开用例附件上传，UI 会暗示用户可变更用例资产。
- 实际权限不足时后端会拦截，造成体验割裂。

将证据写入能力绑定在“评审评论权限”上，更符合角色职责边界。

## 3. 实现原理（代码与数据模型）

> 说明：以下为“实现机制”总结，便于二次开发时复用能力、避免改动核心链路。

### 3.1 附件置灰的实现原理（前端）

附件上传入口位于 `TestCaseEditOtherInfo.vue` 的 `attachment` tab。

- 上传按钮禁用条件为：`readOnly || isCopy`
- 关联文件、删除、取消关联等写操作，也会受到 `readOnly` 影响

结论：附件置灰是明确的 `readOnly` 设计，而非功能缺失。

### 3.2 评论承载证据的实现原理（前端）

评审页使用 `CommentHistory` 展示评审评论：

- `default-type="REVIEW"`
- 通过 `caseId + reviewId` 过滤出当前评审过程的评论

评论内容使用 markdown 编辑器，并支持“图片证据”上传：

- **临时上传**：编辑器先把图片上传到临时目录（避免评论未保存却产生正式资源）。
- **保存归档**：评论保存成功后解析 markdown 内容中的图片引用，将图片从临时目录移入正式目录，并建立“评论-图片”的关联。

关键点：

- 保存归档时的 `resourceId` 使用 **commentId**（评论 ID）。
- 因此图片证据天然归属到“某条评论（某次评审动作）”。

### 3.3 评论承载证据的实现原理（后端 + 数据库）

#### 3.3.1 评论表：`test_case_comment`

字段能力：

- `type`：区分 `CASE / PLAN / REVIEW`
- `belong_id`：评审场景下存放 `reviewId`（用于隔离不同评审过程）

因此可以实现：

- 同一个用例（caseId）在不同评审（reviewId）下的评论独立展示与追溯。

#### 3.3.2 markdown 图片的存储与关联：`file_metadata` + `file_association`

markdown 图片会以文件元数据方式存储，并建立关联关系：

- `file_metadata`：文件元信息
- `file_association`：资源与文件的关联
  - `type = "MD_IMAGE"`
  - `source_id = resourceId`（在评论场景下就是 **commentId**）

带来的能力：

- **证据可追溯**：某张图属于哪条评论明确可查。
- **删除闭环**：删除评论时可按 commentId 清理关联并删除无引用文件。
- **避免脏数据**：支持临时目录上传，评论保存后再归档。

#### 3.3.3 两段式上传（临时目录 -> 归档）

后端提供两段式能力：

- 临时上传：将图片上传到临时目录
- 保存归档：评论保存后，将图片移入正式目录并建立关联

价值在于：

- 以“评论保存成功”为事务边界，避免未提交的证据污染正式资源。

## 4. 边界结论（给产品/研发统一口径）

### 4.1 用例附件（资产层）

- 定位：用例的长期资产
- 写入：仅在用例编辑态、具备用例编辑权限时发生
- 只读页：只允许查看，不允许变更（因此置灰）

### 4.2 评审评论（过程证据层）

- 定位：评审执行过程产生的沟通证据
- 归属：`caseId + reviewId`，证据（图片）绑定到 `commentId`
- 能力：上传（临时）、保存归档、展示、删除清理闭环

## 5. 二次开发建议（如需支持非图片文件证据）

- 优先在“评审评论”维度扩展：为评论增加“文件证据列表”（zip/log/pdf等）。
- 不建议在评审只读页直接放开“用例附件上传”，避免破坏只读语义与资产边界。

按钮置灰不是为了难用，而是为了让用户“知道功能存在但当前只读不可写”，并引导把评审过程证据沉淀到评论里，避免污染用例资产，同时减少用户找入口与误解成本。
