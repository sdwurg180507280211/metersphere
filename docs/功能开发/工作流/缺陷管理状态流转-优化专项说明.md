# 缺陷管理状态流转：优化专项说明

## 一、目标与范围

本文件用于集中记录“缺陷管理状态流转”功能在近期实现过程中做过的所有关键优化，重点说明：

- **为什么要优化**（问题/风险/动机）
- **怎么优化的**（落地方案与改动点）
- **不优化会怎样**（后果/潜在缺陷）
- **如何验证**（可操作的验证步骤）

> 说明：本文面向二次开发与升级搬运场景，强调“原因 + 可回滚 + 可对照”。

## 二、优化清单（按影响面从高到低）

### 2.1 调整：状态流转入口收敛至「缺陷管理页」（关联缺陷列表只读展示）

- 我在做：把“状态流转（状态修改）”的入口**只保留在缺陷管理页**；在“功能用例编辑页/测试计划页”的关联缺陷列表中，状态字段仅作为普通字段展示（只读）。
- 目的是：避免多入口造成权限/规则不一致、以及维护成本过高；让状态变更入口可控、可审计。
- 如果不这样做，就无法实现：工作流的核心价值（规则与权限），并会留下**严重的权限绕过风险**。

#### 2.1.1 优化原因（根因）

- **根因 1：前端下拉使用模板全量选项**
  - 旧实现直接渲染模板状态字段的全量 `options`。
  - 即使后端已有规则，用户也能在 UI 上选择任意状态，造成“可以随便改”的错觉，并可能通过旧接口绕过。
- **根因 2：不同页面实现不一致**
  - 缺陷列表页、用例编辑页、测试计划页各自实现不同，容易出现：
    - 状态下拉可选项不一致
    - 一处修复、另一处遗漏
    - 状态回显方式不一致（导致个别缺陷不显示）

#### 2.1.2 落地方案（怎么改）

- **入口收敛**
  - 缺陷管理页（IssueList）保留状态下拉并调用后端获取可流转状态。
  - 功能用例编辑页/测试计划页的关联缺陷列表：状态字段仅文本展示，不提供任何修改入口。
- **可流转状态获取 + 懒加载（仅缺陷管理页）**
  - 展开状态下拉时请求：`GET /issues/transition/available/{issueId}`（避免列表渲染期触发大量请求）。

#### 2.1.3 已落地入口与文件

- **功能用例编辑页-关联缺陷列表**
  - 文件：`test-track/frontend/src/business/case/components/case/CaseIssueRelate.vue`
  - 改动：状态字段仅文本展示（只读），不提供状态下拉/状态修改。
- **测试计划-功能用例-关联缺陷列表**
  - 文件：`test-track/frontend/src/business/case/components/TestCaseIssueRelate.vue`
  - 改动：状态字段仅文本展示（只读），不提供状态下拉/状态修改。
- **缺陷管理列表页**
  - 文件：`test-track/frontend/src/business/issue/IssueList.vue`
  - 改动：状态下拉仅展示可流转选项，并进一步优化为懒加载（见 2.3）。

#### 2.1.4 如何验证

- **验证 1：入口收敛是否生效**
  - 功能用例编辑页/测试计划页：状态字段应为纯文本展示，鼠标点击不应出现下拉。
  - 缺陷管理页：仍可展开状态下拉并发起流转。
- **验证 2：缺陷管理页接口是否正确**
  - 打开浏览器 Network。
  - 展开缺陷管理页状态下拉时应请求：`GET /issues/transition/available/{issueId}`。

### 2.2 修复：个别缺陷在“关联缺陷列表”中状态不回显（例如：缺陷 100021）

- 我在做：对齐缺陷列表页的状态回显方式，优先使用后端返回的展示值。
- 目的是：解决同一条缺陷在“缺陷列表页能显示状态，但关联缺陷列表页显示为空”的不一致。
- 如果不这样做，就无法实现：历史数据/模板选项不一致时的稳定展示，会出现“状态空白”的体验问题。

#### 2.2.1 优化原因（根因）

- 关联缺陷列表的状态展示曾依赖 `getCustomFieldValue(row, field)`：
  - `select/radio` 类型会在 `field.options` 中按 `value` 匹配展示文案。
  - 当 `row.fields` 的状态值与模板 `options` 不完全一致时，返回空字符串导致页面空白。
- 缺陷列表页更稳：优先展示 `row.displayValueMap['状态']`。

#### 2.2.2 落地方案（怎么改）

- 新增统一展示方法 `getIssueStatusDisplay(row, field)`：
  - 第一优先：`row.displayValueMap['状态']`
  - 第二优先：`getCustomFieldValue(row, field)`
  - 兜底：`issueStatusMap[row.status]` / `row.status` / `--`

#### 2.2.3 涉及文件

- `test-track/frontend/src/business/case/components/case/CaseIssueRelate.vue`
- `test-track/frontend/src/business/case/components/TestCaseIssueRelate.vue`

### 2.3 优化：缺陷列表页状态下拉改为懒加载（仅展开时请求）

- 我在做：把缺陷列表页的“可流转状态”请求从渲染期触发改为展开下拉时触发。
- 目的是：避免列表页一次加载 N 条缺陷就触发 N 次 `available` 请求，降低首屏压力。
- 如果不这样做，就无法实现：状态下拉真正懒加载，网络请求会在用户未操作时提前打爆。

#### 2.3.1 落地方案（怎么改）

- 在状态下拉上监听 `visible-change`：
  - 展开时调用 `loadAvailableTransitionsForIssue(issue)` 发起请求
  - `getAvailableTransitionsForIssue(issue)` 仅返回缓存，不发请求
- 下拉菜单增加加载态：
  - 请求中展示“加载中...”】【禁用项】
  - 已加载且无结果时展示“无可流转状态”

#### 2.3.2 涉及文件

- `test-track/frontend/src/business/issue/IssueList.vue`

### 2.4 风险与待办：后端统一强校验仍需补齐（最终兜底）

- 我在做：明确当前已落地的是“前端入口一致性”，但最终安全兜底必须在后端完成。
- 目的是：防止任何人绕过前端直接调用旧接口修改状态。
- 如果不这样做，就无法实现：真正意义上的“任何入口都必须通过工作流规则”。

#### 2.4.1 当前现状

- 前端页面状态变更目前仍会调用旧接口：`POST /issues/change/status`（封装：`issueStatusChange(param)`）。
- 后端若未在该接口内强制走工作流校验，则存在“接口层绕过”的空间。

#### 2.4.2 建议后端改造点（待落地）

- 在 `IssuesService.changeStatus` 内部统一调用 `IssueStatusTransitionService.transitionStatus(...)` 做：
  - 责任人权限检查
  - 状态流转规则检查
  - 记录历史
- 检查其他可能入口（如 close/编辑/批量操作）是否也会绕开校验。

## 三、回归验证建议（最小集）

- **入口一致性回归**
  - 缺陷管理列表（IssueList）
  - 功能用例编辑页-关联缺陷（CaseIssueRelate）
  - 测试计划-功能用例-关联缺陷（TestCaseIssueRelate）
  - 缺陷管理页：有状态下拉入口。
  - 关联缺陷列表页：状态仅文本展示（无下拉入口）。
- **性能回归（列表页）**
  - 打开缺陷列表页首屏：不应立即出现大量 `GET /issues/transition/available/*` 请求。
  - 仅在展开某条缺陷的状态下拉时，才出现对应的 `available` 请求。
- **状态回显回归**
  - 选择一条历史/特殊数据缺陷（例如编号 100021）
  - 在三个入口中均应能看到状态，不应出现空白。

## 四、可回滚点（升级搬运友好）

- **前端回滚（不推荐）**
  - 将“状态下拉”恢复为模板全量选项（会重新暴露绕过风险）。
  - 将缺陷列表页 `visible-change` 懒加载逻辑移除（会恢复列表渲染期大量请求）。
- **后端回滚（暂无）**
  - 本次专项未落地后端强校验改造，因此不存在后端回滚点。
