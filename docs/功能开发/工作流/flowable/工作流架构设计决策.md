# MeterSphere + Flowable 架构设计决策文档

## 📋 设计理念

> **核心原则**: 在不破坏现有架构的前提下，优雅地引入工作流能力

---

## 一、核心设计决策

### 决策 1: 为什么独立部署工作流服务？

#### 对比方案

**方案 A: 在现有服务中嵌入 Flowable（不推荐）**
```
test-track-service
├── 测试计划管理
├── 测试用例管理
└── Flowable 引擎 ← 嵌入

问题：
❌ 耦合度高：业务逻辑和流程引擎混在一起
❌ 依赖冲突：Flowable 依赖可能与现有依赖冲突
❌ 难以复用：其他服务想用工作流需要重复集成
❌ 维护困难：Flowable 升级影响业务服务
❌ 资源竞争：流程引擎和业务逻辑争抢资源
```

**方案 B: 独立工作流服务（推荐）**
```
workflow-service (8008)
├── Flowable 引擎
├── 流程管理 API
└── 任务管理 API
     ↓ REST API / Kafka
test-track-service (8003)
├── 测试计划管理
└── 调用 workflow-service

优势：
✅ 单一职责：workflow-service 只管流程
✅ 松耦合：通过 API/消息队列通信
✅ 可复用：所有服务都可以使用
✅ 独立演进：可以独立升级 Flowable
✅ 资源隔离：独立的 JVM，互不影响
```

#### 实际场景对比

```
场景：测试计划审批 + 缺陷处理审批 + 接口变更审批

方案 A（嵌入式）：
test-track-service     → 集成 Flowable
api-test-service       → 集成 Flowable
project-mgmt-service   → 集成 Flowable
结果：重复集成 3 次，代码重复，维护成本 ×3

方案 B（独立服务）：
workflow-service ← 只需要集成 1 次
    ↓
test-track, api-test, project-mgmt 都通过 API 调用
结果：代码复用，维护成本 ÷3
```

### 决策 2: 为什么使用独立数据库？

#### 数据库架构对比

**方案 A: 共享数据库（不推荐）**
```
metersphere_db
├── test_plan              # 业务表
├── test_case              # 业务表
├── act_re_deployment      # Flowable 表
├── act_ru_execution       # Flowable 表
└── act_hi_procinst        # Flowable 表
    ... (共 30+ 张 Flowable 表)

问题：
❌ 表混乱：业务表和 Flowable 表混在一起
❌ 权限难控：无法对 Flowable 表单独控制权限
❌ 备份困难：无法单独备份流程数据
❌ 升级风险：Flowable 升级可能影响业务表
❌ 性能问题：Flowable 查询影响业务查询
```

**方案 B: 独立数据库（推荐）**
```
metersphere_db              # 业务数据库
├── test_plan
├── test_case
└── ...

metersphere_workflow        # 工作流数据库
├── act_*                   # Flowable 30+ 张表
└── ms_workflow_*           # 扩展表

优势：
✅ 清晰边界：业务和流程完全分离
✅ 独立备份：可以单独备份流程数据
✅ 独立优化：可以针对 Flowable 表优化
✅ 安全隔离：流程数据库单独控制权限
✅ 独立扩展：流程数据库可以单独扩容
```

#### 数据关联设计

```
如何关联业务数据和流程数据？
──────────────────────────────

方案：通过业务 ID 关联

metersphere_db.test_plan
├── id: "test-plan-001"
├── name: "V2.0 接口测试"
└── status: "PENDING_APPROVAL"

metersphere_workflow.ms_workflow_instance
├── id: "wf-001"
├── process_instance_id: "flowable-proc-001"  # Flowable ID
├── business_id: "test-plan-001"              # 关联业务ID
├── business_type: "TEST_PLAN"                # 业务类型
└── status: "RUNNING"

查询流程：
1. 前端查询测试计划详情
2. 根据 test_plan.id 查询 ms_workflow_instance
3. 根据 process_instance_id 查询 Flowable 流程状态
4. 合并返回

好处：
✅ 两个数据库完全独立
✅ 通过 business_id 关联
✅ 即使工作流服务挂了，业务也能查询基本信息
```

### 决策 3: 为什么选择 REST API + Kafka 混合通信？

#### 通信方式对比

**纯 REST API 方式**
```
优势：
✅ 简单直接
✅ 易于调试
✅ 同步返回结果

劣势：
❌ 同步阻塞
❌ 强耦合
❌ 无法解耦流量峰值
```

**纯 Kafka 消息方式**
```
优势：
✅ 完全解耦
✅ 异步处理
✅ 削峰填谷

劣势：
❌ 延迟高
❌ 调试困难
❌ 需要处理消息顺序
```

**混合方式（推荐）**
```
根据场景选择：

1. 查询操作 → REST API
场景：查询待办任务、查询流程状态
理由：需要立即返回结果

2. 状态变更 → Kafka 消息
场景：流程完成、任务分配
理由：不需要立即返回，允许异步处理

3. 启动流程 → REST API
场景：提交审批
理由：需要知道流程是否启动成功

4. 流程通知 → Kafka 消息
场景：任务到期提醒、流程完成通知
理由：异步通知，不阻塞主流程
```

#### 实际示例

```
场景 1: 用户提交测试计划审批
───────────────────────────────

前端 → test-track-service → workflow-service
        (REST API)             (启动流程)
                                   ↓
                              返回流程实例ID
                                   ↓
                              发送 Kafka 消息
                                   ↓
                              通知相关人员

时序：
1. [同步] 调用 workflow-service 启动流程
2. [同步] 返回流程实例ID给前端（用户立即知道提交成功）
3. [异步] 发送 Kafka 消息通知审批人（不阻塞）
4. [异步] 工作台服务接收消息，更新待办列表

场景 2: 审批人批准任务
─────────────────────

审批人操作 → workflow-service.completeTask()
                     ↓ (同步返回)
                 任务完成成功
                     ↓ (异步消息)
            Kafka: task.completed
                     ↓
         test-track-service 接收消息
                     ↓
            更新测试计划状态为"已批准"
                     ↓
         workstation-service 接收消息
                     ↓
            移除该待办任务
```

## 二、架构模式分析

### 模式 1: 事件驱动架构

**为什么使用事件驱动？**

```
传统方式（同步调用链）
──────────────────────

流程完成
  ↓ (同步)
更新业务状态
  ↓ (同步)
发送通知
  ↓ (同步)
更新统计
  ↓ (同步)
记录日志

问题：
❌ 响应时间长
❌ 某个环节失败，整个链路失败
❌ 扩展困难


事件驱动方式（异步解耦）
───────────────────────

流程完成
  ↓
发布事件: ProcessCompletedEvent
  ↓
  ├─→ [订阅者 1] 更新业务状态
  ├─→ [订阅者 2] 发送通知
  ├─→ [订阅者 3] 更新统计
  └─→ [订阅者 4] 记录日志

优势：
✅ 快速响应（只需要发布事件）
✅ 解耦（订阅者独立）
✅ 可扩展（新增订阅者不影响现有功能）
✅ 容错（某个订阅者失败不影响其他）
```

**事件设计示例：**

```java
// 事件定义
public class ProcessCompletedEvent {
    private String processInstanceId;
    private String businessId;
    private String businessType;
    private String result;  // APPROVED / REJECTED
    private Map<String, Object> variables;
    private LocalDateTime completedAt;
}

// 发布事件（workflow-service）
@Component
public class ProcessEventPublisher {
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    public void publishProcessCompleted(ProcessCompletedEvent event) {
        String topic = "workflow.process.completed";
        String json = JSON.toJSONString(event);
        kafkaTemplate.send(topic, event.getBusinessId(), json);
    }
}

// 订阅事件（test-track-service）
@Component
public class ProcessEventListener {
    
    @KafkaListener(topics = "workflow.process.completed")
    public void onProcessCompleted(String message) {
        ProcessCompletedEvent event = JSON.parseObject(message, ProcessCompletedEvent.class);
        
        if ("TEST_PLAN".equals(event.getBusinessType())) {
            // 更新测试计划状态
            testPlanService.updateStatus(
                event.getBusinessId(), 
                event.getResult().equals("APPROVED") ? "APPROVED" : "REJECTED"
            );
        }
    }
}
```

### 模式 2: CQRS（命令查询职责分离）

**为什么需要 CQRS？**

```
问题场景：
用户频繁查询"我的待办任务"
  ↓
每次都要：
1. 查询 Flowable 任务表
2. 关联查询业务表
3. 过滤权限
4. 排序分页
  ↓
性能差，数据库压力大


CQRS 方案：
──────────

写操作（Command）：
流程创建/更新
  ↓
写入 Flowable 表
  ↓
发送事件
  ↓
更新读模型（Redis 缓存）

读操作（Query）：
查询待办任务
  ↓
直接从 Redis 读取
  ↓
快速返回

实现：
┌──────────────┐     事件     ┌──────────────┐
│ 写模型（源）  │ ──────────→ │ 读模型（缓存）│
│ Flowable DB  │             │    Redis     │
└──────────────┘             └──────────────┘
       ↑                            ↓
       │                            │
   写操作                        读操作
   (少)                         (多)
```

**实现示例：**

```java
// 写操作：任务创建时更新读模型
@Component
public class TaskEventListener {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    @EventListener
    public void onTaskCreated(TaskCreatedEvent event) {
        // 构建任务视图对象
        TaskView view = TaskView.builder()
            .id(event.getTaskId())
            .name(event.getTaskName())
            .assignee(event.getAssignee())
            .businessData(loadBusinessData(event.getBusinessId()))
            .createdAt(event.getCreatedAt())
            .build();
        
        // 存入 Redis
        String key = "user:tasks:" + event.getAssignee();
        redisTemplate.opsForList().rightPush(key, JSON.toJSONString(view));
        
        // 设置过期时间（7 天）
        redisTemplate.expire(key, 7, TimeUnit.DAYS);
    }
}

// 读操作：快速查询
@Service
public class TaskQueryService {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    public List<TaskView> getMyTasks(String userId) {
        String key = "user:tasks:" + userId;
        List<String> jsonList = redisTemplate.opsForList().range(key, 0, -1);
        
        if (CollectionUtils.isEmpty(jsonList)) {
            // 缓存未命中，从数据库加载
            return loadFromDatabase(userId);
        }
        
        return jsonList.stream()
            .map(json -> JSON.parseObject(json, TaskView.class))
            .collect(Collectors.toList());
    }
}
```

### 模式 3: 补偿事务（Saga）

**为什么需要补偿事务？**

```
场景：启动审批流程
──────────────────

步骤 1: 创建流程实例（Flowable）
  ↓ 成功
步骤 2: 更新业务状态（MySQL）
  ↓ 失败！
  
问题：
流程已创建，但业务状态没更新
  ↓
数据不一致


Saga 补偿方案：
────────────────

步骤 1: 创建流程实例
  ↓ 成功
步骤 2: 更新业务状态
  ↓ 失败
  ↓
触发补偿：删除流程实例
  ↓
最终一致性
```

**实现示例：**

```java
@Service
public class TestPlanApprovalSaga {
    
    @Autowired
    private RuntimeService runtimeService;
    
    @Autowired
    private TestPlanService testPlanService;
    
    @Transactional
    public void startApproval(String testPlanId) {
        ProcessInstance processInstance = null;
        
        try {
            // 步骤 1: 启动流程
            processInstance = runtimeService.startProcessInstanceByKey(
                "testPlanApproval",
                Map.of("testPlanId", testPlanId)
            );
            
            // 步骤 2: 更新业务状态
            testPlanService.updateStatus(testPlanId, "PENDING_APPROVAL");
            
            // 步骤 3: 记录关联关系
            saveWorkflowInstance(testPlanId, processInstance.getId());
            
        } catch (Exception e) {
            // 补偿：如果任何步骤失败，回滚
            if (processInstance != null) {
                compensateProcessInstance(processInstance.getId());
            }
            throw new BusinessException("启动审批失败", e);
        }
    }
    
    private void compensateProcessInstance(String processInstanceId) {
        try {
            // 删除流程实例
            runtimeService.deleteProcessInstance(
                processInstanceId, 
                "Saga compensation"
            );
        } catch (Exception e) {
            // 记录补偿失败日志，后续人工处理
            log.error("Compensation failed for process: {}", processInstanceId, e);
        }
    }
}
```

## 三、性能优化策略

### 策略 1: 数据归档

**为什么需要归档？**

```
问题：
Flowable 历史表数据量巨大
  ↓
act_hi_procinst: 100 万条
act_hi_taskinst: 500 万条
act_hi_varinst: 1000 万条
  ↓
查询变慢，影响性能


归档策略：
─────────

当前数据（热数据）
  ├── 近 3 个月的流程
  ├── 存储在主库
  └── 支持快速查询

历史数据（冷数据）
  ├── 3 个月前的流程
  ├── 归档到历史库
  └── 按需查询

好处：
✅ 主库数据量小
✅ 查询速度快
✅ 历史数据可查
```

**实现方案：**

```sql
-- 1. 创建历史库
CREATE DATABASE metersphere_workflow_archive;

-- 2. 创建相同结构的归档表
CREATE TABLE metersphere_workflow_archive.act_hi_procinst_archive 
LIKE metersphere_workflow.act_hi_procinst;

-- 3. 定期归档（每天凌晨执行）
-- 归档 3 个月前的已完成流程
INSERT INTO metersphere_workflow_archive.act_hi_procinst_archive
SELECT * FROM metersphere_workflow.act_hi_procinst
WHERE end_time_ < DATE_SUB(NOW(), INTERVAL 3 MONTH)
  AND end_time_ IS NOT NULL;

-- 4. 删除已归档数据
DELETE FROM metersphere_workflow.act_hi_procinst
WHERE end_time_ < DATE_SUB(NOW(), INTERVAL 3 MONTH)
  AND end_time_ IS NOT NULL;

-- 5. 优化主库
OPTIMIZE TABLE metersphere_workflow.act_hi_procinst;
```

### 策略 2: 查询优化

**慢查询分析：**

```sql
-- 慢查询示例：查询用户的待办任务
SELECT t.*, p.business_key_
FROM act_ru_task t
JOIN act_ru_execution e ON t.execution_id_ = e.id_
JOIN act_re_procdef p ON t.proc_def_id_ = p.id_
WHERE t.assignee_ = 'user-001'
ORDER BY t.create_time_ DESC
LIMIT 20;

问题：
❌ 多表 JOIN
❌ 没有索引
❌ 排序耗时


优化方案：
─────────

1. 添加索引
CREATE INDEX idx_assignee_createtime 
ON act_ru_task(assignee_, create_time_);

2. 使用物化视图（MySQL 8.0）
CREATE VIEW v_user_tasks AS
SELECT 
    t.id_ as task_id,
    t.name_ as task_name,
    t.assignee_,
    t.create_time_,
    p.business_key_,
    p.name_ as process_name
FROM act_ru_task t
JOIN act_ru_execution e ON t.execution_id_ = e.id_
JOIN act_re_procdef p ON t.proc_def_id_ = p.id_;

-- 查询变简单
SELECT * FROM v_user_tasks
WHERE assignee_ = 'user-001'
ORDER BY create_time_ DESC
LIMIT 20;
```

### 策略 3: 缓存策略

```
缓存层次：
─────────

L1: 本地缓存（Caffeine）
  ├── 流程定义
  ├── 用户信息
  └── 过期时间：1 小时

L2: 分布式缓存（Redis）
  ├── 待办任务列表
  ├── 流程实例状态
  └── 过期时间：10 分钟

L3: 数据库
  ├── 完整数据
  └── 最终数据源

查询流程：
L1 有？ → 返回
  ↓ 否
L2 有？ → 返回 + 更新 L1
  ↓ 否
L3 查询 → 返回 + 更新 L2 + 更新 L1
```

## 四、总结

### 核心设计原则

1. **松耦合优先**
   - 独立服务部署
   - 事件驱动通信
   - 清晰的接口定义

2. **渐进式改造**
   - 不破坏现有功能
   - 灰度发布
   - 可回滚

3. **性能优先**
   - CQRS 读写分离
   - 多级缓存
   - 数据归档

4. **可观测性**
   - 完整日志
   - 监控告警
   - 链路追踪

### 关键技术选型理由

| 选型 | 理由 |
|------|------|
| Flowable | Spring Boot 3 完美兼容 |
| 独立服务 | 松耦合，可复用 |
| 独立数据库 | 清晰边界，独立优化 |
| REST + Kafka | 同步查询，异步通知 |
| bpmn-js | 标准化，开箱即用 |
| CQRS | 读写分离，性能优化 |
| Saga | 最终一致性 |

### 实施建议

**DO（推荐）:**
✅ 先做 MVP，快速验证
✅ 分阶段实施，降低风险
✅ 充分测试，确保稳定
✅ 灰度发布，逐步推广
✅ 监控到位，及时发现问题

**DON'T（避免）:**
❌ 一次性开发所有功能
❌ 没有充分测试就上线
❌ 忽略性能优化
❌ 缺少监控告警
❌ 强行推广，不考虑用户接受度
