## 升级路线总览

```
阶段一：直接配置文件
    │  个人项目、学习阶段
    │  1-2 人、快速开发
    ↓
阶段二：配置文件 + 脚本
    │  小团队（3-5人）
    │  需要多环境但还不复杂
    ↓
阶段三：环境变量 + CI/CD
    │  中型团队（5-20人）
    │  正规开发流程、自动化部署
    ↓
阶段四：配置中心
    │  大型团队（20人+）、多项目
    │  微服务集群、动态配置
```

### 快速决策表

| 项目阶段 | 团队规模 | 环境数量 | 推荐方案 | 预计工作量 |
|---------|---------|---------|---------|-----------|
| 个人学习 | 1人 | 1个（本地） | 阶段一 | 5分钟 |
| 初创项目 | 2-3人 | 2个（本地+测试） | 阶段一 | 10分钟 |
| 小团队 | 3-5人 | 3个（本地+测试+生产） | 阶段二 | 1小时 |
| 中型团队 | 5-20人 | 3个+多个微服务 | 阶段三 | 1天 |
| 大型团队 | 20人+ | 多环境+多项目 | 阶段四 | 3-5天 |

## 阶段一：直接配置文件（入门级）

### 适用场景

- ✅ 个人学习、Demo 项目
- ✅ 1-2 人的小团队
- ✅ 本地开发为主，偶尔部署测试
- ✅ 快速迭代，不需要严格流程

### 配置方式

#### 目录结构

```
metersphere/
├── backend/
│   └── src/main/resources/
│       ├── commons.properties              # 默认配置（不改）
│       └── application.yml                  # 默认配置（不改）
└── /opt/metersphere/conf/
    └── metersphere.properties              # 👈 只改这个文件
```

#### 配置示例

**本地开发配置：**

```properties
# /opt/metersphere/conf/metersphere.properties

# ========== 本地开发数据库 ==========
spring.datasource.url=jdbc:mysql://localhost:3306/metersphere_local?autoReconnect=false&useUnicode=true&characterEncoding=UTF-8&characterSetResults=UTF-8&zeroDateTimeBehavior=convertToNull&useSSL=false
spring.datasource.username=root
spring.datasource.password=local123

# ========== Redis（本地） ==========
spring.redis.host=localhost
spring.redis.port=6379
spring.redis.password=

# ========== Kafka（本地） ==========
kafka.bootstrap-servers=localhost:9092

# ========== MinIO（本地） ==========
minio.endpoint=http://localhost:9000
minio.accessKey=minioadmin
minio.secretKey=minioadmin
```

#### 使用步骤

```bash
# 1. 修改配置文件
vim /opt/metersphere/conf/metersphere.properties

# 2. 直接启动（配置会自动加载）
cd backend
mvn spring-boot:run

# 3. 需要切换到测试环境？直接改配置文件
vim /opt/metersphere/conf/metersphere.properties
# 改成测试环境的数据库地址
spring.datasource.url=jdbc:mysql://test-server:3306/metersphere_test?...
```

#### 环境切换方法

**方法一：直接修改（推荐）**

```bash
# 切换到测试环境
vim /opt/metersphere/conf/metersphere.properties
# 手动改数据库地址、用户名、密码

# 重启应用
mvn spring-boot:run
```

**方法二：备份多个配置文件**

```bash
# 备份不同环境的配置
cp metersphere.properties metersphere.properties.local
cp metersphere.properties metersphere.properties.test

# 需要切换环境时
cp metersphere.properties.test /opt/metersphere/conf/metersphere.properties
```

### 优缺点

| 优点 | 缺点 |
|-----|-----|
| ✅ 最简单直接，5分钟搞定 | ❌ 手动操作，容易出错 |
| ✅ 不需要学习额外知识 | ❌ 没有版本控制 |
| ✅ 配置清晰可见 | ❌ 团队协作困难 |
| ✅ 适合快速开发 | ❌ 无法自动化部署 |

### 升级时机

当出现以下情况时，考虑升级到阶段二：

- 🚨 团队扩大到 3 人以上
- 🚨 经常需要在多个环境间切换
- 🚨 曾经误改生产配置导致故障
- 🚨 需要多人协作开发

#### 环境切换脚本

**scripts/switch-env.sh：**

```bash
#!/bin/bash

set -e

# 颜色定义
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
NC='\033[0m' # No Color

# 配置路径
CONFIG_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/../config" && pwd)"
TARGET_CONFIG="/opt/metersphere/conf/metersphere.properties"

# 显示使用帮助
function show_help() {
    echo "用法: ./switch-env.sh [环境名称]"
    echo ""
    echo "可用环境："
    echo "  local    - 本地开发环境"
    echo "  test     - 测试环境"
    echo "  prod     - 生产环境"
    echo ""
    echo "示例："
    echo "  ./switch-env.sh local"
    echo "  ./switch-env.sh test"
}

# 检查参数
if [ $# -eq 0 ]; then
    echo -e "${RED}错误：请指定环境名称${NC}"
    show_help
    exit 1
fi

ENV=$1

# 检查环境配置文件是否存在
CONFIG_FILE="${CONFIG_DIR}/${ENV}.properties"
if [ ! -f "$CONFIG_FILE" ]; then
    echo -e "${RED}错误：配置文件不存在: ${CONFIG_FILE}${NC}"
    exit 1
fi

# 生产环境二次确认
if [ "$ENV" = "prod" ]; then
    echo -e "${YELLOW}⚠️  警告：即将切换到生产环境！${NC}"
    read -p "请输入 'YES' 确认继续: " CONFIRM
    if [ "$CONFIRM" != "YES" ]; then
        echo -e "${RED}已取消切换${NC}"
        exit 1
    fi
fi

# 备份当前配置
if [ -f "$TARGET_CONFIG" ]; then
    BACKUP_FILE="${TARGET_CONFIG}.backup.$(date +%Y%m%d_%H%M%S)"
    cp "$TARGET_CONFIG" "$BACKUP_FILE"
    echo -e "${GREEN}✓ 已备份当前配置到: ${BACKUP_FILE}${NC}"
fi

# 创建目标目录
mkdir -p "$(dirname "$TARGET_CONFIG")"

# 复制新配置
cp "$CONFIG_FILE" "$TARGET_CONFIG"

echo -e "${GREEN}✓ 已切换到 ${ENV} 环境${NC}"
echo ""
echo "当前配置："
echo "  配置文件: ${TARGET_CONFIG}"
echo "  环境: ${ENV}"
echo ""

# 显示数据库配置（用于验证）
DB_URL=$(grep "spring.datasource.url" "$TARGET_CONFIG" | cut -d'=' -f2)
echo "  数据库: ${DB_URL}"

echo ""
echo -e "${YELLOW}提示：请重启应用使配置生效${NC}"
```

**scripts/deploy-local.sh：**

```bash
#!/bin/bash

set -e

echo "========== 本地环境部署 =========="

# 1. 切换环境
./scripts/switch-env.sh local

# 2. 启动依赖服务（Docker）
echo "启动依赖服务..."
docker-compose -f docker-compose.local.yml up -d

# 3. 等待服务启动
echo "等待服务启动..."
sleep 10

# 4. 检查数据库连接
echo "检查数据库连接..."
mysql -h localhost -u root -plocal123 -e "SELECT 1" metersphere_local

# 5. 启动应用
echo "启动应用..."
cd backend
mvn spring-boot:run
```

**scripts/deploy-test.sh：**

```bash
#!/bin/bash

set -e

echo "========== 测试环境部署 =========="

# 1. 检查当前分支
CURRENT_BRANCH=$(git branch --show-current)
if [ "$CURRENT_BRANCH" != "test" ]; then
    echo "错误：请先切换到 test 分支"
    echo "执行: git checkout test"
    exit 1
fi

# 2. 切换环境
./scripts/switch-env.sh test

# 3. 构建项目
echo "构建项目..."
cd backend
mvn clean package -DskipTests

# 4. 部署到测试服务器
echo "部署到测试服务器..."
scp target/metersphere.jar deploy@test-server:/opt/metersphere/

# 5. 远程重启服务
ssh deploy@test-server "sudo systemctl restart metersphere"

echo "✓ 部署完成"
echo "测试环境地址: http://test-server:8080"
```

#### Docker 配置

FROM openjdk:11-jre-slim

WORKDIR /app

# 暴露端口
EXPOSE 8080

# 启动命令（从环境变量读取配置）

**docker/docker-compose.yml：**

```yaml
version: '3.8'

services:
  metersphere:
    build:
      context: ..
      dockerfile: docker/Dockerfile
    ports:
      - "8080:8080"
    env_file:
      - ../.env.${ENV:-local}
    depends_on:
      - mysql
      - redis
      - kafka

  mysql:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: ${DB_PASSWORD}
      MYSQL_DATABASE: ${DB_NAME}
      MYSQL_USER: ${DB_USERNAME}
      MYSQL_PASSWORD: ${DB_PASSWORD}
    ports:
      - "${DB_PORT}:3306"
    volumes:
      - mysql-data:/var/lib/mysql

  redis:
    image: redis:7-alpine
    ports:
      - "${REDIS_PORT}:6379"
    command: redis-server --requirepass ${REDIS_PASSWORD}

volumes:
  mysql-data:

stages:
  - build
  - test

  MAVEN_OPTS: "-Dmaven.repo.local=$CI_PROJECT_DIR/.m2/repository"

  script:

    - docker build -t $DOCKER_REGISTRY/metersphere:test-$CI_COMMIT_SHORT_SHA .

  only:
    - test

# 部署到生产环境
deploy-prod:
  stage: deploy-prod
  image: bitnami/kubectl:latest
  script:
    # 1. 备份数据库
    - ./scripts/backup-database.sh prod

    - docker push $DOCKER_REGISTRY/metersphere:$CI_COMMIT_TAG

    - kubectl apply -f k8s/configmap-prod.yml
    - kubectl apply -f k8s/secret-prod.yml
    
    # 4. 蓝绿部署
    - ./scripts/blue-green-deploy.sh $CI_COMMIT_TAG
  environment:
    name: production
    url: https://metersphere.example.com
  when: manual  # 手动触发
  only:
    - master
    - tags
```

### 动态配置刷新

**在代码中使用动态配置：**

```java
@RestController
@RefreshScope  // 支持配置动态刷新
public class ConfigController {
    
    @Value("${feature.new-ui.enabled:false}")
    private boolean newUIEnabled;
    
    @Value("${performance.max-threads:10}")
    private int maxThreads;
    
    @GetMapping("/config")
    public Map<String, Object> getConfig() {
        Map<String, Object> config = new HashMap<>();
        config.put("newUIEnabled", newUIEnabled);
        config.put("maxThreads", maxThreads);
        return config;
    }
}
```

**在 Nacos 控制台修改配置后，无需重启应用即可生效！**

## 最佳实践总结

### 通用原则

1. **按需升级，不要过度设计**
   - 当前方案够用就不要升级
   - 升级是为了解决实际问题，不是为了追求技术

2. **保持配置简单**
   - 能用环境变量就不用配置中心
   - 能用文件就不用数据库

3. **安全第一**
   - 敏感信息不要提交到 Git
   - 生产配置要加密存储
   - 定期审计配置变更

4. **可回滚**
   - 每次升级前备份配置
   - 保留降级方案
   - 灰度发布，逐步迁移

5. **文档完善**
   - 记录配置说明
   - 记录操作步骤
   - 记录常见问题

### 各阶段关键点

**阶段一：**
- ✅ 定期备份配置文件
- ✅ 不要在生产环境直接改配置

**阶段二：**
- ✅ 脚本要有二次确认（生产环境）
- ✅ 配置模板要及时更新

**阶段三：**
- ✅ Secret 定期轮换
- ✅ 监控配置加载是否成功
- ✅ CI/CD 管道要有人工审批（生产）

**阶段四：**
- ✅ Nacos 要集群部署（生产）
- ✅ 配置变更要有审核流程
- ✅ 关键配置变更要通知相关人员

## 常见问题 FAQ

### Q1: 我应该选择哪个阶段？

**A:** 参考"升级决策指南"部分的决策树，主要看：
- 团队规模（最重要）
- 微服务数量
- 部署频率

**简单判断：**
- 1-2 人 → 阶段一
- 3-5 人 → 阶段二
- 5-20 人 + 需要 CI/CD → 阶段三
- 20+ 人 + 10+ 微服务 → 阶段四

### Q2: 升级会不会很麻烦？

**A:** 从低阶段升级到高阶段是**增量式**的，不是推倒重来：
- 阶段一 → 阶段二：只是把配置文件规范化，加个脚本
- 阶段二 → 阶段三：把配置改成环境变量，加 CI/CD
- 阶段三 → 阶段四：接入配置中心，原配置保留

每次升级都是在原有基础上改进，可以灰度迁移。

### Q3: 现在用阶段一，以后升级麻烦吗？

**A:** 不麻烦！只要你遵循几个原则：
1. **不要硬编码**：不要在代码里写死配置
2. **配置集中**：所有配置放在一个文件
3. **敏感信息分离**：密码等不要提交到 Git

这样以后升级很容易，只需要：
- 把配置文件拆分成多个环境
- 把固定值改成变量占位符

### Q4: 我们是小团队，需要上配置中心吗？

**A:** **不需要！** 小团队（<10人）用阶段二或阶段三完全够了。

配置中心的价值在于：
- 动态配置更新（不重启）
- 多项目配置共享
- 细粒度权限管理

如果你没有这些需求，用配置中心反而增加复杂度。

### Q5: 直接用配置文件有什么问题？

**A:** 对于小团队/个人项目，**没问题**！

但如果团队变大，会遇到：
- ❌ 每个人本地配置不一致
- ❌ 测试环境配置经常被改乱
- ❌ 生产配置可能被误改
- ❌ 配置变更无法追溯

这时候就需要升级了。

### Q6: 能跳过某个阶段吗？

**A:** 可以，但**不推荐**。

建议按顺序升级的原因：
1. **学习曲线平缓**：逐步学习新技术
2. **风险可控**：每次改动小，容易回滚
3. **团队适应**：给团队时间适应新流程

如果你团队有 DevOps 经验，可以直接跳到阶段三。

### Q7: 配置中心挂了怎么办？

**A:** 这是配置中心的**主要风险**。解决方案：

1. **集群部署**：Nacos 3 节点集群，避免单点故障
2. **本地缓存**：应用启动时缓存配置到本地
3. **降级方案**：保留配置文件作为备份

建议配置：
```yaml
spring:
  cloud:
    nacos:
      config:
        refresh-enabled: true
        # 配置中心不可用时，使用本地缓存
        fail-fast: false
```

### Q8: 生产环境配置要加密吗？

**A:** **必须加密！** 特别是数据库密码、API 密钥等。

推荐方案：
- 阶段一/二：使用 Jasypt 加密配置文件
- 阶段三：使用 K8s Secret（Base64，配合 RBAC）
- 阶段四：使用 Nacos 加密配置 + Vault

### Q9: 多个微服务怎么管理配置？

**A:** 根据阶段选择方案：

- **阶段二**：每个服务一个配置目录
  ```
  config/
  ├── service-a/
  │   ├── local.properties
  │   └── test.properties
  ├── service-b/
  │   ├── local.properties
  │   └── test.properties
  ```

- **阶段三**：K8s ConfigMap/Secret 按服务分离
  ```
  k8s/
  ├── service-a-configmap.yml
  ├── service-a-secret.yml
  ├── service-b-configmap.yml
  └── service-b-secret.yml
  ```

- **阶段四**：Nacos 配置分组
  ```
  namespace: test
    ├── group: service-a
    │   └── service-a.yml
    └── group: service-b
        └── service-b.yml
  ```

### Q10: 我要开始一个新项目，应该用哪个方案？

**A:** 建议**从阶段二开始**：

理由：
1. ✅ 初期成本低（1小时搞定）
2. ✅ 为未来扩展留了空间
3. ✅ 团队配置统一
4. ✅ 环境切换方便

**不建议**从阶段一开始，因为：
- 即使是新项目，也会很快需要测试环境
- 从阶段一升级到阶段二，需要重新整理配置

**也不建议**直接上阶段四，因为：
- 过度设计，增加不必要的复杂度
- 需要维护配置中心基础设施

## 总结

### 升级路线一览

```
阶段一：直接配置文件
  ↓  团队 > 3人 OR 需要多环境
  
阶段二：配置文件 + 脚本
  ↓  团队 > 5人 OR 需要 CI/CD
  
阶段三：环境变量 + CI/CD
  ↓  团队 > 20人 OR 微服务 > 10个
  
阶段四：配置中心
```

### 核心理念

> **不要过度设计，按需升级，保持简单。**

- 小项目用简单方案，大项目用复杂方案
- 升级是为了解决问题，不是为了追求技术
- 每个阶段都有价值，选择最适合的

### 最后建议

1. **从简单开始**：个人项目/学习阶段，直接改配置文件
2. **逐步演进**：团队扩大、流程规范，逐步升级
3. **不要跳跃**：除非团队有经验，否则按顺序升级
4. **保持灵活**：可以在不同环境用不同方案

**记住：最好的方案是最适合你的方案，不是最复杂的方案！**
