# Spring Boot 配置解析机制说明

## 配置加载顺序

## 三类配置文件优先级（application / commons / metersphere）

### 三者各自定位

1. `application.properties`（服务内置、随代码发布）
   - 位置：各微服务 `src/main/resources/application.properties`
   - 作用：承载**服务级**默认配置（如 `spring.application.name`、`server.port`、日志路径、是否开启某些功能等）
   - 特点：通常是“代码仓库内的默认值”，适合做**可运行的缺省配置**

2. `commons.properties`（通用默认值、随 SDK 发布）
   - 位置：`framework/sdk-parent/sdk/src/main/resources/commons.properties`
   - 作用：承载**全服务通用**的默认配置（连接池、MyBatis、Flyway 默认开关、Eureka 默认参数等）
   - 特点：是“全局兜底默认值”，尽量保持稳定

3. `metersphere.properties`（外部化配置、随环境变化）
   - 位置：`/opt/metersphere/conf/metersphere.properties`
   - 作用：承载**环境级**配置（数据库地址、账号密码、注册中心地址、对象存储等）
   - 特点：不进代码仓库，适合 Dev/Test/Prod 环境差异化覆盖

### 优先级结论（从高到低）

1. `application.properties`（Spring Boot 默认加载；同目录还包含 profile 文件，如 `application-dev.properties`）
2. `@PropertySource` 显式加载的配置（MeterSphere 里主要是 `commons.properties` 与外部 `metersphere.properties`）

> 注意：`@PropertySource` 添加到 Spring `Environment` 的 property source **优先级通常低于** Spring Boot 默认加载的 `application.properties`。
> 因此如果同一个 key 同时出现在 `application.properties` 与 `metersphere.properties` 中，通常会以 `application.properties` 为准。

### MeterSphere 项目内的实际覆盖关系（commons vs metersphere）

MeterSphere 各微服务启动类中一般类似如下配置：

```java
@PropertySource(value = {
    "classpath:commons.properties",
    "file:/opt/metersphere/conf/metersphere.properties",
}, encoding = "UTF-8", ignoreResourceNotFound = true)
```

因此在 **`@PropertySource` 这组配置内部**，优先级是：

1. `metersphere.properties`（后加载，覆盖前者）
2. `commons.properties`（先加载，作为通用兜底）

> 说明：MeterSphere 各微服务在启动类中使用 `@PropertySource` 显式加载 `commons.properties` 与外部 `metersphere.properties`。
> 对同一个 key，后加载的配置会覆盖先加载的配置（以项目实际加载顺序为准）。

### 三者的替代（覆盖）与占位符解析逻辑

1. **覆盖（替代）规则**
   - 同一个 key 在多个来源同时存在时：**优先级高的来源会覆盖优先级低的来源**。
   - 在 MeterSphere 的 `@PropertySource` 组内：`metersphere.properties` 覆盖 `commons.properties`。

2. **占位符（`${...}`）解析规则**
   - `${x}` 的取值会按 Spring `Environment` 的 property sources 顺序查找（优先级从高到低）。
   - 典型查找顺序（高到低）可理解为：
     - 命令行参数 / 启动时注入的高优先级来源
     - JVM 系统属性（`-Dxxx=yyy` / `System.getProperties()`）
     - 系统环境变量（`System.getenv()`）
     - Spring Boot 默认加载的配置文件（`application.properties` 及 profile 变体）
     - `@PropertySource` 加载的配置（MeterSphere 里的 `metersphere.properties`、`commons.properties`）
     - 默认值（`${x:default}`）

3. **工程实践建议（避免“外部配置不生效”）**
   - 如果希望 `/opt/metersphere/conf/metersphere.properties` 能覆盖某个配置：
     - 尽量不要在 `application.properties` 里写死同名 key 的“最终值”；
     - 更推荐在 `application.properties` 里使用占位符（例如 `spring.datasource.url=${DB_URL:...}`），把环境差异交给 `metersphere.properties` 或环境变量注入。

根据 `@PropertySource` 注解：

```java
@PropertySource(value = {
    "classpath:commons.properties",                    // 1. 先加载
    "file:/opt/metersphere/conf/metersphere.properties" // 2. 后加载（会覆盖前面的）
}, encoding = "UTF-8", ignoreResourceNotFound = true)
```

**加载顺序**：
1. 先加载 `commons.properties`（类路径）
2. 再加载 `metersphere.properties`（外部文件）
3. **后加载的配置会覆盖先加载的配置**（如果 key 相同）

## 配置覆盖机制（重要！）

**关键点**：如果 `metersphere.properties` 中有完整的配置项（如 `spring.datasource.url`），会**直接覆盖** `commons.properties` 中的配置，**不会解析变量**。

### 覆盖规则

```
1. 先加载 commons.properties
   └── spring.datasource.url=jdbc:mysql://${DB_HOST:localhost}:${DB_PORT:3306}/${DB_NAME:metersphere}?...

2. 再加载 metersphere.properties
   └── 如果这里有 spring.datasource.url=...，会直接覆盖上面的配置
   └── 不会去解析 ${DB_NAME} 变量，因为已经被完整配置覆盖了
```

### 变量解析机制（仅在配置项不存在时）

只有当 `metersphere.properties` 中**没有**完整的配置项时，才会解析变量：

**查找顺序**：
```
1. 命令行参数 / 启动时注入的高优先级来源

2. JVM 系统属性（System.getProperties()）

3. 系统环境变量（System.getenv()）

4. Spring Boot 默认加载的配置文件（application / profile）

5. `@PropertySource` 加载的配置（按加载顺序从后往前）
   ├── metersphere.properties（后加载，优先）
   └── commons.properties（先加载）

6. 如果都找不到，使用默认值（如果有）
```

### 示例说明

**场景一：metersphere.properties 中有完整配置（会直接覆盖）**

**commons.properties**：
```properties
spring.datasource.url=jdbc:mysql://${DB_HOST:localhost}:${DB_PORT:3306}/${DB_NAME:metersphere}?...
```

**metersphere.properties**：
```properties
spring.datasource.url=jdbc:mysql://192.168.0.2:3307/metersphere?...
spring.datasource.username=root
spring.datasource.password=Password123@mysql
```

**结果**：直接使用 `metersphere.properties` 中的完整配置，不会解析 `${DB_NAME}` 变量。

**场景二：metersphere.properties 中只有变量（会解析变量）**

**commons.properties**：
```properties
spring.datasource.url=jdbc:mysql://${DB_HOST:localhost}:${DB_PORT:3306}/${DB_NAME:metersphere}?...
```

**metersphere.properties**：
```properties
DB_HOST=192.168.0.2
DB_PORT=3307
DB_NAME=metersphere_test
DB_USERNAME=root
DB_PASSWORD=Password123@mysql
```

**结果**：会解析 `${DB_NAME}` 变量，最终得到 `jdbc:mysql://192.168.0.2:3307/metersphere_test?...`

## 实际使用方式

### 方式一：在 metersphere.properties 中配置完整值（项目原有方式）

**/opt/metersphere/conf/metersphere.properties**：
```properties
# 数据库配置
spring.datasource.url=jdbc:mysql://192.168.0.2:3307/metersphere?autoReconnect=false&useUnicode=true&characterEncoding=UTF-8&characterSetResults=UTF-8&zeroDateTimeBehavior=convertToNull&useSSL=false
spring.datasource.username=root
spring.datasource.password=Password123@mysql
```

**说明**：
- ✅ 这是项目**原有的配置方式**
- ✅ 会直接覆盖 `commons.properties` 中的配置
- ✅ 不会解析变量，直接使用配置的值

**切换环境**：修改 `metersphere.properties` 中的数据库地址、用户名、密码即可。

### 方式二：在 metersphere.properties 中配置变量（使用变量方式）

**/opt/metersphere/conf/metersphere.properties**：
```properties
# 只配置变量，不配置完整的 spring.datasource.url
DB_HOST=192.168.0.2
DB_PORT=3307
DB_NAME=metersphere_test
DB_USERNAME=root
DB_PASSWORD=Password123@mysql
```

**说明**：
- ✅ 会解析 `commons.properties` 中的 `${DB_NAME}` 等变量
- ✅ 只需修改变量值即可切换环境
- ✅ 配置更灵活

**切换环境**：修改 `DB_NAME`、`DB_HOST` 等变量值即可。

### 方式三：通过系统环境变量配置

**前提**：`metersphere.properties` 中**没有**完整的 `spring.datasource.url` 配置，只有变量或没有配置。

**步骤**：
1. 加载环境变量到系统：
   ```bash
   source .env.local  # 或 source scripts/load-env.sh local
   ```

2. 启动应用：
   ```bash
   mvn spring-boot:run
   ```

**优点**：
- ✅ 不需要修改配置文件
- ✅ 可以通过脚本快速切换环境

**缺点**：
- ❌ 需要先加载环境变量
- ❌ 如果忘记加载，会使用默认值

## 关于 .env.test 文件

**重要**：`.env.test` 文件本身**不会被 Spring Boot 自动读取**！

### 正确使用方式

```bash
# 1. 加载环境变量到系统
source .env.test
# 或
source scripts/load-env.sh test

# 2. 验证环境变量已加载
echo $DB_NAME  # 应该输出: metersphere_test

# 3. 启动应用
mvn spring-boot:run
```

### 错误使用方式

```bash
# ❌ 这样不会生效，因为 Spring Boot 不会读取 .env.test 文件
./scripts/load-env.sh test  # 环境变量只在子进程中生效
mvn spring-boot:run         # 新进程无法访问子进程的环境变量
```

## 总结

### 重要理解

1. ✅ **如果 `metersphere.properties` 中有完整的 `spring.datasource.url`，会直接覆盖，不会解析变量**
2. ✅ **如果 `metersphere.properties` 中没有 `spring.datasource.url`，但有 `DB_NAME` 等变量，才会解析变量**
3. ✅ **`.env.test` 文件需要先加载到系统环境变量中才能生效**

### 项目原有配置方式

**直接在 `metersphere.properties` 中配置完整值**：
```properties
spring.datasource.url=jdbc:mysql://192.168.0.2:3307/metersphere?...
spring.datasource.username=root
spring.datasource.password=Password123@mysql
```

这种方式会直接覆盖 `commons.properties` 中的配置，不会解析变量。

### 使用变量方式（需要确保 metersphere.properties 中没有完整配置）

**在 `metersphere.properties` 中只配置变量**：
```properties
DB_HOST=192.168.0.2
DB_PORT=3307
DB_NAME=metersphere_test
DB_USERNAME=root
DB_PASSWORD=Password123@mysql
```

这样才会解析 `commons.properties` 中的 `${DB_NAME}` 变量。
