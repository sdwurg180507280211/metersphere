## 一、概述

MeterSphere中的时间字段统一使用**毫秒时间戳（bigint类型）**存储，需要通过`FROM_UNIXTIME(字段/1000)`转换为可读日期格式。

### 时间戳转换公式

```sql
-- 毫秒时间戳 → 日期时间
FROM_UNIXTIME(time_field / 1000)

-- 日期时间 → 毫秒时间戳
UNIX_TIMESTAMP('2026-01-01 00:00:00') * 1000
```

## 二、test_case 表（功能用例）

### 2.1 字段列表

| 字段名 | 类型 | 说明 | 业务含义 |
|--------|------|------|----------|
| create_time | bigint | 创建时间戳 | 用例首次创建的时间 |
| update_time | bigint | 更新时间戳 | 用例最后一次修改的时间 |
| delete_time | bigint | 删除时间戳 | 用例被移入回收站的时间 |

### 2.2 字段详解

#### create_time（创建时间）

**含义**：用例首次创建的时间，记录用例的诞生时刻

**作用**：
- 追溯用例的创建历史
- 统计某时间段内新增的用例数量
- 分析用例增长趋势

**使用场景**：
```sql
-- 统计2026年1月新创建的用例
SELECT COUNT(*) 
FROM test_case 
WHERE create_time >= UNIX_TIMESTAMP('2026-01-01 00:00:00') * 1000
  AND create_time < UNIX_TIMESTAMP('2026-02-01 00:00:00') * 1000
  AND delete_time IS NULL;
```

**示例数据**：`2025-12-08 06:27:04`

---

#### update_time（更新时间）

**含义**：用例最后一次被修改的时间（包括内容修改、状态变更等）

**作用**：
- 追踪用例的活跃度
- 识别长期未维护的用例
- 统计某时间段内活跃的用例

**使用场景**：
```sql
-- 查找最近30天内有更新的用例
SELECT * 
FROM test_case 
WHERE update_time >= UNIX_TIMESTAMP(DATE_SUB(NOW(), INTERVAL 30 DAY)) * 1000
  AND delete_time IS NULL;
```

**示例数据**：`2025-12-08 06:27:04`

**注意事项**：
- 用例被执行时，`update_time`可能会更新
- 用例关联关系变更时，`update_time`也会更新

---

#### delete_time（删除时间）

**含义**：用例被移入回收站的时间

**作用**：
- 标识用例是否在回收站中
- 实现软删除机制
- 支持用例恢复功能

**使用场景**：
```sql
-- 查询正常用例（排除回收站）
SELECT * 
FROM test_case 
WHERE delete_time IS NULL;

-- 查询回收站中的用例
SELECT * 
FROM test_case 
WHERE delete_time IS NOT NULL;
```

**示例数据**：`2025-12-08 06:29:31`（已删除）或 `NULL`（正常）

**重要提示**：
- `delete_time IS NULL`：正常用例
- `delete_time IS NOT NULL`：回收站中的用例
- 统计用例数量时**必须**添加 `delete_time IS NULL` 条件

## 三、test_plan 表（测试计划）

### 3.1 字段列表

| 字段名 | 类型 | 说明 | 业务含义 |
|--------|------|------|----------|
| create_time | bigint | 创建时间戳 | 测试计划创建时间 |
| update_time | bigint | 更新时间戳 | 测试计划最后修改时间 |
| planned_start_time | bigint | 计划开始时间戳 | 预计开始执行的时间 |
| planned_end_time | bigint | 计划结束时间戳 | 预计完成的时间 |
| actual_start_time | bigint | 实际开始时间戳 | 实际开始执行的时间 |
| actual_end_time | bigint | 实际结束时间戳 | 实际完成的时间 |
| execution_times | int | 执行次数 | 测试计划被执行的次数 |

### 3.2 字段详解

#### create_time（创建时间）

**含义**：测试计划首次创建的时间

**作用**：
- 记录测试计划的创建历史
- 统计某时间段内新建的测试计划数量

**使用场景**：
```sql
-- 统计2026年1月创建的测试计划
SELECT COUNT(*) 
FROM test_plan 
WHERE create_time >= UNIX_TIMESTAMP('2026-01-01 00:00:00') * 1000
  AND create_time < UNIX_TIMESTAMP('2026-02-01 00:00:00') * 1000;
```

**示例数据**：`2026-01-14 03:09:42`

---

#### update_time（更新时间）

**含义**：测试计划最后一次被修改的时间

**作用**：
- 追踪测试计划的变更历史
- 识别活跃的测试计划

**使用场景**：
```sql
-- 查找最近更新的测试计划
SELECT * 
FROM test_plan 
WHERE update_time >= UNIX_TIMESTAMP(DATE_SUB(NOW(), INTERVAL 7 DAY)) * 1000
ORDER BY update_time DESC;
```

**示例数据**：`2026-02-04 09:03:40`

---

#### planned_start_time（计划开始时间）

**含义**：测试计划预计开始执行的时间（用户手动设置）

**作用**：
- 项目进度规划
- 测试资源调度
- 计划与实际对比分析

**使用场景**：
```sql
-- 查询本周计划开始的测试计划
SELECT * 
FROM test_plan 
WHERE planned_start_time >= UNIX_TIMESTAMP(DATE_SUB(NOW(), INTERVAL WEEKDAY(NOW()) DAY)) * 1000
  AND planned_start_time < UNIX_TIMESTAMP(DATE_ADD(DATE_SUB(NOW(), INTERVAL WEEKDAY(NOW()) DAY), INTERVAL 7 DAY)) * 1000;
```

**示例数据**：`NULL`（未设置）或具体时间

**注意事项**：
- 该字段可能为 `NULL`（用户未设置计划时间）
- 仅用于计划管理，不影响实际执行

---

#### planned_end_time（计划结束时间）

**含义**：测试计划预计完成的时间（用户手动设置）

**作用**：
- 项目进度管理
- 测试周期评估
- 延期风险预警

**使用场景**：
```sql
-- 查询已延期的测试计划（计划结束时间已过，但状态不是已完成）
SELECT * 
FROM test_plan 
WHERE planned_end_time < UNIX_TIMESTAMP(NOW()) * 1000
  AND status NOT IN ('Completed', 'Finished', 'Archived')
  AND planned_end_time IS NOT NULL;
```

**示例数据**：`NULL`（未设置）或具体时间

---

#### actual_start_time（实际开始时间）⭐

**含义**：测试计划实际开始执行的时间（系统自动记录）

**作用**：
- 记录测试计划真正开始执行的时刻
- 统计某时间段内执行的测试计划
- 计算实际执行周期

**触发时机**：
- 测试计划状态从 `Prepare`（未开始）变更为 `Underway`（进行中）时自动记录
- 或首次执行测试计划中的用例时记录

**使用场景**：
```sql
-- 统计2026年1月实际执行的测试计划
SELECT COUNT(*) 
FROM test_plan 
WHERE actual_start_time >= UNIX_TIMESTAMP('2026-01-01 00:00:00') * 1000
  AND actual_start_time < UNIX_TIMESTAMP('2026-02-01 00:00:00') * 1000;
```

**示例数据**：`2026-01-14 03:17:20`

**重要提示**：
- 这是**最常用的"执行时间"字段**
- 用于统计"某时间段内执行的测试计划"
- 可能为 `NULL`（测试计划从未执行）

---

#### actual_end_time（实际结束时间）⭐

**含义**：测试计划实际完成的时间（系统自动记录）

**作用**：
- 记录测试计划完成的时刻
- 统计某时间段内完成的测试计划
- 计算实际执行周期

**触发时机**：
- 测试计划状态变更为 `Completed`（已完成）或 `Finished`（已结束）时自动记录

**使用场景**：
```sql
-- 统计2026年1月完成的测试计划
SELECT COUNT(*) 
FROM test_plan 
WHERE actual_end_time >= UNIX_TIMESTAMP('2026-01-01 00:00:00') * 1000
  AND actual_end_time < UNIX_TIMESTAMP('2026-02-01 00:00:00') * 1000;
```

**示例数据**：`2026-02-04 06:30:36` 或 `NULL`（未完成）

**重要提示**：
- 用于统计"某时间段内完成的测试计划"
- 可能为 `NULL`（测试计划未完成）

---

#### execution_times（执行次数）

**含义**：测试计划被执行的次数（可能用于统计重复执行）

**作用**：
- 记录测试计划的执行频率
- 分析测试计划的复用情况

**使用场景**：
```sql
-- 查询执行次数最多的测试计划
SELECT name, execution_times 
FROM test_plan 
WHERE execution_times IS NOT NULL
ORDER BY execution_times DESC
LIMIT 10;
```

**示例数据**：`NULL`（大部分情况）

**注意事项**：
- 该字段在当前版本中**大部分为 NULL**
- 可能是预留字段或特定场景使用
- 不建议用于统计分析

## 四、test_plan_test_case 表（测试计划用例关联）

### 4.1 字段列表

| 字段名 | 类型 | 说明 | 业务含义 |
|--------|------|------|----------|
| create_time | bigint | 创建时间戳 | 用例关联到测试计划的时间 |
| update_time | bigint | 更新时间戳 | 用例执行状态最后更新时间 |

### 4.2 字段详解

#### create_time（创建时间）

**含义**：功能用例被关联到测试计划的时间

**作用**：
- 记录用例加入测试计划的时间
- 追溯测试计划的用例变更历史

**使用场景**：
```sql
-- 查询某测试计划中最近添加的用例
SELECT * 
FROM test_plan_test_case 
WHERE plan_id = 'xxx'
ORDER BY create_time DESC
LIMIT 10;
```

**示例数据**：`2026-01-14 03:13:37`

---

#### update_time（更新时间）⭐⭐⭐

**含义**：用例执行状态最后一次更新的时间（**最重要的执行时间字段**）

**作用**：
- 记录用例被执行的时间
- 统计某时间段内执行的用例数量
- 分析测试执行进度

**触发时机**：
- 用例执行状态从 `Prepare`（未执行）变更为其他状态时更新
- 用例执行结果被修改时更新
- 用例被重新执行时更新

**使用场景**：
```sql
-- 统计2026年1月执行的用例数量（SQL5的核心逻辑）
SELECT COUNT(DISTINCT tptc.id) 
FROM test_plan_test_case tptc
INNER JOIN test_plan tp ON tptc.plan_id = tp.id
WHERE tptc.update_time >= UNIX_TIMESTAMP('2026-01-01 00:00:00') * 1000
  AND tptc.update_time < UNIX_TIMESTAMP('2026-02-01 00:00:00') * 1000
  AND tptc.is_del = 0;
```

**示例数据**：`2026-01-14 03:15:42`

**重要提示**：
- 这是**统计"执行用例数量"的核心字段**
- SQL5系列使用该字段统计时间范围内的执行情况
- 必须配合 `is_del = 0` 条件（排除已删除的关联）

## 六、时间字段使用建议

### 6.1 统计场景与字段选择

| 统计需求 | 推荐字段 | 理由 |
|---------|---------|------|
| 新创建的功能用例 | `test_case.create_time` | 记录用例创建时间 |
| 活跃的功能用例 | `test_case.update_time` | 记录用例修改时间 |
| 被执行的功能用例 | `test_plan_test_case.update_time` | 记录用例执行时间 |
| 新创建的测试计划 | `test_plan.create_time` | 记录计划创建时间 |
| 执行的测试计划 | `test_plan.actual_start_time` | 记录计划开始执行时间 |
| 完成的测试计划 | `test_plan.actual_end_time` | 记录计划完成时间 |
| 新创建的缺陷 | `issues.create_time` | 记录缺陷创建时间 |
| 活跃的缺陷 | `issues.update_time` | 记录缺陷处理时间 |

### 6.2 SQL3和SQL4修改建议

基于以上分析，对于您的需求"对SQL3和SQL4添加起止时间"，我给出以下建议：

#### 方案A：统计"新创建"的数据（保守方案）

```sql
-- SQL3修改：统计某时间段内新创建的用例、计划、缺陷
WHERE test_case.create_time >= 开始时间 AND test_case.create_time <= 结束时间
  AND test_plan.create_time >= 开始时间 AND test_plan.create_time <= 结束时间
  AND issues.create_time >= 开始时间 AND issues.create_time <= 结束时间
```

**适用场景**：统计某时间段内的新增数据

---

#### 方案B：统计"执行/活跃"的数据（推荐方案）⭐

```sql
-- SQL3修改：统计某时间段内被执行的用例、执行的计划、新创建的缺陷
WHERE test_plan_test_case.update_time >= 开始时间 AND test_plan_test_case.update_time <= 结束时间
  AND test_plan.actual_start_time >= 开始时间 AND test_plan.actual_start_time <= 结束时间
  AND issues.create_time >= 开始时间 AND issues.create_time <= 结束时间
```

**适用场景**：统计某时间段内的测试活动（更符合业务逻辑）

---

#### 方案C：统计"完成"的数据

```sql
-- SQL4修改：统计某时间段内完成的测试计划
WHERE test_plan.actual_end_time >= 开始时间 AND test_plan.actual_end_time <= 结束时间
```

**适用场景**：统计某时间段内完成的测试计划

---

### 6.3 注意事项

1. **时间戳格式**：所有时间字段都是毫秒时间戳，需要乘以1000
2. **NULL值处理**：部分时间字段可能为NULL，需要考虑NULL值的影响
3. **回收站过滤**：统计用例时必须添加 `delete_time IS NULL` 条件
4. **关联删除过滤**：统计执行用例时必须添加 `is_del = 0` 条件
5. **业务逻辑**：选择时间字段时要考虑业务含义，避免统计结果不符合预期

## 七、总结

### 7.1 核心时间字段（最常用）

| 表名 | 字段 | 用途 |
|------|------|------|
| test_plan_test_case | update_time | **统计执行的用例数量** |

### 7.2 快速参考

```sql
-- 时间范围过滤模板
WHERE time_field >= UNIX_TIMESTAMP('2026-01-01 00:00:00') * 1000
  AND time_field <= UNIX_TIMESTAMP('2026-01-31 23:59:59') * 1000

-- 时间转换显示
SELECT FROM_UNIXTIME(time_field / 1000) AS '可读时间'

-- 排除回收站
WHERE delete_time IS NULL

-- 排除已删除关联
WHERE is_del = 0
```
